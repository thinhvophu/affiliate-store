# Feature Specification: Automated Content Ingestion Pipeline
**ID:** `F0012`
**Epic/Module:** Content Operations — Catalog & Editorial Growth
**Status:** Draft

---

## 1. Business Objective
The catalog and blog are hand-authored today: every product is one JSON file and every post one MDX file, written by hand. A content audit (2026-07) found categories (`man-hinh-gaming`, `ghe-gaming`) with as few as 1 product and zero blog posts, which reads as thin/incomplete to visitors and search crawlers alike. Scaling to a healthy baseline (~15 products/category, ≥2 posts/category) by hand is slow and error-prone — a wrong affiliate host, a missing local image, or an unregistered category slug currently only surfaces as a `next build` failure after the fact. This feature builds a repeatable, re-runnable ingestion pipeline that sources product candidates (via the `shopee-affiliate` scraper for high-volume commodity categories, or a curated manual list for higher-consideration categories), validates them against the project's existing build-time invariants *before* they become fixtures, stages local product images, and writes conformant `content/products/*.json` files — plus scaffolds MDX post stubs for content-gap categories. It turns "add 50 products and 5 posts" from bespoke manual authoring into a scriptable operation a human then reviews and finishes.

## 2. Scope

### In Scope
* A CLI script (`scripts/ingest-products.ts`, run via `npm run ingest:products`) that accepts a target `category` slug plus a source flag:
  * `--source=scrape --query=<term> --count=<n>` — calls the `shopee-affiliate` scrape tool for candidates, or
  * `--source=file --path=<file>` — reads a human-curated list of candidate products (name + Shopee URL + notes)
* A shared, in-memory **candidate model** that both sources normalize into, so validation / image staging / dedupe / fixture writing run identically regardless of source.
* Per-candidate validation *before* writing any file, reusing existing chokepoints (not re-implementing them):
  * Affiliate URL host via `assertAffiliateUrl` (`lib/affiliate.ts` allow-list: `shopee.vn`, `shopee.ee`, `shope.ee`)
  * `category` via `assertCategoryRegistered` (`lib/categories.ts` registry)
  * Required `Product` fields present (`types/product.ts`: name, brand, price, images, specs, description)
* Local image staging: every candidate image downloaded and saved under `public/static/images/products/<slug>-<n>.<ext>` — never a hotlinked remote URL (`next.config.ts` defines no `images.remotePatterns`).
* Slug generation consistent with existing fixtures (kebab-case, diacritics transliterated) with collision detection.
* Dedupe / idempotency: re-running the same command does not duplicate a product already present in `content/products/*.json` (keyed on `affiliateUrl` first, slug second).
* One `content/products/<slug>.json` file per accepted candidate, conforming exactly to the `Product` interface.
* A dry-run mode that prints a summary (added / skipped-duplicate / rejected-invalid) without writing files, so a human reviews before committing.
* A companion MDX scaffold mode (`npm run scaffold:post`) that generates a post stub for a content-gap category: frontmatter filled in (title placeholder, `category`, `publishedAt` = today, `tags: []`), a unique `coverImage` sourced from one of that post's embedded products, and `<ProductCard slug="…" />` embeds for the products the human names — body prose left for a human.
* Exit non-zero and print the offending candidate/line/slug on any validation failure, mirroring the existing build-time failure style.

### Out of Scope
* Automated blog-post prose generation — the pipeline only scaffolds frontmatter + product embeds; a human writes the article body.
* Automated re-scraping to refresh prices/stock on *existing* products (a future "catalog freshness" feature).
* Automatic creation of new categories — `lib/categories.ts` stays a manually curated registry; the pipeline only ingests into already-registered categories.
* Committing, opening a PR, or deploying — the pipeline only writes local files under `content/` and `public/static/images/products/`; git/PR steps stay manual.
* Curation judgment for hand-picked categories — `--source=file` accepts a human-supplied list, but deciding *which* products to include is a human decision.

## 3. Process Flow
1. An operator runs `npm run ingest:products -- --category=chuot-gaming --source=scrape --query="chuột gaming không dây" --count=8`.
2. The script preflight-checks `--category` against `lib/categories.ts`; if unregistered it exits non-zero immediately, before calling any source.
3. The chosen source (scrape or file) produces raw candidates, each normalized into the shared candidate model (name, brand, price, images, spec text, affiliate URL).
4. For each candidate: derive a slug, check it against existing `content/products/*.json` for a duplicate (by `affiliateUrl` or slug), skip if found.
5. Validate the candidate's affiliate URL host, category, and required fields; reject (with a printed reason) any that fail.
6. For each accepted candidate: download its image(s) to `public/static/images/products/<slug>-<n>.<ext>` and write `content/products/<slug>.json`.
7. The script prints a summary: N added, N skipped (duplicate), N rejected (invalid), each with a reason.
8. The operator runs `next build` locally to confirm the new fixtures pass every existing build-time assertion, then reviews the diff and images before committing.
9. Separately, for a content-gap category, the operator runs the scaffold mode naming 1–2 already-ingested product slugs; the script writes an MDX stub (frontmatter + `<ProductCard>` embeds + a unique cover image), which a human finishes and publishes through the normal flow.

## 4. User Stories & Acceptance Criteria

### US00121: Ingestion CLI foundation & candidate validation
**As a** site operator,
**I want to** a single ingestion command that parses its arguments, validates every candidate against the project's existing build-time invariants, and reports a clear summary,
**So that** both the scrape and file sources share one validation engine and no invalid fixture is ever written.

**Acceptance Criteria:**
* **Scenario 1:** Unregistered category halts before any source runs
    * `Given` a `--category` value with no entry in `lib/categories.ts`
    * `When` the script starts
    * `Then` it exits non-zero immediately with the offending slug named, via `assertCategoryRegistered`, before calling any source or writing any file

* **Scenario 2:** Candidate missing a required field is rejected, not written
    * `Given` a candidate missing a required `Product` field (e.g. no price or no specs)
    * `When` the script validates it
    * `Then` no fixture file is written for that candidate, and the summary lists it as rejected with the missing field named

* **Scenario 3:** Dry-run reports without writing
    * `Given` any valid or invalid set of candidates and the `--dry-run` flag
    * `When` the script runs
    * `Then` it prints the full summary (added / skipped-duplicate / rejected-invalid with reasons) but writes no `content/products/*.json` file and downloads no image

### US00122: Slug generation, dedupe & idempotent re-runs
**As a** site operator,
**I want to** re-run the same ingest command without creating duplicate products,
**So that** I can safely retry after a partial failure or expand a category incrementally over multiple runs.

**Acceptance Criteria:**
* **Scenario 1:** Re-running an identical command is a no-op for already-ingested products
    * `Given` a product already exists in `content/products/*.json` with a matching `affiliateUrl` or slug
    * `When` the same candidate is sourced again in a later run
    * `Then` it is skipped and counted as "duplicate" in the summary, and no existing file is overwritten

* **Scenario 2:** Partial-run recovery
    * `Given` a run that ingested 5 of 8 requested candidates before hitting an error
    * `When` the operator re-runs the same command
    * `Then` the 5 already-ingested products are skipped as duplicates and only the remaining candidates are processed

* **Scenario 3:** Slug collision between distinct candidates never silently overwrites
    * `Given` two different candidates (different `affiliateUrl`) that derive the same slug
    * `When` the second is processed
    * `Then` the existing file is never silently overwritten — the second is either skipped (if the same `affiliateUrl`) or given a disambiguated slug and flagged for human review

### US00123: Local image staging (no hotlinking)
**As a** site operator,
**I want to** every ingested product's images downloaded and stored under `public/static/images/products/`,
**So that** `next/image` can serve them without requiring `images.remotePatterns` changes to `next.config.ts`.

**Acceptance Criteria:**
* **Scenario 1:** Images are downloaded locally
    * `Given` an accepted candidate with one or more remote image URLs
    * `When` the fixture is written
    * `Then` each image has been downloaded to `public/static/images/products/<slug>-<n>.<ext>` and the fixture's `images` array contains only local `/static/images/products/...` paths, never a remote URL

* **Scenario 2:** Image download failure rejects the candidate
    * `Given` a candidate whose image URL 404s or times out
    * `When` the script attempts the download
    * `Then` the candidate is rejected (no partial fixture written) and the summary names the candidate and the failure reason

* **Scenario 3:** Already-staged image is not re-downloaded
    * `Given` an image already exists at the target local path from a previous partial run
    * `When` the script processes the same candidate again
    * `Then` the download is skipped (treated as already-staged) rather than erroring, so a retried run can complete

### US00124: Scrape-sourced product ingestion for commodity categories
**As a** site operator,
**I want to** pull candidate products for a registered category from Shopee via the scripted command,
**So that** I can grow high-volume, low-consideration categories (chuột, bàn phím, tai nghe) without hand-writing each JSON fixture.

**Acceptance Criteria:**
* **Scenario 1:** Successful scrape produces valid fixtures
    * `Given` a registered category slug and `--source=scrape --query=<term> --count=<n>`
    * `When` the operator runs the ingest script
    * `Then` for each accepted candidate a `content/products/<slug>.json` file is written that conforms to the `Product` interface (`types/product.ts`) and passes `next build`

* **Scenario 2:** Invalid affiliate host is rejected, not written
    * `Given` a scraped candidate whose product URL host is not in the `lib/affiliate.ts` allow-list
    * `When` the script validates it via `assertAffiliateUrl`
    * `Then` no fixture is written for that candidate, and the summary lists it as rejected with the offending host named

* **Scenario 3:** Scraper returns fewer candidates than requested
    * `Given` a `--count=8` request for which the scraper returns only 6 usable candidates
    * `When` the script finishes
    * `Then` it is not an error — the script ingests what passed validation and reports the shortfall in the summary (e.g. "requested 8, ingested 6")

### US00125: Curated manual input for higher-consideration categories
**As a** site operator,
**I want to** supply a hand-picked list of specific products (name + Shopee URL) for a category,
**So that** categories where the specific product choice matters (màn hình, ghế gaming) aren't populated by whatever the scraper happens to return.

**Acceptance Criteria:**
* **Scenario 1:** File-sourced candidates go through the same pipeline as scraped ones
    * `Given` a file listing product name/URL pairs and `--source=file --path=<file>`
    * `When` the script runs
    * `Then` each entry is normalized into the same candidate model and passes through identical validation, image staging, dedupe, and fixture output as the scrape path

* **Scenario 2:** Malformed input file is rejected with a clear error
    * `Given` a file missing a required column (e.g. no product URL for an entry)
    * `When` the script parses it
    * `Then` it exits non-zero naming the offending line, before ingesting any entry from that file

### US00126: Blog-post MDX scaffold for content-gap categories
**As a** site operator,
**I want to** generate an MDX stub (frontmatter + product embeds) for a category that has fewer than 2 posts,
**So that** filling the `man-hinh-gaming` / `ghe-gaming` post gap doesn't require hand-copying the frontmatter shape from an existing post each time.

**Acceptance Criteria:**
* **Scenario 1:** Scaffold produces a valid, buildable MDX stub
    * `Given` a category slug and 1–2 already-ingested product slugs
    * `When` the operator runs `npm run scaffold:post`
    * `Then` a new `content/posts/<slug>.mdx` is written with frontmatter (`title` placeholder, `category`, `publishedAt` = current date, `tags: []`, `coverImage` sourced from one of the named products' images) and one `<ProductCard slug="…" />` per named product, and it passes `next build` as-is (empty body is valid MDX)

* **Scenario 2:** Cover images are not reused across scaffolded posts
    * `Given` two scaffold runs for the same category naming different product slugs
    * `When` both stubs are generated
    * `Then` their `coverImage` values differ (each sourced from its own named products), avoiding the pattern of one product's photo reused across every post in a category

* **Scenario 3:** Unknown product slug in scaffold input fails loudly
    * `Given` a product slug that does not exist in `content/products/*.json`
    * `When` the operator names it in the scaffold command
    * `Then` the script exits non-zero naming the unknown slug, before writing any `.mdx` file

## 5. Edge Cases & Error Handling
* **Scraper returns fewer candidates than `--count`:** not an error — ingest what passed validation and report the shortfall in the summary (US00124 Scenario 3).
* **Duplicate slugs from different candidates** (same brand+model scraped twice under slightly different names): collision detected before write; second candidate skipped (same `affiliateUrl`) or disambiguated + flagged — never silently overwritten (US00122 Scenario 3).
* **Scrape tool rate-limited or unreachable:** fail the run with a clear message; fixtures already written earlier in the same run are left in place (partial progress, not rolled back — consistent with the idempotent re-run design).
* **Vietnamese diacritics in scraped product names:** slugs strip/transliterate diacritics consistently with existing fixture slugs (e.g. `chuot-gaming`, not `chuột-gaming`); JSON content preserves diacritics in `name`/`description`.
* **Price or spec fields missing from scraped data:** candidate rejected, not written with placeholder values — the `Product` interface has no optional fields, and guessing a price would misrepresent the product to buyers.
* **Image already staged at the target path from a previous run:** re-download skipped rather than erroring, so a retried run can complete (US00123 Scenario 3).

## 6. Technical Considerations
* **No file outside `lib/affiliate.ts` may parse/validate affiliate URLs, and no file outside `lib/categories.ts` may validate category slugs** (existing project rule) — the ingestion script must *import and call* `assertAffiliateUrl` / `assertCategoryRegistered`, so the pipeline and the build stay in lock-step by construction.
* **`next.config.ts` has no `images.remotePatterns`** — this is why local image staging (US00123) is mandatory, not optional; hotlinking Shopee CDN URLs would break `next/image`.
* **Product schema is closed** (`types/product.ts` has no `?` fields) — the pipeline must fully populate a candidate or reject it; there is no partial/draft product state.
* **Script location:** a new top-level `scripts/` directory (does not currently exist), with `npm run ingest:products` / `npm run scaffold:post` entries in `package.json`. Kept out of `lib/` (closer to a build tool than app code — the "no JSX in lib/" rule notwithstanding, this is tooling) and out of `app/`/`components/` (not part of the rendered site).
* **This is tooling, not a rendered route or component** — no in-browser UI to verify; verification is `next build` succeeding on its output plus manual review of the written JSON/MDX/images before commit.
* **Idempotency key:** dedupe keys off `affiliateUrl` first (the true identity of "is this the same product") and slug second, since two different scrape queries could surface the same product under slightly different scraped names.
* **Shared candidate model:** both the scrape adapter (US00124) and the file adapter (US00125) normalize into one intermediate candidate shape so US00121's validation, US00123's staging, and US00122's dedupe run once, source-agnostic.
* **Relationship to the content plan:** this pipeline is the *mechanism*; the actual targets (~15 products/category, closing the post gap) are a separate content-operations effort. No changes to `app/`, `components/`, or the route map are expected from this feature.
