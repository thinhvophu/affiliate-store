# Feature Specification: SEO & Discoverability
**ID:** `F0009`
**Epic/Module:** Core App — SEO & Discoverability
**Status:** Draft

---

## 1. Business Objective
Make every page on the affiliate storefront rankable, crawlable, and shareable from day one. This feature establishes a consistent metadata foundation (titles, descriptions, canonical URLs, Open Graph / Twitter cards), structured data (JSON-LD for `Product`, `Article`, and `BreadcrumbList`), visible breadcrumb navigation, and crawl infrastructure (`sitemap.xml`, `robots.txt`). As an SEO-first, SSG-only Vietnamese affiliate site, organic search is the primary traffic and revenue channel — these are launch-blocking (P0) capabilities that let Google index the catalog and blog, qualify pages for rich results, and present correct previews when links are shared on social.

## 2. Scope
### In Scope
* Root-level metadata defaults: `metadataBase` (from `NEXT_PUBLIC_SITE_URL`), title template (`%s | <site name>`), default title/description, default Open Graph (type, site_name, locale `vi_VN`), default Twitter card, and a static default OG image asset (`/static/images/og-default.png`).
* Per-page metadata completeness audit across every route: unique `<title>`, `<meta description>` (≤160 chars), absolute canonical URL, Open Graph (`og:title`, `og:description`, `og:image`), Twitter card.
* Visible `<Breadcrumb>` component on product, category, and blog-post pages (semantic `<nav>` + `<ol>`, crawlable links, current page marked `aria-current`).
* JSON-LD structured data: `Product` on `/san-pham/[slug]/`, `Article` on `/bai-viet/[slug]/`, `BreadcrumbList` on every page that renders a visible breadcrumb (sourced from the same breadcrumb items).
* `app/sitemap.ts` generating `/sitemap.xml` covering all static + dynamic routes (home, listings, every product slug, every category, every post slug, policy pages, about) with `lastModified`.
* `app/robots.ts` generating `/robots.txt` allowing all crawlers + pointing to the sitemap.
* Self-referencing canonical strategy for paginated listings (`?page=N`).

### Out of Scope
* `hreflang` / multilingual SEO (v1 is Vietnamese-only).
* Search functionality (deferred to v2).
* Dynamic / per-page generated OG images (single static default image only in v1).
* Sitemap index splitting / news sitemaps / image sitemaps.
* Schema types beyond `Product`, `Article`, `BreadcrumbList` (e.g., `FAQPage`, `ItemList`, `Organization`).
* Click tracking / analytics (owned by F0007).

## 3. Process Flow
*This feature is consumed primarily by search-engine crawlers and social link unfurlers; the human-visible surface is the breadcrumb.*
1. A crawler requests `/robots.txt`, sees "allow all" plus the sitemap pointer.
2. The crawler fetches `/sitemap.xml` and discovers every canonical URL on the site.
3. The crawler visits a page (e.g., a product detail page); the server-rendered HTML carries a unique `<title>`, `<meta description>`, canonical link, Open Graph / Twitter tags, and a JSON-LD `<script>` block.
4. The crawler parses the `Product` / `Article` / `BreadcrumbList` JSON-LD and qualifies the page for rich results.
5. A human visitor lands on a deep page, reads the visible breadcrumb, and navigates up the hierarchy via crawlable links.
6. A user shares the page URL on social; the unfurler reads Open Graph tags and renders a title, description, and the page's OG image (or the site default).

## 4. User Stories & Acceptance Criteria

### US00091: Global metadata foundation
**As a** site owner,
**I want to** define root-level metadata defaults (base URL, title template, default description, default Open Graph & Twitter card, and a default OG image),
**So that** every page inherits correct absolute URLs and a consistent fallback preview without repeating boilerplate per page.

**Acceptance Criteria:**
* **Scenario 1:** Root metadata defaults are applied
    * `Given` the root layout (`app/layout.tsx`) exports a `metadata` object
    * `When` the app is built
    * `Then` it sets `metadataBase` from `NEXT_PUBLIC_SITE_URL`, `title.template = "%s | <site name>"`, a `title.default`, a default `description`, `openGraph` (`type: "website"`, `siteName`, `locale: "vi_VN"`, default image), and `twitter.card = "summary_large_image"`

* **Scenario 2:** A page that declares only a partial title inherits the template
    * `Given` a page exports `title: "Sản phẩm"` and no OG image
    * `When` the page renders
    * `Then` the document title is `"Sản phẩm | <site name>"` and the Open Graph image falls back to the site default

* **Scenario 3:** Default OG image asset exists
    * `Given` no page-specific image is provided
    * `When` the page is shared
    * `Then` `og:image` resolves to an absolute URL for a static asset committed at `/static/images/og-default.png` (≥1200×630)

* **Scenario 4:** Absolute URL resolution
    * `Given` `metadataBase` is set
    * `When` relative canonical/OG URLs are emitted by child pages
    * `Then` Next.js resolves them to absolute URLs against `metadataBase`

### US00092: Per-page metadata completeness audit
**As a** searcher,
**I want to** see a correct, unique title, description, canonical URL, and link preview for every route,
**So that** search results and shared links display accurate, non-duplicated information.

**Acceptance Criteria:**
* **Scenario 1:** Every route declares complete metadata
    * `Given` the routes `/`, `/san-pham/`, `/san-pham/[slug]/`, `/danh-muc/[category]/`, `/bai-viet/`, `/bai-viet/[slug]/`, `/ve-chung-toi/`, `/chinh-sach-bao-mat/`, `/cong-bo-tiep-thi-lien-ket/`
    * `When` each page is rendered
    * `Then` each emits a unique `<title>`, a `<meta name="description">` of ≤160 characters, an absolute canonical URL, and Open Graph + Twitter title/description

* **Scenario 2:** Canonical URLs are consistent and absolute
    * `Given` the previous mixed pattern (`${process.env.NEXT_PUBLIC_SITE_URL ?? ""}/...`)
    * `When` canonical URLs are generated
    * `Then` they resolve through `metadataBase` (relative `alternates.canonical`) and never produce a malformed or root-relative canonical when the env var is missing at build time

* **Scenario 3:** Content pages use their own image for OG
    * `Given` a product detail page or blog post with an image/cover
    * `When` the page metadata is generated
    * `Then` `og:image` is the product image / post cover (absolute URL); pages without an image fall back to the site default from US00091

* **Scenario 4:** Description truncation is graceful
    * `Given` a product/post with a long description
    * `When` the meta description is generated
    * `Then` it is truncated to ≤160 characters at a word/sentence boundary (no mid-word cut, no trailing partial entity)

### US00093: Visible breadcrumb component
**As a** visitor,
**I want to** see a breadcrumb trail on product, category, and blog-post pages,
**So that** I understand where I am in the site hierarchy and can navigate back up.

**Acceptance Criteria:**
* **Scenario 1:** Breadcrumb renders on deep pages
    * `Given` I am on `/san-pham/[slug]/`, `/danh-muc/[category]/`, or `/bai-viet/[slug]/`
    * `When` the page loads
    * `Then` a breadcrumb trail is visible (e.g., `Trang chủ › <Danh mục> › <Trang hiện tại>`) above the main heading

* **Scenario 2:** Semantic, accessible markup
    * `Given` the `<Breadcrumb items={…} />` component
    * `When` it renders
    * `Then` it uses `<nav aria-label="Breadcrumb">` wrapping an ordered list `<ol>`, all ancestor items are crawlable `<a>` links, and the final item is plain text marked `aria-current="page"` (not a link)

* **Scenario 3:** Single source of truth for trail data
    * `Given` the breadcrumb `items` array (label + href per crumb)
    * `When` the page also emits `BreadcrumbList` JSON-LD (US00096)
    * `Then` both the visible breadcrumb and the structured data are derived from the same `items` so they can never diverge

* **Scenario 4:** Vietnamese labels
    * `Given` the root crumb
    * `When` rendered
    * `Then` its label is `"Trang chủ"`, and category crumbs use the registered Vietnamese display name from `lib/categories.ts`

### US00094: Product JSON-LD
**As** Google,
**I want to** read `Product` structured data on product detail pages,
**So that** the pages qualify for product rich results.

**Acceptance Criteria:**
* **Scenario 1:** Valid Product schema is emitted
    * `Given` a product detail page `/san-pham/[slug]/`
    * `When` the page is rendered
    * `Then` a `<script type="application/ld+json">` block contains a `Product` object with `name`, `image` (absolute URL), `description`, `brand`, `url` (canonical), and an `offers` object with `price`, `priceCurrency: "VND"`, and `availability`

* **Scenario 2:** Price is sourced, never hardcoded
    * `Given` the product JSON `price` field
    * `When` JSON-LD is generated
    * `Then` `offers.price` is the numeric product price (no currency symbol, no thousands separator) and is taken from the product data — never a literal in the component

* **Scenario 3:** Build-time generation
    * `Given` SSG-only rendering
    * `When` `next build` runs
    * `Then` the JSON-LD is present in the statically generated HTML (no client-side injection)

### US00095: Article JSON-LD
**As** Google,
**I want to** read `Article` structured data on blog posts,
**So that** the posts qualify for article rich results.

**Acceptance Criteria:**
* **Scenario 1:** Valid Article schema is emitted
    * `Given` a blog post page `/bai-viet/[slug]/`
    * `When` the page is rendered
    * `Then` a `<script type="application/ld+json">` block contains an `Article` object with `headline`, `image` (cover image, absolute URL), `datePublished` (ISO from frontmatter), `author`, `publisher`, and `mainEntityOfPage` (the canonical URL)

* **Scenario 2:** Headline length compliance
    * `Given` a post with a long title
    * `When` `headline` is generated
    * `Then` it reflects the post title (Google truncates display at ~110 chars; we emit the full title but it is sourced from frontmatter, never hardcoded)

* **Scenario 3:** Build-time generation
    * `Given` SSG-only rendering
    * `When` `next build` runs
    * `Then` the JSON-LD is present in the statically generated HTML

### US00096: BreadcrumbList JSON-LD
**As** Google,
**I want to** read `BreadcrumbList` structured data wherever a breadcrumb is shown,
**So that** breadcrumb rich results appear in the SERP and match the visible trail.

**Acceptance Criteria:**
* **Scenario 1:** BreadcrumbList emitted alongside visible breadcrumb
    * `Given` a page renders the visible `<Breadcrumb>` (US00093)
    * `When` the page is rendered
    * `Then` a `<script type="application/ld+json">` block contains a `BreadcrumbList` with an ordered `itemListElement` of `ListItem`s (`position`, `name`, `item` = absolute URL)

* **Scenario 2:** Structured data matches visible trail
    * `Given` the shared breadcrumb `items` array
    * `When` both visible and structured breadcrumbs are generated
    * `Then` the `name` and order of `ListItem`s exactly match the visible crumbs (Google flags mismatches)

* **Scenario 3:** Last item handling
    * `Given` the current page (last crumb)
    * `When` `BreadcrumbList` is built
    * `Then` it is included as the final `ListItem` with its canonical URL as `item`

### US00097: Auto-generated sitemap.xml
**As a** crawler,
**I want to** fetch a complete `/sitemap.xml`,
**So that** every canonical URL on the site is discoverable and indexable.

**Acceptance Criteria:**
* **Scenario 1:** All routes are listed
    * `Given` `app/sitemap.ts` (returning `MetadataRoute.Sitemap`)
    * `When` the site is built
    * `Then` `/sitemap.xml` includes the homepage, `/san-pham/`, every product slug, `/danh-muc/[category]/` for every registered category, `/bai-viet/`, every post slug, `/ve-chung-toi/`, `/chinh-sach-bao-mat/`, and `/cong-bo-tiep-thi-lien-ket/`

* **Scenario 2:** Absolute URLs and freshness
    * `Given` the sitemap entries
    * `When` generated
    * `Then` each `loc` is an absolute URL (built from `NEXT_PUBLIC_SITE_URL`) and carries a `lastModified` (product/post `publishedAt` where available, build date otherwise)

* **Scenario 3:** Data sourced from loaders
    * `Given` the content loaders
    * `When` the sitemap is built
    * `Then` product/post/category lists come from `getAllProducts()`, `getAllPosts()`, and `lib/categories.ts` — adding a product/post/category automatically adds a sitemap entry with no sitemap-file edit

### US00098: robots.txt
**As a** crawler,
**I want to** read a `/robots.txt` that allows crawling and points to the sitemap,
**So that** crawl rules are explicit and the sitemap is discoverable.

**Acceptance Criteria:**
* **Scenario 1:** Allow-all with sitemap pointer
    * `Given` `app/robots.ts` (returning `MetadataRoute.Robots`)
    * `When` the site is built
    * `Then` `/robots.txt` emits `User-agent: *` with `Allow: /` and a `Sitemap:` line pointing to `<NEXT_PUBLIC_SITE_URL>/sitemap.xml`

* **Scenario 2:** Absolute sitemap URL
    * `Given` the env-configured site URL
    * `When` robots.txt is generated
    * `Then` the `Sitemap:` value is an absolute URL

### US00099: Paginated listing canonical strategy
**As a** site owner,
**I want to** each paginated listing page declare a self-referencing canonical,
**So that** duplicate-content signals are avoided and each page is crawled and indexed correctly.

**Acceptance Criteria:**
* **Scenario 1:** Self-referencing canonical per page
    * `Given` a listing route with `?page=N` (`/san-pham/`, `/danh-muc/[category]/`, `/bai-viet/`)
    * `When` page N (N > 1) is rendered
    * `Then` its canonical URL is the same paginated URL (self-referencing, e.g. `/san-pham/?page=2`), not a canonical pointing back to page 1

* **Scenario 2:** Page 1 canonical
    * `Given` the first page of a listing
    * `When` rendered
    * `Then` its canonical is the clean listing URL without a `?page=1` suffix

* **Scenario 3:** Crawlable pagination preserved
    * `Given` the existing `?page=N` pagination links
    * `When` listings render
    * `Then` page links remain real crawlable `<a href>` query-param links (no change to F0004's crawlable pagination)

## 5. Edge Cases & Error Handling
* **Missing `NEXT_PUBLIC_SITE_URL` at build:** `lib/env.ts` already requires it; `metadataBase` and sitemap/robots generation depend on it. A missing value should fail the build loudly rather than emit empty/relative canonical URLs.
* **Product with no images / post with no cover:** OG image and `Product`/`Article` `image` fall back to the site default OG image (the product loader already enforces `images.length ≥ 1`, so product images are guaranteed; posts without a cover use the default).
* **Long titles/descriptions:** descriptions are truncated to ≤160 chars at a word boundary; titles pass through the template untruncated (search engines truncate display-side).
* **Out-of-range `?page=N`:** pagination already guards out-of-range pages (F0004/F0006); the canonical for such a page should still be self-referencing or the listing handles it via its existing empty/redirect behavior — SEO layer must not crash on an invalid page param.
* **Unknown category/slug:** dynamic pages already `notFound()` for unregistered slugs; such URLs are never emitted into the sitemap because the sitemap is built from the same registered/loaded sources.
* **Special characters in Vietnamese titles/descriptions:** JSON-LD must be safely serialized (proper escaping) so diacritics and quotes don't break the `<script>` block.

## 6. Technical Considerations
* **SSG-only:** all metadata, JSON-LD, sitemap, and robots must be generated at build time (`generateMetadata`, `app/sitemap.ts`, `app/robots.ts`). No client-side metadata injection.
* **Single source of truth for the base URL:** canonical, OG, sitemap, and robots URLs all derive from `NEXT_PUBLIC_SITE_URL` via `metadataBase` (per-page `alternates.canonical` should be relative and resolved by Next.js). Replace the existing `${process.env.NEXT_PUBLIC_SITE_URL ?? ""}` string pattern.
* **Breadcrumb is the single source for trail data:** the visible `<Breadcrumb>` and `BreadcrumbList` JSON-LD consume one `items` array so they cannot diverge (Google penalizes mismatch).
* **Category labels:** breadcrumb category crumbs and any category metadata must use `lib/categories.ts` (`getCategoryMeta` / `getCategoryLabels`) — no ad-hoc category lookups (per project rule).
* **Price formatting boundary:** JSON-LD `offers.price` must be the raw numeric value (no `formatVnd()`), since `lib/format.ts` is for human-facing display only; `priceCurrency` is the separate `"VND"` field.
* **JSON-LD rendering:** prefer a small shared helper/component to serialize JSON-LD safely into a `<script type="application/ld+json">` tag; keep schema-building logic out of `lib/` only if it requires JSX, otherwise data-shaping can live in `lib/`.
* **`next/image`:** all images on these pages already route through `next/image` (existing convention); OG image URLs are absolute static asset paths, independent of `next/image` optimization.
* **Validation:** structured data should validate against Google's Rich Results Test / schema.org; titles/descriptions and canonical correctness verified in-browser before marking UI stories done (per project rule).
* **Sitemap scale:** current catalog (~25 products + posts + categories) is well under the 50,000-URL / 50MB single-sitemap limit; no sitemap-index splitting needed in v1.
