# Feature Specification: Editorial Content
**ID:** `F0006`
**Epic/Module:** Content & Blog
**Status:** Draft

---

## 1. Business Objective
Stand up the SEO-driving editorial system: a Vietnamese blog / buying-guide section authored
in MDX. Long-form posts (`/bai-viet/[slug]/`) are the site's primary organic-search surface ‚Äî
they rank on long-tail buyer-intent keywords and funnel readers into Shopee affiliate clicks via
inline product cards. This feature delivers the blog listing, the post page (hero, MDX body,
sticky TOC, related posts, top-of-post affiliate disclosure), and the `<ProductCard slug="‚Ä¶" />`
MDX component that lets writers embed affiliate cards inside prose. It builds on the content
plumbing already shipped in F0001 (`lib/posts.ts`, `types/post.ts`, MDX pipeline) and reuses the
disclosure banner from F0005 and the product card / pagination / shell-layout primitives from
F0002‚ÄìF0004.

## 2. Scope
### In Scope
* Blog post listing `/bai-viet/` ‚Äî posts ordered by `publishedAt` descending, 12 per page via the
  shared `<Pagination>`, with a left-panel category/tag filter.
* Blog post page `/bai-viet/[slug]/` ‚Äî hero image, single `<h1>`, formatted date + byline,
  top-of-post `<AffiliateDisclosure>`, MDX-rendered body, sticky table of contents (P1), related
  posts (3).
* MDX body rendering: headings, tables, bullet/numbered lists, inline images (via `next/image`),
  code blocks; custom component mapping; stable heading anchors that power the TOC.
* `<ProductCard slug="‚Ä¶" />` MDX component ‚Äî build-time product lookup that renders the existing
  whole-card affiliate `<ProductCard>` inline in prose.
* Vietnamese post-date formatting (`02 th√°ng 5, 2026`) as a single chokepoint in `lib/format.ts`.
* Read-time estimate (P2) shown on listing items and the post header.
* Baseline per-page SEO metadata: unique Vietnamese `<title>`, `<meta name="description">`, and a
  canonical URL on the listing and every post page.
* 1‚Äì2 sample fixture MDX posts shipped with the rendering stories solely to verify the pipeline.

### Out of Scope
* **Frontmatter schema validation** ‚Äî already delivered in F0001 (`lib/posts.ts` validates
  `title`, `summary`, `publishedAt`, `category`, `coverImage`, and `tags[]`, dedupes slugs, sorts
  by date). This feature consumes it; any hardening rides along with US00062, not a new story.
* **Full SEO treatment** ‚Äî `Article` JSON-LD, Open Graph / `og:image`, visible breadcrumbs, and
  sitemap inclusion are owned by F0009. These pages get only baseline title/description/canonical
  here (matching the F0005 precedent); they should be picked up by the F0009 sitemap when it ships.
* **Bulk launch content** ‚Äî the ‚â•5 launch-ready posts in the launch checklist are operational
  content authoring (¬ß7.2), not engineering. F0006 ships only verification fixtures.
* Comments, user-generated content, post reactions (v1 out-of-scope).
* Cross-post / site search (deferred to v2).
* A CMS or admin UI ‚Äî publishing stays file-based (add MDX ‚Üí push to `main` ‚Üí Vercel rebuilds).
* Author profiles / multi-author bylines ‚Äî byline is the site name (anonymous), per ¬ß5.6.

## 3. Process Flow
1. A visitor opens `/bai-viet/` and sees the newest posts first (12 per page), each with a cover
   image, title, summary, date, and category tag.
2. The visitor optionally narrows the list using the left-panel category/tag filter; the URL
   updates (`?category=‚Ä¶`, `?tag=‚Ä¶`) and the grid re-renders. On mobile the filter lives in a
   slide-out drawer.
3. The visitor paginates with crawlable `?page=N` links if there are more than 12 matching posts.
4. The visitor clicks a post and lands on `/bai-viet/[slug]/`: hero image, title, date + byline,
   the affiliate-disclosure banner at the top, then the article body.
5. While reading, the visitor uses the sticky table of contents (desktop) to jump between
   sections; inline `<ProductCard slug="‚Ä¶" />` cards link out to Shopee in a new tab.
6. At the bottom, up to 3 related posts (same category) invite the visitor to keep reading.

## 4. User Stories & Acceptance Criteria

### US00061: Vietnamese post-date formatting utility
**As a** content reader,
**I want to** see post dates in Vietnamese long form (`02 th√°ng 5, 2026`),
**So that** dates read naturally and consistently across the blog.

**Acceptance Criteria:**
* **Scenario 1:** Canonical format
    * `Given` an ISO date string such as `"2026-05-02"`
    * `When` `formatPostDate()` is called with it
    * `Then` it returns `"02 th√°ng 5, 2026"` (zero-padded day, Vietnamese `th√°ng` + numeric month,
      full year).

* **Scenario 2:** Single chokepoint
    * `Given` project conventions for `lib/format.ts` (mirrors `formatVnd`)
    * `When` any blog surface (listing item, post header, related-post card) renders a date
    * `Then` it calls `formatPostDate()` ‚Äî no file outside `lib/format.ts` hand-rolls date
      formatting or calls `toLocaleDateString` / `Intl.DateTimeFormat` on a post date.

* **Scenario 3:** Build-environment stability
    * `Given` the formatter must produce identical output across build environments
    * `When` it is implemented
    * `Then` it does not rely on locale-dependent runtime ICU output that could drift between
      environments (same rationale as `formatVnd`); the Vietnamese form is produced deterministically.

### US00062: MDX body rendering with component mapping & heading anchors
**As a** content writer,
**I want to** have my MDX article body rendered with rich formatting and embeddable components,
**So that** posts display headings, tables, lists, images, and affiliate cards correctly.

**Acceptance Criteria:**
* **Scenario 1:** Rich MDX renders (happy path)
    * `Given` a post whose `content` includes headings, tables, bullet/numbered lists, inline
      images, and code blocks
    * `When` the post body is rendered
    * `Then` all of those elements render as semantic HTML with the site's theme styling.

* **Scenario 2:** Images go through `next/image`
    * `Given` an inline image in the MDX body
    * `When` it renders
    * `Then` it is emitted via `next/image` (per the SEO image rule), not a raw `<img>`.

* **Scenario 3:** Heading anchors for the TOC
    * `Given` `h2`/`h3` headings in the body
    * `When` rendered
    * `Then` each receives a stable, unique `id` (slug derived from the heading text) so the TOC
      (US00068) can link to it and the URL hash deep-links to the section.

* **Scenario 4:** Custom component mapping
    * `Given` the MDX renderer
    * `When` a post embeds a mapped component (e.g. `<ProductCard slug="‚Ä¶" />` from US00063)
    * `Then` the component resolves and renders inline; the mapping integrates with the existing
      `@next/mdx` setup and follows the current Next.js docs (per `AGENTS.md`).

* **Scenario 5:** Verification fixtures
    * `Given` the pipeline needs to be verified
    * `When` this story ships
    * `Then` 1‚Äì2 sample MDX posts exercising the above elements are committed under
      `content/posts/`, sufficient to verify rendering in a browser.

### US00063: `<ProductCard slug="‚Ä¶" />` inline MDX affiliate card
**As a** content writer,
**I want to** drop `<ProductCard slug="‚Ä¶" />` into an MDX post,
**So that** a real affiliate product card appears inline without re-entering product data.

**Acceptance Criteria:**
* **Scenario 1:** Slug resolves to a card (happy path)
    * `Given` an MDX post containing `<ProductCard slug="logitech-g102-lightsync" />`
    * `When` the post is built and rendered
    * `Then` the existing whole-card affiliate `<ProductCard>` (from US00042) renders inline with
      that product's image, name, price (via `formatVnd`), and Shopee CTA.

* **Scenario 2:** Routes through the canonical affiliate seam
    * `Given` the inline card
    * `When` rendered
    * `Then` its click target is the canonical `<AffiliateLink>` (`target="_blank"`,
      `rel="noopener noreferrer sponsored"`, the F0003‚ÜîF0007 `data-*` attributes) ‚Äî no raw affiliate
      anchor is introduced.

* **Scenario 3:** Unknown slug fails loudly at build
    * `Given` an MDX post referencing a slug with no matching product JSON
    * `When` `next build` runs
    * `Then` the build fails with an error naming the offending slug (and ideally the post),
      consistent with the build-time validation pattern used elsewhere ‚Äî never a silent blank or a
      runtime 404.

* **Scenario 4:** Build-time lookup only
    * `Given` SSG conventions
    * `When` the component resolves a product
    * `Then` it looks the product up via `lib/products.ts` (`getProductBySlug`) at build time ‚Äî no
      client-side fetch, and product-data parsing stays in `lib/`.

## 5. Edge Cases & Error Handling
* **Unknown post slug:** `/bai-viet/does-not-exist/` calls `notFound()` and renders the Vietnamese
  404 ‚Äî never a blank page or a build crash.
* **Unknown `<ProductCard slug>`:** fails `next build` with the offending slug named (US00063) ‚Äî a
  broken inline card must never reach production.
* **Empty blog:** with zero posts, `/bai-viet/` shows a Vietnamese empty state and still builds
  (`getAllPosts()` already returns `[]` safely); the post route simply pre-renders no slugs.
* **Filtered no-results:** a category/tag combination matching nothing shows a "no posts" message,
  distinct from the all-empty state, while keeping the filter panel usable.
* **Post with no headings:** the TOC is omitted gracefully (US00068) and the article body renders
  normally.
* **Malformed / missing frontmatter:** already fails the build with the offending file named via
  `lib/posts.ts` (F0001); F0006 must not swallow or bypass that validation.
* **Missing cover image / broken image path:** the page should still render the post; a missing
  `coverImage` must not crash the build (graceful degrade ‚Äî e.g. omit the hero frame).
* **Out-of-range `?page`:** a page number beyond the result set behaves consistently with the
  product listing (clamp/empty), not a crash.
* **Missing `NEXT_PUBLIC_SITE_URL`:** canonical generation follows existing project behavior and
  must not crash the build.
* **Disclosure-text drift:** the top-of-post banner must read `AFFILIATE_DISCLOSURE_VI`; no
  copy-pasted literal may be reintroduced.
* **Trailing-slash consistency:** `/bai-viet/` and `/bai-viet/[slug]/` use the trailing-slash form
  and matching canonical URLs, consistent with the rest of the site.

## 6. Technical Considerations
* **Stack alignment:** SSG only (App Router). Listing and post pages are static Server Components
  with a `metadata` export (or `generateMetadata` for the dynamic post route).
* **Read Next.js docs first:** per `AGENTS.md`, find and read the relevant docs in
  `node_modules/next/dist/docs/` before implementing MDX rendering and dynamic-route generation ‚Äî
  training data is outdated.
* **MDX rendering:** body rendering must integrate with the existing `@next/mdx` +
  `remark-frontmatter` setup (`next.config.ts`) and `@mdx-js/react`. The chosen approach must
  support a per-post component map (for `<ProductCard slug>`), heading anchors (rehype-slug-style),
  and `next/image` for inline images. `gray-matter` already separates frontmatter from body in
  `lib/posts.ts`; rendering consumes `Post.content`.
* **Routes & slugs (keep Vietnamese, trailing slash):**
  * `/bai-viet/` ‚Üí `app/bai-viet/page.tsx`
  * `/bai-viet/[slug]/` ‚Üí `app/bai-viet/[slug]/page.tsx` (+ `not-found.tsx`)
* **Reuse, don't rebuild:**
  * `<AffiliateDisclosure>` (US00051) for the top-of-post banner.
  * `<ProductCard>` (US00042) + `<AffiliateLink>` (US00031‚Äì34) for inline cards.
  * Shared `<Pagination>` (US00043/44) with `basePath=/bai-viet`.
  * `<ShellLayout>` / `<Drawer>` (US00024/25) for the left panel + mobile collapse.
  * `lib/categories.ts` for category display labels (no new taxonomy registry).
* **Filter state lives in the URL:** `?category=` and `?tag=`, round-tripped through a helper in
  the F0004 style (extend `lib/filters.ts` or add a sibling loader) ‚Äî comma-separated multi-value,
  unknown values ignored, filter change strips `?page`. No Context / `localStorage`. Per `CLAUDE.md`,
  new filter dimensions must follow this same pattern.
* **Single chokepoints:** dates via `formatPostDate()` in `lib/format.ts`; prices via `formatVnd()`;
  affiliate URLs via `lib/affiliate.ts` / `<AffiliateLink>`; disclosure via `AFFILIATE_DISCLOSURE_VI`.
* **Build-time data only:** `getAllPosts()` / `getPostBySlug()` (posts) and `getProductBySlug()`
  (inline cards) are read at build time; no client fetching, no JSX in `lib/`.
* **SEO baseline only:** provide unique title/description/canonical now; defer `Article` JSON-LD,
  Open Graph / `og:image`, visible breadcrumbs, and sitemap inclusion to F0009 (posts should be
  picked up by the F0009 sitemap when it ships).
* **Components:** PascalCase, Server Components by default (`"use client"` only where interaction
  requires it ‚Äî e.g. the filter reader/writer and any TOC scroll behavior), with co-located
  `<Name>.module.css`.
* **Dependencies:** F0001 (posts loader, types, MDX pipeline), F0002 (shell, drawer, theme),
  F0004 (`<ProductCard>`, `<Pagination>`, `lib/categories.ts`), F0005 (`<AffiliateDisclosure>`).
* **Verification:** per project rules, verify in a browser before marking UI stories done ‚Äî
  listing renders newest-first with working pagination/filters, a post renders with hero +
  disclosure + MDX body + TOC + related posts, inline `<ProductCard>` opens Shopee in a new tab,
  and unknown slugs 404.
