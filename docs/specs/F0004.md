# Feature Specification: Product Catalog

**ID:** `F0004`
**Epic/Module:** Storefront Core
**Status:** Approved

---

## 1. Business Objective

Turn the JSON product files in `content/products/` into a browsable catalog: a paginated all-products listing, SEO-focused category pages, and rich product detail pages — each surface delivering visitors to a Shopee affiliate URL via the canonical `<AffiliateLink>` primitive shipped in F0003. Together these are the storefront's primary monetization surfaces. The Product Catalog is also the visual home of the reusable `<ProductCard />` component, which subsequent features (F0006 inline blog embeds, F0008 homepage featured picks) compose to surface products in their own contexts. Without F0004, the JSON content has no rendering layer, no SEO-rankable category pages exist, and no detail page exists to host the prominent "Mua trên Shopee" CTA.

## 2. Scope

### In Scope

- Vietnamese price-formatting helper (`formatVnd()`) in `lib/format.ts` — single chokepoint that renders any `number` as `"₫1.200.000"` (dot thousands separator, ₫ prefix)
- Reusable `<ProductCard />` Server Component used by the listing grid, category grid, related-products row, and (by F0006/F0008 wrapping) inline MDX embeds and homepage featured picks — implemented as a whole-card `<AffiliateLink>` per F0003's US00032 pattern
- Product listing page `/san-pham/` — paginated grid of all products, 24 per page, responsive 2 / 3 / 4 columns, crawlable `?page=N` pagination
- Listing filters & sort (left panel, client-side, URL-driven) — category, brand, price bucket, and sort (relevance / price asc / price desc / newest); filtered views are shareable via the URL but canonicalize back to `/san-pham/`
- Category page `/danh-muc/[category]/` — statically generated per distinct category value in product JSON; renders a 100–200 word Vietnamese intro paragraph above the filtered grid and inherits the same pagination strategy
- Product detail page `/san-pham/[slug]/` — `<h1>` product name, brand, formatted price, image gallery, short description, specs table, prominent button-styled `<AffiliateLink>` CTA, and the related-products section
- Related products (3–4) — same-category siblings rendered on the product detail page via `<ProductCard />`; deterministic order (no randomness in SSG output)
- Pagination canonical strategy: page 1 of any listing canonicalizes to the bare path (no `?page=1`); pages 2+ self-canonicalize so each paginated page is discoverable and rankable independently
- Vietnamese microcopy for empty filter results, single-image vs. multi-image gallery presentation, and category intro copy

### Out of Scope

- Free-text search across products (deferred to v2 — explicitly **not** part of the filter UI)
- The MDX `<ProductCard slug="..." />` wrapper used by blog posts (F0006 — Editorial; it composes F0004's `<ProductCard />` under the hood)
- Homepage featured-picks layout (F0008 — Homepage; it composes F0004's `<ProductCard />`)
- JSON-LD `Product` schema, breadcrumb JSON-LD, Open Graph tags, per-page `<title>`/meta description (F0009 — SEO & Discoverability)
- The `sitemap.xml` and `robots.txt` files (F0009)
- GA4 `affiliate_click` event emission (F0007 — the seam is already provided by `<AffiliateLink>` per F0003)
- Real-time price sync from Shopee, stock indicators, user-saved products, wishlists, reviews (v2+)
- Custom product imagery beyond the `images: string[]` JSON field (zoom-on-hover, lightbox, video, fullscreen modal)
- Cross-category recommendation algorithms (related is strictly same-category in v1)
- Server-side rendering or ISR — every catalog route is statically generated

## 3. Process Flow

_Two "users" here: the visitor browsing the catalog, and a developer/content editor publishing a new product._

1. A content editor drops a new product JSON file into `content/products/<slug>.json` (one file per product, conforming to the `Product` type) and pushes to `main`.
2. The next build runs. `lib/products.ts` reads every JSON file, validates each `affiliateUrl` against the Shopee allow-list via F0003's `assertAffiliateUrl()` helper, and exposes `getAllProducts()` and `getProductBySlug()`.
3. The build statically generates `/san-pham/` (paginated), `/danh-muc/[category]/` (one route per distinct category, also paginated), and `/san-pham/[slug]/` (one route per product) using Next.js `generateStaticParams`.
4. A visitor lands on `/san-pham/`, sees the paginated grid in the F0002 two-column shell, and uses the left-panel filters to narrow by category / brand / price / sort. URL updates as filters change so the view is shareable.
5. Alternatively, the visitor lands on `/danh-muc/chuot-gaming/` from organic search, reads the intro paragraph, and scrolls the filtered grid.
6. The visitor clicks any `<ProductCard />`. The whole-card `<AffiliateLink>` opens the Shopee destination in a new tab; the storefront tab remains.
7. Or the visitor clicks the product name / image inside the card and navigates to `/san-pham/[slug]/` for the detail view (gallery, specs, big "Mua trên Shopee" CTA, related products).
8. On the detail page, the visitor clicks the prominent CTA — same `<AffiliateLink>` semantics, opens in a new tab. They may also scroll to the related-products row and click through to a sibling product.

## 4. User Stories & Acceptance Criteria

### US00041: Vietnamese price formatter (`formatVnd()`)

**As a** developer rendering a price on any product surface,
**I want to** call a single shared helper that returns the canonical Vietnamese price string,
**So that** every card, detail page, and inline embed renders prices identically and a future change (e.g., locale, separator) touches one file.

**Acceptance Criteria:**

- **Scenario 1:** Helper exists at the canonical location
  - `Given` the codebase
  - `When` reviewed
  - `Then` `lib/format.ts` exports a named function `formatVnd(amount: number): string`

- **Scenario 2:** Standard amount renders with dot thousands separator and ₫ prefix
  - `Given` `formatVnd(1200000)`
  - `When` called
  - `Then` it returns the string `"₫1.200.000"`

- **Scenario 3:** Sub-thousand amounts render without separator
  - `Given` `formatVnd(390)`
  - `When` called
  - `Then` it returns `"₫390"`

- **Scenario 4:** Zero renders as `₫0`
  - `Given` `formatVnd(0)`
  - `When` called
  - `Then` it returns `"₫0"`

- **Scenario 5:** Non-integer input is rounded to the nearest integer dong
  - `Given` `formatVnd(390000.7)`
  - `When` called
  - `Then` it returns `"₫390.001"` (the dong is the smallest unit; no decimal places ever render)

- **Scenario 6:** Single chokepoint convention
  - `Given` the codebase
  - `When` reviewed
  - `Then` no file outside `lib/format.ts` does ad-hoc price formatting (no inline `toLocaleString` calls on price values, no string concatenation with `"₫"`); all price renders flow through `formatVnd()`

- **Scenario 7:** Pure and SSG-safe
  - `Given` the helper
  - `When` invoked during static generation
  - `Then` it depends on no browser APIs, no `Intl` runtime locale resolution that would vary between build environments, and produces identical output for identical input across machines

### US00042: Reusable `<ProductCard />` component

**As a** developer building any grid or row of products,
**I want to** drop in a single card component that renders the standard product summary as a whole-card affiliate link,
**So that** every product surface looks consistent and we never re-solve link semantics, accessibility, or layout per call site.

**Acceptance Criteria:**

- **Scenario 1:** Component renders the standard summary
  - `Given` `<ProductCard product={product} />` where `product` is a valid `Product`
  - `When` the page renders
  - `Then` the card contains: the primary image (via `next/image`), the product name, the brand, the formatted price (via `formatVnd()`), the category badge, and a visually styled "Mua ngay" CTA element

- **Scenario 2:** Whole card is a single affiliate link
  - `Given` the card is rendered
  - `When` the DOM is inspected
  - `Then` the card subtree is wrapped by exactly one `<AffiliateLink>` (from F0003) using the `affiliateStyles.card` pattern, the `<a>` element carries the F0003 ↔ F0007 `data-affiliate-link` / `data-product-name` / `data-product-category` / `data-destination-url` attributes, and there are zero nested `<button>` or `<a>` elements inside

- **Scenario 3:** CTA is a non-interactive styled element
  - `Given` the rendered card
  - `When` the "Mua ngay" CTA is inspected
  - `Then` it is a styled `<span>` (or equivalent non-interactive element), never `<button>` or `<a>` — preserving the single-link tab stop required by F0003's US00032

- **Scenario 4:** Image uses `next/image` with appropriate sizing
  - `Given` the card on any grid (2 / 3 / 4 cols depending on viewport)
  - `When` the image renders
  - `Then` it uses `next/image` with `sizes` configured so the browser requests the appropriate width per breakpoint (no fixed huge image at mobile sizes)

- **Scenario 5:** Server Component by default
  - `Given` the `<ProductCard />` source file
  - `When` reviewed
  - `Then` it does not contain `"use client"`; the card ships zero hydration cost

- **Scenario 6:** Accessible product context
  - `Given` a screen-reader user encounters the card
  - `When` it is announced
  - `Then` it conveys: product name, brand, price, and the F0003 accessible label ("Mở [productName] trên Shopee (mở tab mới)"); decorative card chrome and the styled "Mua ngay" element do not produce duplicate announcements

- **Scenario 7:** Responsive layout
  - `Given` the card is used inside a grid container that switches between 2 / 3 / 4 columns at the F0001/F0002 breakpoints
  - `When` viewed
  - `Then` image, text, and CTA reflow without overflow, text truncates instead of wrapping past two lines for the name, and the card maintains a consistent height per row

- **Scenario 8:** Renders zero "Mua ngay" CTA padding regressions
  - `Given` the card uses the `affiliateStyles.card` whole-card pattern from F0003
  - `When` keyboard-focused
  - `Then` the focus-visible indicator covers the whole card surface, not just the inner CTA, and respects `prefers-reduced-motion` for any hover/focus transitions

### US00043: Product listing page `/san-pham/` (paginated grid)

**As a** visitor,
**I want to** browse every product in the storefront on a single paginated listing page,
**So that** I can scan the full catalog without committing to a specific category up front.

**Acceptance Criteria:**

- **Scenario 1:** Route exists and renders the F0002 two-column shell
  - `Given` the visitor navigates to `/san-pham/`
  - `When` the page renders
  - `Then` it uses F0002's two-column shell (`<ShellLayout />`) with a left panel slot (filters; see US00044) and main content slot (the grid), inside the F0002 Header/Footer chrome

- **Scenario 2:** Grid renders up to 24 products per page in a responsive layout
  - `Given` more than 24 products exist
  - `When` page 1 renders
  - `Then` exactly 24 `<ProductCard />` instances appear, arranged as 2 columns on viewports <768px, 3 columns on 768–1279px, and 4 columns on ≥1280px

- **Scenario 3:** Page is statically generated
  - `Given` the build runs
  - `When` `/san-pham/` is generated
  - `Then` the page is pre-rendered HTML at build time (no `dynamic = "force-dynamic"` or runtime server rendering); the product list is read from `lib/products.ts` during build

- **Scenario 4:** Pagination via `?page=N` query params
  - `Given` more than 24 products exist
  - `When` the visitor scrolls to the bottom of page 1
  - `Then` a pagination control renders with anchor links to subsequent pages using `?page=2`, `?page=3`, etc., each a real `<a href="…">` so it is crawlable

- **Scenario 5:** Page N renders the correct slice
  - `Given` the visitor navigates to `/san-pham/?page=3`
  - `When` the page renders
  - `Then` it shows products at offset 48–71 (the third 24-product slice), in the same order as page 1's underlying sort

- **Scenario 6:** Out-of-range page numbers return 404
  - `Given` only 50 products exist total (3 pages)
  - `When` the visitor navigates to `/san-pham/?page=99`
  - `Then` the page returns Next.js `notFound()` (the F0002 chrome still wraps it; only the main content is the 404 surface)

- **Scenario 7:** Pagination canonical strategy
  - `Given` page 1
  - `When` inspected
  - `Then` the canonical URL is the bare path `/san-pham/` (not `/san-pham/?page=1`); `Given` pages 2+, `Then` each page's canonical points at itself (`/san-pham/?page=N`) so each paginated page is independently rankable

- **Scenario 8:** Empty catalog renders an empty state
  - `Given` zero products exist in `content/products/`
  - `When` `/san-pham/` renders
  - `Then` the grid is replaced by a Vietnamese empty-state message (e.g., "Chưa có sản phẩm nào.") and the build does not fail

- **Scenario 9:** Default product order is stable
  - `Given` two builds run against the same product JSON content
  - `When` both pages render
  - `Then` the order of products on `/san-pham/` is identical between builds (e.g., sorted by `publishedAt` desc with `slug` as tiebreaker — choose one deterministic order)

### US00044: Listing filters & sort (left panel, client-side, URL-driven)

**As a** visitor on `/san-pham/`,
**I want to** narrow the catalog by category, brand, and price, and re-sort the results,
**So that** I can find products that match my criteria without paging through everything.

**Acceptance Criteria:**

- **Scenario 1:** Filter UI renders in the F0002 left panel
  - `Given` `/san-pham/` is rendered on a desktop viewport
  - `When` the left panel slot is inspected
  - `Then` it contains filter groups for Category (checkboxes), Brand (checkboxes), Price (preset buckets as radio or checkboxes), and a Sort control (select or radio group)

- **Scenario 2:** Filter options are derived from product JSON
  - `Given` the build runs
  - `When` the filter UI is generated
  - `Then` the Category and Brand options reflect the distinct `category` and `brand` values across all products in `content/products/` (no hand-maintained list); price buckets are predefined Vietnamese-shopper-appropriate ranges (e.g., `<200k`, `200–500k`, `500k–1tr`, `1tr–3tr`, `>3tr`)

- **Scenario 3:** Sort options are exactly the four documented choices
  - `Given` the sort control
  - `When` inspected
  - `Then` it offers exactly four options: `Mặc định` (default — same stable order as US00043 Scenario 9), `Giá tăng dần`, `Giá giảm dần`, `Mới nhất` (by `publishedAt` desc)

- **Scenario 4:** Filtering is client-side
  - `Given` the page is statically generated with all products in the HTML payload
  - `When` the visitor toggles a filter
  - `Then` the grid re-renders in the browser without a network request, filtering in-memory; no additional product data is fetched

- **Scenario 5:** Filter state is reflected in the URL
  - `Given` the visitor selects Category=`chuot-gaming` and Brand=`Logitech`
  - `When` the selections are applied
  - `Then` the URL updates to include query params (e.g., `/san-pham/?category=chuot-gaming&brand=Logitech`) without a full page reload, and reloading the URL restores the same filter state

- **Scenario 6:** Filtered view canonicalizes to the unfiltered listing
  - `Given` any filter is active
  - `When` the page's canonical URL is inspected
  - `Then` it points at the bare `/san-pham/` (or `/san-pham/?page=N` for the underlying pagination of the unfiltered listing) — never at the filter combination — so Google consolidates rank to the listing, not to filtered variants

- **Scenario 7:** Pagination resets when filters change
  - `Given` the visitor is on `/san-pham/?page=3`
  - `When` they apply or change a filter
  - `Then` the view returns to page 1 of the filtered set (the `?page` query param is dropped or set to 1)

- **Scenario 8:** Mobile: filters live in the F0002 drawer
  - `Given` a viewport <768px
  - `When` the visitor opens the F0002 left-panel drawer (US00025)
  - `Then` the same filter UI is rendered inside the drawer; selections take effect immediately and persist when the drawer closes

- **Scenario 9:** Empty filter result renders a Vietnamese empty state
  - `Given` the visitor's filter combination yields zero products
  - `When` the grid would render
  - `Then` it is replaced by a Vietnamese empty-state message (e.g., "Không có sản phẩm phù hợp với bộ lọc.") and a "Xóa bộ lọc" action that clears all filters

- **Scenario 10:** No free-text search input
  - `Given` the filter UI
  - `When` inspected
  - `Then` there is no `<input type="search">` or `<input type="text">` for filtering by name; filtering operates exclusively on enumerated category / brand / price values (search is v2)

- **Scenario 11:** Active filter count and clear action
  - `Given` one or more filters are active
  - `When` the left panel is inspected
  - `Then` a count or chip set indicates which filters are active and a single "Xóa bộ lọc" action clears them all in one interaction

### US00045: Category page `/danh-muc/[category]/`

**As a** visitor arriving from organic search for a specific category,
**I want to** see a dedicated, content-rich page for that category with a filtered grid,
**So that** I land on a focused surface that answers my intent (and ranks well for the category query).

**Acceptance Criteria:**

- **Scenario 1:** One route per distinct category value
  - `Given` the product JSON contains distinct `category` values (e.g., `chuot-gaming`, `tai-nghe`, `ban-phim`)
  - `When` the build runs
  - `Then` `app/danh-muc/[category]/page.tsx` uses `generateStaticParams` to emit one static route per distinct category — and only those categories

- **Scenario 2:** Unknown category returns 404
  - `Given` the visitor navigates to `/danh-muc/khong-ton-tai/`
  - `When` the page would render
  - `Then` it returns `notFound()` (Next.js 404 surface) rather than an empty grid

- **Scenario 3:** 100–200 word Vietnamese intro paragraph
  - `Given` the category page renders
  - `When` inspected above the grid
  - `Then` it contains a Vietnamese intro paragraph between 100 and 200 words for the specific category — sourced from a per-category content map (`lib/categories.ts` or similar) so adding a new category requires adding an intro entry

- **Scenario 4:** Filtered grid uses `<ProductCard />`
  - `Given` the category page renders
  - `When` the grid is inspected
  - `Then` it contains only products whose `category` exactly matches the route param, rendered as `<ProductCard />` instances in the same responsive 2 / 3 / 4-col grid as `/san-pham/`

- **Scenario 5:** Pagination inherits the listing strategy
  - `Given` the category contains more than 24 products
  - `When` the page renders
  - `Then` it paginates via `?page=N` exactly like US00043 — same 24-per-page slice, same canonical rules (page 1 → bare path, pages 2+ → self-canonical), same crawlable anchor links

- **Scenario 6:** Empty category surfaces an empty state, build still passes
  - `Given` a category JSON entry exists in the intro map but no products are tagged with that category yet
  - `When` the page renders
  - `Then` the grid shows the Vietnamese empty-state message and the build does not fail

- **Scenario 7:** Category page lives inside the F0002 shell
  - `Given` the page renders
  - `When` inspected
  - `Then` it uses `<ShellLayout />` (F0002's US00024) with the Header/Footer chrome; the left panel slot may surface category-specific navigation (e.g., other categories, or brand sub-filter — implementation detail) or remain empty

- **Scenario 8:** Visible heading hierarchy
  - `Given` the category page
  - `When` inspected
  - `Then` it has exactly one `<h1>` containing the Vietnamese category name (e.g., "Chuột gaming"), and the intro paragraph follows; the grid does not introduce nested `<h1>` elements via `<ProductCard />`

### US00046: Product detail page `/san-pham/[slug]/`

**As a** visitor who clicked a product name or image,
**I want to** see a focused page with the product's images, specs, price, and a prominent "Mua trên Shopee" CTA,
**So that** I have enough information to decide and a clear path to the affiliate destination.

**Acceptance Criteria:**

- **Scenario 1:** One static route per product slug
  - `Given` each product JSON file in `content/products/`
  - `When` the build runs
  - `Then` `app/san-pham/[slug]/page.tsx` uses `generateStaticParams` to emit one route per product slug; unknown slugs return `notFound()`

- **Scenario 2:** Page surfaces the standard product fields
  - `Given` a valid product slug
  - `When` the page renders
  - `Then` it shows: the product name as the single `<h1>`, the brand, the formatted price (via `formatVnd()`), the short description, a specs table built from `product.specs` (key/value rows), and the prominent affiliate CTA

- **Scenario 3:** Affiliate CTA uses `<AffiliateLink>` (not the whole-card pattern)
  - `Given` the detail page renders
  - `When` the CTA is inspected
  - `Then` it is a `<AffiliateLink>` styled as a prominent button (e.g., orange band, white text, large hit target), with visible label "Mua trên Shopee", that bakes in the F0003 link semantics and emits the F0003 ↔ F0007 data attributes; the page surface itself is **not** wrapped as a whole-card link

- **Scenario 4:** Image gallery — single-image case
  - `Given` `product.images` has exactly one entry
  - `When` the page renders
  - `Then` a single `next/image` renders prominently without thumbnail chrome; the image has a non-empty `alt` (the product name)

- **Scenario 5:** Image gallery — multi-image case
  - `Given` `product.images` has more than one entry
  - `When` the page renders
  - `Then` the first image renders as the main image, the remaining images render as a thumbnail row beneath; clicking a thumbnail swaps the main image without a full page navigation

- **Scenario 6:** Specs table renders only when specs exist
  - `Given` `product.specs` is empty or missing
  - `When` the page renders
  - `Then` the specs section is omitted entirely (no empty table, no zero-row header); when specs exist, each key/value pair renders as a row with the spec name on the left

- **Scenario 7:** Related-products row appears at the bottom
  - `Given` the page renders
  - `When` scrolled to the bottom of the main content
  - `Then` the related-products section (US00047) renders below the primary content, before the F0002 Footer

- **Scenario 8:** Page lives inside the F0002 chrome but does not need the two-column shell
  - `Given` the detail page renders
  - `When` inspected
  - `Then` Header and Footer wrap the page normally; the detail layout uses the full container width inside `<main>` (no `<ShellLayout />`) so the gallery and specs have room to breathe

- **Scenario 9:** Statically generated with no client hydration in the default path
  - `Given` the build runs
  - `When` the page is generated
  - `Then` the route renders as static HTML; only the thumbnail-swap behavior (multi-image gallery) introduces client JS, scoped to a small client component (e.g., `<ProductGallery />`) — the rest of the page is Server Components

- **Scenario 10:** Vietnamese microcopy for CTA and related-products heading
  - `Given` the page renders
  - `When` inspected
  - `Then` the CTA label is "Mua trên Shopee", the related-products section heading is "Sản phẩm liên quan" (or equivalent), and all visitor-facing strings on the page are Vietnamese

### US00047: Related products section (3–4 same-category)

**As a** visitor on a product detail page,
**I want to** see a small selection of similar products at the bottom,
**So that** I can compare alternatives without going back to the listing.

**Acceptance Criteria:**

- **Scenario 1:** Same-category siblings are surfaced
  - `Given` a product detail page for slug `X` with category `chuot-gaming`
  - `When` the related-products section renders
  - `Then` it shows up to 4 products whose `category` equals `chuot-gaming` and whose slug is **not** `X`, rendered as `<ProductCard />` instances

- **Scenario 2:** Deterministic ordering
  - `Given` more than 4 same-category siblings exist
  - `When` the related-products section renders
  - `Then` the selection and order are deterministic across builds (e.g., the 4 nearest by `publishedAt` desc, with `slug` as tiebreaker) — no `Math.random()` or build-time randomness

- **Scenario 3:** Minimum of 3 — fallback when fewer same-category siblings exist
  - `Given` only 2 same-category siblings exist (excluding the current product)
  - `When` the related-products section renders
  - `Then` it fills the row to at least 3 cards by adding the most recently published products from any other category, still excluding the current product; the order remains deterministic

- **Scenario 4:** Section is omitted only when the catalog has fewer than 3 other products total
  - `Given` the catalog contains fewer than 3 products other than the current one
  - `When` the detail page renders
  - `Then` the related-products section is omitted entirely (no empty heading, no single-card row)

- **Scenario 5:** Responsive row layout
  - `Given` the related-products section renders
  - `When` viewed
  - `Then` cards arrange as 2 columns on viewports <768px, 3 columns on 768–1279px, and 3–4 columns on ≥1280px (depending on count) — visually consistent with the listing grid

- **Scenario 6:** Server-rendered, no extra client cost
  - `Given` the section
  - `When` inspected
  - `Then` it ships as Server Component output; the selection is computed at build time from `getAllProducts()`, not at runtime in the browser

## 5. Edge Cases & Error Handling

- **Product with missing or empty `images` array.** The card and detail page must fail the build with a clear error naming the offending slug — every product surface depends on at least one image. (Validate in `lib/products.ts` alongside F0003's `assertAffiliateUrl()`.)
- **Product with missing `price` (zero or negative).** Treat zero as a valid display (renders `₫0`), but negative numbers should fail the build. Missing field should fail the build too.
- **Product whose `category` does not match any entry in the per-category intro map.** Fail the build with a clear error so a new product cannot ship without a category page intro; or — alternative — the per-category intro is *required* for any category that appears in product JSON, enforced at build via `lib/categories.ts`.
- **Filter combination yielding zero products.** Show empty-state message + "Xóa bộ lọc" action (US00044 Scenario 9); do **not** show "no results" without an escape hatch.
- **Pagination + filter interaction.** Changing any filter resets `?page` to 1 (US00044 Scenario 7); changing page does not alter filter selections.
- **Same product appearing on `/san-pham/?page=2` and `/danh-muc/chuot-gaming/?page=1`.** Expected — these are different surfaces with different canonical destinations. No deduplication needed.
- **Multi-image gallery with broken thumbnails (image file missing).** `next/image` should surface a build-time error since images are referenced from the static asset path; do not silently fall through to a broken image at runtime.
- **Specs table with very long values.** Long values should wrap inside the row, not horizontally overflow the container.
- **Visitor with JavaScript disabled.**
  - `/san-pham/` and `/danh-muc/[category]/`: the grid + pagination + affiliate-card clicks all work (links are real `<a>` elements). Filters and sort do not work (acceptable — the URL canonicalizes to the unfiltered listing anyway).
  - `/san-pham/[slug]/`: page, CTA, and specs all work; thumbnail-swap on multi-image gallery does not (the first image still renders, which is sufficient).
- **`prefers-reduced-motion`.** Card hover/focus transitions, thumbnail-swap, any filter-applied animations must respect the user preference (instant state changes, no fade/slide).
- **Touch-target sizes on mobile.** Filter checkboxes, sort options, pagination links, and the gallery thumbnails must each have ≥44×44 px touch targets per F0002 conventions.
- **Long product names.** Card truncates after two lines (US00042 Scenario 7); detail page `<h1>` wraps naturally with no truncation.
- **Duplicate slugs in product JSON.** Build must fail with a clear error naming the offending file pair — slug is the primary key for the detail route.
- **Build time vs. content scale.** With v1 product counts (≤200 products expected), `generateStaticParams` for `/san-pham/[slug]/` is fine. If the catalog grows past a few thousand, revisit (out of scope for this story).
- **Filter param tampering.** A URL like `/san-pham/?category=<script>` should not cause a hydration error or render the malicious string into HTML; React's default escaping handles this, but the client-side filter logic must compare exact-match strings, not eval-style operations.
- **Visitor lands on a paginated category URL after the underlying products change.** A bookmarked `/danh-muc/chuot-gaming/?page=4` may 404 after products were removed; acceptable v1 behavior. The pagination control on page 1–3 will not link to a stale page 4.

## 6. Technical Considerations

- **SSG everywhere.** Every route in this feature is statically generated via `generateStaticParams`. No `dynamic = "force-dynamic"`. The grid pages may use small client components for filter / sort / gallery thumbnail-swap, but the document is pre-rendered HTML.
- **F0003 is a hard dependency.** Every `<ProductCard />`, every detail-page CTA, every related-product card uses `<AffiliateLink>`. No raw `<a href="https://shope.ee/…">` anywhere — that's blocked by the convention in F0003's US00034.
- **F0002 is a hard dependency.** Listing and category pages use `<ShellLayout />` (US00024). All catalog routes inherit Header (US00021) and Footer (US00022) via root layout. Mobile filters use F0002's drawer (US00025).
- **F0001's product loader is the data source.** `getAllProducts()` and `getProductBySlug()` from `lib/products.ts` already exist and already call `assertAffiliateUrl()` per US00034. F0004 must not duplicate or sidestep them.
- **`formatVnd()` is the single price chokepoint.** Per F0003's pattern of one-helper-one-job (`assertAffiliateUrl`), prices flow through `formatVnd()` and only `formatVnd()`. No `Intl.NumberFormat` calls scattered through components.
- **Pagination canonical rules are SEO-load-bearing.** Page 1 of `/san-pham/` and `/danh-muc/[category]/` canonicalizes to the bare path; pages 2+ self-canonicalize. Filter combinations canonicalize to the bare listing. Implementing this requires per-page metadata generation (`generateMetadata`) — coordinate the exact API with F0009 when it lands, but ship the canonical attribute now.
- **Filter UI is a client component, but a thin one.** The component receives the full product set from a Server Component parent (already in the HTML), reads the URL query params on mount, filters in-memory, and writes URL updates back via Next.js `useRouter().replace` (no full page reload). Avoid pulling in a state-management library.
- **Filter options derive from product data at build time.** Build a `lib/filters.ts` (or co-located helper) that computes distinct categories, brands, and the static price buckets. Per-category intro copy lives in `lib/categories.ts` (or similar) — both are pure functions / constants, no I/O after build.
- **Image gallery on multi-image products is a small client component.** `<ProductGallery />` (`"use client"`) accepts the image array and manages local state for which thumbnail is active. The single-image path renders pure server HTML — most products are expected to ship one image.
- **Related products are computed at build time, not runtime.** For each product, `getRelatedProducts(currentProduct)` returns 3–4 deterministic siblings as a Server Component computation; the detail page passes them to `<ProductCard />`.
- **Empty states have Vietnamese microcopy.** "Chưa có sản phẩm nào.", "Không có sản phẩm phù hợp với bộ lọc.", "Xóa bộ lọc", "Mua trên Shopee", "Sản phẩm liên quan" — treat as canonical Vietnamese user-facing copy. Centralize in a `lib/copy.ts` if any string appears in two places.
- **No new heavy dependencies.** Filters, gallery, pagination — all implementable in vanilla React + Next.js. No carousel library, no UI kit, no state library.
- **`getStaticPaths` semantics.** For `/san-pham/[slug]/` and `/danh-muc/[category]/`, use `generateStaticParams` (App Router API) returning the full product/category list; rely on Next.js `notFound()` for any path not in that list.
- **`<h1>` discipline.** Listing pages have one `<h1>` ("Tất cả sản phẩm" or similar); category pages have one `<h1>` (the Vietnamese category name); detail pages have one `<h1>` (the product name). `<ProductCard />` must never use `<h1>` for the product name — use a non-heading element or appropriate sub-heading depending on context.
- **Component file locations.** `components/ProductCard.tsx` + `components/ProductCard.module.css`; `components/ProductGallery.tsx` (client); `components/CatalogFilters.tsx` (client); `components/Pagination.tsx`. New helpers: `lib/format.ts`, `lib/categories.ts`, optionally `lib/filters.ts`.
- **English identifiers, Vietnamese user-facing copy.** Per project conventions. Filter labels, sort labels, empty-state strings, CTA labels are Vietnamese; component names, prop names, file names are English.
- **F0009 will harden SEO on top of F0004.** Open Graph tags, `Product` / `BreadcrumbList` JSON-LD, sitemap entries, and breadcrumb UI are F0009's job. F0004 ships the routes and the pagination/canonical strategy that F0009 will build on.
- **F0008 will consume `<ProductCard />` for the homepage's featured picks.** F0004 must produce a card that is composable in non-grid contexts (a small horizontal row, an inline blog embed) without layout assumptions that only work inside the full listing grid.
