# Feature Specification: Site Shell & Theme

**ID:** `F0002`
**Epic/Module:** Core UI / Global Chrome
**Status:** Draft

---

## 1. Business Objective

Deliver the consistent global chrome (header, footer, layout shell) that every page on the affiliate storefront inherits, with a visual identity that immediately evokes the Shopee marketplace — bold orange brand band on top, white nav and logo, light-gray page surface with white content cards, hairline borders, and dense compact spacing. The shell sets the trust and familiarity cues Vietnamese shoppers expect from Shopee-adjacent affiliate sites, while remaining fast (system font stack only — no external fonts) and accessible (semantic landmarks, skip link, keyboard-navigable drawer). Subsequent features (F0004 product catalog, F0006 editorial, F0008 homepage, F0010 about) plug into this shell rather than re-solving header/footer/layout per page.

## 2. Scope

### In Scope

- Site Header: solid orange brand band (using `--color-primary`, default `#EE4D2D`), white logo + site name + primary nav (`Trang chủ · Sản phẩm · Bài viết · Về chúng tôi`), mobile hamburger that opens a drawer
- Site Footer: multi-column layout (`Về chúng tôi` · `Chính sách` · `Liên hệ` · `Theo dõi`), thin orange top-accent line, Vietnamese affiliate-disclosure block, links to Privacy Policy and Affiliate Disclosure pages (destinations ship in F0005), copyright with current year computed at build
- Root layout integration: Header + `<main>` + Footer wrap every route via `app/layout.tsx`, with a skip-to-main-content link; preserves `<html lang="vi">`, `<SpeedInsights />`, and the `globals.css` import added by prior stories
- Additive design tokens for Shopee surfaces: `--color-page-bg` (`#F5F5F5`), `--color-surface` (`#FFFFFF`), `--color-border-hairline` (`#E5E5E5`), `--color-on-primary` (`#FFFFFF`), `--radius-sm` (`2px`), `--radius-md` (`4px`)
- Reusable two-column shell layout component (left panel ~280px + main content), with the left panel rendered as a white "card" surface with hairline border and small radius, content container max-width ~1280px
- Mobile-responsive left panel: collapses to a slide-out drawer with dimmed overlay below 768px, keyboard accessible (Esc to close, focus trap, focus restoration)
- Sticky-on-scroll header (P2): header becomes sticky after a small scroll threshold with a subtle drop shadow

### Out of Scope

- Privacy Policy and Affiliate Disclosure destination pages (F0005) — F0002 only links to their reserved paths
- About page content at `/ve-chung-toi/` (F0010)
- Concrete page bodies: product listing, product detail, blog listing, blog post, homepage (F0004 / F0006 / F0008)
- Search bar in the header (search is deferred to v2)
- Top utility strip / category quick-links band under the header (not in v1)
- Dark mode (v2+)
- Cookie consent banner (deferred until GA goes live)
- External font loading — system font stack only
- Re-defining the base CSS reset, breakpoint tokens, or `--color-primary` (already shipped via F0001's US00016 — this feature only extends tokens additively)

## 3. Process Flow

_The "user" here is any visitor arriving on any page, plus a developer building a new page on top of the shell._

1. Visitor lands on any route (homepage, product page, blog post, etc.).
2. The orange Shopee-style Header renders at the top with logo, site name, and primary nav links; the body of the page sits on a light-gray page surface (`--color-page-bg`).
3. On routes that opt into the two-column shell (product listing, blog listing, blog post), the left panel renders as a white card to the left of the main content; on routes that don't (homepage, About), main content spans full container width.
4. The visitor scrolls — at the P2 sticky threshold, the Header pins to the top with a subtle drop shadow.
5. On a viewport <768px, the visitor sees a hamburger / filter trigger in the Header. Tapping it opens a slide-out drawer containing the nav (or the left panel filters, depending on the page) with a dimmed overlay; focus moves into the drawer and Esc / overlay click closes it, returning focus to the trigger.
6. The visitor scrolls to the bottom and sees the multi-column Footer with the affiliate-disclosure block, Privacy / Affiliate Disclosure links, contact info, and current-year copyright.
7. A developer building a new page imports the two-column shell layout component (or simply relies on root layout) and never re-implements Header, Footer, or chrome.

## 4. User Stories & Acceptance Criteria

### US00021: Shopee-style Site Header

**As a** visitor,
**I want to** see a consistent, branded orange header on every page with clear navigation,
**So that** I instantly recognize where I am, trust the site, and can reach any major section.

**Acceptance Criteria:**

- **Scenario 1:** Header renders with Shopee brand band
  - `Given` any page on the site
  - `When` the page loads
  - `Then` the Header renders as a solid orange band using `var(--color-primary)` as its background, with white text/icons on top using `var(--color-on-primary)`

- **Scenario 2:** Header contains the required primary nav
  - `Given` the Header is rendered
  - `When` inspected on a desktop viewport
  - `Then` it displays the site logo, the site name, and exactly the four primary nav links in order: `Trang chủ`, `Sản phẩm`, `Bài viết`, `Về chúng tôi`

- **Scenario 3:** Nav reflects the current route
  - `Given` the visitor is on a route matching one of the nav targets
  - `When` the Header renders
  - `Then` the matching nav link is visually distinguished (e.g., underline or higher-contrast state) without breaking the white-on-orange contrast

- **Scenario 4:** Header is semantic and keyboard accessible
  - `Given` the Header is rendered
  - `When` inspected
  - `Then` it uses a `<header>` element containing a `<nav>` element; all nav links are reachable and operable via keyboard with a visible focus ring

- **Scenario 5:** Mobile hamburger toggle
  - `Given` a viewport narrower than 768px
  - `When` the Header renders
  - `Then` the desktop nav links collapse behind a white hamburger icon that, when activated, opens the navigation drawer (drawer behavior defined in US00025)

### US00022: Multi-column Site Footer

**As a** visitor,
**I want to** see a consistent footer with affiliate disclosure, policy links, and contact info on every page,
**So that** I understand how the site is monetized and can find legal / contact information.

**Acceptance Criteria:**

- **Scenario 1:** Footer renders site-wide
  - `Given` any page
  - `When` the page loads
  - `Then` the Footer renders inside a `<footer>` element below the main content

- **Scenario 2:** Footer uses the multi-column Shopee layout
  - `Given` a desktop viewport
  - `When` the Footer renders
  - `Then` it shows distinct columns with the section titles `Về chúng tôi`, `Chính sách`, `Liên hệ`, and `Theo dõi`, separated by hairline rules using `var(--color-border-hairline)`

- **Scenario 3:** Thin orange top-accent line
  - `Given` the Footer is rendered
  - `When` inspected
  - `Then` a thin top accent line in `var(--color-primary)` separates the Footer from the content above it

- **Scenario 4:** Affiliate disclosure block is present
  - `Given` the Footer is rendered
  - `When` inspected
  - `Then` it contains the Vietnamese affiliate-disclosure text exactly as specified in `CLAUDE.md` ("Các liên kết sản phẩm trên trang này là liên kết tiếp thị liên kết...")

- **Scenario 5:** Policy links and contact present
  - `Given` the Footer is rendered
  - `When` inspected
  - `Then` it contains a link to `/chinh-sach-bao-mat/` (Privacy Policy) and `/cong-bo-tiep-thi-lien-ket/` (Affiliate Disclosure) — destinations ship in F0005 — and at least one contact channel (email or social handle)

- **Scenario 6:** Copyright shows current year at build time
  - `Given` the project is built
  - `When` the Footer renders the copyright line
  - `Then` it displays the year from the build timestamp (not a hardcoded literal)

- **Scenario 7:** Mobile stacking
  - `Given` a viewport narrower than 768px
  - `When` the Footer renders
  - `Then` columns stack vertically, the disclosure block remains fully visible, and all links remain tappable (min 44×44 touch target)

### US00023: Root layout — shell wiring + Shopee surface tokens

**As a** developer,
**I want to** have Header and Footer mounted through `app/layout.tsx` and the Shopee surface tokens available globally,
**So that** every page automatically inherits the chrome and shared theming without per-page wiring.

**Acceptance Criteria:**

- **Scenario 1:** Header and Footer wrap every route
  - `Given` `app/layout.tsx`
  - `When` any route renders
  - `Then` the Header renders above the route's content and the Footer renders below it, with the route's content placed inside a single `<main>` element

- **Scenario 2:** Skip-to-main-content link
  - `Given` the root layout
  - `When` a keyboard user presses Tab on initial focus
  - `Then` a "Skip to main content" link becomes visible and, when activated, moves focus to the `<main>` element

- **Scenario 3:** Existing root-layout invariants are preserved
  - `Given` the root layout edited by this story
  - `When` inspected
  - `Then` `<html lang="vi">` is retained, `globals.css` is still imported once, and `<SpeedInsights />` (mounted by US00115) is still rendered

- **Scenario 4:** Shopee surface tokens are defined
  - `Given` the global stylesheet
  - `When` inspected
  - `Then` it defines, additively to F0001's US00016 tokens, the following CSS custom properties on `:root`: `--color-page-bg: #F5F5F5`, `--color-surface: #FFFFFF`, `--color-border-hairline: #E5E5E5`, `--color-on-primary: #FFFFFF`, `--radius-sm: 2px`, `--radius-md: 4px`

- **Scenario 5:** Body uses the page background token
  - `Given` the global stylesheet
  - `When` any page renders
  - `Then` the `<body>` background resolves to `var(--color-page-bg)` so content cards visually float on a light-gray surface

- **Scenario 6:** Tokens are overridable
  - `Given` a parent scope redefines `--color-primary` or any new token
  - `When` a descendant element reads the variable
  - `Then` the descendant picks up the overridden value (no hardcoded hex outside the token definition)

### US00024: Two-column shell layout (Shopee card surfaces)

**As a** developer,
**I want to** have a reusable two-column shell component that renders a ~280px left panel and a main content area,
**So that** product-listing, blog-listing, and blog-post pages can opt into the standard layout without re-implementing it.

**Acceptance Criteria:**

- **Scenario 1:** Component renders the two-column structure on desktop
  - `Given` a page that uses the shell layout component
  - `When` rendered on a viewport ≥768px
  - `Then` the left panel renders at ~280px width on the left and the main content fills the remaining container width

- **Scenario 2:** Left panel uses a white card surface
  - `Given` the shell layout is rendered
  - `When` the left panel is inspected
  - `Then` it has a background of `var(--color-surface)`, a 1px border of `var(--color-border-hairline)`, and a `var(--radius-md)` corner radius

- **Scenario 3:** Container max-width and centering
  - `Given` a viewport wider than the container
  - `When` the shell layout renders
  - `Then` the combined left-panel + main content sits inside a container with a max-width of ~1280px, centered horizontally, with consistent gutter padding

- **Scenario 4:** Slot-based API
  - `Given` a page imports the component
  - `When` the developer passes content to the `leftPanel` and main `children` slots
  - `Then` each slot renders in its respective column without the component dictating the slot's internal markup

- **Scenario 5:** Opt-in usage
  - `Given` a page (homepage, About) does not use the shell layout component
  - `When` it renders
  - `Then` its content occupies the full container width inside `<main>` with no left panel — confirming the shell is opt-in, not forced by the root layout

### US00025: Mobile-responsive left panel (drawer)

**As a** visitor on a phone,
**I want to** access the same filters / nav that desktop visitors see, via a slide-out drawer,
**So that** I retain full functionality without the panel consuming half my screen.

**Acceptance Criteria:**

- **Scenario 1:** Panel collapses below 768px
  - `Given` a page using the shell layout
  - `When` the viewport is narrower than 768px
  - `Then` the inline left panel is hidden and a trigger button (e.g., filter / menu icon styled white-on-orange) is rendered to open the drawer

- **Scenario 2:** Drawer opens with overlay
  - `Given` the trigger button is activated
  - `When` the drawer opens
  - `Then` it slides in from the side, a dimmed overlay covers the rest of the page, and the drawer surface uses `var(--color-surface)` with hairline borders

- **Scenario 3:** Esc and overlay click close the drawer
  - `Given` the drawer is open
  - `When` the visitor presses Esc, or clicks/taps the overlay outside the drawer
  - `Then` the drawer closes

- **Scenario 4:** Focus management
  - `Given` the drawer opens
  - `When` it is fully rendered
  - `Then` focus moves into the drawer, focus is trapped within it while open, and on close focus returns to the trigger button that opened it

- **Scenario 5:** Body scroll lock while drawer is open
  - `Given` the drawer is open
  - `When` the visitor attempts to scroll the page behind the overlay
  - `Then` the underlying page does not scroll

- **Scenario 6:** Breakpoint sourced from tokens
  - `Given` the responsive switch logic
  - `When` reviewed
  - `Then` the 768px breakpoint is read from the breakpoint tokens defined in F0001's US00016, not hardcoded inline

### US00026: Sticky-on-scroll Header (P2)

**As a** visitor scrolling through a long blog post or product grid,
**I want to** keep the navigation visible without scrolling back to the top,
**So that** I can jump to another section at any time.

**Acceptance Criteria:**

- **Scenario 1:** Header becomes sticky past a small threshold
  - `Given` a long page
  - `When` the visitor scrolls past a small threshold (e.g., the header's own height)
  - `Then` the Header remains fixed at the top of the viewport for the remainder of the scroll

- **Scenario 2:** Subtle shadow appears once sticky
  - `Given` the Header is in its sticky state
  - `When` rendered
  - `Then` a subtle drop shadow / bottom-edge separator appears to visually lift the orange band from the page content; the shadow is absent in the at-top state

- **Scenario 3:** Graceful no-JS / reduced-motion fallback
  - `Given` JavaScript is disabled or the visitor has `prefers-reduced-motion: reduce`
  - `When` the page loads
  - `Then` the Header is still fully visible at the top of the page (either statically positioned or sticky without animated transitions) — no broken / hidden state

- **Scenario 4:** Sticky does not break the skip-link
  - `Given` the sticky Header is active
  - `When` a keyboard user activates the skip-to-main-content link
  - `Then` `<main>` receives focus and scrolls into view without being obscured by the sticky Header (account for header height in the scroll offset)

## 5. Edge Cases & Error Handling

- **Very long site name:** Header must truncate or constrain the site-name label so it does not push the nav off the orange band on narrow desktop viewports; min-width of nav links is preserved.
- **Active route under deeply nested paths:** Current-route detection in the nav should match by prefix (e.g., `/san-pham/[slug]/` highlights `Sản phẩm`), not exact equality only.
- **Missing translations / typos in nav labels:** Nav labels are hardcoded Vietnamese strings in v1 — any change must be a deliberate code edit. No runtime fallback needed.
- **Drawer opened then viewport resized past 768px:** When crossing back above the mobile breakpoint while the drawer is open, the drawer should close cleanly and the inline left panel reappear; no orphaned overlay or trapped focus.
- **Drawer with no content (empty filters / empty nav slot):** Drawer must still close cleanly and not crash on focus-trap when no focusable children exist (fall back to focusing the close button).
- **Sticky header on very short pages:** If the page is shorter than the viewport, the header should not enter the sticky state (no jitter); the static top position is sufficient.
- **Reduced-motion preference:** Drawer slide animation and sticky-header transitions must respect `prefers-reduced-motion: reduce` — instant state changes instead of animated ones.
- **Build year vs. timezone:** Footer copyright year uses the build environment's UTC year to avoid Jan-1 boundary flips between local and Vercel build timezone.
- **Privacy / Affiliate Disclosure links 404 before F0005 ships:** Acceptable in the interim — links point to the canonical paths so that when F0005 lands, no link update is needed.
- **Skip link visible only on focus:** The skip-to-main link must be visually hidden by default and become visible (and contrast-compliant against the orange Header) only when focused.
- **Token drift:** Hardcoded `#EE4D2D`, `#F5F5F5`, `#FFFFFF`, or `#E5E5E5` outside the token definition block should be avoided (and flagged in code review); all consumers go through CSS custom properties.

## 6. Technical Considerations

- **Server Components by default.** Header and Footer are Server Components; only the mobile drawer and sticky-scroll behavior require `"use client"` — keep client boundaries as small as possible (`components/HeaderMobileDrawer.tsx`, `components/StickyHeader.tsx` or similar).
- **System font stack only.** Per `CLAUDE.md`, no external fonts. Shopee's visual feel is approximated via weight/size/spacing on the system stack — do not import Google Fonts or load custom font files.
- **Additive tokens only.** F0001's US00016 already defines the CSS reset, `--color-primary`, and breakpoint tokens. F0002 only adds new tokens (`--color-page-bg`, `--color-surface`, `--color-border-hairline`, `--color-on-primary`, `--radius-sm`, `--radius-md`) and must not redefine existing ones.
- **Breakpoints sourced from tokens.** The 768px switch (and any 1280px container behavior) must read from the breakpoint tokens established in F0001; no inline magic numbers.
- **Root layout invariants.** `app/layout.tsx` must retain `<html lang="vi">`, the `globals.css` import, and the `<SpeedInsights />` mount added by US00115. This story extends the layout — it does not replace it.
- **A11y baseline.** Semantic landmarks (`<header>`, `<nav>`, `<main>`, `<footer>`), single `<h1>` per page (provided by individual pages, not the shell), visible focus indicators, drawer focus trap, skip link, `prefers-reduced-motion` support.
- **SSG compatibility.** Per project constraints, every route is statically generated. The sticky header and drawer must work entirely on the client after hydration — no SSR-only assumptions.
- **No new dependencies for the drawer.** Implement focus trap, scroll lock, and overlay with vanilla React + small inline logic rather than adding a UI library. If complexity escalates, raise it in PR rather than silently introducing a dependency.
- **Disclosure copy is canonical.** The Vietnamese affiliate-disclosure text in the Footer must match `CLAUDE.md` verbatim; treat that string as a single source of truth (consider extracting to a constant in `lib/` if it is referenced again by F0005 or blog posts).
- **Reserved policy paths.** `/chinh-sach-bao-mat/` and `/cong-bo-tiep-thi-lien-ket/` are the canonical Vietnamese slugs used by the Footer links; F0005 must honor them. If F0005 chooses different slugs, F0002's Footer must be updated in the same PR.
- **Sticky header is P2.** Ship the non-sticky Header first if needed; US00026 may land in a follow-up PR without blocking F0002 closure. The other five stories are P0.
- **English identifiers, Vietnamese user-facing copy.** Per project working conventions — only nav labels, footer column titles, disclosure text, and policy link labels are Vietnamese; component names, props, file names stay English.
