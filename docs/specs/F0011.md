# Feature Specification: Build & Deploy Pipeline

**ID:** `F0011`
**Epic/Module:** Core Infrastructure / DevOps
**Status:** Draft

---

## 1. Business Objective

Establish a reliable, automated build and deploy pipeline so that every subsequent feature ships to production with zero manual intervention. Push to `main` triggers a Vercel Production deploy; a weekly GitHub Actions cron rebuilds the static site to refresh affiliate metadata; CI validates PRs (typecheck, lint, build) before merge so broken code never reaches `main`. Without this foundation, every later feature would need bespoke deploy steps and quality gates, and the team would lose the speed advantage of an SSG-only architecture.

## 2. Scope

### In Scope

- Vercel project connected to the GitHub repository with `main` as the Production branch
- Production build configured for SSG output (per F0001)
- `NEXT_PUBLIC_SITE_URL` and `NEXT_PUBLIC_GA_MEASUREMENT_ID` configured in Vercel per environment (Production, Preview)
- GitHub Actions cron workflow `.github/workflows/scheduled-deploy.yml` running Mon 02:00 UTC
- `workflow_dispatch` trigger for on-demand rebuild
- `VERCEL_DEPLOY_HOOK_URL` stored as a GitHub repository secret and consumed by the cron workflow
- PR validation CI workflow `.github/workflows/ci.yml` running typecheck, lint, and build on PRs and non-main pushes
- Branch protection on `main` requiring CI jobs to pass before merge
- Core Web Vitals monitoring via Vercel Speed Insights enabled on the project
- `.env.example` documenting required env-var keys

### Out of Scope

- Custom domain registration / DNS configuration (deferred to launch prep — see launch checklist §9)
- Multi-environment staging beyond default Vercel Production + Preview
- Custom CDN or edge configuration
- Deploy notifications (Slack / email) on success or failure
- Branch deploy preview gating policies
- E2E or visual regression tests in CI
- Build cache tuning beyond Vercel / GitHub defaults
- Performance budgets that fail CI on regression
- Cookie consent banner and any GA4 wiring (F0007)

## 3. Process Flow

1. Developer opens a PR from a feature branch.
2. PR validation CI runs typecheck, lint, and build. Vercel creates a Preview deploy and posts its URL on the PR.
3. PR is reviewed; required status checks must pass before it can merge.
4. PR merges to `main`. Vercel detects the push and runs a Production deploy using the configured env vars.
5. Vercel publishes the static output to its CDN; Speed Insights starts collecting Core Web Vitals.
6. Every Monday 02:00 UTC, the scheduled GitHub Actions workflow fires, POSTs to the Vercel deploy hook, and triggers a fresh Production rebuild.
7. Operator can also trigger the scheduled workflow on demand from the Actions tab via `workflow_dispatch`.

## 4. User Stories & Acceptance Criteria

### US00111: Connect Vercel project for auto-deploy on `main`

**As a** site operator,
**I want to** connect the repository to Vercel so every push to `main` triggers a Production deploy,
**So that** changes ship to production without manual intervention.

**Acceptance Criteria:**

- **Scenario 1:** Production branch is `main`
  - `Given` the Vercel project is linked to the GitHub repository
  - `When` the operator inspects the project's Git settings
  - `Then` the Production branch is configured as `main`

- **Scenario 2:** Push to `main` triggers a Production deploy
  - `Given` the Vercel/GitHub integration is connected
  - `When` a commit is pushed to `main`
  - `Then` Vercel automatically starts a Production deployment for that commit

- **Scenario 3:** Build produces SSG output
  - `Given` the Next.js project is configured for SSG per F0001
  - `When` Vercel runs the build
  - `Then` the build completes and the deployment serves static output from Vercel's CDN

- **Scenario 4:** Failed build does not replace the live deployment
  - `Given` a deploy is running and the build fails
  - `When` the deploy completes with an error
  - `Then` the previously successful deployment remains live and the site does not break

- **Scenario 5:** Preview deploys on non-main branches
  - `Given` a PR is opened from a non-main branch
  - `When` Vercel processes the push
  - `Then` a Preview deployment is created and its URL is posted on the PR

### US00112: Configure environment variables in Vercel

**As a** site operator,
**I want to** configure `NEXT_PUBLIC_SITE_URL` and `NEXT_PUBLIC_GA_MEASUREMENT_ID` in Vercel per environment,
**So that** the typed env module (US00018) validates correctly and analytics can be enabled later.

**Acceptance Criteria:**

- **Scenario 1:** `NEXT_PUBLIC_SITE_URL` set in Production
  - `Given` the Vercel project's Environment Variables settings
  - `When` the operator inspects the Production environment
  - `Then` `NEXT_PUBLIC_SITE_URL` is set to a valid absolute URL

- **Scenario 2:** `NEXT_PUBLIC_GA_MEASUREMENT_ID` slot exists
  - `Given` the Vercel project's Environment Variables settings
  - `When` the operator inspects Production and Preview
  - `Then` `NEXT_PUBLIC_GA_MEASUREMENT_ID` is defined (may be empty per F0007)

- **Scenario 3:** Preview environment has matching env vars
  - `Given` Preview deploys are enabled
  - `When` a Preview build runs
  - `Then` it receives the same env-var keys, scoped to Preview values where they differ from Production

- **Scenario 4:** Missing required env var fails the deploy
  - `Given` `NEXT_PUBLIC_SITE_URL` is removed from the Production environment
  - `When` a deploy runs
  - `Then` the build fails per US00018 with a clear error naming the missing variable, and the previously live deployment remains served

- **Scenario 5:** No secrets committed to the repo
  - `Given` the repository
  - `When` reviewed
  - `Then` no `.env.production` (or equivalent) file with real values is committed, and an `.env.example` documenting the required keys is present at the repo root

### US00113: PR validation CI workflow

**As a** developer,
**I want to** have a CI workflow that runs typecheck, lint, and build on every PR and non-main push,
**So that** broken code is caught before it reaches `main` (which auto-deploys).

**Acceptance Criteria:**

- **Scenario 1:** Workflow file exists
  - `Given` the repository
  - `When` inspected
  - `Then` `.github/workflows/ci.yml` (or equivalent) exists and is triggered on `pull_request` and on `push` to non-main branches

- **Scenario 2:** Typecheck job runs
  - `Given` a PR is opened
  - `When` CI runs
  - `Then` a typecheck job (e.g. `tsc --noEmit`) executes and fails if any TypeScript errors are present

- **Scenario 3:** Lint job runs
  - `Given` a PR is opened
  - `When` CI runs
  - `Then` ESLint runs across the project and fails on any violation per US00017

- **Scenario 4:** Build job runs
  - `Given` a PR is opened
  - `When` CI runs
  - `Then` the production build runs and the job fails if it does not complete successfully

- **Scenario 5:** Node version matches the project
  - `Given` the CI workflow
  - `When` reviewed
  - `Then` it pins the same Node major version as `package.json`'s `engines` / `.nvmrc` (≥ 22.11) and matches the version Vercel uses

- **Scenario 6:** Lint/typecheck/build commands come from `package.json` scripts
  - `Given` the workflow
  - `When` reviewed
  - `Then` it invokes the canonical `npm run` (or equivalent) scripts rather than re-implementing the commands inline

- **Scenario 7:** Status checks gate merging to `main`
  - `Given` the repository's branch protection rules
  - `When` configured
  - `Then` the CI jobs are listed as required status checks and a PR cannot merge to `main` while any are failing

- **Scenario 8:** CI does not require deploy secrets
  - `Given` a PR opened from a fork (no access to repository secrets)
  - `When` CI runs
  - `Then` typecheck, lint, and build all complete without reading `VERCEL_DEPLOY_HOOK_URL` or any other secret

### US00114: Scheduled rebuild via GitHub Actions cron

**As a** site operator,
**I want to** have a GitHub Actions workflow rebuild the site every Monday 02:00 UTC by calling a Vercel deploy hook,
**So that** the static site refreshes weekly without a code push.

**Acceptance Criteria:**

- **Scenario 1:** Workflow file exists at the canonical path
  - `Given` the repository
  - `When` inspected
  - `Then` `.github/workflows/scheduled-deploy.yml` exists on the default branch

- **Scenario 2:** Cron schedule matches the spec
  - `Given` the workflow file
  - `When` reviewed
  - `Then` it includes `schedule: - cron: "0 2 * * 1"` (Mon 02:00 UTC)

- **Scenario 3:** Manual trigger is available
  - `Given` the workflow file
  - `When` the operator opens the Actions tab
  - `Then` the workflow exposes a `workflow_dispatch` "Run workflow" button

- **Scenario 4:** Workflow calls the Vercel deploy hook
  - `Given` `VERCEL_DEPLOY_HOOK_URL` is configured as a repository secret
  - `When` the workflow runs (cron or manual)
  - `Then` it issues a POST request to the deploy hook URL and Vercel begins a Production rebuild

- **Scenario 5:** Deploy hook URL is not exposed in logs
  - `Given` the workflow uses the secret
  - `When` it runs
  - `Then` the deploy hook URL is referenced via `secrets.VERCEL_DEPLOY_HOOK_URL` and is not echoed in plaintext anywhere in the workflow logs

- **Scenario 6:** Workflow uses minimal permissions
  - `Given` the workflow file
  - `When` reviewed
  - `Then` its `permissions:` block grants no more than `contents: read` (sufficient for a webhook POST)

## 5. Edge Cases & Error Handling

- **Vercel deploy hook URL leak:** If the deploy hook URL is committed or echoed in logs, anyone can trigger production rebuilds. It must live only in `secrets.VERCEL_DEPLOY_HOOK_URL` and never be printed.
- **Cron skew / missed run:** GitHub Actions cron can drift or skip occasional runs. The workflow must be idempotent; missing one Monday is acceptable, and `workflow_dispatch` is the manual fallback.
- **Build fails on `main`:** A bad commit reaches `main` and the deploy fails. The previously successful deployment must remain live. The fix path is revert or roll-forward; no manual intervention in Vercel is required to keep the site up.
- **Missing required env var in Vercel:** Per US00018, the build must fail loudly. Operator updates the env var in Vercel and triggers a redeploy.
- **PR opened from a fork:** GitHub does not expose repository secrets to forked-PR runners. CI jobs must not require any secrets to run.
- **Vercel free-tier build minutes:** Long or frequent builds can exceed free-tier caps. Build duration must be observable so the operator can prune dependencies/content before considering a plan upgrade. Out of scope for the initial setup.
- **Concurrent deploys (push + cron firing close together):** Vercel queues deploys; the latest commit ends up live. No special handling required.
- **Cron workflow not on default branch:** GitHub Actions only honors `schedule:` triggers from workflows on the repository's default branch. The workflow must be merged to `main` for the cron to fire.
- **Branch protection misconfigured:** If status checks are not required, a broken PR can land on `main` and auto-deploy a broken build. Branch protection must list each CI job by name as a required check.
- **Empty or invalid `NEXT_PUBLIC_SITE_URL` value (not just missing):** Build still fails per US00018's URL validation; same recovery path as missing.

## 6. Technical Considerations

- **Vercel project ownership and access** must be settled before configuring secrets — the deploy hook URL is generated from the Vercel UI and pasted into GitHub repository secrets.
- **GitHub repository secret name is `VERCEL_DEPLOY_HOOK_URL`.** Do not rename without updating the workflow file.
- **Cron string is `0 2 * * 1`** (Monday 02:00 UTC). GitHub Actions cron runs in UTC. Vietnam local time is UTC+7, so this fires at 09:00 ICT Monday — note for operator visibility.
- **Workflow `permissions:` block** should be `contents: read` (or stricter). The cron workflow only POSTs to a webhook; it does not push commits, comment on PRs, or read other resources.
- **CI Node version must match `package.json`'s `engines` (≥ 22.11)** and align with the version Vercel uses for the build, to avoid runtime drift between CI and Production.
- **Lint / typecheck / build commands must come from `package.json` scripts**, not be re-implemented in YAML — single source of truth.
- **Vercel's "Ignore Build Step" feature** can be configured later to skip rebuilds for doc-only changes (P2 — not required at launch).
- **`.env.example`** documents `NEXT_PUBLIC_SITE_URL` and `NEXT_PUBLIC_GA_MEASUREMENT_ID`; real values live only in Vercel and (optionally) a developer's gitignored `.env.local`.
- **Deploy hooks over CLI deploys:** This pipeline uses Vercel deploy hooks for the scheduled path so no `VERCEL_TOKEN` needs to be stored in GitHub. Smaller blast radius if the secret leaks.
- **Speed Insights activation:** May require installing `@vercel/speed-insights` and mounting the component in the root layout. Confirm the current activation method from official Next.js / Vercel docs at implementation time (per AGENTS.md rule).
- **No custom domain in this feature.** The site is reachable at the Vercel-assigned `*.vercel.app` URL until launch prep handles DNS. `NEXT_PUBLIC_SITE_URL` should reflect whichever URL is canonical at the time and be updated when the custom domain is added.
- **Scheduled cron only runs on the default branch.** The cron workflow must be merged to `main` for the schedule to start firing — until then only `workflow_dispatch` will work.
- **Branch protection is configured in GitHub repo settings**, not in YAML. The required checks must list each CI job by name (e.g. `typecheck`, `lint`, `build`) — generic "all checks" gating is not sufficient if jobs are renamed.
- **All identifiers, code, and code comments stay in English** per project working conventions; this feature ships no user-facing copy.
