# Feature Specification: Technical Foundation & Bootstrap

**ID:** `F0001`
**Epic/Module:** Core Infrastructure
**Status:** Draft

---

## 1. Business Objective

Establish the project scaffolding and shared building blocks that every other feature depends on. Without this foundation, no page, component, or content pipeline can be built. The goal is a Next.js (App Router) + TypeScript project that compiles cleanly, supports static generation, parses MDX with typed frontmatter, exposes typed content loaders and env vars, and ships with consistent linting, formatting, and base design tokens — so subsequent features (F0004+) start from a predictable, validated baseline rather than re-solving plumbing.

## 2. Scope

### In Scope

- Next.js (App Router) + TypeScript project initialization, configured for SSG
- Canonical folder structure: `app/`, `components/`, `content/`, `lib/`, `static/`, `types/`
- Path aliases: `@/components`, `@/lib`, `@/content`, `@/types`
- MDX pipeline with frontmatter parsing and support for custom MDX components
- Shared TypeScript types: `Product`, `Post`, frontmatter shapes
- Content loader utilities for products (JSON) and posts (MDX) with runtime validation
- Base global styles: CSS reset, CSS variables (`--color-primary` and design tokens), breakpoint tokens (768px, 1280px)
- System font stack only — no external fonts
- ESLint + Prettier configuration
- Typed environment variable access (`NEXT_PUBLIC_SITE_URL`, `NEXT_PUBLIC_GA_MEASUREMENT_ID`)

### Out of Scope

- Any concrete page, route, or UI component (homepage, product list, blog, etc.) — belongs to F0004+
- Site shell / header / footer (F0002)
- SEO metadata wiring, sitemap, JSON-LD (F0009)
- Analytics integration (F0007)
- Vercel project setup and deploy pipeline (F0011)
- Search functionality (deferred to v2)
- Dark mode, multilingual (deferred to v2+)

## 3. Process Flow

_This feature has no end-user flow — the "user" is a developer setting up or building on the foundation._

1. Developer clones the repository fresh.
2. Developer runs the install command and the project's dependencies resolve.
3. Developer runs the dev server — Next.js (App Router) starts cleanly with TypeScript strict mode active.
4. Developer adds a `.json` product file under `content/products/` or a `.mdx` post under `content/posts/` — content loaders parse and validate it, returning typed objects.
5. Developer imports a shared type or utility using a `@/...` path alias — it resolves in both the IDE and the bundler.
6. Developer runs the lint command — ESLint + Prettier rules apply consistently.
7. Developer runs the production build — env vars are validated; missing required vars (`NEXT_PUBLIC_SITE_URL`) fail the build with a clear message.

## 4. User Stories & Acceptance Criteria

### US00011: Initialize Next.js + TypeScript project

**As a** developer,
**I want to** initialize a Next.js (App Router) + TypeScript project with SSG defaults,
**So that** all subsequent feature work has a working baseline that builds and runs.

**Acceptance Criteria:**

- **Scenario 1:** Project boots in development
  - `Given` a fresh clone of the repository with dependencies installed
  - `When` the developer runs the dev server
  - `Then` Next.js starts using the App Router (not Pages Router) on the default port without errors

- **Scenario 2:** Production build succeeds
  - `Given` the project is initialized
  - `When` the developer runs the production build
  - `Then` the build completes successfully and produces static output suitable for SSG

- **Scenario 3:** TypeScript strict mode is enforced
  - `Given` `tsconfig.json` is committed
  - `When` the project compiles
  - `Then` `strict: true` is active and the build fails on type errors

- **Scenario 4:** Node version is pinned
  - `Given` the repository contains an `.nvmrc` file or `engines` field in `package.json`
  - `When` a new developer onboards
  - `Then` the required Node version is unambiguous

### US00012: Establish folder structure & path aliases

**As a** developer,
**I want to** have the canonical folder layout and `@/*` path aliases configured,
**So that** imports are stable and every feature lands in a predictable place.

**Acceptance Criteria:**

- **Scenario 1:** Canonical folders exist
  - `Given` the repository is checked out
  - `When` the developer inspects the project root
  - `Then` `app/`, `components/`, `content/`, `lib/`, `static/`, and `types/` all exist (with `.gitkeep` if empty)

- **Scenario 2:** Path aliases resolve in TypeScript
  - `Given` `tsconfig.json` defines `@/components`, `@/lib`, `@/content`, `@/types` aliases
  - `When` a file imports using one of these aliases
  - `Then` the IDE and the TypeScript compiler resolve the import without error

- **Scenario 3:** Path aliases resolve in the bundler
  - `Given` the aliases are configured in the build pipeline
  - `When` the project is built
  - `Then` the bundler resolves alias imports identically to the TS compiler — no "module not found" errors

### US00013: Configure MDX pipeline with frontmatter parsing

**As a** developer,
**I want to** have `.mdx` files parseable with typed frontmatter and custom-component support,
**So that** the editorial feature (F0006) can author posts in MDX without re-solving plumbing.

**Acceptance Criteria:**

- **Scenario 1:** MDX file compiles
  - `Given` a `.mdx` file with valid frontmatter and Markdown body
  - `When` the project is built or the file is imported
  - `Then` the file compiles successfully and exposes both its rendered content and its frontmatter

- **Scenario 2:** Frontmatter is extracted as a typed object
  - `Given` an MDX file with a YAML frontmatter block
  - `When` the loader processes the file
  - `Then` the frontmatter is returned as a typed object matching the `Post` frontmatter type from US00014

- **Scenario 3:** Custom MDX components can be injected
  - `Given` an MDX file referencing a component such as `<ProductCard slug="..." />`
  - `When` MDX provider configuration is supplied
  - `Then` the custom component is rendered as part of the MDX output (foundation for F0006's inline `<ProductCard />`)

- **Scenario 4:** Invalid frontmatter surfaces a clear error
  - `Given` an MDX file with malformed YAML frontmatter
  - `When` the file is loaded
  - `Then` the build (or loader call) fails with an error message identifying the offending file and the nature of the issue

### US00014: Define shared TypeScript types

**As a** developer,
**I want to** have authoritative `Product`, `Post`, and frontmatter types under `types/`,
**So that** every feature consumes the same shape and breakages are caught at compile time.

**Acceptance Criteria:**

- **Scenario 1:** `Product` type matches the canonical JSON shape
  - `Given` the product JSON shape defined in `CLAUDE.md`
  - `When` the developer inspects the `Product` type
  - `Then` it includes `slug`, `name`, `category`, `brand`, `price`, `affiliateUrl`, `images`, `description`, `specs`, `publishedAt`, and `featured` with appropriate types

- **Scenario 2:** `Post` type covers MDX frontmatter
  - `Given` the MDX frontmatter shape defined in `CLAUDE.md`
  - `When` the developer inspects the `Post` (or post-frontmatter) type
  - `Then` it includes `title`, `summary`, `publishedAt`, `category`, `tags`, and `coverImage` with appropriate types

- **Scenario 3:** Types are exported from a single barrel
  - `Given` the `types/` folder is configured
  - `When` a feature imports a shared type
  - `Then` it can be imported from one canonical entry point (e.g. `@/types`)

- **Scenario 4:** No `any` in shared types
  - `Given` the shared type definitions
  - `When` reviewed
  - `Then` no field uses `any`; unknown fields use `unknown` or a discriminated union

### US00015: Build content loader utilities

**As a** developer,
**I want to** have `lib/` helpers that read products from JSON and posts from MDX, validate them, and return typed objects,
**So that** pages call one function instead of re-implementing filesystem access.

**Acceptance Criteria:**

- **Scenario 1:** Product loaders return typed data
  - `Given` valid product JSON files in `content/products/`
  - `When` the developer calls `getAllProducts()` or `getProductBySlug(slug)`
  - `Then` they receive `Product` objects (or arrays thereof) typed per US00014

- **Scenario 2:** Post loaders return typed data
  - `Given` valid MDX files in `content/posts/`
  - `When` the developer calls `getAllPosts()` or `getPostBySlug(slug)`
  - `Then` they receive `Post` objects (or arrays thereof) with frontmatter typed per US00014

- **Scenario 3:** Loaders reject malformed content
  - `Given` a product JSON file missing a required field, or an MDX file with missing required frontmatter
  - `When` the loader processes the content
  - `Then` it surfaces a clear validation error identifying the file and the missing/invalid field

- **Scenario 4:** Loaders are build-safe (no client bundle leakage)
  - `Given` a loader function that uses Node `fs`
  - `When` the project is built
  - `Then` no Node-only code is bundled into the client output

- **Scenario 5:** `getProductBySlug` / `getPostBySlug` handle "not found"
  - `Given` a slug that does not match any file
  - `When` the loader is called
  - `Then` it returns a clearly nullable result (or throws a documented error) — behavior is consistent and documented

### US00016: Base global styles & design tokens

**As a** developer,
**I want to** have a CSS reset, CSS variables (including `--color-primary: #EE4D2D`), and breakpoint tokens applied globally,
**So that** every component inherits consistent defaults without re-declaring them.

**Acceptance Criteria:**

- **Scenario 1:** Global stylesheet is loaded once
  - `Given` the root layout
  - `When` any page renders
  - `Then` the global stylesheet (reset + tokens) is imported exactly once at the root and applies to all pages

- **Scenario 2:** Theme tokens are available as CSS variables
  - `Given` the global tokens file
  - `When` a component uses `var(--color-primary)`
  - `Then` it resolves to `#EE4D2D` by default and can be overridden via a parent scope

- **Scenario 3:** Breakpoint tokens are documented
  - `Given` the design tokens
  - `When` a developer needs to use the tablet (768px) or desktop (1280px) breakpoint
  - `Then` the value is available as a documented token (CSS custom property or tokens module) — not hardcoded per component

- **Scenario 4:** No external font requests
  - `Given` the project's network requests during page load
  - `When` inspected in dev tools
  - `Then` no Google Fonts / external font CDNs are loaded — only the system font stack is used

### US00017: Lint & format configuration

**As a** developer,
**I want to** have ESLint + Prettier configured with sensible Next.js/TypeScript defaults,
**So that** code style is enforced automatically and doesn't become a per-PR debate.

**Acceptance Criteria:**

- **Scenario 1:** Lint passes on a clean checkout
  - `Given` a freshly cloned repository with dependencies installed
  - `When` the developer runs the lint command
  - `Then` it exits successfully with no errors

- **Scenario 2:** ESLint extends recommended Next.js + TypeScript rules
  - `Given` the ESLint configuration file
  - `When` reviewed
  - `Then` it extends the official Next.js and TypeScript recommended configurations

- **Scenario 3:** Prettier configuration is committed
  - `Given` the repository
  - `When` the developer inspects the project root
  - `Then` a Prettier config file is present and applied consistently across `.ts`, `.tsx`, `.mdx`, `.json`, and `.css`

- **Scenario 4:** Lint catches a deliberate violation
  - `Given` the configured rules
  - `When` a file with an obvious violation (e.g. unused variable) is committed
  - `Then` the lint command fails with a clear message

### US00018: Typed environment variable access

**As a** developer,
**I want to** have a single typed module that exposes `NEXT_PUBLIC_SITE_URL` and `NEXT_PUBLIC_GA_MEASUREMENT_ID`,
**So that** missing or malformed env vars fail fast at build time rather than at runtime in production.

**Acceptance Criteria:**

- **Scenario 1:** Required env var is validated as a URL
  - `Given` `NEXT_PUBLIC_SITE_URL` is set to a valid URL
  - `When` the typed env module loads
  - `Then` it exposes the value as a typed string and the build proceeds

- **Scenario 2:** Missing required env var fails the build
  - `Given` `NEXT_PUBLIC_SITE_URL` is not set
  - `When` the project is built
  - `Then` the build fails with an error message naming the missing variable

- **Scenario 3:** Optional env var allows empty
  - `Given` `NEXT_PUBLIC_GA_MEASUREMENT_ID` is unset or empty
  - `When` the typed env module loads
  - `Then` the value is exposed as an optional/empty string and the build still succeeds (analytics disabled per F0007)

- **Scenario 4:** No raw `process.env.*` access elsewhere
  - `Given` the codebase outside the typed env module
  - `When` reviewed (and enforced via lint rule where practical)
  - `Then` no other file reads `process.env.*` directly — all access flows through the typed module

## 5. Edge Cases & Error Handling

- **Empty content folders:** `content/products/` and `content/posts/` contain only `.gitkeep` at bootstrap. Loaders must return an empty array (not throw) when no content exists.
- **Malformed JSON product file:** Loader must report the offending filename and the parse/validation error; build fails rather than silently skipping the file.
- **MDX file with missing required frontmatter (e.g. no `title`):** Loader must fail with a message naming the file and the missing field.
- **Invalid `NEXT_PUBLIC_SITE_URL` (not a valid URL):** Build fails with a clear validation error pointing to the env module.
- **Path-alias drift:** If `tsconfig.json` paths and bundler aliases drift apart, builds may pass but runtime imports fail. Both must be configured from the same source of truth (or kept in lockstep and verified).
- **Mixed line endings / formatting drift on Windows:** Prettier + an `.editorconfig` (or equivalent) should normalize line endings to prevent CRLF/LF churn.
- **Accidental client bundling of Node-only loader code:** Loaders that touch `fs` must be marked / kept on the server boundary so the bundler does not include them in the client bundle.
- **Duplicate product slugs / post slugs across files:** Loaders should detect and surface duplicate slugs as an error at load time (slugs are used as URL keys downstream).

## 6. Technical Considerations

- **App Router only.** Pages Router is explicitly not used. All routing lives under `app/`.
- **SSG-only target.** No SSR, no ISR — every route must be statically generatable. This constrains how loaders are invoked (build time only) and rules out runtime DB calls.
- **TypeScript strict mode is non-negotiable.** All shared types must be strict; `any` is disallowed in shared modules.
- **Single source of truth for env vars.** All runtime/build access goes through `lib/env.ts` (or equivalent). Direct `process.env.*` reads are disallowed outside that module.
- **No external fonts.** System font stack only, per `CLAUDE.md`. Any later styling work must respect this.
- **Default `--color-primary` is `#EE4D2D`** (Shopee orange), but it must be overridable via CSS variable scoping — do not hardcode the hex elsewhere.
- **Breakpoints are 768px (tablet) and 1280px (desktop), mobile-first.** These values live in tokens and must not be re-declared per component.
- **Content paths are fixed:** `content/products/*.json` and `content/posts/*.mdx`. The loader API is the contract that downstream features (F0004, F0006) depend on — changes here ripple, so the API surface should be deliberate.
- **No CMS, no database.** Content is filesystem-only. Loaders run at build time.
- **`@/*` path alias must be configured in both `tsconfig.json` AND the bundler/Next.js config** to avoid silent drift.
- **MDX provider** must be set up in a way that allows F0006 to register `<ProductCard />` later without re-architecting the pipeline.
- **All identifiers, code, and code comments stay in English** per project working conventions; only user-facing copy is Vietnamese (no user-facing copy is shipped in F0001).
