# Feature Specification: Affiliate Link Routing

**ID:** `F0003`
**Epic/Module:** Monetization Core
**Status:** Draft

---

## 1. Business Objective

Every product CTA, product card, and inline blog `<ProductCard />` on the site exists to send a visitor to a Shopee affiliate URL in a new tab. This feature delivers the single canonical primitive that all of those surfaces route through — `<AffiliateLink>` — with correct link semantics baked in (`target="_blank"`, `rel="noopener noreferrer sponsored"`), an accessible label, support for whole-card clickability, and a predefined seam that F0007 (Analytics & Click Tracking) plugs into without touching any call site. Without F0003, F0004 (Product Catalog) and F0006 (Editorial) would each re-solve link semantics, click tracking would require per-site edits later, and the risk of a broken or untracked affiliate anchor sneaking into production would grow with every new page.

## 2. Scope

### In Scope

- Canonical `<AffiliateLink>` component used by every affiliate destination on the site (product cards, product detail CTA, inline MDX `<ProductCard />`, related products, homepage featured picks)
- `target="_blank"` and `rel="noopener noreferrer sponsored"` baked into the primitive — never specified at the call site
- Accessible label for screen readers ("Mở [productName] trên Shopee (mở tab mới)") via visually hidden text or `aria-label`
- Whole-card clickability pattern: the primitive supports wrapping arbitrary children (text, button-styled span, full card surface) without producing nested interactive markup
- Tracking-hook seam: `data-affiliate-link` plus structured `data-product-name` / `data-product-category` / `data-destination-url` attributes so F0007 can attach a delegated click handler at the layout level with zero per-site changes
- `lib/affiliate.ts` URL helper that validates affiliate destinations against the Shopee host allow-list (`shopee.vn`, `shopee.ee`, `shope.ee`) and surfaces clear build-time errors for any product JSON with an unrecognized host
- Documented usage convention (in `CLAUDE.md` and the component's example) prohibiting raw `<a href="https://shope.ee/...">` outside the primitive

### Out of Scope

- The visual product-card component itself (F0004 — Product Catalog)
- The product detail page and its CTA layout (F0004)
- The inline MDX `<ProductCard slug="..." />` component (F0006 — Editorial)
- Implementation of the GA4 `affiliate_click` event itself (F0007 — Analytics & Click Tracking) — F0003 only provides the seam
- Click-redirect interstitials (e.g., `/go/[slug]` server route) — links go straight to Shopee
- Server-side link cloaking / URL shortening
- Custom ESLint rule enforcing the no-raw-anchor convention (P2 follow-up; convention + code review suffice at launch)
- Multi-network affiliate support (Lazada, Tiki, etc.) — v1 is Shopee-only

## 3. Process Flow

_Two "users" here: the visitor clicking through to Shopee, and the developer building a page that surfaces a product._

1. A developer building a page that surfaces a product imports `<AffiliateLink>` from `@/components/AffiliateLink`.
2. The developer passes the product's `affiliateUrl`, `name`, and `category` as props and provides the visual content as children (a button label, a card layout, or inline text).
3. At build time, `lib/affiliate.ts` validates the destination URL against the Shopee host allow-list; an unrecognized host fails the build with the offending product slug named.
4. The page renders. The `<a>` element is emitted with `target="_blank"`, `rel="noopener noreferrer sponsored"`, the accessible label, and the `data-affiliate-link` / structured product attributes.
5. The visitor clicks anywhere on the rendered surface (button, card, inline link).
6. The browser opens the Shopee destination in a new tab. The original tab remains on the affiliate-storefront page.
7. (After F0007 ships) A delegated click handler attached at the layout level catches the click via the `data-affiliate-link` selector, reads the structured attributes, and fires the GA4 `affiliate_click` event — without `<AffiliateLink>` or any call site being modified.

## 4. User Stories & Acceptance Criteria

### US00031: Canonical `<AffiliateLink>` primitive

**As a** developer building any product surface,
**I want to** wrap affiliate destinations in a single component that bakes in the correct link semantics and accessibility,
**So that** call sites never reimplement `target`/`rel`/aria and we have one place to evolve the contract.

**Acceptance Criteria:**

- **Scenario 1:** Primitive renders a correctly-attributed anchor
  - `Given` a developer renders `<AffiliateLink href="https://shope.ee/abc" productName="..." productCategory="...">Mua ngay</AffiliateLink>`
  - `When` the page is inspected
  - `Then` the rendered DOM contains an `<a>` element with `href` set to the destination, `target="_blank"`, and `rel="noopener noreferrer sponsored"` — and the visible children are inside the anchor

- **Scenario 2:** Accessible label includes product name and "opens in new tab"
  - `Given` an `<AffiliateLink>` instance with `productName="Chuột Logitech G102"`
  - `When` a screen reader announces the link
  - `Then` it conveys the equivalent of "Mở Chuột Logitech G102 trên Shopee (mở tab mới)" — via visually hidden text or `aria-label` on the anchor

- **Scenario 3:** Required props are typed
  - `Given` the `<AffiliateLink>` component
  - `When` a developer inspects its TypeScript signature
  - `Then` `href`, `productName`, `productCategory`, and `children` are required props with non-optional, non-`any` types

- **Scenario 4:** No `target`/`rel` allowed at the call site
  - `Given` the component's prop signature
  - `When` a developer attempts to pass `target` or `rel` as a prop
  - `Then` the type system rejects the override (or the prop is silently ignored in favor of the baked-in values) — call sites cannot weaken link semantics

- **Scenario 5:** Server Component by default
  - `Given` the `<AffiliateLink>` source file
  - `When` reviewed
  - `Then` it does not contain `"use client"` — a plain anchor renders on the server with zero hydration cost

- **Scenario 6:** Children render unchanged inside the anchor
  - `Given` arbitrary JSX is passed as children (text, a styled `<span>`, a multi-element card layout)
  - `When` the component renders
  - `Then` the children appear inside the `<a>` without the component injecting wrapper elements that would interfere with layout

### US00032: Whole-card clickability pattern (no nested interactives)

**As a** developer building product cards (F0004) and inline `<ProductCard />` (F0006),
**I want to** wrap an entire card surface as a single affiliate link using `<AffiliateLink>`,
**So that** the whole card is clickable and we never produce nested `<a>`/`<button>` markup.

**Acceptance Criteria:**

- **Scenario 1:** Whole-surface wrap is supported
  - `Given` a developer wraps a card-shaped JSX subtree (image + name + price + decorative "Mua ngay" element) as children of `<AffiliateLink>`
  - `When` the page renders
  - `Then` clicking anywhere within the card surface follows the affiliate destination, and tab-focus stops once on the card as a single link (not multiple times for inner elements)

- **Scenario 2:** No nested interactive elements
  - `Given` the documented usage example for whole-card clickability
  - `When` reviewed
  - `Then` the "Mua ngay" CTA inside the card is rendered as a styled `<span>` (or equivalent non-interactive element), never as `<button>` or a nested `<a>`

- **Scenario 3:** Focus-visible state applies to the card
  - `Given` a keyboard user tabs to a product card wrapped by `<AffiliateLink>`
  - `When` the link receives focus
  - `Then` a visible focus indicator applies to the entire card surface (not just an inner element), meeting the contrast requirements of the surrounding context

- **Scenario 4:** Hover state applies to the card
  - `Given` a pointer hovers over any part of the card
  - `When` inspected
  - `Then` the hover state (e.g., subtle elevation or border emphasis) treats the card as one interactive surface — not just the inner CTA

- **Scenario 5:** Documented usage example is checked in
  - `Given` the `<AffiliateLink>` component file (or a co-located README / Storybook-style doc)
  - `When` a developer reads it
  - `Then` it includes a copyable whole-card example that F0004's product card and F0006's `<ProductCard />` can adopt directly

### US00033: Tracking-hook seam ready for F0007 (inert until wired)

**As a** developer,
**I want to** a predefined, structured integration point for affiliate-click tracking on every `<AffiliateLink>`,
**So that** F0007 can light up GA4 `affiliate_click` events across the entire site without per-call-site changes.

**Acceptance Criteria:**

- **Scenario 1:** Structured data attributes are emitted
  - `Given` an `<AffiliateLink>` renders
  - `When` the DOM is inspected
  - `Then` the `<a>` element has `data-affiliate-link` (presence flag), `data-product-name`, `data-product-category`, and `data-destination-url` attributes populated from the corresponding props

- **Scenario 2:** Seam is inert until F0007 ships
  - `Given` F0007 has not yet been implemented (no GA env var, no delegated handler attached)
  - `When` a visitor clicks an `<AffiliateLink>`
  - `Then` no JavaScript event handler fires from F0003 code, no analytics call is attempted, and the new-tab navigation proceeds normally

- **Scenario 3:** No `preventDefault` on click
  - `Given` any future tracking handler wired through the seam
  - `When` the visitor clicks the link
  - `Then` the handler never calls `event.preventDefault()` — new-tab navigation is the source of truth and tracking is fire-and-forget

- **Scenario 4:** Call sites need no changes when tracking turns on
  - `Given` F0007 later attaches a delegated click listener at the root layout that reads the `data-affiliate-link` attributes
  - `When` F0007 ships
  - `Then` every existing `<AffiliateLink>` call site begins emitting `affiliate_click` events with no edit to the call site, no edit to `<AffiliateLink>` itself, and no rebuild of consumer components

- **Scenario 5:** Attribute names are stable and documented
  - `Given` the seam contract
  - `When` documented (in the component file's example and in `CLAUDE.md` codebase-structure section)
  - `Then` the four attribute names (`data-affiliate-link`, `data-product-name`, `data-product-category`, `data-destination-url`) are listed as the F0003 ↔ F0007 contract and treated as a public API of the primitive

### US00034: Affiliate URL helper & no-raw-anchor convention

**As a** developer,
**I want to** a tiny URL helper plus a documented convention preventing raw affiliate `<a>` tags,
**So that** broken, malformed, or untracked affiliate links can't sneak into production.

**Acceptance Criteria:**

- **Scenario 1:** Helper validates Shopee hosts
  - `Given` `lib/affiliate.ts` exposes a validation helper (e.g., `assertAffiliateUrl(url, productSlug)`)
  - `When` called with a URL whose host is `shopee.vn`, `shopee.ee`, or `shope.ee`
  - `Then` it returns / passes through without error

- **Scenario 2:** Unrecognized host fails the build
  - `Given` a product JSON file with an `affiliateUrl` pointing to a non-Shopee host (e.g., `https://example.com/...`)
  - `When` the product loader (F0001's `getAllProducts()`) processes the file
  - `Then` the build fails with a clear error naming the offending product slug and the disallowed host

- **Scenario 3:** Malformed URL fails the build
  - `Given` an `affiliateUrl` that is not a valid absolute URL (missing scheme, empty string, garbage text)
  - `When` the helper validates it
  - `Then` the build fails with a clear error naming the offending product slug

- **Scenario 4:** Helper is the single normalization point
  - `Given` the codebase
  - `When` reviewed
  - `Then` no file outside `lib/affiliate.ts` performs ad-hoc parsing, trimming, or rewriting of affiliate URLs — all access flows through the helper

- **Scenario 5:** Convention is documented
  - `Given` `CLAUDE.md` codebase-structure section and the `<AffiliateLink>` component's example
  - `When` reviewed
  - `Then` both explicitly state: "All affiliate destinations on the site must route through `<AffiliateLink>`. Raw `<a href="https://shope.ee/...">` (or any Shopee-host anchor) outside `<AffiliateLink>` is disallowed."

- **Scenario 6:** Convention is enforceable in review
  - `Given` a hypothetical PR adding `<a href="https://shope.ee/xyz" target="_blank">` to a component
  - `When` reviewed
  - `Then` the convention gives reviewers a clear, citeable rule to block the change on — without requiring lint tooling (custom ESLint rule is P2)

## 5. Edge Cases & Error Handling

- **Empty or missing `affiliateUrl` in product JSON:** The build must fail at load time (US00034) with the offending product slug named — the product must not silently ship without a working CTA.
- **Affiliate URL with tracking-bait query params we don't want:** Normalization belongs in `lib/affiliate.ts` (single chokepoint) — never per call site. If we later decide to strip a param, only one file changes.
- **Product name containing quotes or special characters:** The accessible label and `data-product-name` attribute must safely encode these — use plain text interpolation through React (which escapes by default), not string concatenation into HTML.
- **Visitor middle-clicks, Cmd/Ctrl-clicks, or context-menu-opens the link:** Browser default behavior (open in new tab / window) wins; the tracking seam (if active) must still fire — or, if it cannot fire reliably on alt-click, the failure is silent and the navigation still succeeds.
- **Children include an `<a>` or `<button>` (programmer error):** Ideally the component's documented usage rules surface this in code review; runtime detection is not required, but the convention doc must call it out explicitly.
- **Visitor has JavaScript disabled:** The link is a plain anchor — navigation works fully without JS. Tracking is silently absent. No degraded UX.
- **`prefers-reduced-motion`:** Hover / focus visuals on the wrapping card (US00032) must respect reduced-motion preferences — no animated transitions that violate the user setting.
- **Touch-target size on mobile:** When the primitive wraps a small inline element (e.g., a one-line product mention inside MDX), call sites are responsible for ensuring a min 44×44 px touch target. The primitive itself does not enforce a min size (would interfere with inline use).
- **Tracking seam attribute name collision:** `data-affiliate-link` is the contract name; no other component in the codebase may emit that attribute. Document this in the component file.
- **F0007 not yet shipped at v1 launch:** Acceptable. The seam is inert (US00033 Scenario 2). Links work fully; clicks are simply unmeasured until F0007 lands.

## 6. Technical Considerations

- **Server Component by default.** A plain `<a>` does not need hydration. `<AffiliateLink>` ships zero client JavaScript — every product surface on the site benefits.
- **No `"use client"` in `<AffiliateLink>`.** Click tracking is implemented in F0007 via a delegated listener attached at the layout level, not a per-anchor `onClick`. This keeps the primitive lightweight and SSG-friendly.
- **F0003 ↔ F0007 contract is the `data-affiliate-link` attribute set.** Both features depend on these names being stable. Changing them is a breaking change requiring both stories to update in the same PR.
- **`<AffiliateLink>` lives at `components/AffiliateLink.tsx`** per the project's `components/<Name>.tsx` convention. Styles, if any, co-locate as `AffiliateLink.module.css`. The primitive should be visually neutral (no opinions on display, padding, color) — consumers style the children.
- **`lib/affiliate.ts` is the single URL chokepoint.** Use `new URL(url)` for parsing (throws on malformed input — catch and rethrow with the product slug). Maintain the Shopee host allow-list as a `const` array.
- **Helper is called from `lib/products.ts`** (F0001's loader) during build-time validation, so unrecognized hosts fail the build, not at runtime.
- **Accessibility:** The accessible-label string ("Mở [productName] trên Shopee (mở tab mới)") is Vietnamese user-facing copy — treat as canonical. Identifier names and component code stay English per project conventions.
- **`rel` order matters less than presence.** `rel="noopener noreferrer sponsored"` — `noopener` is the security baseline, `noreferrer` strengthens privacy, `sponsored` is Google's required affiliate annotation. All three are non-negotiable.
- **`target="_blank"` is non-negotiable.** Every product click must open in a new tab per project convention; do not expose a prop to override this.
- **No new dependencies.** This entire feature is achievable with React + a small TypeScript helper. No tracking library, no link-management library.
- **F0004 and F0006 are blocked on this primitive.** Schedule F0003 ahead of (or alongside) F0004's product card and F0006's `<ProductCard />` — both will import `<AffiliateLink>` directly.
- **Lint rule is P2.** A custom ESLint rule that flags `<a href="https://shope.ee/...">` outside `<AffiliateLink>` would be ideal, but is deferred. Convention documented in `CLAUDE.md` + code review is the v1 enforcement mechanism.
- **English identifiers, Vietnamese user-facing copy.** The only Vietnamese strings introduced by this feature are the accessible label and (later, by consumers) the visible "Mua ngay" CTA — both per project working conventions.
