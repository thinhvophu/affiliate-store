# Implementation Plan — US00033: Tracking-hook seam ready for F0007 (inert until wired)

**Status:** Approved
**Feature:** F0003 — Affiliate Link Routing
**Spec ref:** `docs/specs/F0003.md` § US00033
**Priority:** P0 — ships as part of F0003's launch wave (cannot ship F0003 without the seam in place, because the F0003 ↔ F0007 contract is what makes F0007 a per-call-site-free integration).

**Predecessors (hard deps):**

- **US00031** — Canonical `<AffiliateLink>` Server Component. Per the Q3 design decision, **US00033 is folded into US00031's PR (`feat/US00031`)** and does not ship as an independent branch. US00031's plan absorbs both this story's runtime emit (the four `data-*` attributes — they are inseparable from the anchor render) and this story's contract documentation (the new "Affiliate links" section in `CLAUDE.md`). This plan remains as the story-level acceptance record and as the spec of the contract; the implementation tasks live in `docs/plans/US00031.md`.

**Sibling consistency:**

- **US00032** — Whole-card clickability pattern. The four `data-*` attributes added by this story must continue to render unchanged when `<AffiliateLink>` wraps a multi-element card subtree. The seam contract attaches to the `<a>` element regardless of the child shape.
- **US00034** — Affiliate URL helper + no-raw-anchor convention. The convention enforces that **every** Shopee-host anchor on the site flows through `<AffiliateLink>`, which is the only way the F0007 delegated listener can rely on `data-affiliate-link` being present on every trackable click target. US00034 carries the convention text; US00033 references the same seam in the same `CLAUDE.md` section.

**Future consumer (out of scope but governed by this contract):**

- **F0007 — Analytics & Click Tracking.** Will attach a single delegated click listener at the root layout, match via `[data-affiliate-link]`, read the three structured attributes, and fire the GA4 `affiliate_click` event. This story explicitly does **not** design or implement that listener.

## 1. Scope

Strictly:

1. Emit a presence flag `data-affiliate-link` (no value, or empty string — see §3 D1) on the rendered `<a>` element of every `<AffiliateLink>` instance.
2. Emit three structured payload attributes on the same `<a>`:
   - `data-product-name` ← `productName` prop
   - `data-product-category` ← `productCategory` prop
   - `data-destination-url` ← `href` prop
3. Verify the seam is **inert in F0003**: the component file contains no `"use client"` directive, no `onClick` handler, no event listener wiring, no analytics import, no `useEffect`, no reference to `window` / `document` / `gtag` / `dataLayer`.
4. Document the four attribute names as the **F0003 ↔ F0007 public contract** in two places:
   - Inline in `components/AffiliateLink.tsx` (header docblock + a copyable usage example showing the rendered DOM).
   - In `CLAUDE.md` § Codebase structure under the `components/` listing (and a short F0003 ↔ F0007 contract note).
5. Make stability of the attribute names an explicit, citeable rule: any rename is a breaking change and must update both `<AffiliateLink>` and (future) F0007 listener in the same PR.
6. Add an automated verification step in this plan's AC (§7) that confirms the rendered HTML for a representative `<AffiliateLink>` call site carries all four attributes with the expected values, and that the component file passes a `grep` check for forbidden client-side primitives (`"use client"`, `onClick`, `useEffect`, `addEventListener`).

**Explicitly out of scope** (owned by sibling stories or downstream features):

- The `<AffiliateLink>` component's anchor semantics — `target`, `rel`, the visually hidden accessible label, the Server-Component declaration, the prop type signatures, the children pass-through, the "no `target`/`rel` at the call site" type guard. All owned by **US00031**. This story assumes the anchor is already rendering correctly per US00031 and simply adds four `data-*` attributes to it.
- Whole-card clickability example and hover/focus states. Owned by **US00032**.
- `lib/affiliate.ts` URL helper, Shopee host allow-list, no-raw-anchor convention text. Owned by **US00034**. This story cites US00034's convention but does not write it.
- F0007's delegated click listener at the layout level, the GA4 `affiliate_click` event payload mapping, env-gating on `NEXT_PUBLIC_GA_MEASUREMENT_ID`, `gtag` plumbing, the layout wiring of `<script>` tags. All owned by **F0007**.
- A runtime check that `data-affiliate-link` is unique to `<AffiliateLink>` (no other component emits it). Convention-only at v1; documented in the component file and `CLAUDE.md`.
- A custom ESLint rule that flags raw `<a href="https://shope.ee/...">` or that flags `data-affiliate-link` outside `<AffiliateLink>`. P2 per F0003 spec §2.
- Migration of any existing raw `<a>` Shopee anchors in the codebase to `<AffiliateLink>`. None exist today (no product / blog routes shipped yet — verified against the route map in `CLAUDE.md`).

## 3. Approved Decisions

| #   | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| D1  | Presence flag encoding | **Emit `data-affiliate-link=""` (i.e., as a JSX attribute `data-affiliate-link=""`).** Selector `[data-affiliate-link]` matches either an empty value or any non-null value, so F0007 can use the cleanest possible selector. | A pure boolean attribute in JSX would need `data-affiliate-link` set to a string literal anyway (React serialises booleans differently across versions). `""` is the smallest stable serialisation, matches `[data-affiliate-link]` selectors, and reads as "presence flag, no payload" to a reviewer. The exact runtime serialisation (empty string vs. attribute-only) is implementation detail and not part of the F0003 ↔ F0007 contract — the **selector** is what's contracted. |
| D2  | Payload attribute serialisation | **Pass `productName`, `productCategory`, and `href` straight through as React string children of the `data-*` attribute.** React handles HTML-encoding (quotes, ampersands, Vietnamese diacritics) automatically. | The spec's Edge Cases §5 explicitly call out "product name containing quotes or special characters" — relying on React's built-in escaping is the safe, idiomatic answer. No custom `escapeHtml` helper. |
| D3  | Attribute name choice | **`data-affiliate-link`, `data-product-name`, `data-product-category`, `data-destination-url`.** These four names exactly. | Names are dictated verbatim by the F0003 spec (§2 + US00033 AC Scenario 1 + AC Scenario 5). Treating any deviation as a breaking change keeps the contract trivial to reason about. The kebab-case form follows HTML `data-*` convention and aligns with React's required attribute casing for custom DOM attributes. |
| D4  | Where the attributes are emitted | **Directly on the rendered `<a>` element inside `components/AffiliateLink.tsx`.** Not on a wrapper `<span>`, not on the children. | A delegated listener at the layout level uses `event.target.closest('[data-affiliate-link]')` to find the nearest matching ancestor. The attribute **must** be on the anchor so that the closest match is the anchor itself, and so the listener reads the anchor's `href` if it ever needs the actual destination value (already mirrored in `data-destination-url`, but co-location prevents drift). |
| D5  | Inert verification | **Static-source guard:** the component's source file must not contain any of the strings `"use client"`, `onClick`, `useEffect`, `addEventListener`, `gtag`, `dataLayer`, `window.`, or `document.`. A short `grep -E` invocation in §7's AC table verifies this at PR time. | The spec's AC Scenario 2 ("no JS handler from F0003 fires") and Scenario 3 ("no `preventDefault`") translate naturally to a source-level invariant. We don't need to run the page and intercept events to prove absence — proving the symbols are not in the file is stronger and faster. |
| D6  | Documentation location: component file | **JSDoc header on the exported component + a fenced markdown code block in a `/** Example */` doc-comment** showing both the JSX call-site and the rendered DOM, including all four `data-*` attributes. The doc-block also carries a one-line "F0003 ↔ F0007 contract" note naming the four attributes as a public API. | The closest possible point to the source of truth — anyone editing `<AffiliateLink>` will see the contract in the same file. Co-locating the example with the implementation keeps it from drifting. |
| D7  | Documentation location: `CLAUDE.md` | **Create a new top-level "Affiliate links" section** (placed between "UI / theme rules" and "Required disclosures") containing an "F0003 ↔ F0007 contract" paragraph that names the four attributes as a public API. Plus a one-line annotation on the `components/AffiliateLink.tsx` row in the Codebase-structure tree. **The new section is also the future home for US00034's no-raw-anchor convention bullet** (per the Q4 design decision — one section, two paragraphs, not two scattered bullets). | Per CLAUDE.md's own rule: "Update this section whenever a story adds/moves/renames files or introduces new conventions." A new section beats sprinkling affiliate rules across "Working conventions" and "UI / theme rules" — F0003 is load-bearing and deserves discoverability. |
| D8  | No new client JS, no new dependency | The seam is **only** four `data-*` attributes on an SSR'd `<a>`. No new package, no new import, no new bundle delta. | Confirmed against the spec §6: "F0003 ships zero client JavaScript" and "no new dependencies". |
| D9  | No runtime guard that the attribute is unique to `<AffiliateLink>` | Convention-only at v1. The component file's docblock and `CLAUDE.md` both state: "No other component in the codebase may emit `data-affiliate-link`." | Per spec §5 Edge Cases ("Tracking seam attribute name collision"). Enforcement is review-time, not runtime. A custom ESLint rule is P2. |
| D10 | Branch / commit target | **Folded into `feat/US00031`** as a single primitive-PR. PR title: `feat(US00031,US00033): canonical AffiliateLink primitive + F0003↔F0007 tracking seam`. The four `data-*` attributes ship as part of the anchor's render (Scenario 1's "correctly-attributed anchor" is inseparable from the seam attrs); the contract documentation ships as part of the same PR's JSDoc + CLAUDE.md edits. | Per the Q3 design decision: splitting US00031 and US00033 would yield two trivial diffs against the same file (the second one almost entirely `.md`). One coherent PR is the cleaner unit of review and merge. |
| D11 | Test framework | **None added.** AC §7 uses (a) `grep` on the source and built output, (b) `npm run dev` + DevTools inspection of a representative product CTA, (c) the existing `npm run typecheck && npm run lint && npm run build` chain. | The codebase has no test runner today. Introducing one for a 4-attribute change is out of proportion. F0007 will likely justify adding a test runner; until then, source-level + DOM-inspection verification is sufficient. |
| D12 | Where the "inert under JS-off" guarantee comes from | **Structural, not behavioural.** Because `<AffiliateLink>` is a Server Component that emits a plain `<a>` with only `data-*` attributes, there is literally no JS path that could fire on click from F0003 code. JS-off behaviour = JS-on behaviour for this primitive: a normal new-tab anchor click. | Spec AC Scenario 2 is satisfied by construction; no explicit JS-disable test is needed beyond the source-grep guard from D5. |

## 5. Concrete diff sketch

### 5.1 `components/AffiliateLink.tsx` — illustrative shape (final structure owned by US00031)

The exact JSX skeleton and prop signature are US00031's contract. This story contributes the bolded lines plus the docblock:

```tsx
// Server Component — NO "use client". See US00031 + US00033.

type AffiliateLinkProps = {
  href: string;
  productName: string;
  productCategory: string;
  children: React.ReactNode;
  // US00031 owns the rest of the prop shape; target/rel deliberately excluded.
};

/**
 * <AffiliateLink> — canonical affiliate anchor.
 *
 * Renders a Server-side <a> with target="_blank" rel="noopener noreferrer sponsored"
 * and a Vietnamese accessible label. See US00031 for the anchor contract,
 * US00032 for the whole-card clickability pattern, US00034 for the URL helper
 * and no-raw-anchor convention.
 *
 * --- F0003 ↔ F0007 contract (US00033) ---------------------------------
 * Every rendered anchor carries FOUR stable data-* attributes that F0007's
 * delegated click listener relies on. These names are a PUBLIC API of this
 * primitive — renaming any of them is a breaking change and must update the
 * F0007 listener in the same PR.
 *
 *   data-affiliate-link              presence flag (selector hook)
 *   data-product-name                from productName prop
 *   data-product-category            from productCategory prop
 *   data-destination-url             from href prop (mirrors the href)
 *
 * No other component in the codebase may emit `data-affiliate-link`.
 *
 * Example rendered DOM:
 *
 *   <a href="https://shope.ee/abc"
 *      target="_blank"
 *      rel="noopener noreferrer sponsored"
 *      aria-label="Mở Chuột Logitech G102 trên Shopee (mở tab mới)"
 *      data-affiliate-link=""
 *      data-product-name="Chuột Logitech G102"
 *      data-product-category="chuot-gaming"
 *      data-destination-url="https://shope.ee/abc">
 *     Mua ngay
 *   </a>
 *
 * Until F0007 ships, the seam is INERT: no JS handler is attached from F0003,
 * the click triggers a normal new-tab navigation, and analytics are silently
 * absent. JS-off behaviour is identical (the anchor is a plain server-rendered
 * <a>).
 * -----------------------------------------------------------------------
 */
export function AffiliateLink({ href, productName, productCategory, children }: AffiliateLinkProps) {
  return (
    <a
      href={href}
      target="_blank"
      rel="noopener noreferrer sponsored"
      aria-label={`Mở ${productName} trên Shopee (mở tab mới)`}
      // US00033 — F0003 ↔ F0007 contract attributes. Do not rename without
      // updating the F0007 delegated listener in the same PR.
      data-affiliate-link=""
      data-product-name={productName}
      data-product-category={productCategory}
      data-destination-url={href}
    >
      {children}
    </a>
  );
}
```

Notes:

- The bolded contributions of US00033 are the four `data-*` attributes, the `// US00033 …` comment, and the `--- F0003 ↔ F0007 contract ---` section of the JSDoc.
- The exact accessible-label string and the children-pass-through structure are illustrative; US00031 is authoritative for those.
- No `"use client"`. No `useEffect`. No `onClick`. No `addEventListener`. By construction.

## 10. Risks / open items

- **Coordination with US00031.** This story cannot land in isolation — it edits a file US00031 creates. The implementer must verify US00031's branch state at start (step 1 in §6) and either fold the diff in or rebase appropriately. If US00031 ends up using slightly different prop names (e.g., `name` instead of `productName`), this story's attribute-payload sources must follow suit and the JSDoc example updated to match. The four **attribute names** are non-negotiable; the prop names sourcing them are owned by US00031.
- **Documentation drift between the file and `CLAUDE.md`.** The four attribute names appear in two places by design (component file JSDoc + `CLAUDE.md` working-conventions paragraph). Any future rename must update both — listed in the contract docblock explicitly to prevent silent drift.
- **F0007 selector contract.** This plan assumes F0007 will use `event.target.closest('[data-affiliate-link]')` to find the anchor. If F0007 later prefers, e.g., `[data-affiliate-link="true"]`, the value form of the presence flag (D1) becomes contractual. Recommend F0007's planner adopt the attribute-presence selector (`[data-affiliate-link]`) rather than a value-equality selector to keep this freedom.
- **MDX edge case.** F0006's `<ProductCard slug="..." />` (out of scope here) will render via the React MDX renderer and ultimately delegate to `<AffiliateLink>`. The four `data-*` attributes will appear on the resulting anchor exactly as for any other call site — but reviewers of F0006 should explicitly confirm this when that story lands. The convention paragraph in `CLAUDE.md` makes the expectation citeable.
- **No automated regression test for attribute presence.** With no test runner in the codebase today, a future careless edit could silently remove one of the four attributes. Mitigation: the JSDoc example block in the component file shows the rendered DOM verbatim, which acts as a code-review checklist. F0007's own test suite (when it lands) will exercise the selector and would fail loudly if any attribute went missing.
- **Selector collision risk.** If a future component (e.g., a third-party widget, an embedded affiliate from a non-Shopee network in some hypothetical v2) emits `data-affiliate-link`, F0007's listener would fire incorrectly. The convention forbids this; the documented enforcement is review-time. A custom ESLint rule remains the P2 follow-up.
- **No client-JS-disabled regression in CI.** The "JS-off behaviour is identical" guarantee from D12 is structural, not exercised by a test. Acceptable at v1 because the component is provably plain SSR HTML; a regression would require introducing `"use client"` or an event handler, which the source-grep guard catches.
