**Status:** Approved

**Spec ref:** `docs/specs/F0002.md` § US00025
**Predecessors (hard):**

## 1. Scope

Strictly:

1. Below 768px, hide the inline left panel produced by US00024's shell layout and render a **trigger button** (filter / menu icon, white-on-orange) in its place.
2. Activating the trigger opens a **slide-out drawer** containing the same content that would render inline at desktop width, with a dimmed full-page **overlay** behind it.
3. Drawer surface uses `var(--color-surface)` with `1px var(--color-border-hairline)` borders and `var(--radius-md)` corner radius.
4. Closing the drawer:
   - **Esc** key,
   - **click/tap on overlay** (outside the drawer panel),
   - explicit **close button** inside the drawer header.
5. **Focus management:**
   - On open: focus moves to the first focusable element inside the drawer; if none exist, focus the close button (edge case from spec §5).
   - While open: focus is trapped — Tab and Shift+Tab cycle inside the drawer only.
   - On close: focus returns to the trigger button that opened it.
6. **Body scroll lock** while the drawer is open (no underlying scroll behind the overlay), including iOS Safari quirks.
7. **Viewport resize past 768px while open**: the drawer closes cleanly, scroll lock releases, focus restores to a sensible target, and the inline left panel re-appears.
8. **Breakpoint sourced from tokens (not hardcoded inline)**: a single shared constant (`BREAKPOINT_TABLET_PX`) in `lib/breakpoints.ts` mirrors the documented value of `--breakpoint-tablet` from `app/globals.css`. JS uses `window.matchMedia(\`(max-width: ${BREAKPOINT_TABLET_PX - 0.02}px)\`)`. A comment in both files cross-references the other so drift is obvious in review.
9. **`prefers-reduced-motion: reduce`**: drawer slide animation and overlay fade are disabled — instant open/close.
10. Use a **shared `<Drawer>` primitive** under `components/Drawer.tsx` (`"use client"`) that hosts: state, Esc handler, focus trap, scroll lock, overlay, slide panel. The `ShellLayoutDrawer` (for the left panel) and the future header nav drawer (US00021) both consume this primitive.

**Explicitly out of scope** (owned by sibling stories or downstream features):

- The two-column shell scaffold itself (`ShellLayout`) — **US00024**.
- The Header (orange band, logo, nav links, hamburger trigger wiring) — **US00021**. This story exports the `<Drawer>` primitive that US00021 will consume; US00021 owns the *trigger* in the header and the nav content passed in.
- The Footer — **US00022**.
- The actual filter UI / nav links that *go inside* the drawer — those are page-owned content (F0004 product filters, etc.); this story only provides the chrome.
- Sticky-on-scroll header — **US00026**.
- New design tokens — US00016/US00023 already cover all tokens this story reads. No new tokens added here.
- No `aria-live` announcements or skip-to-drawer link — drawer is a self-contained modal-like region; standard focus trap + `role="dialog"` + `aria-modal="true"` is sufficient.

## 3. Approved Decisions

| #   | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| D1  | Shared primitive vs. two separate drawers | **One `<Drawer>` primitive in `components/Drawer.tsx`** consumed by both this story's `ShellLayoutDrawer` and US00021's header nav drawer. | Spec §5 calls out "drawer behavior defined in US00025"; reuse keeps focus trap / scroll lock logic in one place. The primitive accepts `side: "left" \| "right"`, `open: boolean`, `onClose: () => void`, `triggerRef: RefObject`, `labelledBy?: string`, `children`. Two consumers, one implementation. |
| D2  | Slot vs. wrapper for shell integration | **`ShellLayout` (US00024) renders both an inline panel (≥768px) and a `<ShellLayoutDrawer>` (always present, only visible <768px via CSS) over the same `leftPanel` content.** | Avoids prop-drilling drawer state into every consumer page. Pages still only pass `leftPanel` once. The CSS shows/hides the desktop panel and the mobile trigger via `@media`. No SSR/client mismatch — both render unconditionally, CSS decides visibility. |
| D3  | Where the trigger button lives | **Inside `<ShellLayoutDrawer>`, rendered above main content on mobile** (sticky-top or top-of-main area). | The trigger is mobile-only chrome of the shell layout itself, not the header. Per spec §1 Scenario 1: "a trigger button (e.g., filter / menu icon styled white-on-orange) is rendered to open the drawer". Stays adjacent to the content it filters. |
| D4  | Breakpoint source in JS | **Single shared constant in `lib/breakpoints.ts` that mirrors the documented value of `--breakpoint-tablet`.** Cross-referenced via comments in both files. | Spec Scenario 6 forbids "hardcoded inline" but pragmatically allows a shared constant or `getComputedStyle()`. `getComputedStyle(document.documentElement).getPropertyValue('--breakpoint-tablet')` returns a string like `" 768px"` that needs parsing every read — and it can't run during SSR. A constant is simpler, type-safe, and trivially audited. The comment in `breakpoints.ts` and `globals.css` makes drift visible. |
| D5  | Focus trap implementation | **Hand-rolled: on Tab/Shift+Tab keydown inside the drawer, compute the focusable elements list and redirect focus when it would escape.** Use a small util in `lib/focus-trap.ts` (or inline in `Drawer.tsx`). | No external dep allowed (spec §6). Standard pattern: query `[href], button, input, select, textarea, [tabindex]:not([tabindex="-1"])` inside the drawer root, filter visible/enabled, wrap focus on first/last. |
| D6  | Scroll lock implementation | **Toggle `overflow: hidden` on `<body>` and pad-right by scrollbar width to prevent layout shift.** Restore previous inline style on cleanup. | Vanilla approach, no dep. iOS quirk: also set `body { position: fixed; width: 100%; top: -<scrollY>px }` and restore scroll on unlock. Spec §5 only requires "underlying page does not scroll" — full iOS treatment is bonus; document the iOS branch in Section 5. |
| D7  | Animation strategy | **CSS `transition: transform 200ms ease, opacity 200ms ease`** on the drawer panel (translate) and overlay (opacity). Wrapped in `@media (prefers-reduced-motion: reduce) { transition: none; }`. | Pure CSS. Matches the rest of the project (no JS animation libs). Reduced-motion compliance per spec §5. |
| D8  | Drawer side | **Left** (slides in from the left edge) for the shell-layout filter drawer, matching the inline panel's desktop position. The header nav drawer (US00021) can choose its own side via the `side` prop. | Visual continuity: filters are on the left at desktop, so they should slide in from the left at mobile. |
| D9  | A11y semantics | **`role="dialog"`, `aria-modal="true"`, `aria-labelledby="drawer-title"`** on the drawer root. The drawer has a visually-shown title (e.g., "Bộ lọc" for filters, "Điều hướng" for nav) provided by the consumer. | Standard modal pattern; screen readers correctly identify the trapped region. Title text is consumer-supplied (Vietnamese, owned by the consumer page). |
| D10 | Initial render in SSG | **Drawer renders in DOM at all times but is `display: none` (or `visibility: hidden` + `pointer-events: none`) until `open === true` AND viewport <768px.** | Avoids hydration mismatch (no conditional client render). State changes only after hydration. Initial server HTML for ≥768px users contains the markup but it's CSS-hidden — zero visual impact. |
| D11 | matchMedia listener | **Subscribe via `mql.addEventListener('change', …)` inside `useEffect` and force-close the drawer when the query no longer matches.** Unsubscribe on unmount. | Implements spec §5 "viewport resized past 768px while open" edge case cleanly. |
| D12 | Trigger button placement & icon | **Rendered as the first child of the mobile shell area; uses an inline SVG hamburger / filter icon (no icon library).** White stroke on `var(--color-primary)` background, `var(--radius-sm)` corners, ≥44×44 touch target. | No new dep. | D13 | Branch strategy | **Feature branch `feat/US00025`, PR to `main`.** Plan reviewer (main agent) bundles all six F0002 plans before any implementation kicks off. | Follows project convention. |
| D14 | Trigger label text | **Vietnamese: "Bộ lọc" for filter drawer (default), overridable via prop `triggerLabel`.** | User-facing copy in Vietnamese per CLAUDE.md. Default fits the filter-list use case; US00021 will override to "Điều hướng" or similar. |

### 5.1 `lib/breakpoints.ts` (new)

```ts
/**
 * Breakpoint constants — single source of truth for JS-side responsive logic.
 *
 * IMPORTANT: these values MUST mirror the breakpoint CSS custom properties in
 * `app/globals.css` (US00016). CSS spec forbids `var()` inside `@media`
 * queries, so JS and CSS each need a copy. If you change one, change the
 * other and grep for usages.
 *
 *   CSS:  --breakpoint-tablet: 768px;   --breakpoint-desktop: 1280px;
 *   JS:   BREAKPOINT_TABLET_PX = 768;   BREAKPOINT_DESKTOP_PX = 1280;
 */
export const BREAKPOINT_TABLET_PX = 768;
export const BREAKPOINT_DESKTOP_PX = 1280;

/** matchMedia string that matches mobile (strictly below tablet). */
export const MOBILE_MEDIA_QUERY = `(max-width: ${BREAKPOINT_TABLET_PX - 0.02}px)`;
```

(The `- 0.02` avoids the half-pixel overlap with `min-width: 768px` queries used elsewhere — standard practice.)

### 5.2 `components/Drawer.tsx` (new — sketch, not production-ready)

```tsx
"use client";

import { useEffect, useRef, type ReactNode, type RefObject } from "react";
import styles from "./Drawer.module.css";

interface DrawerProps {
  open: boolean;
  onClose: () => void;
  side?: "left" | "right";
  labelledBy: string;             // id of the visible heading inside `children`
  triggerRef: RefObject<HTMLElement | null>;
  children: ReactNode;
}

export function Drawer({
  open, onClose, side = "left", labelledBy, triggerRef, children,
}: DrawerProps) {
  const panelRef = useRef<HTMLDivElement>(null);
  const closeButtonRef = useRef<HTMLButtonElement>(null);

  // 1. Esc to close
  useEffect(() => {
    if (!open) return;
    const onKey = (e: KeyboardEvent) => {
      if (e.key === "Escape") onClose();
    };
    document.addEventListener("keydown", onKey);
    return () => document.removeEventListener("keydown", onKey);
  }, [open, onClose]);

  // 2. Body scroll lock
  useEffect(() => {
    if (!open) return;
    const prev = document.body.style.overflow;
    const scrollbarW = window.innerWidth - document.documentElement.clientWidth;
    document.body.style.overflow = "hidden";
    if (scrollbarW > 0) document.body.style.paddingRight = `${scrollbarW}px`;
    return () => {
      document.body.style.overflow = prev;
      document.body.style.paddingRight = "";
    };
  }, [open]);

  // 3. Focus management: move focus in on open; restore on close
  useEffect(() => {
    if (!open) return;
    const panel = panelRef.current;
    if (!panel) return;

    const focusables = panel.querySelectorAll<HTMLElement>(
      'a[href], button:not([disabled]), input:not([disabled]), select:not([disabled]), textarea:not([disabled]), [tabindex]:not([tabindex="-1"])'
    );
    const first = focusables[0] ?? closeButtonRef.current;
    first?.focus();

    return () => {
      // Restore focus to the trigger that opened us
      triggerRef.current?.focus();
    };
  }, [open, triggerRef]);

  // 4. Focus trap on Tab / Shift+Tab
  const onKeyDownPanel = (e: React.KeyboardEvent<HTMLDivElement>) => {
    if (e.key !== "Tab") return;
    const panel = panelRef.current;
    if (!panel) return;
    const focusables = Array.from(
      panel.querySelectorAll<HTMLElement>(
        'a[href], button:not([disabled]), input:not([disabled]), select:not([disabled]), textarea:not([disabled]), [tabindex]:not([tabindex="-1"])'
      )
    ).filter((el) => el.offsetParent !== null);
    if (focusables.length === 0) {
      e.preventDefault();
      closeButtonRef.current?.focus();
      return;
    }
    const first = focusables[0];
    const last = focusables[focusables.length - 1];
    const active = document.activeElement;
    if (e.shiftKey && active === first) {
      e.preventDefault();
      last.focus();
    } else if (!e.shiftKey && active === last) {
      e.preventDefault();
      first.focus();
    }
  };

  return (
    <div
      className={`${styles.root} ${open ? styles.open : ""}`}
      aria-hidden={!open}
    >
      <div className={styles.overlay} onClick={onClose} />
      <div
        ref={panelRef}
        className={`${styles.panel} ${styles[`panel--${side}`]}`}
        role="dialog"
        aria-modal="true"
        aria-labelledby={labelledBy}
        onKeyDown={onKeyDownPanel}
        tabIndex={-1}
      >
        <button
          ref={closeButtonRef}
          type="button"
          className={styles.closeButton}
          onClick={onClose}
          aria-label="Đóng"
        >
          ×
        </button>
        {children}
      </div>
    </div>
  );
}
```

### 5.3 `components/ShellLayoutDrawer.tsx` (new — sketch)

```tsx
"use client";

import { useEffect, useId, useRef, useState, type ReactNode } from "react";
import { Drawer } from "./Drawer";
import { MOBILE_MEDIA_QUERY } from "@/lib/breakpoints";
import styles from "./ShellLayoutDrawer.module.css";

interface ShellLayoutDrawerProps {
  leftPanel: ReactNode;
  title?: string;          // visible heading inside the drawer; defaults to "Bộ lọc"
  triggerLabel?: string;   // accessible label / button text; defaults to "Bộ lọc"
}

export function ShellLayoutDrawer({
  leftPanel,
  title = "Bộ lọc",
  triggerLabel = "Bộ lọc",
}: ShellLayoutDrawerProps) {
  const [open, setOpen] = useState(false);
  const triggerRef = useRef<HTMLButtonElement>(null);
  const titleId = useId();

  // Auto-close when viewport crosses back to ≥768px (spec §5 edge case)
  useEffect(() => {
    if (!open) return;
    const mql = window.matchMedia(MOBILE_MEDIA_QUERY);
    const onChange = (e: MediaQueryListEvent) => {
      if (!e.matches) setOpen(false);
    };
    mql.addEventListener("change", onChange);
    return () => mql.removeEventListener("change", onChange);
  }, [open]);

  return (
    <>
      <button
        ref={triggerRef}
        type="button"
        className={styles.trigger}
        onClick={() => setOpen(true)}
        aria-haspopup="dialog"
        aria-expanded={open}
      >
        {/* inline hamburger SVG */}
        <svg aria-hidden="true" width="20" height="20" viewBox="0 0 20 20">
          <path d="M3 5h14M3 10h14M3 15h14" stroke="currentColor" strokeWidth="2" strokeLinecap="round" />
        </svg>
        <span>{triggerLabel}</span>
      </button>

      <Drawer
        open={open}
        onClose={() => setOpen(false)}
        side="left"
        labelledBy={titleId}
        triggerRef={triggerRef}
      >
        <h2 id={titleId} className={styles.title}>{title}</h2>
        {leftPanel}
      </Drawer>
    </>
  );
}
```

### 5.6 `components/ShellLayout.tsx` (US00024) — integration point

export function ShellLayout({ leftPanel, children }: ShellLayoutProps) {
  return (
    <div className={styles.container}>

CSS: `.leftPanelDesktop { display: none; } @media (min-width: 768px) { .leftPanelDesktop { display: block; } }`. The `leftPanel` node is rendered twice — once as RSC output for desktop, once passed as a child to the client `ShellLayoutDrawer` for mobile. **This is a coordination point with US00024's plan — flagged in §10.**

> Note on double-rendering `leftPanel`: passing the same React node twice is supported by React. If the `leftPanel` contains stateful Client Components (filters with internal state), each rendering will own independent state — a known footgun. F0004's filter story should hoist filter state to URL search params (already the project convention per CLAUDE.md SEO rules: `?page=N`-style crawlable params), so the two renderings remain in sync via the URL. Document this in the F0004 plan when it lands; out of scope here.

## 10. Risks / open items

- **Coordination with US00024:** This plan modifies the `ShellLayout.tsx` component that US00024 creates. The two plans must be reviewed together so US00024's plan does not render `leftPanel` *only* inline. **Action:** the F0002 batch reviewer should cross-check US00024's plan exposes `leftPanel` as a slot that can be passed unchanged into `<ShellLayoutDrawer>`.
- **Coordination with US00021:** US00021 (header) wants its own mobile nav drawer triggered by a header hamburger. The recommended pattern is for US00021 to import and reuse `components/Drawer.tsx` directly, owning its own `open` state and `triggerRef` inside a `HeaderMobileNav` client component. **Action:** the US00021 plan should explicitly call out "Reuse `<Drawer>` from US00025; do not implement a parallel primitive". If the US00021 plan lands first or independently, raise it in PR.
- **Double-rendering `leftPanel`:** As noted in §5.6, the `leftPanel` node renders twice (once as the desktop aside, once inside the mobile drawer). For purely static markup this is harmless. For stateful filter UIs (F0004), state must be hoisted to URL search params or a shared client store — flag for F0004's plan.
- **iOS Safari scroll lock:** The simple `overflow: hidden` approach can be defeated by iOS's overscroll behavior. The sketch in §5.2 documents a fuller `position: fixed` + scroll-restore fallback. Decide at implementation time whether to ship the simple version (covers desktop + Android, acceptable per spec wording) or the full iOS treatment. Recommend: ship simple, add iOS treatment only if real-device QA shows underlying scroll bleeding through.
- **No external focus-trap lib:** The hand-rolled trap covers the common cases but does *not* handle dynamic content (focusables added after mount), shadow DOM, or `inert` attribute on background. For this project's use case (static filter UI, nav links) the simple trap is sufficient. If a future story adds dynamic content inside the drawer, revisit.
- **`role="dialog"` vs. `aria-modal`:** Some screen readers misbehave with `aria-modal="true"` on non-truly-modal regions. Mitigation: in addition to `aria-modal="true"` on the panel, apply `inert` (or fallback `aria-hidden="true"`) to the rest of the page when the drawer is open. Out of scope for v1 (the spec only requires focus trap), but flag for follow-up.
- **`useId()` SSR stability:** React 18+ `useId()` returns stable IDs across server and client. Confirmed in React 19 (project's version). No hydration mismatch expected.
- **CSS `display: none` on `.root` at ≥768px:** Provides defense-in-depth so the drawer is invisible on desktop even if state-management bug leaks. But `display: none` also disables focus inside — fine, since focus shouldn't be there at desktop anyway. The `useEffect` in `ShellLayoutDrawer` *also* listens for media-query changes and force-closes the state; the two together cover all paths.
- **Hardcoded `200ms` animation duration in CSS:** Not strictly a token. If F0002 or a downstream story introduces an `--anim-duration-*` token suite, refactor. For now, 200ms is inline-acceptable.
- **Drawer width `min(85vw, 320px)`:** Chosen to leave a visible strip of overlay for "tap-outside-to-close" affordance on small phones. Not a token. Acceptable inline magic number.
