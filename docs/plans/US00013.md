# Implementation Plan — US00013: Configure MDX pipeline with frontmatter parsing

**Status:** Approved
**Feature:** F0001 — Technical Foundation & Bootstrap
**Spec ref:** `docs/specs/F0001.md` § US00013
**Predecessors:** US00011 (Next.js + TS bootstrap), US00012 (canonical folders + `@/*` alias)

---

## 1. Scope

Strictly:

1. Install and configure `@next/mdx` so `.mdx` files can be imported as React components in the App Router (Turbopack-friendly).
2. Add `remark-frontmatter` to strip the YAML block at MDX compile time so it doesn't render as visible content.
3. Add `gray-matter` as the read-time frontmatter parser (the *dependency* lands here; the loader that consumes it is US00015).
4. Create the root-level `mdx-components.tsx` required by `@next/mdx` + App Router, with an empty components map (F0003 will register `<ProductCard />` here later).
5. Prove the pipeline end-to-end with a touchstone `.mdx` file (frontmatter + body + custom-component reference + malformed-frontmatter error case), then **revert** the touchstone before commit.

**Explicitly out of scope** (owned by sibling stories in F0001):

- `Post` / `Product` / frontmatter shared types → **US00014**
- `getAllPosts()` / `getPostBySlug()` loader functions → **US00015**
- Real `<ProductCard />` implementation → **F0002**
- Global styles / design tokens → **US00016**
- ESLint / Prettier → **US00017**
- `lib/env.ts` → **US00018**

No real MDX content, no loader code, no shared types land here. We're wiring the pipeline.

## 3. Approved Decisions

| # | Decision | Choice | Rationale |
|---|---|---|---|
| D1 | MDX library | **`@next/mdx`** | Official, recommended by `node_modules/next/dist/docs/01-app/02-guides/mdx.md`. `next-mdx-remote` is for runtime/CMS-sourced MDX — not our static-files use case. |
| D2 | Frontmatter strategy | **`gray-matter` (read-time) + `remark-frontmatter` (strip YAML at compile)** | gray-matter gives the US00015 loader typed frontmatter cheaply for listing pages (no MDX compile needed). remark-frontmatter ensures the YAML block doesn't appear in rendered output. `remark-mdx-frontmatter` is **not** needed because frontmatter consumption flows through gray-matter. |
| D3 | `pageExtensions` | **Default — do NOT add `'mdx'`** | Posts live in `content/posts/`, routes live in `app/`. We never want a stray `.mdx` in `app/` to become a route. |
| D4 | `mdx-components.tsx` location | **Project root** (sibling of `app/`) | Required by `@next/mdx` + App Router (per Next docs). Single map is what F0003 / F0006 will extend. |
| D5 | TS module declaration for `*.mdx` | **Add `types/mdx.d.ts`** | `@next/mdx` does not auto-emit a global type for `import Post from "./foo.mdx"`. 5 lines in `types/` (which exists from US00012) future-proofs imports for US00015. |
| D6 | Custom-component touchstone | A tiny inline `<Touchstone />` React component, mapped in `mdx-components.tsx` *during verification only* | Demonstrates the injection mechanism without pre-empting F0002's `<ProductCard />`. Reverted before commit. |
| D7 | Verification surface | **Touchstone MDX at `content/posts/_touchstone.mdx` + temp edit to `app/page.tsx`** | Mirrors US00012 pattern (inject → verify → revert). No dedicated route needed. |
| D8 | AC4 (clear error on bad frontmatter) | Demonstrate via a one-shot `node -e "require('gray-matter')(...)"` against a deliberately corrupted file; confirm the error mentions the file. | Final loader-level error wrapping is US00015's job. US00013 only proves the underlying parser fails clearly. |
| D9 | Turbopack plugin format | Pass `'remark-frontmatter'` as a **string** in `createMDX({ options: { remarkPlugins: [...] } })` | Turbopack (default in Next 16) requires string-form plugins — functions can't cross the JS↔Rust boundary. |
| D10 | `content/posts/` subfolder | **Create `content/posts/.gitkeep`** (replace top-level `content/.gitkeep`) | The MDX pipeline targets this path; US00015 expects it. Cheap and right-sized. |
| D11 | Branch / commit target | **Commit directly to `main`** | Matches US00011 / US00012 convention. |

## 5. Concrete diff sketch

### 5.1 `next.config.ts`

```ts
import type { NextConfig } from "next";
import createMDX from "@next/mdx";

const nextConfig: NextConfig = {
  // pageExtensions intentionally left default — .mdx files do NOT become routes.
};

const withMDX = createMDX({
  options: {
    remarkPlugins: ["remark-frontmatter"],
    rehypePlugins: [],
  },
});

export default withMDX(nextConfig);
```

## 8. Risks / open items

- **Turbopack plugin string form.** `remark-frontmatter` is passed as a string per Next 16 + Turbopack constraints. Future plugins must follow the same form.
- **`@types/mdx` ↔ React 19 drift.** Some `@types/mdx` versions are typed against React 18; if the components map type complains, a `satisfies MDXComponents` cast handles it. The local `*.mdx` declaration in D5 isolates downstream code from this.
- **`content/posts/.gitkeep` lifetime.** US00015 will populate the folder with real `.mdx` files; `.gitkeep` should be removed in that story.
- **No real `<ProductCard />`.** F0002 owns the component. The map ships empty here.
- **`mdx-components.tsx` root position is load-bearing.** Required by `@next/mdx` + App Router. Moving it is a breaking change.
- **Frontmatter schema is not yet enforced.** Validation against the `Post` type is US00015's job. US00013 only proves parsing works.
