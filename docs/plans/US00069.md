# US00069 — Read-time estimate (P2)

**Feature:** F0006 (Editorial Content)
**Priority:** P2 (nice-to-have, cleanly deferrable)
**Status:** Approved

---

## 1. Summary

Surface an estimated read time (e.g. `5 phút đọc`) for each blog post, derived
from the post body word count at a fixed words-per-minute rate, computed
deterministically at build time. The estimate is shown **alongside the date** on
both the listing items (US00064) and the post header (US00066). Because this is
a P2 nicety, it must be an additive, drop-in addition: the listing and post
pages must render correctly with or without it, with no layout breakage and no
hard dependency forced onto the P0 stories.

This is the smallest story in F0006: one pure helper plus two one-line render
additions. No new route, no new component, no new data model.

## 3. Where the computation lives, and why

**Decision: add `readingTimeVi(content: string): string` to `lib/format.ts`.**

Rationale:
- F0006 §6 explicitly designates `lib/format.ts` as the home for the parallel
  Vietnamese chokepoint formatters (`formatVnd`, `formatPostDate`). Read-time is
  the same shape: pure string → Vietnamese display string, deterministic,
  React-free.
- A separate `lib/reading-time.ts` would fragment a tiny (≈10 line) formatter
  away from its siblings for no benefit and add a new structure-doc entry to
  maintain. The chokepoint convention favours consolidation of display
  formatters.
- Keeping it in `lib/format.ts` means the structure-doc "Last updated" note and
  the `formatVnd` chokepoint paragraph extend naturally rather than spawning a
  new module row.

**Build-time computation:** Yes. The helper is a pure function of `Post.content`,
called from Server Components (`app/bai-viet/page.tsx`,
`app/bai-viet/[slug]/page.tsx`) during SSG. No runtime/client computation, no
client fetch — consistent with *"Build-time data only"* (F0006 §6). The word
count is recomputed per build from the MDX body; no value is persisted into
frontmatter or `Post`.

### Helper contract

```ts
const WORDS_PER_MINUTE = 200; // reasonable adult VI/long-form reading rate

/**
 * Estimate read time from a post's raw MDX body and render it in Vietnamese,
 * e.g. "5 phút đọc". Always returns at least "1 phút đọc" for non-empty bodies.
 */
export function readingTimeVi(content: string): string {
  const words = countWords(content);
  const minutes = Math.max(1, Math.round(words / WORDS_PER_MINUTE));
  return `${minutes} phút đọc`;
}
```

Word counting (`countWords`, module-private):
- Strip MDX/markdown noise that would inflate the count: fenced code blocks
  (```` ``` ````), inline JSX/HTML tags (`<ProductCard slug="…" />`, `<img …>`),
  markdown image/link syntax, and heading/list/emphasis punctuation markers.
- Split the cleaned text on whitespace runs (`/\s+/`), filter empties, count.
- Deterministic and locale-independent (same rationale as `formatVnd`): no ICU,
  no `Intl`, no locale-dependent segmentation — byte-identical across build
  pools. (Vietnamese is space-segmented, so whitespace splitting is correct;
  CJK word-segmentation concerns do not apply.)
- Empty / whitespace-only body → `0` words → clamps to `"1 phút đọc"` (acceptable
  floor; verification fixtures always have substantial bodies).

Rate (200 WPM) is a single named constant so it can be tuned in one place.

## 4. Consumer wiring (additive, non-breaking)

### Listing item (US00064)
In the listing card markup, render the chip next to the formatted date:

```tsx
<span className={styles.meta}>
  {formatPostDate(post.publishedAt)} · {readingTimeVi(post.content)}
</span>
```

`post.content` is already on the `Post` objects returned by `getAllPosts()` — no
loader change. Use the same separator/typography the US00064 card already uses
for its date so the addition is visually consistent and does not reflow the card.

### Post header (US00066)
In the post header meta row (date + byline), append the read-time beside the
date:

```tsx
<p className={styles.postMeta}>
  {formatPostDate(post.publishedAt)} · {SITE_NAME} · {readingTimeVi(post.content)}
</p>
```

Both sites already have `post.content` in scope (the post page passes it to the
MDX renderer; the listing maps over `getAllPosts()`).

### Deferability guarantee (Scenario 3)
- The chip is rendered inside the existing meta line; if this story is omitted,
  the meta line still renders date (+ byline) with no empty container, no
  reserved-but-blank space, no broken grid.
- No P0 component gains a required prop from this story. The read-time call is
  added inline in the consumer's own JSX, so removing it is a one-line revert.
- If `readingTimeVi` were somehow absent (story not shipped), US00064/US00066
  compile and render unchanged — there is no import or interface coupling until
  the wiring lines are added.

## 6. Edge cases

- **Empty / whitespace-only body:** floors to `"1 phút đọc"` (never `"0 phút
  đọc"` or a crash). F0006 §5 edge cases otherwise unaffected.
- **Body heavy in code blocks / inline components:** `countWords` strips fences
  and JSX so a code-dump post is not over-counted into an inflated read time.
- **Very short post:** clamps to a 1-minute floor — reads naturally.
- **No frontmatter/type change:** consumes existing `Post.content`; cannot
  disturb the F0001 frontmatter validation in `lib/posts.ts`.
- **SSG determinism:** no `Intl`, no locale APIs, no runtime clock — output is
  byte-identical across Vercel build pools (same constraint as `formatVnd`).

## 8. Acceptance criteria (for `/implement-story`)

- **AC1 (Scenario 1):** `readingTimeVi(content)` returns a Vietnamese string of
  the form `"<n> phút đọc"`, where `n` is the body word count divided by a fixed
  ~200 WPM rate, rounded, floored at 1. Computed at build time from
  `Post.content`. Deterministic across environments (no `Intl`/locale APIs).
- **AC2 (Scenario 2):** The read-time estimate renders alongside the formatted
  date on (a) each `/bai-viet/` listing item and (b) the `/bai-viet/[slug]/`
  post header.
- **AC3 (Scenario 3):** With the read-time omitted, both pages render correctly
  — no layout breakage, no empty reserved slot, no required-prop coupling forced
  onto US00064/US00066.
- **AC4:** Word counting excludes code blocks and inline JSX/HTML so the estimate
  reflects prose, not markup.

## Resolved decisions

1. **WPM rate — 200** (C5). Conservative for long-form Vietnamese; better to slightly
   over-estimate than under-promise reading time.
2. **Meta separator — defer to US00064/US00066** (C6). This story owns the *value* (the
   "`N phút đọc`" string), not the visual treatment. The listing-item meta line (US00064) and
   post-header meta line (US00066) decide how to compose date · read-time (middot, chip, or
   whatever they settle on); US00069 plugs the string into the slot they expose.
