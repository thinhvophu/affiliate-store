# Plan: US00092 — Per-page metadata completeness audit

**Story:** US00092 (Feature F0009 — SEO & Discoverability)
**Status:** Done
**Spec:** `docs/specs/F0009.md` §US00092 (+ §5 Edge Cases, §6 Technical Considerations)
**Depends on:** US00091 (Global metadata foundation — `metadataBase`, title template, default OG, `lib/seo.ts` helpers)

---

## 1. Objective

Audit every route's `generateMetadata` / `metadata` export and bring it to a single, consistent shape so search results and shared-link previews always show: a unique `<title>` (via the US00091 template), a `<meta description>` ≤160 chars truncated at a word/sentence boundary, an absolute canonical URL resolved from `metadataBase` via a **relative** `alternates.canonical`, Open Graph + Twitter title/description, and a per-page OG image (product image / post cover, else the US00091 default).

This story is the "fill in the gaps" pass that consumes the foundation laid in US00091. It does **not** add JSON-LD (US00094/95/96), breadcrumbs (US00093), sitemap/robots (US00097/98), or pagination canonical logic (US00099).

## 4. Design

### 4.1 Shared SEO helper (extends US00091's `lib/seo.ts`)

Add these pure functions to `lib/seo.ts`. (US00091 created the file; this story appends.)

```ts
// lib/seo.ts (additions)

/** Maximum length for <meta description>. SERP snippets are typically ~150–160 chars. */
export const MAX_META_DESCRIPTION_LENGTH = 160;

/**
 * Truncate a description to ≤MAX_META_DESCRIPTION_LENGTH chars at a sentence/word boundary.
 * Order of preference: full text → sentence boundary (last ". ") ≥ 80 chars in → last
 * word boundary → hard slice + ellipsis. Never cuts a word in half; never leaves a
 * dangling HTML/Unicode entity (input is plain text, never markup).
 */
export function truncateMetaDescription(
  text: string,
  max: number = MAX_META_DESCRIPTION_LENGTH,
): string { /* … */ }

/** Build per-page metadata used by every route. */
export interface PageMetadataInput {
  /** Page title (raw — title template adds " | <site name>"). */
  title: string;
  /** Long-form description; truncated for <meta description>. */
  description: string;
  /** Absolute-from-root path WITH trailing slash, e.g. "/san-pham/". */
  path: string;
  /** Optional per-page OG image (absolute or root-relative). Falls back to default. */
  ogImage?: string;
  /** Optional OG image alt text. */
  ogImageAlt?: string;
  /** OG type. Defaults to "website"; product/post details pass "article". */
  ogType?: "website" | "article";
}

export function buildPageMetadata(input: PageMetadataInput): Metadata;
```

Rules baked into `buildPageMetadata`:
- `title` is returned as a plain string — the root layout's `title.template` (US00091) prepends ` | ${SITE_NAME}`. No page hand-rolls a `" | aff-store"` suffix.
- `description` is run through `truncateMetaDescription` once before being assigned to both `description` and `openGraph.description` / `twitter.description`.
- `alternates.canonical` = `path` (relative — Next.js resolves via `metadataBase` from US00091). All paths are normalized to trailing slash to match `app/` directory layout.
- `openGraph` = `{ title, description: <truncated>, url: path, type, images: [{ url: ogImage ?? DEFAULT_OG_IMAGE, width: 1200, height: 630, alt }] }`. (`DEFAULT_OG_IMAGE` constant lives in `lib/seo.ts` from US00091.)
- `twitter` = `{ card: "summary_large_image", title, description: <truncated>, images: [ogImage ?? DEFAULT_OG_IMAGE] }`. (Root layout already sets `twitter.card` default in US00091, but per-page builder restates it for clarity; Next.js merges per its rules.)

### 4.3 Canonical normalization

All canonical paths terminate with `/` to match Next.js's default `trailingSlash: false` URLs being normalized as Next sees fit (Next.js normalizes single/double slashes against `metadataBase`). The category page currently emits `/danh-muc/<slug>` (no trailing slash); we standardize to `/danh-muc/<slug>/` to match every other route's canonical and the sitemap URLs that US00097 will emit. This is internal-consistency only — Next.js redirects don't enter the picture because we render the page itself.

## 9. Risks & open questions for the reviewer

1. **Homepage title strategy.** US00091's spec says the root layout sets `title.default = SITE_NAME` (or similar) and `title.template = "%s | <site name>"`. The current homepage title is `"${SITE_NAME} — Đồ chơi công nghệ & gaming Việt Nam"`. Two options:
   - **(a) Remove the homepage `metadata.title`** and let US00091's `title.default` own it (cleanest, but loses the editorial tagline).
   - **(b) Pass `title: { absolute: "aff-store — Đồ chơi công nghệ & gaming Việt Nam" }`** so the template doesn't append ` | aff-store` (avoids `"aff-store — … | aff-store"`).
   - Recommend (b). Awaiting reviewer call.
2. **Trailing slash on canonical.** `next.config.ts` does not set `trailingSlash`, so Next defaults to `false`. We still emit canonicals with trailing slashes because the `app/<vietnamese-slug>/page.tsx` directory layout produces URLs that historically rendered with trailing slashes in this codebase (e.g., `/chinh-sach-bao-mat/`, `/cong-bo-tiep-thi-lien-ket/`). Confirm with reviewer that this matches the existing sitemap intent for US00097.
3. **OG image dimensions.** The `images[0]` of a product is the gallery hero (sized for the product card / detail page, not 1200×630). We still pass it because Open Graph allows non-1200×630 images; unfurlers will use it as-is. If reviewer wants a strict 1200×630 fallback per-page, that requires a generated-image pipeline (explicitly out of scope per F0009 §2 "Dynamic / per-page generated OG images" — single static default only in v1). Recommend: pass the product image as-is and accept whatever dimensions are on disk.
4. **Twitter restatement.** Root layout (US00091) already sets `twitter.card = "summary_large_image"`. Repeating it in `buildPageMetadata` is defensive but not strictly needed; Next.js merges per-page metadata over layout metadata. Tentatively keep it — easier mental model. Confirm.

### Implementation resolutions

1. **Homepage title:** went with **(b)**. `app/page.tsx` spreads `buildPageMetadata(...)` then overrides `title: { absolute: HOME_TITLE }`, where `HOME_TITLE` is the same string as the root layout's `title.default`. Verified via local Next.js docs (`generate-metadata.md`): `title.default` only applies when a child segment sets **no** `title` at all and is never passed through `title.template`; since the homepage *does* set OG/Twitter via `buildPageMetadata`, an `absolute` override was needed to avoid `"… | aff-store | aff-store"` doubling on the literal `<title>` tag. Confirmed in browser: `<title>aff-store — Đồ chơi công nghệ & gaming Việt Nam</title>` (no doubling), `og:title`/`twitter:title` match.
2. **Trailing slash:** confirmed — all canonicals end in `/` except the homepage, which Next.js itself resolves to the bare origin (`https://example.com`, no trailing slash) for root-path `alternates.canonical`/`openGraph.url` — this is deliberate Next.js framework behavior in `resolveAbsoluteUrlWithPathname` (`resolve-url.js`), not a bug in `lib/seo.ts`.
3. **OG image dimensions:** implemented as recommended — `product.images[0]` / `post.coverImage` passed through as-is, no resizing.
4. **Twitter restatement:** kept `twitter.card` in `buildPageMetadata`'s per-page output.
