# Implementation Plan — US00124: Scrape-sourced product ingestion for commodity categories

**Status:** Draft
**Feature:** F0012 — Automated Content Ingestion Pipeline
**Spec ref:** `docs/specs/F0012.md` § US00124
**Predecessors:** **US00121** (candidate model + validation engine + CLI skeleton). Best landed alongside **US00122** (slug/dedupe) and **US00123** (image staging) so a scraped candidate can go all the way to a written fixture — but the scrape *adapter* itself only depends on US00121's `Candidate` model.
**Successors:** none direct; US00125 is the sibling file adapter.

---

## 1. Scope

Strictly:

1. Add the **scrape source adapter** (`scripts/ingest/sources/scrape.ts`) that produces `Candidate[]` for `--source=scrape`.
2. **Bridge decision (approved):** the `shopee-affiliate` MCP tool is **not callable from a Node process** — it is an agent-side MCP tool that *saves results to `data/deals/YYYY-MM-DD.json`*. So the adapter **reads that JSON file**, filters by `--query`, and takes the top `--count`. The operator (or the agent, in-session) runs the scrape tool first; the script consumes its output. This is the honest, testable architecture.
3. New flags for the scrape source: `--query=<term>`, `--count=<n>`, `--date=<YYYY-MM-DD>` (defaults to today; selects which `data/deals/<date>.json` to read).
4. **Normalize** each raw deal record into the shared `Candidate` shape (name, brand, price, description, specs, `imageUrls`, `affiliateUrl`, `category` injected from `--category`, `sourceRef="scrape#<i>"`).
5. Route every scraped candidate through US00121's **identical** validation → US00123 staging → US00122 dedupe → writer. No scrape-specific validation.
6. **Shortfall is not an error** (Scenario 3): if fewer usable candidates than `--count` survive, ingest what passed and report `requested N, ingested M`.
7. **Invalid host rejected** (Scenario 2): delegated entirely to `assertAffiliateUrl` via US00121's engine — the summary names the offending host.

**Explicitly out of scope:**

- Invoking the MCP scrape tool from Node (impossible; see §3 D1). The pipeline consumes `data/deals/*.json`; producing it is an operator step.
- Image download → **US00123**. Slug + dedupe → **US00122**. (This story assumes those seams exist or are stubbed; if US00124 lands first, its written fixtures use a provisional slug and no staging until those stories complete — but the recommended order is US00121 → US00122/US00123 → US00124.)
- Curated file source → **US00125**.
- Re-scraping to refresh existing prices (future "catalog freshness").

## 2. Pre-conditions / current state

- US00121 merged: `Candidate` model, `validateCandidate`, arg parser, reporter, writer, and a `loadCandidates(args)` seam in `scripts/ingest-products.ts`.
- The MCP tool `mcp__shopee-affiliate__scrape_products` (agent-side) has signature `{ keywords: string[], top_n?: number }` and, per its description, **"saves to `data/deals/YYYY-MM-DD.json`"** and returns structured results. **The exact JSON schema of that file is not yet known** — it must be inspected at implementation time by running one scrape and reading the file (see §10 risk R1). This plan defines the adapter against an **assumed** shape and isolates the mapping in one function so a schema surprise is a one-function fix.
- `data/deals/` does **not** exist yet; it is created the first time the scrape tool runs. Per the approved decision, `data/` is **committed** (audit trail of what each run was fed).
- `next.config.ts` has **no** `images.remotePatterns` → scraped remote image URLs must be staged locally (US00123), never written into a fixture.

## 3. Approved Decisions

| # | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| D1 | Scrape bridge | Adapter **reads `data/deals/<date>.json`** produced by a prior MCP scrape run; the Node script never calls the MCP tool | (User-approved.) A `node`/`tsx` process cannot invoke an agent-side MCP tool. The tool's own output file is the contract. Keeps the CLI a pure, offline-testable Node process. |
| D2 | Two-step operator flow | (1) agent/operator runs `scrape_products` → writes `data/deals/<date>.json`; (2) `npm run ingest:products -- --source=scrape --date=<date> --query=<term> --count=<n>` | Documented in `scripts/ingest/README.md`. Matches spec §3 step 1 semantics without pretending Node can call MCP. |
| D3 | `--query` semantics | Filter the deal records to those whose keyword/name matches `--query` (the scrape file may hold multiple keywords' results) | The scrape tool accepts `keywords[]`; one `data/deals` file can mix categories. `--query` scopes to the intended term. |
| D4 | `--count` semantics | Cap the number of **accepted** candidates at `<n>`; take deals in the file's given order (assumed = deal rank) | Spec "top N". Shortfall (fewer than N survive) is reported, not fatal (Scenario 3). |
| D5 | `--date` default | Defaults to **today** (`new Date()` → `YYYY-MM-DD`); overridable to read an older dump | The tool names files by scrape date; the operator usually ingests same-day. |
| D6 | Raw→Candidate mapping isolated | One `mapDealToCandidate(deal, category, i)` function; all schema assumptions live there | R1: the deal JSON schema is unconfirmed. A single mapping site means a schema surprise is a localized fix, not a scatter. |
| D7 | Category is injected, never scraped | `category` comes from `--category`, not from the deal record | The operator declares intent; the registry (`assertCategoryRegistered`) validates it. Guessing category from scraped text would be unreliable. |
| D8 | Brand/specs extraction | Best-effort from the deal record; if brand or specs are absent → **reject** the candidate (do not guess) | Spec §5 "Price or spec fields missing → candidate rejected, not written with placeholder values." The `Product` interface is closed. |
| D9 | Missing scrape file | Fatal, non-zero, naming the expected path | If `data/deals/<date>.json` is absent the operator forgot step 1; fail loudly rather than silently ingest nothing. |

### 4.3 Adapter sketch

```ts
// scripts/ingest/sources/scrape.ts
export async function loadScrapeCandidates(args: IngestArgs): Promise<Candidate[]> {
  const date = args.date ?? isoToday();                       // D5
  const file = path.join(process.cwd(), "data", "deals", `${date}.json`);
  if (!fs.existsSync(file))                                    // D9 fatal
    throw new FatalIngestError(`scrape source: no deals file at ${file}. Run the shopee-affiliate scrape tool first.`);
  const deals = parseDeals(fs.readFileSync(file, "utf-8"));    // deal-schema guard
  const matched = deals.filter((d) => matchesQuery(d, args.query)); // D3
  return matched
    .slice(0, args.count ?? matched.length)                   // D4
    .map((d, i) => mapDealToCandidate(d, args.category, i));  // D6/D7/D8
}
```

Shortfall is computed downstream: `requested = args.count`, `ingested = accepted.length`.

## 5. Step-by-step execution order

1. Branch `feat/US00124` off `main` (after US00121/US00122/US00123 merged).
2. **Confirm the deal JSON schema (R1):** in-session, run `scrape_products` for one keyword, inspect the written `data/deals/<date>.json`, and lock `RawDeal` in `deal-schema.ts` to the real shape. Adjust `mapDealToCandidate` accordingly.
3. Create `deal-schema.ts` + `scrape.ts` + fixtures + tests.
4. Wire the `scrape` branch into `loadCandidates`.
5. Add `--query/--count/--date` to the args parser + tests.
6. `npm run typecheck && npm run lint && npm run test`.
7. **End-to-end dry-run** against a real `data/deals` file: `npm run ingest:products -- --category=chuot-gaming --source=scrape --query="chuột gaming" --count=8 --dry-run` → verify summary.
8. Real run (no `--dry-run`) → `next build` on the output → review JSON + staged images.
9. Update README/docs; commit `feat(US00124): scrape-sourced product ingestion via data/deals`; PR.

## 9. Risks / open items

- **R1 — Unknown deal JSON schema (highest).** The `data/deals/*.json` shape is unconfirmed. **Mitigation:** step 2 inspects one real file before locking `RawDeal`; all mapping lives in `mapDealToCandidate`. Do **not** guess the schema from this plan.
- **R2 — Scrape data may lack `brand`/`specs`/`price`.** Per D8/spec §5 those candidates are rejected, not defaulted. A category may therefore ingest fewer than requested — acceptable (Scenario 3), but the operator may need multiple queries to hit a target count.
- **R3 — `affiliateUrl` presence.** If the deal record's affiliate link field is empty, the candidate fails `assertAffiliateUrl` (unparseable URL) and is rejected — correct behavior.
- **R4 — Rate limiting / tool unreachable** (spec §5): this is an operator-side concern during step 1 (running the MCP tool), *outside* the Node script. The script only fails if the resulting file is missing (D9).
- **R5 — Stale `data/deals` file.** `--date` defaults to today; a leftover old file could be silently read if the operator forgets to re-scrape. README calls this out; consider warning if the file's mtime is older than N days (nice-to-have, not required).
