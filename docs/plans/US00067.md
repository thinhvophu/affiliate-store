# US00067 — Related posts ("Bài viết liên quan")

**Feature:** F0006 (Editorial Content)
**Status:** Approved
**Depends on:** US00066 (post page `/bai-viet/[slug]/`), US00061 (`formatPostDate`), US00064 (post-listing card — see component-reuse note), US00062 (MDX body it sits below)

---

## 1. Story summary

At the bottom of a blog post page (`/bai-viet/[slug]/`), render a **"Bài viết liên quan"** section
showing **up to 3 other posts in the same category** (post frontmatter `category`), **newest first**,
each a linked card (title, date, cover/summary), **excluding the current post**.

- If fewer than 3 other posts share the category, show however many exist (0–2) — **no padding from
  other categories** (this is the key divergence from `getRelatedProducts`, which DOES top-up cross-category).
- If **zero** other same-category posts exist, **omit the section entirely** — no empty heading.

This is a Server Component attached to the US00066 post page.

## 3. Component-reuse decision (PostCard)

US00064 (blog listing) is specced to render post cards (*"cover image, linked title, 2–3 sentence summary,
formatted date, category tag"*) but **no `PostCard` component exists in `components/` yet** (verified:
`ls components/ | grep -iE "post|blog"` → none) and `app/bai-viet/` does not exist yet.

**Decision:**
- **Preferred:** reuse the `PostCard` component that US00064 introduces, exactly as US00047's `RelatedProducts`
  reuses `<ProductCard>`. If US00064 ships a `components/PostCard.tsx` before this story is implemented,
  `RelatedPosts` consumes it (`<PostCard post={p} />`), and this story adds **no new card**.
- **Fallback (if US00064's card is not yet available at implementation time):** introduce a minimal
  `components/PostCard.tsx` + `.module.css` here (linked card: `next/image` cover, `<h3>` linked title to
  `/bai-viet/[slug]/`, `formatPostDate(date)`, summary clamp) and flag it for US00064 to adopt rather than
  duplicate. **Do not** hand-roll a second card inside `RelatedPosts`.

`RelatedPosts` itself stays card-agnostic: it owns the section chrome + grid only, mirroring `RelatedProducts`.

## 4. Selection logic — `getRelatedPosts()` in `lib/posts.ts` (build-time)

Add a sibling to `getRelatedProducts`, but with **same-category-only** semantics (no cross-category top-up):

```ts
/**
 * getRelatedPosts — US00067 / F0006
 * Up to 3 OTHER posts in the same category, newest first, excluding `current`.
 * No cross-category padding (unlike getRelatedProducts): if < 3 same-category
 * posts exist, return however many (0–2). Empty array => section omitted.
 */
export function getRelatedPosts(
  current: Post,
  all: Post[] = getAllPosts(),
): Post[] {
  return all
    .filter((p) => p.slug !== current.slug && p.category === current.category)
    .slice(0, 3); // getAllPosts() is already publishedAt-desc; ties keep file order
}
```

- Lives in `lib/` (no JSX), read at **build time** — consistent with SSG + the `getRelatedProducts` precedent.
- Relies on `getAllPosts()`'s existing newest-first sort, so no re-sort needed (matches US00061/US00064 ordering).
- Accepts an optional pre-loaded `all` array so the US00066 page can pass its already-loaded posts and avoid a
  second disk read (same ergonomics as `getRelatedProducts`).

## 5. `RelatedPosts` component

`components/RelatedPosts.tsx` (Server Component) + `components/RelatedPosts.module.css`, mirroring
`RelatedProducts.*`:

```tsx
import type { Post } from "@/types";
import { PostCard } from "@/components/PostCard"; // reuse (see §3)
import styles from "./RelatedPosts.module.css";

export interface RelatedPostsProps {
  posts: Post[];
}

export function RelatedPosts({ posts }: RelatedPostsProps) {
  if (posts.length === 0) return null; // omit entirely (Scenario 2)

  return (
    <section className={styles.section} aria-labelledby="related-posts-heading">
      <h2 id="related-posts-heading" className={styles.heading}>
        Bài viết liên quan
      </h2>
      <div className={styles.grid} style={{ "--related-count": posts.length }}>
        {posts.map((p) => (
          <PostCard key={p.slug} post={p} />
        ))}
      </div>
    </section>
  );
}
```

**Divergence from `RelatedProducts`:** the guard is `posts.length === 0` (omit only when empty), NOT
`< 3` — the story explicitly wants 1–2 related posts shown rather than padded/hidden.

**CSS** (`RelatedPosts.module.css`): copy the `RelatedProducts.module.css` grid (2 / 3 / responsive cols,
token-based spacing `--space-*`, no new transitions). Cap columns at `min(var(--related-count), 3)` since
the max is 3 here. `<h2>` because the page already has one `<h1>` (the post title) — preserves heading
hierarchy.

## 9. Acceptance criteria (implementer-facing)

- **AC1 (Scenario 1):** On a post whose category has ≥3 other posts, the bottom of the article shows a
  "Bài viết liên quan" `<section>` with exactly 3 linked cards — other posts, same category, newest first,
  current post excluded; each card shows title (linked to `/bai-viet/[slug]/`), date via `formatPostDate`,
  and cover/summary.
- **AC2 (Scenario 2a):** When 1–2 other same-category posts exist, exactly that many cards render — none
  borrowed from other categories.
- **AC3 (Scenario 2b):** When 0 other same-category posts exist, the section (and its heading) is **not
  rendered** — `RelatedPosts` returns `null`.
- **AC4:** Selection is computed at build time in `lib/posts.ts::getRelatedPosts`; dates render via
  `formatPostDate()` only; no client fetch; Server Component (no `"use client"`).
