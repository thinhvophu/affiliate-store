## 1. Scope

Strictly:

1. Make the Site Header pin to the top of the viewport after the visitor scrolls past a small threshold (~the header's own height) and stay pinned for the remainder of the scroll.
2. Render a subtle bottom-edge drop shadow on the Header only while it is in the "stuck" state; absent at the at-top resting state.
3. Provide a graceful fallback when JavaScript is disabled **or** the visitor has `prefers-reduced-motion: reduce`: the Header remains fully visible and usable at the top of the page, with no animated transition into the stuck state.
4. Preserve the skip-to-main-content behaviour added by US00023: when a keyboard user activates the skip link, `<main>` must scroll into view without being obscured by the sticky Header. Implemented via `scroll-margin-top` on `<main>` keyed off the header height.
5. Suppress sticky behaviour on pages shorter than the viewport (per spec §5) so the Header does not "jitter" between sticky and non-sticky states — using `position: sticky` makes this automatic.
6. Keep the client-JS footprint minimal: only the shadow-state toggle needs `"use client"`; the pin itself is CSS-only.

**Explicitly out of scope** (owned by sibling stories or downstream features):

- Building the Header structure / brand band / nav links — **US00021**.
- Building the Footer — **US00022**.
- Root layout wiring, surface tokens, skip-link element itself — **US00023**.
- Two-column shell layout component — **US00024**.
- Mobile drawer (open/close/overlay/focus-trap/scroll-lock) — **US00025**. This story does not touch drawer behaviour, but the drawer must continue to work when the header is in its sticky state (covered by smoke check, not by code in this story).
- Search bar in the Header — explicitly out of scope for v1 per F0002 §2.
- A "shrink on scroll" header height transition — out of scope; only the shadow changes between resting and stuck states.

## 3. Approved Decisions

| #   | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| D1  | Pinning mechanism | **CSS `position: sticky; top: 0;` on `<header>`** | Native, zero-JS, automatically respects shorter-than-viewport pages (no jitter — the header simply never leaves its slot if there's nothing to scroll), and works under JS-off / reduced-motion exactly as required by Scenario 3. The alternative (`position: fixed` toggled by a JS scroll listener) would require manual layout offsets, manual short-page detection, and a JS-failure fallback — strictly worse. |
| D2  | Shadow-state mechanism | **Tiny `"use client"` `<HeaderStickyShadow />` sibling that uses `IntersectionObserver` on a 1px sentinel rendered just above the header, toggling a `data-stuck="true"` attribute on the host `<header>`** | Pure CSS `:has()` would also work in modern browsers but has spotty support before Chrome/Safari 2023; the sentinel + IO pattern is the standard, well-supported way to detect "element has left the top of the viewport" without polling `scroll`. Total client JS is ~20 lines. Under JS-off the attribute stays absent and the header simply never gets the shadow — acceptable per Scenario 3. |
| D3  | Where the sentinel + observer live | **Inside `components/HeaderStickyShadow.tsx`** which the Server Component `<Header />` renders as its **first child** (before the brand band markup). The component renders both the 1px sentinel `<div aria-hidden>` and a small inline `useEffect` that observes it and toggles `document.querySelector("header[data-site-header]")` attribute. | Keeps the Server/Client boundary as small as possible per spec §6 and `CLAUDE.md` conventions ("`"use client"` only when needed"). The Header itself stays a Server Component. |
| D4  | How the Header is targeted by the toggle | **`<header data-site-header>` attribute set in `components/Header.tsx` (added by US00021).** The shadow component uses `document.querySelector('header[data-site-header]')` once on mount. | Avoids re-renders / context wiring / refs across the server-client boundary. The data attribute is a single contract the two components share. If US00021 has not added it, this story's implementation includes a one-line edit to `components/Header.tsx` to add the attribute. |
| D5  | At-top vs. stuck visual differentiation | **Drop shadow only.** No background colour change, no height change. | Spec Scenario 2 only requires "a subtle drop shadow / bottom-edge separator". Anything else (height shrink, colour shift) would be a separate design decision and is explicitly excluded by this plan to keep scope tight and CLS = 0. |
| D6  | Shadow token | **New token `--shadow-header: 0 2px 4px rgba(0, 0, 0, 0.08);`** declared on `:root` in `app/globals.css`. | Per spec §6 "additive tokens only" and project rule "no hardcoded hex outside the token definition". The shadow is a single declared value reused by the stuck-state rule. |
| D7  | Header-height token | **New token `--header-height: 64px;`** declared on `:root` in `app/globals.css`. Mobile and desktop share one value to avoid CLS on viewport resize and to keep `scroll-margin-top` simple. | The skip-link offset must be predictable; `scroll-margin-top: var(--header-height)` on `<main>` is the cleanest implementation. Using a single value (rather than responsive) is consistent with the dense / compact Shopee vibe described in F0002 §1 and avoids a media-query'd `scroll-margin-top`. The Header itself sets `height: var(--header-height)` (added by US00021 if not already; this story's implementation includes that one-line edit if missing). |
| D8  | Reduced-motion handling | **Wrap the shadow transition (`transition: box-shadow 150ms ease`) inside `@media (prefers-reduced-motion: no-preference)`.** The default rule (outside the media query) sets the shadow instantly, with no transition. | Matches the pattern that US00025 will use for the drawer slide animation. | D9  | Skip-link offset | **`main { scroll-margin-top: var(--header-height); }` in `app/globals.css`.** | One-line, framework-agnostic, correctly handles both the initial skip and any subsequent in-page anchor navigation that may land near the top. |
| D10 | Short-page guard | **Implicit via `position: sticky`** — there is no "engage" event; the header simply stays in flow when there's nothing to scroll. | Per spec §5 "very short pages: no jitter". `position: sticky` is jitter-free by construction. | D11 | Z-index | **`--z-header: 100;` token plus `header[data-site-header] { z-index: var(--z-header); }`.** | The sticky header must stack above page content (product cards, blog images) but below the mobile drawer overlay (US00025 will declare a higher `--z-drawer`). ### 5.1 `app/globals.css` — additive block

Insert after the F0002 surface tokens added by US00023 (do **not** redefine existing tokens):

```css
/* --------------------------------------------------------------------------
   F0002 / US00026 — Sticky Header tokens + rules
   -------------------------------------------------------------------------- */

:root {
  --header-height: 64px;
  --shadow-header: 0 2px 4px rgba(0, 0, 0, 0.08);
  --z-header: 100;
}

header[data-site-header] {
  position: sticky;
  top: 0;
  z-index: var(--z-header);
  height: var(--header-height);
  /* Default (at-top): no shadow. Stuck state adds it via data-stuck. */
  box-shadow: none;
}

header[data-site-header][data-stuck="true"] {
  box-shadow: var(--shadow-header);
}

/* Animate the shadow only for users who haven't requested reduced motion.
   When reduced-motion is preferred, the shadow toggles instantly. The
   sticky behaviour itself is unaffected (position: sticky stays). */
@media (prefers-reduced-motion: no-preference) {
  header[data-site-header] {
    transition: box-shadow 150ms ease;
  }
}

/* Skip-link / in-page anchor offset so the focused <main> isn't hidden
   under the sticky header. */
main {
  scroll-margin-top: var(--header-height);
}
```

### 5.2 `components/HeaderStickyShadow.tsx` — new file

```tsx
"use client";

import { useEffect } from "react";

/**
 * Sticky-shadow toggle for the site Header.
 *
 * Renders a 1px sentinel above the header and uses IntersectionObserver to
 * toggle `data-stuck="true"` on `<header data-site-header>` once the
 * sentinel scrolls out of view. Pure CSS handles the actual pinning
 * (`position: sticky`); this component only governs the shadow transition.
 *
 * Under JS-off, the sentinel is inert and the header simply never gets a
 * shadow — that's the documented graceful fallback (spec §US00026 Scenario 3).
 */
export function HeaderStickyShadow() {
  useEffect(() => {
    const header = document.querySelector<HTMLElement>("header[data-site-header]");
    const sentinel = document.getElementById("header-sticky-sentinel");
    if (!header || !sentinel) return;

    const observer = new IntersectionObserver(
      ([entry]) => {
        header.dataset.stuck = entry.isIntersecting ? "false" : "true";
      },
      { threshold: 0 },
    );
    observer.observe(sentinel);
    return () => observer.disconnect();
  }, []);

  return <div id="header-sticky-sentinel" aria-hidden="true" style={{ height: 1 }} />;
}
```

Notes:

- The component returns only the sentinel `<div>`. The observer wires up in `useEffect` on mount.
- `aria-hidden="true"` keeps assistive tech from announcing the empty element.
- `style={{ height: 1 }}` is intentional inline — the sentinel is purely structural and a one-property inline style is clearer than adding a global class.
- Cleanup disconnects the observer on unmount (relevant during HMR / fast refresh in dev).

### 5.3 `components/Header.tsx` — edit by US00021's owner OR by this story

If US00021 has not added `data-site-header` and the height/sentinel placement, this story includes the edit. Conceptually:

```tsx
import { HeaderStickyShadow } from "@/components/HeaderStickyShadow";

export function Header() {
  return (
    <>
      <HeaderStickyShadow />
      <header data-site-header>
        {/* …brand band + nav from US00021 stays exactly as-is… */}
      </header>
    </>

}
```

Important: `HeaderStickyShadow` renders the sentinel **outside** the `<header>` (as a previous sibling) so the IO can observe a node that is **above** the sticky element. If the sentinel were inside the sticky `<header>`, it would scroll with the header and never report `isIntersecting: false`.

## 10. Risks / open items

- **Sentinel placement.** If `Header.tsx` returns a `<header>` element directly (no wrapper), the `<HeaderStickyShadow />` sentinel **must** be rendered as a previous sibling (e.g. inside a React fragment with `<header>` second). If it's rendered as a child, the observer will never fire because the sentinel scrolls with the sticky element. The §5.3 sketch makes this explicit; reviewers should double-check at PR time.
- **Header height drift.** Hardcoding `--header-height: 64px` means any future visual redesign that changes the header's vertical metrics must update the token in one place. There is no runtime measurement. This is intentional (CLS = 0 and simpler `scroll-margin-top`); raise a follow-up if a responsive header height becomes necessary.
- **`position: sticky` ancestor pitfall.** `position: sticky` silently degrades to relative if any ancestor of `<header>` has `overflow: hidden | scroll | auto`. The plan does not change ancestors, but if US00024's shell layout (or any future global wrapper) adds such an overflow, sticky will break silently. Verify in the AC round and add a regression smoke check.
- **`prefers-reduced-motion` granularity.** The plan gates only the `transition` (shadow fade). If a future story adds further sticky-related animation (e.g., a slide-down on re-stick), that animation must also be gated by the same media query. Document this expectation in the F0002 close-out notes.
- **No additional Web-API guards.** `IntersectionObserver` is universally supported in our target browsers (Vercel Speed Insights audience). No polyfill needed. Confirm at implementation time against current `browserslist` defaults.
- **Coupling to US00021's markup.** This story assumes `Header.tsx` renders a single root `<header>` element to which `data-site-header` can be added cleanly. If US00021 chose a different structure (e.g., a fragment with multiple root elements), the integration must be revisited. Surface in PR if so.
- **Future search bar / category band.** F0002 §2 excludes those for v1. If/when they ship, they will likely be rendered inside the sticky `<header>` and inherit the shadow toggle for free — no rework needed here, but the contract (`data-site-header`, `--header-height`) is the seam to honour.
