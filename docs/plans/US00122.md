# Implementation Plan — US00122: Slug generation, dedupe & idempotent re-runs

**Status:** Draft
**Feature:** F0012 — Automated Content Ingestion Pipeline
**Spec ref:** `docs/specs/F0012.md` § US00122
**Predecessors:** **US00121** (candidate model + accept flow). Provides the `slug` that **US00123** needs for image filenames, so land these together (US00122 before or with US00123).
**Successors:** US00124 / US00125 fixtures rely on this for slug + dedupe.

---

## 1. Scope

Strictly:

1. Add a **Vietnamese-aware slug generator** (`scripts/ingest/slug.ts`) that produces kebab-case ASCII slugs consistent with existing fixtures (`chuot-gaming`, not `chuột-gaming`), i.e. diacritics **transliterated/stripped**, `đ→d`, spaces→`-`, lowercase.
2. Add **dedupe / idempotency** (`scripts/ingest/dedupe.ts`) keyed **`affiliateUrl` first, slug second** against existing `content/products/*.json`:
   - Re-running an identical command is a **no-op** for already-ingested products (Scenario 1) — counted "duplicate," no file overwritten.
   - **Partial-run recovery** (Scenario 2): already-ingested products skip as duplicates; only the rest process.
3. **Slug-collision safety** (Scenario 3): two *distinct* candidates (different `affiliateUrl`) that derive the same slug **never silently overwrite** — the second is either skipped (same `affiliateUrl`) or **disambiguated** (`-2`, `-3`, …) and **flagged for human review** in the summary.

**Explicitly out of scope:**

- The image filename `<slug>-<n>` scheme — owned by US00123 (it consumes the slug from here).
- Editing/refreshing existing products — dedupe only *skips*; it never mutates existing fixtures.

---

## 2. Pre-conditions / current state

- No product slug generator exists today — slugs are hand-baked into JSON. `github-slugger` (used for MDX headings in `lib/mdx-slug.ts`) is **not suitable**: it strips combining diacritics rather than transliterating (`chuột`→`chut`), so it would diverge from the existing `chuot-…` fixtures. A dedicated transliterating slugifier is required.
- Existing fixtures confirm the target style: `aoc-c27g2`, `logitech-g102-lightsync`, `keychron-k2-v2` — lowercase, kebab, ASCII, diacritics folded.
- `lib/products.ts::getAllProducts()` already enforces **build-time slug uniqueness** (throws on duplicate). The pipeline must never produce a colliding file, or the whole build breaks — hence Scenario 3's hard guarantee.
- `getAllProducts()` is importable (via `@/lib/products`) to read the existing catalog for dedupe — reuse it rather than re-reading the dir, so dedupe sees exactly what the build sees.

## 3. Approved Decisions

| # | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| D1 | Slugify algorithm | `name` → NFD normalize → strip combining marks (`̀-ͯ`) → `đ/Đ→d` → lowercase → non-`[a-z0-9]`→`-` → collapse/trim `-` | Produces `chuot-gaming` from `Chuột Gaming`, matching existing fixtures. Deterministic, no locale API (SSG-safe mindset). |
| D2 | Slug source | Derived from `candidate.name` (brand already usually in the name); if the result is empty (e.g. all-symbol name) → **reject** the candidate | A product with no sluggable name can't get a stable URL key. |
| D3 | Dedupe key priority | **`affiliateUrl` first, then `slug`** | Spec §6 idempotency key: `affiliateUrl` is the true product identity; two scrape queries may surface the same product under different names (→ same URL, different slug). |
| D4 | `affiliateUrl` match | Exact string match against existing fixtures' `affiliateUrl` (no normalization — that's `lib/affiliate.ts`'s domain, and it currently doesn't normalize) | If the same product is sourced again, its URL matches → skipped as duplicate (Scenario 1/2). Conservative: differing tracking params count as distinct until `lib/affiliate.ts` normalizes them (future). |
| D5 | Existing catalog read | Use `getAllProducts()` from `@/lib/products` once at startup; build two indexes: `Map<affiliateUrl, slug>` and `Set<slug>` | Single source of truth = what `next build` sees; avoids a second, divergent dir walk. |
| D6 | Within-run dedupe | Also track affiliateUrls/slugs **accepted earlier in the same run**, so two identical candidates in one source don't both write | A curated file or scrape result may list the same product twice. |
| D7 | Collision resolution | If derived slug already taken by a **different** `affiliateUrl`: append `-2` (then `-3`…) until free, mark the candidate `needsReview: true`, and list it in a dedicated **"disambiguated — review"** summary group | Scenario 3: never silently overwrite; surface for human judgment (maybe it's a genuine variant, maybe a mis-scrape). |
| D8 | Never overwrite | `writeFixture` refuses to write if the target `content/products/<slug>.json` already exists (throws) | Defense-in-depth: even if dedupe logic had a bug, an existing file is never clobbered. Belt-and-suspenders for Scenario 1/3. |
| D9 | Duplicate ≠ error | A skipped duplicate is exit-0, counted in `skippedDuplicate` | Idempotent re-runs are the *point*; a re-run that finds everything already there is a success. |

### 4.3 Slugify sketch

```ts
// scripts/ingest/slug.ts
export function slugifyProductName(name: string): string {
  return name
    .normalize("NFD").replace(/[̀-ͯ]/g, "")   // strip diacritics
    .replace(/đ/g, "d").replace(/Đ/g, "D")              // Vietnamese đ
    .toLowerCase()
    .replace(/[^a-z0-9]+/g, "-")
    .replace(/^-+|-+$/g, "");
}
export function disambiguate(base: string, taken: (s: string) => boolean): string {
  if (!taken(base)) return base;
  for (let i = 2; ; i++) { const c = `${base}-${i}`; if (!taken(c)) return c; }
}
```

### 4.4 Dedupe/classify sketch

```ts
// per accepted candidate, in ingest-products.ts
const base = slugifyProductName(c.name);
if (index.byUrl.has(c.affiliateUrl)) { skippedDuplicate.push(c); continue; }   // D3/D4 Scenario 1/2
let slug = base, needsReview = false;
if (index.slugs.has(base)) { slug = disambiguate(base, s => index.slugs.has(s)); needsReview = true; } // D7 Scenario 3
// register in-run so later candidates see it (D6)
index.byUrl.set(c.affiliateUrl, slug); index.slugs.add(slug);
// → stage images (US00123) with slug → writeFixture (refuses if file exists, D8)
```

## 5. Step-by-step execution order

1. Branch `feat/US00122` off `main` (with/after US00121).
2. Create `slug.ts` + tests; verify against a handful of existing fixture names to match their slugs.
3. Create `dedupe.ts` + tests; build indexes from `getAllProducts()`.
4. Wire slug+classify into the accept path; add the writer refuse-if-exists guard; add the report group.
5. `npm run typecheck && npm run lint && npm run test`.
6. **Idempotency check:** run an ingest twice with the same input → second run reports all as duplicates, `git status` shows no changes on the second run.
7. **Collision check:** craft two file entries with different URLs but names that slugify identically → second is disambiguated + flagged.
8. Docs; commit `feat(US00122): slug generation, dedupe & idempotent re-runs`; PR.

## 9. Risks / open items

- **R1 — Slug parity with existing fixtures.** Verify the slugifier reproduces several existing fixture slugs exactly (step 2). If a legacy fixture used a non-derivable slug, that's fine (dedupe keys on URL too), but the *derivation* should match the common case.
- **R2 — Tracking-param drift in `affiliateUrl`.** Exact-match dedupe (D4) treats `shope.ee/X?utm=a` and `shope.ee/X?utm=b` as distinct. Acceptable now (conservative); if it causes false "new" products, the fix is URL normalization **in `lib/affiliate.ts`** (its domain), then dedupe benefits automatically. Do not normalize in `scripts/`.
- **R3 — Disambiguated slug is usually a smell.** `needsReview` items often mean a mis-scrape or a genuine variant. Summary must make them visible so the operator decides before commit; they are not auto-trusted.
- **R4 — In-run vs on-disk index sync.** The in-run index (D6) must be seeded from `getAllProducts()` and updated as candidates are accepted; a bug here could double-write. D8's refuse-if-exists is the backstop.
