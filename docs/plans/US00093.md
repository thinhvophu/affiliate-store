## 1. Story summary

Add a visible breadcrumb trail above the `<h1>` on the three deep pages:

- `/san-pham/[slug]/` — `Trang chủ › Sản phẩm › <Vietnamese category name> › <Product name>`
- `/danh-muc/[category]/` — `Trang chủ › Sản phẩm › <Vietnamese category name>`
- `/bai-viet/[slug]/` — `Trang chủ › Bài viết › <Vietnamese category name> › <Post title>`

This story **owns two shared seams** other F0009 stories depend on:

1. **`<Breadcrumb items={...} />`** Server Component (`components/Breadcrumb.tsx` + co-located CSS Module): semantic `<nav aria-label="Breadcrumb">` wrapping an ordered `<ol>`. Ancestor crumbs are crawlable `<a>` links (Next `<Link>`); the final crumb is plain text marked `aria-current="page"`.
2. **A single `BreadcrumbItem` type + builder helper** in `lib/breadcrumbs.ts`. The same `items` array feeds both the visible component (US00093) and the `BreadcrumbList` JSON-LD (US00096). Google penalizes mismatch between visible and structured breadcrumbs — sharing one builder makes divergence impossible by construction.

The visible separator character is `›` (U+203A) rendered as a CSS `::after` decoration on `<li>` (so it's invisible to assistive tech and to crawlers that read text content).

## 3. Data model — `BreadcrumbItem` and the builder helper

### 3.1 The shared type

```ts
// lib/breadcrumbs.ts

/**
 * One crumb in a breadcrumb trail. Shared seam between:
 *   - components/Breadcrumb.tsx (US00093, visible markup)
 *   - the BreadcrumbList JSON-LD emitter in F0009 / US00096
 *
 * Both consumers MUST receive the same items array so the visible trail
 * and the structured data can never diverge. Adding fields here is fine
 * (e.g., future "current: boolean"); renaming/removing is a breaking
 * change to US00096.
 */
export interface BreadcrumbItem {
  /** Vietnamese display label rendered as link text or current-page text. */
  label: string;
  /**
   * Site-relative path with leading slash and trailing slash to match the
   * existing route convention (e.g. "/san-pham/chuot-gaming/").
   * The final crumb still carries its own href so US00096 can use it as
   * the JSON-LD `item` value for the current page. The visible component
   * does NOT render the last crumb as a link.
   */
  href: string;
}
```

### 3.2 Builder helpers

All three builders return `BreadcrumbItem[]` and always include `Trang chủ` as the first item. They route category-slug → Vietnamese label through `getCategoryMeta()` only.

```ts
// lib/breadcrumbs.ts (continued)
import { getCategoryMeta } from "@/lib/categories";
import type { Product } from "@/types";
import type { Post } from "@/types";

const ROOT_CRUMB: BreadcrumbItem = { label: "Trang chủ", href: "/" };
const PRODUCTS_CRUMB: BreadcrumbItem = { label: "Sản phẩm", href: "/san-pham/" };
const POSTS_CRUMB: BreadcrumbItem = { label: "Bài viết", href: "/bai-viet/" };

/** /san-pham/[slug]/ */
export function buildProductBreadcrumbs(product: Product): BreadcrumbItem[] {
  const cat = getCategoryMeta(product.category);
  // cat is guaranteed to resolve at build time because lib/products.ts already
  // calls assertCategoryRegistered() during getAllProducts(). Defensive fallback
  // protects local edits: if it ever misses, omit the category crumb rather
  // than crash the render.
  const items: BreadcrumbItem[] = [ROOT_CRUMB, PRODUCTS_CRUMB];
  if (cat) items.push({ label: cat.name, href: `/danh-muc/${cat.slug}/` });
  items.push({ label: product.name, href: `/san-pham/${product.slug}/` });
  return items;
}

/** /danh-muc/[category]/ */
export function buildCategoryBreadcrumbs(categorySlug: string): BreadcrumbItem[] {
  const cat = getCategoryMeta(categorySlug);
  if (!cat) return [ROOT_CRUMB, PRODUCTS_CRUMB];
  return [
    ROOT_CRUMB,
    PRODUCTS_CRUMB,
    { label: cat.name, href: `/danh-muc/${cat.slug}/` },
  ];
}

/** /bai-viet/[slug]/ */
export function buildPostBreadcrumbs(post: Post): BreadcrumbItem[] {
  const cat = getCategoryMeta(post.category);
  // lib/posts.ts already asserts the post category is registered at build time.
  const items: BreadcrumbItem[] = [ROOT_CRUMB, POSTS_CRUMB];
  if (cat) items.push({ label: cat.name, href: `/danh-muc/${cat.slug}/` });
  items.push({ label: post.title, href: `/bai-viet/${post.slug}/` });
  return items;
}
```

**Why a builder helper, not inline construction in each page?**

- US00096 will import these exact builders (or the resulting arrays). Centralizing them makes the "single source of truth for trail data" guarantee enforceable in code, not just by convention.
- Adding a new deep route (e.g., a future tag page) becomes a one-file change.
- Builders are pure → unit-testable without spinning up React (see §6 Step 2).

### 3.3 Constants — keep DRY with `lib/nav-items.ts`

`NAV_ITEMS` already typifies the four primary nav labels. To avoid duplicating the strings `"Sản phẩm"` and `"Bài viết"`, the builder file imports them once:

```ts
const POSTS_LABEL = NAV_ITEMS.find((n) => n.href === "/bai-viet/")!.label;
```

## 4. UI component structure

### 4.1 `components/Breadcrumb.tsx`

```tsx
import Link from "next/link";
import type { BreadcrumbItem } from "@/lib/breadcrumbs";
import styles from "./Breadcrumb.module.css";

export interface BreadcrumbProps {
  /** Ordered crumbs from root → current page. Min length = 2. */
  items: BreadcrumbItem[];
}

export function Breadcrumb({ items }: BreadcrumbProps) {
  if (items.length < 2) return null; // defensive — never render a 1-item trail

  return (
    <nav aria-label="Breadcrumb" className={styles.nav}>
      <ol className={styles.list}>
        {items.map((item, idx) => {
          const isLast = idx === items.length - 1;
          return (
            <li key={item.href} className={styles.item}>
              {isLast ? (
                <span aria-current="page" className={styles.current}>
                  {item.label}
                </span>
              ) : (
                <Link href={item.href} className={styles.link}>
                  {item.label}
                </Link>
              )}
            </li>
          );
        })}
      </ol>
    </nav>
  );
}
```

Notes:

- **Server Component** — no `"use client"`. The breadcrumb has no state, no events, no client APIs.
- **`<Link>`** vs `<a>` — use Next's `<Link>` so internal navigations stay client-side; the rendered DOM is still a crawlable `<a>`. Spec says "crawlable `<a>` links" — `<Link>` satisfies this (it emits `<a href>`).
- **Separator** is rendered by CSS (`::after`) — not by an emitted text node — so it isn't read by screen readers and isn't parsed by crawlers as part of the link text. This is the WAI-ARIA recommended pattern for breadcrumbs.
- **`aria-current="page"`** is on the final `<span>`, not the parent `<li>` — per ARIA APG.
- **Defensive empty render** — if `items.length < 2`, render nothing. This is a "should never happen at build time" guard for hand-edited routes.

### 4.3 Wire-up on each page

#### `app/san-pham/[slug]/page.tsx`


Because the existing `.detail` is a CSS `grid` (2-col on desktop), inserting `<Breadcrumb>` as the first grid child would put it in column 1. Two options:

- **Option A (recommended):** Move `<Breadcrumb>` **outside** `<article>` into the page-level `<div className={styles.container}>`, before `<article>`. Add a small `margin-bottom` to `.nav` (already there) and the breadcrumb spans the full container width above the 2-col detail layout. Simpler, no grid override.
- Option B: Keep `<Breadcrumb>` inside `<article>` and set `grid-column: 1 / -1` on it so it spans both columns.

Plan adopts **Option A**.

## 7. Acceptance criteria (verbatim mapping from F0009 §US00093)

- **Scenario 1 — Breadcrumb renders on deep pages:** Navigating to `/san-pham/[slug]/`, `/danh-muc/[category]/`, or `/bai-viet/[slug]/` shows a visible breadcrumb trail `Trang chủ › <Danh mục> › <Trang hiện tại>` rendered above the `<h1>` (Option A places it above `<article>` for product/category, inside `<article>` above `<header>` for post — both yield "above the main heading").
- **Scenario 2 — Semantic, accessible markup:** Rendered DOM is exactly `<nav aria-label="Breadcrumb"><ol>…</ol></nav>`; ancestor crumbs are `<a href="…">` (via Next `<Link>`), the final crumb is `<span aria-current="page">label</span>` — not a link. Verified in DevTools.
- **Scenario 3 — Single source of truth for trail data:** All three pages call a `build*Breadcrumbs` function from `lib/breadcrumbs.ts` that returns `BreadcrumbItem[]`. The exported `BreadcrumbItem` type and the three builder functions are public seam APIs that US00096 will import to construct `BreadcrumbList` JSON-LD from the same array. Demonstrated by: in the US00096 PR, the JSON-LD emitter calls `buildProductBreadcrumbs(product)` (etc.) and maps the result.
- **Scenario 4 — Vietnamese labels:** Root crumb label is the literal `"Trang chủ"` (not `"Home"`, not `SITE_NAME`). Category crumb labels come from `getCategoryMeta(slug).name` (verified by inspecting `lib/categories.ts` — e.g., `"Chuột gaming"`, `"Bàn phím gaming"`, `"Tai nghe gaming"`, `"Màn hình gaming"`, `"Ghế gaming"`).

## 9. Edge cases & guards

- **Unknown category slug in builder input.** The product/post loaders already call `assertCategoryRegistered()` at build time, so this should never reach the builder at runtime. The builder still falls back to omitting the category crumb (rather than crashing) — this matches `notFound()` semantics for the page itself.
- **Very long product names / post titles.** CSS truncates the current-page label with ellipsis on mobile, wraps on tablet+. Crawlers and screen readers still get the full text.
- **Special characters / diacritics in labels.** All labels are plain React children — no JSX-string interpolation into `data-*` attributes, no `dangerouslySetInnerHTML`. Vietnamese diacritics render natively.
- **Trailing slash discipline.** All builder hrefs use trailing slashes (`/san-pham/`, `/danh-muc/.../`, `/bai-viet/.../`) to match the existing route convention. US00096 will reuse these hrefs as-is for the JSON-LD `item` URLs, but **must** also concatenate `metadataBase` for the absolute form — that conversion lives in US00096, not here.
- **Cohesion with `aria-current="page"` placement.** ARIA APG recommends `aria-current` on the link **or** on the text node that is "the current location". Because the last item is intentionally **not** a link, `aria-current="page"` lives on the `<span>`. This matches Google's breadcrumb guidance too.
- **Browser back-button behavior.** Each ancestor crumb is a real navigation, not `history.back()`. This is intentional — breadcrumbs are positional, not historical.
- **Separator and crawlers.** Using `::after { content: "›" }` keeps the separator out of the link text (search-engine breadcrumb anchor text stays clean) and out of the accessibility tree.
- **No client-side JS.** Component is server-rendered; no `"use client"` hooks. This means no theme-sensing or interactive truncation — pure CSS truncation only.
