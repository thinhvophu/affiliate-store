## 1. Scope

Strictly:

1. Define the authoritative `Product` type matching the canonical JSON shape in `CLAUDE.md`.
2. Define the authoritative `PostFrontmatter` type matching the MDX frontmatter shape in `CLAUDE.md`, plus a `Post` type that wraps frontmatter with slug and compiled content metadata.
3. Export all shared types from a single barrel (`types/index.ts`) so downstream code imports from `@/types`.
4. Ensure no field uses `any` — use `Record<string, string>` for open-ended maps, `string[]` for arrays, etc.

**Explicitly out of scope** (owned by sibling stories in F0001):

- MDX pipeline / `mdx-components.tsx` — **US00013**
- Content loader functions (`getAllProducts`, `getPostBySlug`, etc.) — **US00015**
- Runtime validation / Zod schemas for content — **US00015**
- Global styles & design tokens — **US00016**
- ESLint / Prettier — **US00017**
- Typed env var module (`lib/env.ts`) — **US00018**

No loader code, no validation logic, no React components land here. We are defining the type contracts only.

## 3. Approved Decisions

| #   | Decision                               | Choice                                                                                             | Rationale                                                                                                                                                                                                                                                                                          |
| --- | -------------------------------------- | -------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| D1  | File organization                      | **One file per domain type** (`types/product.ts`, `types/post.ts`) + **barrel** (`types/index.ts`) | Keeps each type focused and independently navigable. Barrel satisfies AC Scenario 3 ("single canonical entry point"). Matches the convention documented in CLAUDE.md: "Types in `types/<domain>.ts`".                                                                                              |
| D2  | `Product.specs` field type             | **`Record<string, string>`**                                                                       | The canonical JSON shape shows `specs` as a flat key-value map (`"DPI": "200-8000"`, etc.). Spec names and values are always strings. `Record<string, string>` is precise, avoids `any`, and is extensible.                                                                                        |
| D3  | `Product.images` field type            | **`string[]`** (non-empty at the type level; runtime validation of min-length is US00015's job)    | Array of image paths. Could use `[string, ...string[]]` for compile-time non-empty enforcement, but that creates friction with JSON parsing where arrays always come in as `string[]`. Runtime validation in US00015 is the right place to enforce min-length.                                     |
| D4  | `Product.price` type                   | **`number`**                                                                                       | Spec shows `390000` (integer VND). No currency object needed; formatting (`₫1.200.000`) is a display concern handled by utility functions in `lib/`.                                                                                                                                               |
| D5  | `Post` vs `PostFrontmatter` separation | **Two types: `PostFrontmatter` (pure YAML fields) + `Post` (slug + frontmatter + content)**        | `PostFrontmatter` is what `gray-matter` extracts and what US00015 validates. `Post` adds the derived `slug` (from filename) and `content` (raw MDX string body for rendering). This separation keeps the frontmatter shape clean for validation while giving page components everything they need. |
| D6  | `Post.content` type                    | **`string`**                                                                                       | The raw MDX body after gray-matter strips frontmatter. Downstream rendering (compiling MDX to React) is handled at the page level, not in the type.                                                                                                                                                |
| D7  | Date fields (`publishedAt`)            | **`string`** (ISO date format, e.g. `"2026-05-01"`)                                                | Stored as strings in both JSON and YAML. Parsing to `Date` objects is a runtime concern for display utilities. The type contract says "it's a date string".                                                                                                                                        |
| D8  | `Product.featured` optionality         | **Required `boolean`**                                                                             | The canonical JSON shape shows it as a top-level field. Products must declare whether they are featured or not. Defaulting to `false` if missing is a validation/loader concern (US00015).                                                                                                         |
| D9  | `.gitkeep` cleanup                     | **Remove `types/.gitkeep`** when adding real files                                                 | The folder is no longer empty. Matches the convention noted in US00012 Section 8.                                                                                                                                                                                                                  |
| D10 | Branch / commit target                 | **Commit directly to `main`**                                                                      | Matches US00011/US00012/US00013 convention.                                                                                                                                                                                                                                                        |

## 5. Concrete diff sketch

### 5.1 `types/product.ts`

```ts
/**
 * Canonical product shape — mirrors content/products/*.json.
 *
 * Source of truth: CLAUDE.md "Product JSON shape".
 * Runtime validation is handled by the content loader (US00015).
 */
export interface Product {
  /** URL-safe identifier, used as the route key: /san-pham/[slug] */
  slug: string;

  /** Vietnamese product name (e.g. "Chuot Gaming Logitech G102 Lightsync") */
  name: string;

  /** Category slug for filtering and routing (e.g. "chuot-gaming") */
  category: string;

  /** Brand name (e.g. "Logitech") */
  brand: string;

  /** Price in VND (integer, e.g. 390000). Display formatting handled by lib/. */
  price: number;

  /** Shopee affiliate URL — always opened with target="_blank" rel="noopener noreferrer sponsored" */
  affiliateUrl: string;

  /** Product image paths (relative to static/, e.g. ["/static/images/products/logitech-g102.jpg"]) */
  images: string[];

  /** Vietnamese product description */
  description: string;

  /** Technical specs — flat key-value pairs (e.g. { "DPI": "200-8000", "Sensor": "Optical" }) */
  specs: Record<string, string>;

  /** ISO date string when the product was published (e.g. "2026-05-01") */
  publishedAt: string;

  /** Whether this product should appear in featured/promoted sections */
  featured: boolean;
}
```

### 5.2 `types/post.ts`

```ts
/**
 * MDX post frontmatter — mirrors the YAML block in content/posts/*.mdx.
 *
 * Source of truth: CLAUDE.md "MDX post frontmatter".
 * Runtime validation is handled by the content loader (US00015).
 */
export interface PostFrontmatter {
  /** Post title in Vietnamese */
  title: string;

  /** Short summary for meta description and listing cards */
  summary: string;

  /** ISO date string when the post was published (e.g. "2026-05-02") */
  publishedAt: string;

  /** Category slug matching product categories (e.g. "tai-nghe") */
  category: string;

  /** Tags for filtering and SEO (e.g. ["tai nghe gaming", "budget", "review"]) */
  tags: string[];

  /** Cover image path (e.g. "/static/images/blog/tai-nghe-gaming-500k.png") */
  coverImage: string;
}

/**
 * Full post object returned by content loaders.
 *
 * Combines the frontmatter with derived fields (slug from filename)
 * and the raw MDX body content.
 */
export interface Post extends PostFrontmatter {
  /** URL-safe identifier derived from the MDX filename: /bai-viet/[slug] */
  slug: string;

  /** Raw MDX body content (frontmatter stripped). Compiled to React at render time. */
  content: string;
}
```

### 5.3 `types/index.ts`

```ts
export type { Product } from "./product";
export type { Post, PostFrontmatter } from "./post";
```

## 10. Risks / open items

- **`Post` extends `PostFrontmatter`.** This means `slug` and `content` are top-level fields alongside frontmatter fields. If a future frontmatter field collides with `slug` or `content`, the `extends` relationship would need restructuring. Low risk — both names are deliberate and unlikely to appear as frontmatter keys.
- **`specs` as `Record<string, string>`.** This assumes all spec values are strings. If a future product needs numeric or boolean spec values (e.g., `"weight_g": 85`), the type would need to become `Record<string, string | number>`. For now, the canonical shape shows all string values, so `Record<string, string>` is correct.
- **No runtime validation here.** These are compile-time contracts only. US00015 owns the runtime validation (likely with Zod schemas that mirror these interfaces). If the Zod schemas drift from these types, bugs will be silent. US00015 should use `z.infer<>` or `satisfies` to keep them locked.
- **`Post.content` is raw MDX string.** Downstream pages will need to compile this to React. The type does not encode what "compiled" looks like — that is intentionally left to the rendering layer.
- **No `id` field on `Product`.** The canonical shape uses `slug` as the unique identifier. If a future data source adds numeric IDs, a new field would need to be added. Not a concern for filesystem-based content.
