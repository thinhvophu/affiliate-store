# US00081 — Homepage shell, hero/intro section & baseline metadata

**Status:** Approved
**Feature:** F0008 — Homepage
**Spec source:** `docs/specs/F0008.md` → `### US00081: Homepage shell, hero/intro section & baseline metadata`

## 1. Summary

Replace the placeholder `app/page.tsx` (`<h1>aff-store</h1>`) with a real, statically pre-rendered Vietnamese homepage Server Component at `/`. This story delivers **only the shell + hero/intro section + baseline SEO metadata**. The page is full-width (no left panel, **does not** use `<ShellLayout>`), styled exclusively with F0002 theme tokens and the system font stack already in `app/globals.css`. The hero communicates what the site is and offers two CTAs to `/san-pham/` and `/bai-viet/`. Baseline `metadata` exports a unique Vietnamese title, meta description, and a canonical URL resolving to `/` that respects `NEXT_PUBLIC_SITE_URL` with a graceful fallback (no build-time throw if it is unset in dev/preview).

The other F0008 sections — featured products (US00082), category highlights (US00083), latest posts (US00084) — are **out of scope** here, but `app/page.tsx` is laid out so each section can later be inserted as an independent Server Component child of the page's `<article>`/`<div>` shell without touching the hero.

## 3. Key decisions

### 3.1 Page shape — single `app/page.tsx` Server Component + optional `<HomeHero />` child

The story scope is small (hero + two CTAs + metadata), but F0008 will compose four sections on the same page. **Decision:** keep `app/page.tsx` as the **page-level composition shell** (the file that owns `metadata`, the outer `<div className={styles.container}>`, and the section ordering) and extract the hero into its own Server Component **`components/HomeHero.tsx` + `components/HomeHero.module.css`**. This is consistent with the project convention ("Each homepage section may be its own Server Component under `components/` with a co-located `<Name>.module.css`" — F0008 §6 Technical Considerations) and keeps `page.tsx` purely declarative when US00082–84 land:

```tsx
// app/page.tsx (target end state for US00081)
return (
  <div className={styles.container}>
    <HomeHero />
    {/* US00082 placeholder: <FeaturedProducts /> */}
    {/* US00083 placeholder: <CategoryHighlights /> */}
    {/* US00084 placeholder: <LatestPosts /> */}
  </div>
);
```

The `{/* US00082 placeholder: … */}` comments are deliberate: they encode the planned section order so the next implementer drops a component reference into the same spot. They are **not** TODOs that block this story; the page renders correctly with only `<HomeHero />`.

### 3.2 Canonical URL — `process.env.NEXT_PUBLIC_SITE_URL ?? ""` fallback, root path `/`

Two existing canonical patterns in the repo (already noted in `docs/plans/US00052.md` §3.1):

| Page                                | Canonical expression                                          | Behavior if env unset |
| ----------------------------------- | ------------------------------------------------------------- | --------------------- |
| `app/san-pham/page.tsx`             | `` `${process.env.NEXT_PUBLIC_SITE_URL}/san-pham/` ``         | emits `undefined/san-pham/` literal — bad |
| `app/cong-bo-tiep-thi-lien-ket/page.tsx` | `` `${process.env.NEXT_PUBLIC_SITE_URL ?? ""}/cong-bo-tiep-thi-lien-ket/` `` | emits `/cong-bo-tiep-thi-lien-ket/` relative path — degrades cleanly |

The spec (Scenario 2 + Edge Case "`NEXT_PUBLIC_SITE_URL` unset in dev: canonical falls back to the relative `/` path; build-time metadata must not throw") explicitly requires the **second** pattern. **Decision:** emit

```ts
canonical: `${process.env.NEXT_PUBLIC_SITE_URL ?? ""}/`,
```

so that:
- With env set (`https://aff-store.vercel.app`): canonical → `https://aff-store.vercel.app/` (absolute, trailing slash on root, exactly what Scenario 2 mandates).
- With env unset (local dev): canonical → `/` (relative, no `undefined/` garbage, no build error).

Reading `process.env.NEXT_PUBLIC_SITE_URL` directly (rather than going through `lib/env.ts`'s `requireUrl()`) is the **correct** choice here: `lib/env.ts` throws on unset, which would break the F0008 §5 edge case. The metadata module is the one place we trade the strict-env guarantee for the graceful fallback the spec demands. We do **not** retrofit other pages in this story — that is a separate consistency PR.

`next.config.ts` declares **no `metadataBase`**, so the absolute string is required for the production canonical to be a fully-qualified URL (per `node_modules/next/dist/docs/01-app/03-api-reference/04-functions/generate-metadata.md`: "If a `metadata` field provides an absolute URL, `metadataBase` will be ignored" and "Using a relative path in a URL-based `metadata` field without configuring a `metadataBase` will cause a build error" — but a fully-relative `/` is the documented graceful behavior for path-only canonicals at the root).

### 3.3 Page title — `aff-store — Đồ chơi công nghệ & gaming Việt Nam`

Brand consistency: `lib/site.ts` exports `SITE_NAME = "aff-store"`; the visible `<Header />` shows `aff-store`. The homepage title should **not** use the `| aff-store` suffix convention used by sub-pages (`Chính sách bảo mật | aff-store`) — at the root of the site, the brand goes **first**. Format:

```
aff-store — Đồ chơi công nghệ & gaming Việt Nam
```

Title is unique (not duplicated on any other page — sub-pages all use the `<Page name> | aff-store` suffix). Length under 60 characters to avoid SERP truncation. Constructed at module-load using template literal interpolation with `SITE_NAME` so a future brand rename is one-line.

### 3.4 Meta description — Vietnamese, ≤160 chars, mentions both surfaces

```
Đánh giá, hướng dẫn mua và liên kết Shopee cho chuột, bàn phím, tai nghe gaming & gadget công nghệ — chọn nhanh sản phẩm tốt, đọc bài viết bằng tiếng Việt.
```

≈155 chars. Mentions: product category breadth, affiliate context (Shopee), editorial content (bài viết), and Vietnamese-language hook for SEO.

### 3.5 Hero structure — single `<h1>`, two CTA links, semantic `<section>`

```tsx
<section className={styles.hero} aria-labelledby="home-hero-heading">
  <p className={styles.eyebrow}>{SITE_NAME}</p>
  <h1 id="home-hero-heading" className={styles.heading}>Đồ chơi công nghệ &amp; gaming Việt Nam</h1>
  <p className={styles.tagline}>Tagline + orienting copy (Vietnamese, 2–3 sentences).</p>
  <div className={styles.ctaGroup}>
    <Link href="/san-pham/" className={styles.ctaPrimary}>Khám phá sản phẩm</Link>
    <Link href="/bai-viet/" className={styles.ctaSecondary}>Đọc bài viết &amp; hướng dẫn</Link>
  </div>
</section>
```

- **Exactly one `<h1>`** on the page (spec Scenario 1). The brand text above it is a `<p>` "eyebrow", not a heading, so it does not pollute the heading outline.
- **CTAs are `next/link`** (`Link` components) — not `<a>`. Internal links must be `Link` per Next.js convention (client-side nav, prefetch). They route to `/san-pham/` and `/bai-viet/` exactly as the spec mandates (Scenario 3); trailing slashes match the canonicals of the destination pages.
- **CTAs are visually differentiated** (primary = filled orange `--color-primary`/`--color-on-primary`; secondary = outlined or token-text) so the visitor's "first click" expectation is the catalog, with the editorial path one tap away.
- **No client interactivity** — no `"use client"` anywhere in the hero. Hover/focus styling is pure CSS via `:hover` / `:focus-visible`, plus `@media (prefers-reduced-motion: no-preference)` for any transition (mirrors the `AffiliateLink.module.css` reduced-motion gate).
- `aria-labelledby="home-hero-heading"` ties the section to its heading for AT navigation. The `id` is unique.

### 3.6 Hero copy (Vietnamese, real prose — not lorem ipsum)

| Slot       | Copy |
| ---------- | ---- |
| Eyebrow    | `aff-store` (from `SITE_NAME`) |
| Heading    | `Đồ chơi công nghệ & gaming Việt Nam` |
| Tagline    | `Đánh giá ngắn gọn, hướng dẫn mua thực tế, và liên kết Shopee cho chuột, bàn phím, tai nghe và phụ kiện công nghệ — chọn nhanh sản phẩm phù hợp với ngân sách của bạn.` |
| Primary CTA   | `Khám phá sản phẩm` → `/san-pham/` |
| Secondary CTA | `Đọc bài viết & hướng dẫn` → `/bai-viet/` |

Spec mandates "site name, a Vietnamese tagline, short orienting copy, and two CTAs". The eyebrow covers "site name" without forcing it into the `<h1>`; the heading + tagline together form the tagline + orienting copy. Tagline ≈170 chars — short enough to scan above the fold on mobile (≤4 lines at 16px / ~360px-wide column).

### 3.7 Full-width layout (no `<ShellLayout>`)

Spec Scenario 1: "full-width homepage (no left panel)". Spec §6: "The homepage is full-width; it does not use `<ShellLayout>`."

**Decision:** the hero (and the future US00082–84 sections) sits inside a plain page-scoped `.container` div that:

- spans the full viewport width up to `--max-width` (1280px), centered with `margin-inline: auto`;
- has horizontal padding (`var(--spacing-md)` on mobile, `var(--spacing-xl)` ≥1280px) so content doesn't kiss the viewport edges;
- has **no** background panel — it sits directly on `--color-page-bg`. Individual sections may opt into a `--color-surface` card later (US00082/84) if the visual rhythm calls for it; the hero in this story is borderless and lives on the page background.

The hero specifically uses generous vertical padding (`var(--spacing-2xl)` block-padding) so it visually owns the top of the page without needing a hero image — system font + token color is enough per F0002 minimalist rules ("Minimalist, low visual noise. Color: `--color-primary`. Font: system font stack only.").

### 3.8 Responsive behavior

- **Mobile (<768px):** hero CTAs stack vertically (flex column), `font-size` of `<h1>` ≈ `1.5rem` (24px), tagline 1rem. Container padding `var(--spacing-md)`. Hero block-padding `var(--spacing-xl)`.
- **Tablet (≥768px):** CTAs sit side-by-side (flex row, `gap: var(--spacing-md)`). `<h1>` scales up to ≈ `2rem` (32px).
- **Desktop (≥1280px):** Container hits `--max-width: 1280px` cap, hero block-padding `var(--spacing-2xl)`, `<h1>` ≈ `2.5rem` (40px).

All breakpoints use the literal `768px` / `1280px` values (per `app/globals.css` comment: "Breakpoints… reference only — cannot be used inside `@media` queries. Use these raw values in media queries"). No new tokens.

### 3.9 No new lib helpers, no new types

The hero copy is page-local; CTA labels are page-local; the only shared constant the page imports is `SITE_NAME` from `@/lib/site.ts`. No new file in `lib/`. No new file in `types/`. No new path alias.

## 10. Open questions for reviewer

1. **Env strictness for the homepage build.** `lib/env.ts` `requireUrl()` throws if `NEXT_PUBLIC_SITE_URL` is unset, and `app/layout.tsx` imports `@/lib/env`. The spec Edge Case says "build-time metadata must not throw" when the env is unset — I have interpreted this as **the metadata module itself must not throw**, satisfied by the `?? ""` fallback. The layout-level validation still throws globally, which means `next build` overall **does** require the env var. Is that the intended interpretation, or should US00081 also loosen the global env requirement? My recommendation: leave `lib/env.ts` alone — the env var is genuinely required for canonical URLs in production, and dev/preview can set it in `.env.local`. If the spec intends "buildable with zero env config", that is a separate (and larger) change.
2. **Title format — `aff-store — …` vs `… | aff-store`.** I chose the root-style `aff-store — Đồ chơi công nghệ & gaming Việt Nam` to differentiate the homepage from sub-pages that use the `| aff-store` suffix. Acceptable? If you prefer the sub-page convention `Đồ chơi công nghệ & gaming Việt Nam | aff-store`, swap §3.3 / §5 accordingly — both are valid.
3. **Hero heading copy.** `Đồ chơi công nghệ & gaming Việt Nam` is my best guess at a tagline that scans well on Vietnamese SERPs. If you have a preferred tagline already drafted (e.g., from marketing or a brand doc), feed it in and I will swap §3.6 verbatim.
4. **Secondary CTA wording.** `Đọc bài viết & hướng dẫn` is concise; alternatives: `Bài viết & hướng dẫn`, `Xem bài viết`. The primary CTA `Khám phá sản phẩm` matches the F0008 Process Flow §3 framing.
5. **Section placeholder comments.** Plan §3.1 sketches them as inline JSX comments. If you would rather US00081 ship with **no** placeholder comments (and let US00082/83/84 each add their own) for a cleaner diff, that is fine — call it out and §3.1 / §4 adjust.

## 11. Risks & mitigations

- **Risk:** Stale `<h1>aff-store</h1>` cached by a CDN/edge after replacement.
  **Mitigation:** Vercel auto-purges on push to `main`; no manual cache action needed.
- **Risk:** Adding `<section>` inside the root `<main>` causes a duplicate-landmark a11y warning if someone later adds `role="main"` here.
  **Mitigation:** the hero uses `<section aria-labelledby="home-hero-heading">`, not `<main>` — explicitly called out in AC1.
- **Risk:** CTA Link prefetch traffic on a homepage with high bounce.
  **Mitigation:** acceptable — `next/link` prefetch is lightweight and both targets are SSG-cached. No mitigation needed for v1.
- **Risk:** Title length truncation on SERPs if a future brand rename makes `SITE_NAME` longer.
  **Mitigation:** title is template-literal-built from `SITE_NAME`; if length becomes a concern, swap the constant or hard-code a new title — both one-line changes.
