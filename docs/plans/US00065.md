# US00065 — Listing left-panel category/tag filter (`/bai-viet/`)

**Feature:** F0006 (Editorial Content)
**Status:** Approved
**Depends on:** US00064 (blog listing page `/bai-viet/`), F0002 (`<ShellLayout>` / `<Drawer>` / `<ShellLayoutDrawer>`), F0004 (`lib/filters.ts`, `<Pagination>`, `lib/categories.ts`)

---

## 1. Summary

Add a left-panel **category + tag** filter to the blog listing at `/bai-viet/`. The filter narrows the post list (newest-first) and writes its state to the URL as `?category=…&?tag=…` (comma-separated multi-value). The URL is the **only** source of truth — no local/Context/`localStorage` state. Changing a filter strips `?page` (resets to page 1). On mobile the panel collapses into the existing slide-out `<Drawer>` via `<ShellLayout>`. This is the F0004 product-filter pattern re-applied to posts.

This story builds **directly on US00064's listing page**. US00064 renders the post grid + pagination inside the page; US00065 wraps that grid with a URL-aware filter and feeds it the filtered/sorted post slice. The plan below assumes US00064 ships a client grid component analogous to `CatalogGrid`; if US00064 ships only a server-rendered grid, this story converts the post-grid render into a `"use client"` component (`PostListingGrid`) that reads `useSearchParams()` — mirroring `components/CatalogGrid.tsx`.

## 2. Codebase touchpoints (quoted from `CLAUDE.md` structure doc)

- **Module convention — Catalog filter state:** *"Catalog filter state lives in the URL (`?category=`, `?brand=`, `?price=`, `?sort=`). The URL is the only source of truth — no local state, no Context, no `localStorage`. … Multi-value params use comma-separated values … Filter changes strip `?page` to reset pagination. Future stories adding filter dimensions must follow this same pattern."* — US00065 is exactly such a future story; `?tag=` is a new dimension.
- **`lib/filters.ts`** — *"PRICE_BUCKETS, SORT_OPTIONS, getFilterOptions, parseFilterParams, serializeFilterParams, applyFilters, compareDefault, countActiveFilters (US00044)."* These are **product-shaped** (`Product`, `price`, `brand`). Posts need post-shaped equivalents (see §4 decision).
- **`lib/categories.ts`** — *"CATEGORIES map + getCategoryMeta + assertCategoryRegistered … getCategoryLabels()."* Reused read-only for category display labels. No new registry.
- **`lib/posts.ts`** — `getAllPosts()` returns `Post[]` sorted `publishedAt` desc; `Post` has `category: string` and `tags: string[]`.
- **`components/ShellLayout.tsx` / `ShellLayoutDrawer.tsx` / `Drawer.tsx`** — *"opt-in two-column shell (leftPanel + children)"* + *"generic slide-out drawer primitive (Esc, focus trap, scroll lock, overlay)."* The shell already renders `leftPanel` inline on desktop AND inside a `<Drawer>` on mobile; passing the post filter as `leftPanel` gets mobile collapse for free. **No new mobile-trigger component needed** (unlike F0004's interim `CatalogFiltersMobileTrigger`, which carries a `TODO(US00025-drawer)` to migrate to `<Drawer>` — we adopt the drawer pattern from the start).
- **`components/Pagination.tsx`** — *"accepts basePath + extraParams for filter-aware URLs."* Reused with `basePath="/bai-viet/"` and `extraParams` carrying `category`/`tag`.
- **Route map:** `/bai-viet` → `app/bai-viet/page.tsx` (created by US00064).
- **`@/*` path alias** (`tsconfig.json`) — all new imports use it.

## 3. Acceptance criteria (from F0006 §US00065)

1. **Filtering narrows the list** — selecting a category and/or tag re-renders the grid to only matching posts, newest-first.
2. **URL is the source of truth** — state written to `?category=…`, `?tag=…`; multi-value comma-separated; unknown values silently ignored; no local/Context/`localStorage`; URL round-trips (share/reload restores the view).
3. **Filter change resets pagination** — any filter change strips `?page`.
4. **Display labels reuse existing registry** — category labels from `lib/categories.ts` (`getCategoryLabels()`); tags are free-form strings shown as-is.
5. **Mobile behavior** — left panel collapses into the existing `<Drawer>` / `ShellLayout` pattern, not pushed down.

Edge cases (F0006 §5): **Filtered no-results** shows a distinct "no posts match" message (different from the all-empty state) while keeping the filter usable; **out-of-range `?page`** clamps/empties consistently with the product listing.

## 4. Key design decision — extend `lib/filters.ts` vs. sibling loader

**Decision: add a separate `lib/post-filters.ts` sibling module** (do NOT overload `lib/filters.ts`).

Rationale:
- `lib/filters.ts` is tightly coupled to the **product domain**: it imports `Product`, defines `PRICE_BUCKETS`/`SORT_OPTIONS`, and its `ActiveFilters`/`applyFilters`/`getFilterOptions`/`parseFilterParams` all operate on `price`/`brand`/`category`. Posts have **no price, no brand, no relevance sort** — they sort solely by `publishedAt` desc. Forcing post logic through the same functions would require either polluting the `Product`-typed signatures with generics or branching on a discriminator, both of which muddy a module the structure doc describes as the single product-filter chokepoint.
- A sibling `lib/post-filters.ts` keeps the **convention** (the structure doc mandates the *URL-as-source-of-truth pattern*, not a single shared file) while isolating the post-specific shape. F0006 §6 explicitly sanctions this: *"round-tripped through a helper in the F0004 style (extend `lib/filters.ts` **or add a sibling loader**)."*
- Mirrors the existing precedent of one loader per domain (`lib/products.ts` vs `lib/posts.ts`).

`lib/post-filters.ts` exports (parallel to `lib/filters.ts` names, post-shaped):

```ts
import type { Post } from "@/types";

export interface PostFilterOptions {
  categories: string[]; // distinct post.category slugs, sorted vi-locale
  tags: string[];       // distinct post.tags, sorted vi-locale
}

export interface ActivePostFilters {
  categories: string[];
  tags: string[];
}

export function getPostFilterOptions(posts: readonly Post[]): PostFilterOptions;
export function parsePostFilterParams(
  params: URLSearchParams,
  options: PostFilterOptions,
): ActivePostFilters;            // reads ?category / ?tag; comma-split; trims; drops unknown values
export function serializePostFilterParams(f: ActivePostFilters): URLSearchParams;
export function applyPostFilters(posts: readonly Post[], f: ActivePostFilters): Post[];
                                  // AND across dimensions; preserves getAllPosts() newest-first order
export function countActivePostFilters(f: ActivePostFilters): number;
```

`parsePostFilterParams` uses the same `readList` split-trim-filter logic as `parseFilterParams` (comma-separated; unknown values dropped against the `options` allow-lists — so a `?tag=` value not present in any post is silently ignored). `applyPostFilters` is intersection (a post must match the selected category set AND contain at least one selected tag), and re-uses the `getAllPosts()` ordering rather than re-sorting (posts are already `publishedAt` desc; a `compareDefault`-style tie-break on slug can be added for stability).

## 5. Resolved decision — taxonomy & option sourcing

**Posts share the product category taxonomy in `lib/categories.ts`** (A4). To prevent silent
drift between post frontmatter and the registry:

- **Extend `lib/posts.ts`** to call `assertCategoryRegistered(post.category, post.slug)` for every
  post at build time, mirroring how the product loader guards `Product.category`. A post that
  declares an unregistered category slug **fails `next build`** with the offending post slug
  named — same loud-build pattern as products.
- Under that guard, `getCategoryLabels()` is guaranteed to return a label for every post
  category. The `?? slug` fallback in `CatalogFilters` is kept only as defense-in-depth (in case
  a future code path bypasses the loader); it must not be relied on as the normal display path
  for posts.
- **Tags** stay as-authored: `Post.tags` is a free-form `string[]`, options are the distinct
  union across all posts, sorted vi-locale alphabetical. No registry, no normalization, no cap
  (re-evaluate if post count explodes).

Derive both option lists at build time in the page (`getPostFilterOptions(getAllPosts())`) and
pass down as props, exactly as `app/san-pham/page.tsx` does with `getFilterOptions`.

**Cross-story note:** the `lib/posts.ts` change technically belongs in whichever post-loader
story lands first (US00064 or US00066). If both land before this one, the assert is already in
place and US00065 only consumes it. If neither has shipped, US00065 owns the change. Track in
implementation sequencing.
