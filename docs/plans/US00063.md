# Implementation Plan — US00063: `<ProductCard slug="…" />` inline MDX affiliate card

**Status:** Approved
**Feature:** F0006 — Editorial Content
**Spec ref:** `docs/specs/F0006.md` § US00063 (+ §5 Edge Cases, §6 "Reuse, don't rebuild")
**Predecessors / siblings / consumers:**

- **Upstream dep — US00042** (`components/ProductCard.tsx`). This story does **not** build a new card. It exposes the existing whole-card affiliate `<ProductCard product={…} />` to MDX by wrapping it in a slug-resolving adapter. No restyle, no new variant.
- **Upstream dep — US00031–34** (`components/AffiliateLink.tsx` + `lib/affiliate.ts`). The inline card's click target is the canonical `<AffiliateLink>` transitively (via `<ProductCard>`). The F0003 ↔ F0007 `data-*` contract and the "no raw affiliate anchors" rule are preserved with zero new anchors.
- **Upstream dep — F0001** (`lib/products.ts` → `getProductBySlug()`). Single build-time data source. Adapter does the lookup; product-data parsing stays in `lib/`.
- **Upstream dep / sibling — US00062** (MDX body rendering + component map in `mdx-components.tsx`). US00062 owns the `useMDXComponents()` map and the rendering pipeline; the existing `mdx-components.tsx` already carries the placeholder comment `// F0003 will register <ProductCard /> here.`. **US00063 registers the entry; US00062 owns the surrounding map and the body renderer.** These two stories must agree on the map key (see §3 D2 — name collision).
- **Consumers:** every MDX post under `content/posts/*.mdx` that drops `<ProductCard slug="…" />` into prose (US00066 post page renders the body).

This story is the **author-facing inline-card seam**: it lets a writer type one tag and get a real, data-backed, affiliate-tracked product card inline — without re-entering image/name/price/URL — while keeping all product parsing in `lib/` and all affiliate semantics in the F0003 primitive.

## 1. Scope

Strictly:

1. Create a thin **MDX adapter component** `components/MdxProductCard.tsx` (Server Component, no `"use client"`):
   - Props: `{ slug: string }`.
   - Calls `getProductBySlug(slug)` from `@/lib/products` at render time (build time — posts are SSG).
   - On hit: renders the existing `<ProductCard product={product} />` (US00042) unchanged → image, name, `formatVnd(price)`, "Mua ngay" CTA, whole-card `<AffiliateLink>`.
   - On miss (`null`): **throws** an `Error` naming the offending slug (and, where available, the post) so `next build` fails loudly (Scenario 3 / §5 Edge Cases). Never renders a blank or defers to a runtime 404.
2. Register the adapter in the MDX component map (`mdx-components.tsx`) under the **author-facing JSX name `ProductCard`** so writers type `<ProductCard slug="logitech-g102-lightsync" />` in `.mdx`. The map value is `MdxProductCard` (the slug adapter), **not** the prop-based `@/components/ProductCard`. This resolves the name collision (§3 D2). If US00062 has already created the map scaffold, US00063 only adds this one entry; otherwise US00063 adds it to the existing placeholder.
3. Verification: 1 fixture MDX post with a valid `<ProductCard slug>` (renders) + a deliberate unknown-slug build to confirm the build fails with the slug named. Fixture removed before commit (or folded into US00062's verification fixture, coordinated).
4. **Structure-doc update:** add `components/MdxProductCard.tsx` to the `components/` tree in `CLAUDE.md` + `README.md`, note the MDX-map convention, bump "Last updated".

**Explicitly out of scope** (owned elsewhere):

- The MDX rendering pipeline, heading anchors, base typography map, `next/image` mapping → **US00062**.
- The post page chrome, hero, disclosure, SSG/`generateStaticParams`, 404 → **US00066**.
- Any change to `<ProductCard>` visuals/props (US00042) or `<AffiliateLink>` (US00031–34).
- A `compact`/inline card variant — the inline card is the **same** card; v1 ships no variant.
- Caching/memoizing `getProductBySlug()` (acceptable at v1 scale; see §10).

## 3. Approved Decisions

| # | Decision | Choice | Rationale |
| - | -------- | ------ | --------- |
| D1 | Where the adapter lives | New file `components/MdxProductCard.tsx` (+ no own CSS — it delegates all chrome to `<ProductCard>`). | Follows the `components/<Name>.tsx` convention. A separate file keeps the slug→product resolution distinct from the pure presentational card. |
| **D2** | **Name-collision resolution** | The existing component is `ProductCard(product: Product)`. The MDX one takes `slug`. **Do not** overload `ProductCard` to accept `slug \| product` (ambiguous prop union, two render paths, and an import-name clash in the map). Instead create a distinctly-named adapter `MdxProductCard({ slug })` and map it to the **author-facing JSX key `ProductCard`** inside `mdx-components.tsx`. Writers still type `<ProductCard slug="…" />`; the map redirects that name to `MdxProductCard`, which internally renders the real `<ProductCard product={…} />`. The two React identifiers never collide because the MDX-facing name is a *map key* (a string), not an imported symbol. | Keeps `<ProductCard>` single-responsibility (presentational, prop-based) and the adapter single-responsibility (resolve slug → render card → fail on miss). The author-facing tag stays the spec-mandated `<ProductCard slug="…" />`. No prop union, no runtime branch in the card. |
| D3 | Build-time lookup only | `MdxProductCard` calls `getProductBySlug(slug)` directly in the Server Component body. No `fetch`, no client component, no `useEffect`. | Scenario 4: build-time only, parsing stays in `lib/`. Posts are SSG so this runs during `next build`. |
| D4 | Unknown slug → loud build failure | On `null`, `throw new Error(\`[mdx] <ProductCard slug="\${slug}" />: no product JSON matches this slug.\`)`. Where the host post slug is available, include it. | Scenario 3 + §5 Edge Cases: must fail `next build` with the offending slug named — never a blank card or runtime 404. A thrown error during SSG render aborts the build for that route with the message. |
| D5 | Affiliate seam | Inherited verbatim from `<ProductCard>` → `<AffiliateLink>`. The adapter introduces **zero** anchors and **zero** `data-*` attributes of its own. | Scenario 2 + "No raw affiliate anchors (US00034)". The F0003↔F0007 contract is preserved transparently. |
| D6 | Map ownership / coordination with US00062 | US00062 owns `mdx-components.tsx` as the body-render map. US00063 contributes the single `ProductCard` entry. If US00062 lands first, US00063 edits its map; if US00063 lands first, it replaces the placeholder comment and US00062 builds on top. Document the seam in both plans. | Avoids two stories overwriting the same map. The placeholder comment already reserves the seam. |
| D7 | No own CSS module | `MdxProductCard` renders `<ProductCard>` directly; any inline-in-prose spacing (margin around the card) is owned by the MDX prose styles in **US00062**, not here. | Single source of card chrome (US00042); prose layout is US00062's concern. |
| D8 | Server Component | No `"use client"`. | Pure build-time render; matches `<ProductCard>` (also a Server Component). |
| D9 | Verification | One temporary fixture `.mdx` post + one deliberate bad-slug build. No new test runner (consistent with F0003/F0004 precedent). | Matches project verification convention. |
| D10 | Branch | `feat/US00063` → PR to `main`. PR is the review gate. | CLAUDE.md working conventions. |

## 10. Resolved decisions & residual risks

**Resolved:**

- **MDX component map ownership** (A2). US00062 ships the map with a `ProductCard` **stub** (a
  labelled placeholder box). US00063 swaps the stub for the real `MdxProductCard` adapter as a
  one-line edit to `components/mdx/mdx-components.tsx`. Lets US00062 verify its fixtures
  end-to-end without waiting on US00063, and keeps the map registration the only file both
  stories touch. Whichever lands second rebases the single line.
- **Host-post slug in build errors** (A3). Rely on Next.js's build stack trace, which names the
  failing route (`/bai-viet/<post-slug>`) when the adapter throws. The post is implicitly
  identified. **Do not** plumb post context through the MDX scope just to enrich the error
  message — the route name is enough. Prefix the thrown error with `[mdx ProductCard]` and the
  product slug; the route in the stack trace points to the host post.

**Residual risks:**

- **Throw-in-render abort granularity.** A thrown error in an SSG render aborts the build for
  that route. Verify during implementation against the Next.js MDX docs that the message surfaces
  clearly in `next build` output rather than being swallowed by a generic page-render error.
- **Empty catalog during development.** `content/products/` may be sparse; the fixture post must
  reference a slug that actually exists, or the verification build will (correctly) fail. Use a
  known seeded slug.
- **No memoization of `getProductBySlug`.** Each inline card re-reads/sorts all products
  (`getAllProducts` under the hood). At v1 scale (≤ a few hundred products, few cards per post)
  this is negligible; revisit only if build time regresses.
