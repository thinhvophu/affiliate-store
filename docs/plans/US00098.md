---

## 1. Story summary

Create `app/robots.ts` that exports a default function returning `MetadataRoute.Robots` so Next.js serves `/robots.txt` at the site root. The output must contain:

- `User-agent: *`
- `Allow: /`
- `Sitemap: <NEXT_PUBLIC_SITE_URL>/sitemap.xml` — an **absolute** URL sourced from the central env helper (`lib/env.ts`), never a hand-rolled `process.env...` fallback string.

A missing `NEXT_PUBLIC_SITE_URL` must **fail the build loudly** (already enforced by `requireUrl()` in `lib/env.ts`) — we explicitly want the strict-fail behavior here, in contrast to the `?? ""` fallback some legacy pages use for canonicals. Crawl rules + sitemap pointer must be correct or unset; an empty/relative `Sitemap:` line is worse than no robots.txt at all.

This story is **robots-only**. Sitemap content (US00097), root metadata (US00091), and per-page canonicals (US00092/US00099) are out of scope.

## 3. Existing convention to reuse (do not invent a new one)

`lib/env.ts` already exposes a frozen `env` object whose `NEXT_PUBLIC_SITE_URL` is validated by `requireUrl()` — it throws with a named error if the env var is missing or not a valid URL. Quoted:

```ts
export const env = {
  NEXT_PUBLIC_SITE_URL: requireUrl("NEXT_PUBLIC_SITE_URL"),
  …
} as const;
```

**Decision:** `app/robots.ts` reads the base URL via `import { env } from "@/lib/env"` and concatenates `"/sitemap.xml"` — it does **not** call `process.env.NEXT_PUBLIC_SITE_URL` directly, and it does **not** use a `?? ""` fallback. The strict-fail behavior is the whole point — this is the cohort `lib/env.ts` was designed for. (Future stories US00091/US00092 are expected to migrate canonical generation to the same `env` helper.)

The base URL value is not normalized for a trailing slash inside `lib/env.ts` today. We deliberately do **not** widen that helper's contract in this story; instead we trim a single trailing slash locally so concatenation produces exactly `<host>/sitemap.xml` whether the env var is set as `https://example.com` or `https://example.com/`. See §5 for the exact one-liner.

## 5. Exact file to add

```ts
// app/robots.ts
import type { MetadataRoute } from "next";
import { env } from "@/lib/env";

export default function robots(): MetadataRoute.Robots {
  const baseUrl = env.NEXT_PUBLIC_SITE_URL.replace(/\/$/, "");
  return {
    rules: {
      userAgent: "*",
      allow: "/",
    },
    sitemap: `${baseUrl}/sitemap.xml`,
  };
}
```

Notes on every line:

- **`import type { MetadataRoute }`** — type-only import from `next`, matches the Next.js docs example exactly. Erased at compile, no runtime cost.
- **`import { env } from "@/lib/env"`** — module-evaluation time call to `requireUrl()`. If `NEXT_PUBLIC_SITE_URL` is missing or malformed, `lib/env.ts` throws during build (when Next.js evaluates `app/robots.ts`), which is the loud-fail behavior the spec demands.
- **`baseUrl = env.NEXT_PUBLIC_SITE_URL.replace(/\/$/, "")`** — trims at most one trailing slash so concatenation never produces `//sitemap.xml`. Cheap and local; no helper change.
- **`rules: { userAgent: "*", allow: "/" }`** — single-rule object form (the `Robots` type accepts a single object OR an array; we use the object form because we want one universal rule). No `disallow` — spec scenario 1 calls for `Allow: /` only.
- **`sitemap: \`${baseUrl}/sitemap.xml\``** — absolute URL string. The `Robots` type allows `string | string[]`; we use a single string.
- **No `host` field** — not part of the spec, and `host` directives are non-standard.
- **No `Disallow:` line** — even an empty `Disallow:` is unnecessary; spec scenario 1 specifies "`Allow: /`" without a `Disallow:` counterpart.

## 10. Edge cases & guards

- **Missing `NEXT_PUBLIC_SITE_URL`:** loud build failure via `lib/env.ts` — the spec explicitly wants this. Do **not** introduce a `?? ""` fallback here even though some older pages use one for canonicals; that pattern is a hold-over scheduled for cleanup by US00091/US00092.
- **Trailing slash in env var:** the local `replace(/\/$/, "")` handles both `https://example.com` and `https://example.com/` without producing `//sitemap.xml`. We do not widen `lib/env.ts` to do this normalization globally — out of scope.
- **Conflict with `public/robots.txt`:** Next.js will serve `public/robots.txt` instead of the file-convention route if both exist. Confirm `public/robots.txt` is absent before merging (it is, as of planning).
- **Casing in output:** Next.js docs emit `User-Agent:` (capital A); the spec writes `User-agent: *`. Both are valid per the Robots Exclusion Standard (case-insensitive directive names). Do not try to override the output casing — accept Next.js's emission.
- **Caching:** Per the Next.js robots docs, the route is "cached by default unless it uses a Request-time API or dynamic config option." Our implementation has neither — it is fully static, which matches the SSG-only project rule.
- **Multiple sitemaps in the future:** the `Robots` type accepts `string | string[]` for `sitemap`. Today we ship a single string. If a future story adds e.g. an image sitemap, we promote to an array — not a concern for this story.
- **Special characters / encoding:** `NEXT_PUBLIC_SITE_URL` is validated as a `new URL(...)` in `lib/env.ts`, so percent-encoding/Punycode/etc. are already normalized by the WHATWG URL parser before we read it. No further escaping needed.
