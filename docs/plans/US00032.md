# Implementation Plan — US00032: Whole-card clickability pattern (no nested interactives)

**Status:** Done
**Feature:** F0003 — Affiliate Link Routing
**Spec ref:** `docs/specs/F0003.md` § US00032
**Priority:** P0 within F0003 (unblocks F0004 product cards and F0006 inline `<ProductCard />`).
**Predecessors (hard deps):**

- **US00031** — `components/AffiliateLink.tsx` must exist as a Server Component that emits a single `<a>` wrapping arbitrary children, with `target="_blank"`, `rel="noopener noreferrer sponsored"`, an accessible label, and the `data-affiliate-link` / `data-product-name` / `data-product-category` / `data-destination-url` attributes baked in. This story does **not** re-specify those concerns — it builds on top of them.

**Sibling consistency:**

- **US00033** — defines the tracking-hook seam. The whole-card pattern in this story must not weaken the seam; the wrapping anchor (and its `data-affiliate-link` attributes) is the only interactive element the visitor can target, so delegated click handlers attached by F0007 keep working unchanged whether the children are a one-line label or a full card subtree.
- **US00034** — defines the `lib/affiliate.ts` helper and no-raw-anchor convention. The whole-card example added by this story must obey that convention (the only anchor in the example is the `<AffiliateLink>` itself).
- **US00026** — established the `prefers-reduced-motion: no-preference` gating pattern in `app/globals.css`. This story re-uses the same pattern (gate `transition` declarations, never the static end-state) for the card hover/focus visuals so the codebase has one consistent reduced-motion strategy.

## 1. Scope

Strictly:

1. Add the **documented whole-card usage pattern** for `<AffiliateLink>` so F0004 (product card) and F0006 (inline `<ProductCard />`) can wrap a full card subtree (image + name + price + decorative CTA) as the children of a single `<AffiliateLink>` and have the entire surface behave as one link. The consumer pattern is: `import affiliateStyles from "@/components/AffiliateLink.module.css"; <AffiliateLink className={affiliateStyles.card} …>` — leveraging US00031's `className` prop. **No new prop is added to `<AffiliateLink>` by this story.**
2. Add **CSS surface styles** that style the card-shaped anchor consistently — specifically the `:hover` and `:focus-visible` states on the anchor — so that the whole card (not just the inner CTA) responds to pointer hover and to keyboard focus. The hover state is a subtle elevation/border emphasis; the focus-visible state is a visible 2 px outline that meets WCAG contrast against the card surface.
3. Add a **`prefers-reduced-motion: reduce` guard** that disables the hover/focus transition while keeping the end-state visuals intact. Pattern matches US00026 — gate only the `transition` declaration inside `@media (prefers-reduced-motion: no-preference)`, never the static styles themselves.
4. Document the **no-nested-interactives rule** both inline in the component file's JSDoc example and in `CLAUDE.md`'s "Affiliate links" section (created by US00031): any CTA rendered inside an `<AffiliateLink>` card subtree must be a non-interactive element (styled `<span>`), never `<button>` or a nested `<a>`. The card-styled CTA used in the example is a `<span>` with `data-affiliate-cta` styling.
5. Provide a **copyable JSX example** in the component's JSDoc and (optionally) at the top of the new `AffiliateLink.module.css` so a downstream developer can paste it directly into a product card or `<ProductCard />` MDX consumer. The example imports the CSS Module and passes `className={affiliateStyles.card}`.
6. Verify focus-visible tab-order: a keyboard user tabs **once** to the wrapping anchor and the focus indicator outlines the whole card, never multiple times for inner elements (which would indicate accidental nested interactives).

**Explicitly out of scope** (owned by sibling stories or downstream features):

- The `<AffiliateLink>` primitive itself — link semantics, `target`/`rel`, accessible label, required props, Server-Component-ness — **US00031**.
- The tracking-hook seam (`data-affiliate-link` and related data attributes) — **US00033**. This story neither emits nor reads those attributes; it only relies on US00031 having put them on the same `<a>`.
- The `lib/affiliate.ts` URL helper and the no-raw-anchor convention text — **US00034**. This story's example obeys the convention but does not author the convention paragraph in `CLAUDE.md`.
- The visual product card component (`components/ProductCard.tsx` or equivalent) — **F0004**. This story documents the pattern; F0004 implements the consumer.
- The MDX `<ProductCard slug="..." />` component — **F0006**. Same reasoning.
- A custom ESLint rule banning nested `<a>`/`<button>` inside an `<AffiliateLink>` subtree — explicitly P2 per F0003 §2; convention + JSDoc + code review is the v1 enforcement mechanism.
- The CTA's exact Vietnamese copy ("Mua ngay") and full visual treatment — the example uses placeholder styling; F0004 owns the canonical look.

## 5. Concrete diff sketch

### 5.1 `components/AffiliateLink.module.css` — append to existing file

US00031 ships this file with only the `.srOnly` class. This story appends:

```css
/* ==========================================================================
   AffiliateLink — whole-card surface styles (US00032)
   --------------------------------------------------------------------------
   Applied to the wrapping <a> when a consumer passes
   className={affiliateStyles.card}. Inline usage (no className) is
   unaffected — these rules only match when the class is present.

   No new :root tokens — composes from F0002 surface tokens and the
   F0001 spacing scale.
   ========================================================================== */

/* Block-level anchor that wraps a card-shaped subtree (image + name + price
   + decorative CTA). Children own their own layout (grid/flex). */
.card {
  display: block;
  background: var(--color-surface);
  border: 1px solid var(--color-border-hairline);
  border-radius: var(--radius-md);
  padding: var(--spacing-md);
  color: inherit;
  text-decoration: none;
  /* Static end-state — no transition outside the prefers-reduced-motion:
     no-preference block below. Hover/focus visuals snap for reduced-motion
     users. */
  box-shadow: none;
}

/* Hover state: subtle elevation + border emphasis. transform is GPU-only
   so neighbouring cards do not reflow. */
.card:hover {
  border-color: var(--color-primary);
  transform: translateY(-1px);
  box-shadow: 0 4px 8px rgba(0, 0, 0, 0.06);
}

/* Focus-visible state: 2 px Shopee-orange outline against the white surface.
   outline (not box-shadow) survives forced-colors mode. Applied to the
   wrapping anchor — the whole card surface — never to inner elements. */
.card:focus-visible {
  outline: 2px solid var(--color-primary);
  outline-offset: 2px;
  border-radius: var(--radius-md);
}

/* Non-interactive CTA element used inside the card subtree. Renders as a
   styled <span>; NEVER as <button> or a nested <a>. See AffiliateLink.tsx
   JSDoc example for the canonical markup. */
.cta {
  display: inline-block;
  margin-top: var(--spacing-sm);
  padding: var(--spacing-xs) var(--spacing-md);
  background: var(--color-primary);
  color: var(--color-on-primary);
  border-radius: var(--radius-sm);
  font-weight: 600;
}

/* Animate the hover/focus visuals only for users who haven't requested
   reduced motion. Mirrors the US00026 pattern: gate transitions, never
   end-state styles. */
@media (prefers-reduced-motion: no-preference) {
  .card {
    transition:
      border-color 150ms ease,
      transform 150ms ease,
      box-shadow 150ms ease;
  }
}
```

/**

 *    inline example showing the rendered DOM) appears above...

## 10. Risks / open items

- **Dependency on US00031's `className` prop.** This story's whole-card pattern relies on US00031 having shipped the `className?: string` escape hatch on `<AffiliateLink>`. If US00031 ever removes that prop, this story's example breaks at compile time. Mitigation: US00031's plan (D6/D7) explicitly commits to the prop; any future removal would be a coordinated breaking change.
- **Tab-focus regression in F0004 consumers.** If a downstream F0004 product card author accidentally includes a `<button>` (e.g., a "Save for later" placeholder) inside the card subtree, the no-nested-interactives rule is violated. The JSDoc + `CLAUDE.md` paragraph are the v1 enforcement; an ESLint rule is the right P2 follow-up.
- **`:focus-visible` browser support.** Universally supported in our target browsers (Chrome 86+, Safari 15.4+, Firefox 85+). No fallback needed. Confirm against current `browserslist` defaults at implementation time.
- **Hover-state `transform` on touch devices.** On iOS Safari, `:hover` fires on first touch and persists until the user taps elsewhere — meaning the card may stay in its lifted state momentarily after a tap. This is standard mobile-Safari behaviour for any `:hover` rule and not unique to this story. Acceptable per spec; document in F0004's plan if it becomes a UX issue.
- **CTA-class adoption.** The `.cta` class in the CSS Module is offered as a convenience for F0004; consumers may choose to define their own CTA styling. Both paths satisfy the no-nested-interactives rule as long as the inner element stays non-interactive (`<span>`).
- **Reduced-motion edge case — outline animation.** The `:focus-visible` outline is not transitioned (no `outline-*` in the transition list) — it appears instantly in both motion modes. This is deliberate: focus indicators should never be animated even for normal-motion users (sudden focus feedback is a usability win, not motion noise).
- **Forced-colors mode.** `outline` (not `box-shadow`) is used for the focus-visible ring so the indicator survives Windows High Contrast / forced-colors mode. The border-emphasis hover state will not survive forced-colors (the OS picks border colours), which is acceptable — focus accessibility is the priority.
- **Bundle size of `.module.css` for inline-only consumers.** Next.js's CSS handling loads the module CSS on any route that imports `<AffiliateLink>`, even if that route only uses the inline (no-className) form. The module is ~30 lines (~0.5 KB gzipped) so this overhead is negligible; revisit only if F0006 inline use becomes the dominant call site and the inline-only routes show measurable CSS bloat.
