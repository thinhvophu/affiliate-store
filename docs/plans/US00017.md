## 1. Scope

Strictly:

1. Install and configure **ESLint 9** with the official Next.js + TypeScript recommended rules, using the **flat config** format (`eslint.config.mjs`).
2. Install and configure **Prettier** with a committed config file (`.prettierrc`) covering `.ts`, `.tsx`, `.mdx`, `.json`, and `.css`.
3. Add `eslint-config-prettier` to disable ESLint formatting rules that conflict with Prettier.
4. Add `lint` and `format` npm scripts to `package.json`.
5. Add an `.editorconfig` file to normalize line endings (LF) and basic editor settings across platforms (Windows CRLF mitigation per F0001 edge cases).
6. Verify lint passes on a clean checkout and catches a deliberate violation (unused variable).

**Explicitly out of scope** (owned by sibling stories in F0001):

- MDX pipeline → **US00013**
- Shared TS types → **US00014**
- Content loaders → **US00015**
- Global styles & design tokens → **US00016**
- Typed env var module → **US00018**

No application code changes land here. We are wiring tooling only.

## 5. Concrete diff sketch

### 5.1 `eslint.config.mjs`

```js
import { defineConfig, globalIgnores } from "eslint/config";
import nextVitals from "eslint-config-next/core-web-vitals";
import nextTs from "eslint-config-next/typescript";
import prettier from "eslint-config-prettier/flat";

const eslintConfig = defineConfig([
  ...nextVitals,
  ...nextTs,
  prettier,
  globalIgnores([".next/**", "out/**", "build/**", "next-env.d.ts"]),
]);

export default eslintConfig;
```

This follows the exact pattern from the Next.js 16 ESLint documentation at `node_modules/next/dist/docs/01-app/03-api-reference/05-config/03-eslint.md`.

Key points:

- `core-web-vitals` includes base Next.js rules + React + React Hooks, with CWV rules upgraded to errors.
- `typescript` adds `@typescript-eslint/recommended` rules.
- `prettier` (last) disables all ESLint formatting rules that would conflict with Prettier.
- `globalIgnores` overrides the defaults from `eslint-config-next` to ensure build artifacts are excluded.

### 5.2 `.prettierrc`

```json
{
  "semi": true,
  "singleQuote": false,
  "tabWidth": 2,
  "trailingComma": "all",
  "printWidth": 100,
  "endOfLine": "lf",
  "overrides": [
    {
      "files": "*.mdx",
      "options": {
        "proseWrap": "preserve"
      }
    }
  ]
}
```

Notes on choices:

- `semi: true`, `singleQuote: false` — Prettier defaults, consistent with Next.js scaffolded code.
- `trailingComma: "all"` — cleaner diffs.
- `printWidth: 100` — slightly wider than default 80 for readability in TSX.
- `endOfLine: "lf"` — enforces LF line endings (Windows CRLF mitigation).
- MDX override with `proseWrap: "preserve"` — avoids rewrapping prose in blog posts.

### 5.3 `.prettierignore`

```
.next/
out/
build/
node_modules/
package-lock.json
*.tsbuildinfo
next-env.d.ts
```

### 5.4 `.editorconfig`

```ini
root = true

[*]
indent_style = space
indent_size = 2
end_of_line = lf
charset = utf-8
trim_trailing_whitespace = true
insert_final_newline = true

[*.md]
trim_trailing_whitespace = false

[*.mdx]
trim_trailing_whitespace = false
```

Notes:

- `end_of_line = lf` is the primary reason for this file — normalizes line endings for Windows contributors (F0001 edge case).
- Markdown/MDX trailing whitespace is preserved (significant in Markdown for line breaks).

## 6. Step-by-step execution order

1. Confirm on `main` branch with clean tree (`git rev-parse --abbrev-ref HEAD`, `git status`).
2. Install dev dependencies:
   ```
   npm install -D eslint eslint-config-next eslint-config-prettier prettier
   ```
3. Create `eslint.config.mjs` per section 5.1.
4. Create `.prettierrc` per section 5.2.
5. Create `.prettierignore` per section 5.3.
6. Create `.editorconfig` per section 5.4.
7. Add `lint`, `lint:fix`, `format`, `format:fix` scripts to `package.json` per section 5.5.
8. Run `npm run format:fix` to auto-format all existing files to the new Prettier config (ensures clean baseline).
9. Run `npm run lint` to verify lint passes on the clean codebase (AC Scenario 1).
10. Run `npm run format` to verify all files are formatted (exit 0).
11. **AC verification round** (Section 7) — inject deliberate violation, verify lint catches it, revert.
12. `git status` to confirm only the intended files changed.
13. Commit on `main`:
    `chore: configure ESLint + Prettier + EditorConfig (US00017)`
14. `git push origin main`.

## 10. Risks / open items

- **US00013 (MDX pipeline) may not be merged yet.** If `mdx-components.tsx` or MDX-related files exist when this story runs, ESLint may flag issues in those files. The implementer should run `npm run lint` after all changes and fix any issues in committed files. If US00013 is not merged, this is a non-issue since the files won't exist.
- **Prettier reformatting existing files.** Running `prettier --write .` may reformat `app/layout.tsx`, `app/page.tsx`, `next.config.ts`, `tsconfig.json`, and `package.json`. These are whitespace-only changes and are intentional -- they establish the formatting baseline. The commit message should note this.
- **`eslint-config-next` version pinning.** We pin `^16` to match the installed Next.js 16.x. Major version mismatches between Next.js and `eslint-config-next` can cause compatibility issues.
- **`eslint-config-prettier/flat` export.** This is the flat-config-aware export. If `eslint-config-prettier` version < 9.1.0 does not expose it, fall back to the default import. Current stable (v10.x) supports it.
- **Future lint rules.** US00018 may want to add a custom rule or override to ban `process.env.*` outside `lib/env.ts`. That rule belongs to US00018, not US00017. The ESLint config is designed to be extended.
- **No `lint-staged` or pre-commit hooks.** These are useful but not in the AC. Can be added as a follow-up if desired.
