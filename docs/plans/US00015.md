# Implementation Plan — US00015: Build content loader utilities

**Status:** Approved
**Feature:** F0001 — Technical Foundation & Bootstrap
**Spec ref:** `docs/specs/F0001.md` § US00015
**Predecessors:** US00011 (Next.js + TS bootstrap), US00012 (folders + `@/*` alias), US00013 (MDX pipeline + `gray-matter`), US00014 (shared `Product` / `Post` types)

---

## 1. Scope

Strictly:

1. Create `lib/products.ts` — product loader utilities that read JSON files from `content/products/`, validate them against the `Product` type from US00014, and return typed results.
2. Create `lib/posts.ts` — post loader utilities that read MDX files from `content/posts/`, parse frontmatter via `gray-matter` (installed by US00013), validate against the `Post` type from US00014, and return typed results.
3. Both modules detect and reject duplicate slugs.
4. Both modules handle empty content folders gracefully (return `[]`, not throw).
5. Both modules handle "not found" slugs gracefully (return `null`).
6. Both modules include clear error messages naming the offending file and field when validation fails.
7. All loader code is build-safe: uses Node `fs` / `path` and runs only at build time — never bundled into the client.
8. Create `content/products/.gitkeep` so the canonical product folder is committed.

**Explicitly out of scope** (owned by sibling stories):

- `Product` / `Post` type definitions themselves → **US00014**
- MDX compilation pipeline (`@next/mdx`, `remark-frontmatter`, `mdx-components.tsx`) → **US00013**
- Real product JSON files or MDX posts → **F0002 / F0003**
- Global styles / design tokens → **US00016**
- ESLint / Prettier → **US00017**
- Typed env var module → **US00018**

## 3. Approved Decisions

| # | Decision | Choice | Rationale |
|---|---|---|---|
| D1 | Loader file organization | **Two files: `lib/products.ts` and `lib/posts.ts`** | Clean separation of concerns. Each file owns one content type. Matches the spec's API surface (`getAllProducts`, `getProductBySlug`, `getAllPosts`, `getPostBySlug`). |
| D2 | Validation strategy | **Manual runtime validation functions** (not Zod or another schema lib) | The project has no schema validation dependency. Adding Zod for four fields is over-engineering. A hand-written `validateProduct()` / `validatePost()` function with descriptive errors is simpler, dependency-free, and sufficient for the file count we expect. |
| D3 | "Not found" return type | **Return `null`** (not throw) for `getProductBySlug` / `getPostBySlug` when slug doesn't match | Lets page-level code decide whether to call `notFound()` or handle differently. Consistent, documented, and idiomatic for Next.js `generateStaticParams` patterns. |
| D4 | Duplicate slug detection | **Detect at `getAll*()` time; throw with a message naming both files** | Slugs are URL keys; duplicates would silently shadow each other. Failing loudly at build time is the only safe option. |
| D5 | Empty folder behavior | **Return `[]`** from `getAll*()` when the content folder is empty or contains only `.gitkeep` | Per spec edge case: "Empty content folders return [] (not throw)." `.gitkeep` files are filtered out by extension check. |
| D6 | Build-safety mechanism | **`"use server"` is NOT used** (that's for Server Actions). Instead, loaders import `fs` and `path` from `node:` — Next.js tree-shakes these from client bundles automatically because they are only called in `getStaticProps`-equivalent contexts (App Router `generateStaticParams`, server components, `page.tsx` data fetching). We verify via build output inspection. | `server-only` package is an option but adds a dependency for a guarantee Next.js already provides for our SSG-only use case. If desired, it can be added later as a belt-and-suspenders measure. |
| D7 | Content directory paths | **Hardcoded constants** at the top of each loader: `PRODUCTS_DIR = path.join(process.cwd(), 'content', 'products')` and `POSTS_DIR = path.join(process.cwd(), 'content', 'posts')` | Per spec: "Content paths are fixed." No configurability needed. Using `process.cwd()` is the standard Next.js pattern for filesystem access at build time. |
| D8 | Post content return shape | **`getPostBySlug` returns frontmatter metadata + raw MDX content string** (not compiled React). Compilation to React components happens at the page level via `@next/mdx`. | Loaders are data loaders. MDX compilation is a rendering concern owned by the page that consumes the post. This keeps loaders fast (no compile step) and lets listing pages (`getAllPosts`) skip compilation entirely. |
| D9 | `content/products/.gitkeep` | **Create in this story** | US00012 created `content/.gitkeep`. US00013 will create `content/posts/.gitkeep`. This story creates `content/products/.gitkeep` to complete the content folder structure. |
| D10 | Sort order | **`getAllProducts` sorted by `publishedAt` descending (newest first); `getAllPosts` sorted by `publishedAt` descending** | Sensible default for listing pages. Callers can re-sort if needed. |
| D11 | Branch / commit target | **Commit directly to `main`** | Matches US00011–US00013 convention. |

## 6. Concrete diff sketch

### 6.1 `lib/products.ts`

```ts
import fs from "node:fs";
import path from "node:path";
import type { Product } from "@/types";

const PRODUCTS_DIR = path.join(process.cwd(), "content", "products");

/**
 * Validates that a parsed JSON object has all required Product fields.
 * Throws a descriptive error naming the file and the issue.
 */
function validateProduct(data: unknown, filePath: string): Product {
  if (typeof data !== "object" || data === null) {
    throw new Error(
      `[content] ${filePath}: file does not contain a valid JSON object.`
    );
  }

  const obj = data as Record<string, unknown>;
  const requiredStrings: (keyof Product)[] = [
    "slug",
    "name",
    "category",
    "brand",
    "affiliateUrl",
    "description",
  ];
  for (const field of requiredStrings) {
    if (typeof obj[field] !== "string" || (obj[field] as string).trim() === "") {
      throw new Error(
        `[content] ${filePath}: missing or invalid required string field "${field}".`
      );
    }
  }

  if (typeof obj.price !== "number" || obj.price < 0) {
    throw new Error(
      `[content] ${filePath}: "price" must be a non-negative number.`
    );
  }

  if (!Array.isArray(obj.images)) {
    throw new Error(
      `[content] ${filePath}: "images" must be an array of strings.`
    );
  }

  if (typeof obj.specs !== "object" || obj.specs === null || Array.isArray(obj.specs)) {
    throw new Error(
      `[content] ${filePath}: "specs" must be a plain object (Record<string, string>).`
    );
  }

  if (typeof obj.publishedAt !== "string") {
    throw new Error(
      `[content] ${filePath}: missing or invalid "publishedAt" date string.`
    );
  }

  if (typeof obj.featured !== "boolean") {
    throw new Error(
      `[content] ${filePath}: "featured" must be a boolean.`
    );
  }

  return data as Product;
}

/**
 * Reads and validates all product JSON files from content/products/.
 * Returns [] if the folder is empty or contains only .gitkeep.
 * Throws on malformed files or duplicate slugs.
 * Results sorted by publishedAt descending (newest first).
 */
export function getAllProducts(): Product[] {
  if (!fs.existsSync(PRODUCTS_DIR)) {
    return [];
  }

  const files = fs
    .readdirSync(PRODUCTS_DIR)
    .filter((f) => f.endsWith(".json"));

  if (files.length === 0) {
    return [];
  }

  const products: Product[] = [];
  const slugMap = new Map<string, string>(); // slug -> filePath (for duplicate detection)

  for (const file of files) {
    const filePath = path.join(PRODUCTS_DIR, file);
    let raw: string;
    try {
      raw = fs.readFileSync(filePath, "utf-8");
    } catch (err) {
      throw new Error(`[content] ${filePath}: unable to read file. ${err}`);
    }

    let parsed: unknown;
    try {
      parsed = JSON.parse(raw);
    } catch (err) {
      throw new Error(`[content] ${filePath}: invalid JSON. ${err}`);
    }

    const product = validateProduct(parsed, filePath);

    // Duplicate slug detection
    const existing = slugMap.get(product.slug);
    if (existing) {
      throw new Error(
        `[content] Duplicate product slug "${product.slug}" found in:\n` +
          `  - ${existing}\n` +
          `  - ${filePath}\n` +
          `Slugs must be unique because they are used as URL keys.`
      );
    }
    slugMap.set(product.slug, filePath);

    products.push(product);
  }

  // Sort by publishedAt descending (newest first)
  products.sort(
    (a, b) =>
      new Date(b.publishedAt).getTime() - new Date(a.publishedAt).getTime()
  );

  return products;
}

/**
 * Returns a single product by slug, or null if not found.
 */
export function getProductBySlug(slug: string): Product | null {
  const products = getAllProducts();
  return products.find((p) => p.slug === slug) ?? null;
}
```

### 6.2 `lib/posts.ts`

```ts
import fs from "node:fs";
import path from "node:path";
import matter from "gray-matter";
import type { Post } from "@/types";

const POSTS_DIR = path.join(process.cwd(), "content", "posts");

/**
 * Validates that parsed MDX frontmatter has all required Post fields.
 * Throws a descriptive error naming the file and the issue.
 */
function validatePostFrontmatter(
  data: Record<string, unknown>,
  filePath: string
): void {
  const requiredStrings: string[] = [
    "title",
    "summary",
    "publishedAt",
    "category",
    "coverImage",
  ];
  for (const field of requiredStrings) {
    if (typeof data[field] !== "string" || (data[field] as string).trim() === "") {
      throw new Error(
        `[content] ${filePath}: missing or invalid required frontmatter field "${field}".`
      );
    }
  }

  if (!Array.isArray(data.tags)) {
    throw new Error(
      `[content] ${filePath}: frontmatter "tags" must be an array of strings.`
    );
  }
}

/**
 * Reads and validates all MDX post files from content/posts/.
 * Returns [] if the folder is empty or contains only .gitkeep.
 * Throws on malformed files or duplicate slugs.
 * Results sorted by publishedAt descending (newest first).
 */
export function getAllPosts(): Post[] {
  if (!fs.existsSync(POSTS_DIR)) {
    return [];
  }

  const files = fs
    .readdirSync(POSTS_DIR)
    .filter((f) => f.endsWith(".mdx"));

  if (files.length === 0) {
    return [];
  }

  const posts: Post[] = [];
  const slugMap = new Map<string, string>(); // slug -> filePath

  for (const file of files) {
    const filePath = path.join(POSTS_DIR, file);
    let raw: string;
    try {
      raw = fs.readFileSync(filePath, "utf-8");
    } catch (err) {
      throw new Error(`[content] ${filePath}: unable to read file. ${err}`);
    }

    let parsed: matter.GrayMatterFile<string>;
    try {
      parsed = matter(raw);
    } catch (err) {
      throw new Error(
        `[content] ${filePath}: failed to parse frontmatter. ${err}`
      );
    }

    validatePostFrontmatter(
      parsed.data as Record<string, unknown>,
      filePath
    );

    // Derive slug from filename (e.g. "my-post.mdx" -> "my-post")
    const slug = path.basename(file, ".mdx");

    // Duplicate slug detection
    const existing = slugMap.get(slug);
    if (existing) {
      throw new Error(
        `[content] Duplicate post slug "${slug}" found in:\n` +
          `  - ${existing}\n` +
          `  - ${filePath}\n` +
          `Slugs must be unique because they are used as URL keys.`
      );
    }
    slugMap.set(slug, filePath);

    const frontmatter = parsed.data as Record<string, unknown>;

    posts.push({
      slug,
      title: frontmatter.title as string,
      summary: frontmatter.summary as string,
      publishedAt: frontmatter.publishedAt as string,
      category: frontmatter.category as string,
      tags: frontmatter.tags as string[],
      coverImage: frontmatter.coverImage as string,
      content: parsed.content,
    });
  }

  // Sort by publishedAt descending (newest first)
  posts.sort(
    (a, b) =>
      new Date(b.publishedAt).getTime() - new Date(a.publishedAt).getTime()
  );

  return posts;
}

/**
 * Returns a single post by slug, or null if not found.
 */
export function getPostBySlug(slug: string): Post | null {
  const posts = getAllPosts();
  return posts.find((p) => p.slug === slug) ?? null;
}
```

## 10. Structure-doc impact

No changes to `CLAUDE.md` or `docs/spec.md`. The loader API surface defined here becomes the contract that F0002 and F0003 depend on:

- **F0002** (Product pages): calls `getAllProducts()` for listing, `getProductBySlug()` for detail, and `getAllProducts().map(p => p.slug)` for `generateStaticParams`.
- **F0003** (Blog pages): calls `getAllPosts()` for listing, `getPostBySlug()` for detail, and `getAllPosts().map(p => p.slug)` for `generateStaticParams`.
- **F0004** (Category pages): calls `getAllProducts()` and filters by `category`.

If the `Post` or `Product` type shape from US00014 differs from what is sketched in Section 4.1, the validation functions must be updated to match. The API signatures (`getAllProducts`, `getProductBySlug`, `getAllPosts`, `getPostBySlug`) and their return types are the stable contract.

## 11. Risks / open items

- **Dependency on US00013 + US00014.** This story cannot be implemented until both are done. If the `Post` type from US00014 differs from the shape assumed here (e.g., `content` field is named differently, or slug is included in frontmatter rather than derived from filename), the `lib/posts.ts` mapping logic must adapt.
- **`getProductBySlug` / `getPostBySlug` performance.** Both call `getAll*()` internally and linear-scan for the slug. For the expected content volume (tens to low hundreds of files), this is fine. If content grows to thousands of files, a cached Map could be added — but that's premature optimization for an SSG site.
- **Product slug source.** Products store `slug` inside the JSON file. Posts derive `slug` from the filename. This asymmetry is intentional (matches spec) but worth documenting. If a product's `slug` field doesn't match its filename, the `slug` field wins — the filename is irrelevant.
- **`gray-matter` types.** `gray-matter` ships its own TS declarations. The `data` property is typed as `{ [key: string]: any }`. Our `validatePostFrontmatter` function narrows this to the `Post` shape with runtime checks.
- **No caching across calls.** Each `getAll*()` call re-reads the filesystem. In Next.js SSG, this happens at build time only, so the cost is negligible. Next.js may also deduplicate calls within a single build via its internal fetch cache for server components, but we don't rely on that.
- **`content/products/.gitkeep` cleanup.** Should be deleted by the story that first adds real product JSON files (F0002).
