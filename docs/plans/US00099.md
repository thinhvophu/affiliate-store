# Implementation Plan — US00099: Paginated listing canonical strategy

> ⛔ **DEFERRED out of the F0009 implementation batch (decision 2026-06-21).**
> Reading `searchParams` in `generateMetadata` flips the 3 listing routes from SSG (`○`) to dynamic (`ƒ`), which conflicts with the project's **SSG-only** rule (CLAUDE.md). The owner chose to keep SSG and postpone this story rather than accept the carve-out. This plan is retained for a future iteration — likely via **path-based pagination** (e.g. `/san-pham/trang/2/`) that stays statically generatable. **Do NOT implement in the current F0009 batch.**

## 1. Goal (one sentence)

Make each of the three paginated listing routes (`/san-pham/`, `/danh-muc/[category]/`, `/bai-viet/`) emit a **self-referencing** canonical URL that reflects the current `?page=N` value (or the clean listing URL when `N === 1` / no param), so Google indexes each paginated page on its own merits and the catalog stops sending duplicate-content signals.

## 2. Codebase touchpoints (quoted from `CLAUDE.md` "Codebase structure")

This plan touches the three listing route files and one library file:

- `app/san-pham/page.tsx` — *"Product listing — SSG, wires CatalogFilters + CatalogGrid + mobile trigger (US00043/44)"*
- `app/danh-muc/[category]/page.tsx` — *"Category page — SSG per registered category, `<CategoryNav>` left panel (US00045)"*
- `app/bai-viet/page.tsx` — *"Blog listing — SSG, ShellLayout + PostFilters left panel + PostListingGrid; filter options derived at build time (US00064, US00065)"*
- `lib/seo.ts` (new in **US00091**) — *(planned)* will expose a `buildCanonicalPath(pathname, searchParams)` helper. US00099 consumes it; if US00091 has not yet introduced the helper, this story adds a `buildPaginatedCanonical(pathname, page)` function alongside it.
- `lib/env.ts` (existing) — `env.NEXT_PUBLIC_SITE_URL` continues to flow through `metadataBase` at the root layout level (US00091 wiring). US00099 does **not** read it directly; canonicals stay relative and Next.js resolves them via `metadataBase`.

The grids (`CatalogGrid.tsx`, `CategoryPageClient.tsx`, `PostListingGrid.tsx`) and `Pagination.tsx` are **read but not modified** — F0004/F0006's crawlable `?page=N` link pattern is preserved verbatim.

## 3. Critical Next.js context (from `node_modules/next/dist/docs/`)

Read: `01-app/03-api-reference/04-functions/generate-metadata.md` and `01-app/03-api-reference/03-file-conventions/page.md`.

Key facts that shape this plan:

1. **`searchParams` shape:** in Next.js 16, the signature is
   ```ts
   { params: Promise<…>, searchParams: Promise<{ [k: string]: string | string[] | undefined }> }
   ```
   It must be `await`-ed. It is available in both the page component and in `generateMetadata` (page-segment only — not layouts).

2. **`searchParams` is a Request-time API.** Quoting `page.md`:
   > `searchParams` is a **Request-time API** whose values cannot be known ahead of time. Using it will opt the page into **dynamic rendering** at request time.

   This is the load-bearing constraint for US00099. CLAUDE.md states the project is **SSG-only**. Reading `searchParams` in `generateMetadata` (and/or the page body) on these three routes will flip them from prerendered to dynamically-rendered.

3. **No Cache Components.** `next.config.ts` does not enable `experimental.cacheComponents`, so the "streaming metadata" / "DynamicMarker" affordances described in `generate-metadata.md` §"With Cache Components" do **not** apply. The plain rule applies: touch `searchParams` → page is dynamic at request time.

4. **Where the page reads `?page` today:** all three pages currently delegate to **client components** (`CatalogGrid`, `CategoryPageClient`, `PostListingGrid`) that call `useSearchParams()`. The `page.tsx` server files do **not** receive or read `searchParams` today. That keeps the routes prerenderable; the URL state is read client-side after hydration.

### Resolution — accepted trade-off

We accept that reading `searchParams` inside `generateMetadata` on these three listings will move them from fully prerendered to dynamically rendered on request. Justification:

- The canonical strategy *requires* knowing the current `?page=N`. Computing it on the client is not allowed: the canonical and JSON-LD must live in the initial HTML (CLAUDE.md "SEO non-negotiables" + F0009 §6 "SSG-only").
- Listing data itself is still derived at build time (`getAllProducts()`, `getAllPosts()`, `lib/categories.ts`) — no runtime fetches. Dynamic rendering here costs only the cost of slicing an in-memory array per request.
- Vercel's edge cache will satisfy repeat requests for the same `?page` value. The cardinality is tiny: `Math.ceil(totalItems / PAGE_SIZE)` per listing, currently ~2 pages each.
- We do **not** mark the route `export const dynamic = "force-static"` — that flag forbids dynamic APIs and would throw when `searchParams` is read.
- The page body itself continues to **not** read `searchParams` (only `generateMetadata` does). The grid components keep reading via `useSearchParams()` client-side; F0004/F0006 contracts are unchanged.

This trade-off must be **stated in the PR description** so reviewers see the SSG-only departure on three specific routes is intentional.

> **Open question for reviewer:** Are you comfortable with the three listing routes becoming dynamically rendered? Alternative is to drop self-referencing canonicals on listings and ship only the clean-URL canonical for page 1 (Google's older "rel=prev/next is deprecated" guidance accepts this), but it weakens the indexability of pages 2+. F0009 §4 US00099 Scenario 1 explicitly mandates the self-referencing canonical, so dropping it would require a spec change.

## 4. Shared helper: `lib/seo.ts`

Assumption: US00091's plan introduces `lib/seo.ts` with at minimum a relative-canonical helper. US00099 adds (or proposes to US00091) the following pure function:

```ts
// lib/seo.ts
/**
 * Build a self-referencing canonical path for a paginated listing.
 *
 * - Returns the bare pathname when page is 1, undefined, or invalid.
 * - Returns `${pathname}?page=N` for N ≥ 2.
 * - Never throws — invalid input collapses to the page-1 canonical.
 *
 * Path is RELATIVE; the root layout's `metadataBase` resolves it.
 */
export function buildPaginatedCanonical(
  pathname: string,           // e.g. "/san-pham/"
  pageRaw: string | string[] | undefined,
): string {
  const first = Array.isArray(pageRaw) ? pageRaw[0] : pageRaw;
  const n = Number.parseInt(first ?? "1", 10);
  if (!Number.isFinite(n) || n < 2) return pathname;
  return `${pathname}?page=${n}`;
}
```

Notes:
- Plain integer parsing — anything non-numeric, negative, zero, or `NaN` falls through to the page-1 canonical. This satisfies Scenario 1 (self-ref for N>1), Scenario 2 (clean URL for page 1 / missing), and the edge-case requirement "must not crash on out-of-range/invalid page params" (F0009 §5).
- **Upper-bound clamping is intentionally NOT done in this helper.** The page count varies per listing/category/filter combination, so computing it would require the helper to know item counts. We accept that an out-of-range `?page=99` on a 2-page listing emits `canonical="/san-pham/?page=99"`. The grid already shows an empty/out-of-range UI and Google will see the canonical → page 99 mapping is harmless because Google won't index a 99 it has never linked to. If the reviewer prefers clamping, we can pass `totalPages` in and clamp; flagged below.
- Trailing slash on `pathname` is preserved as-passed; callers supply `/san-pham/`, `/danh-muc/<slug>`, `/bai-viet/` — matching the existing canonicals in those files today.

> **Open question for reviewer:** Clamp the canonical to `totalPages` when `?page > totalPages`? Doing so would require each page.tsx to compute totalPages server-side (it already has the data via `getAllProducts()` / `getAllPosts()`), which is fine but adds 3 small computations. Default proposal: do NOT clamp; emit the literal page param. Cheap and matches the F0009 spec wording.

## 5. Per-route changes

For each route, the diff is small: convert `export const metadata` → `export async function generateMetadata({ searchParams })`, read `?page`, and call `buildPaginatedCanonical`. The page component is **not** changed in any way that affects rendering or data flow.

### 5.1 `app/san-pham/page.tsx`

Today:
```tsx
export const metadata: Metadata = {
  title: "Tất cả sản phẩm | TechShop",
  description: "Khám phá toàn bộ sản phẩm gaming và công nghệ với giá tốt nhất.",
  alternates: { canonical: `${process.env.NEXT_PUBLIC_SITE_URL}/san-pham/` },
};
```

After:
```tsx
import { buildPaginatedCanonical } from "@/lib/seo";

type SP = { [k: string]: string | string[] | undefined };

export async function generateMetadata(
  { searchParams }: { searchParams: Promise<SP> },
): Promise<Metadata> {
  const sp = await searchParams;
  return {
    title: "Tất cả sản phẩm | TechShop",  // US00092 may revisit copy; not in scope here.
    description: "Khám phá toàn bộ sản phẩm gaming và công nghệ với giá tốt nhất.",
    alternates: {
      canonical: buildPaginatedCanonical("/san-pham/", sp.page),
    },
  };
}
```

Notes:
- Drops the `${process.env.NEXT_PUBLIC_SITE_URL}/...` string concatenation; relies on US00091's `metadataBase` to absolutize. (Aligned with US00092 Scenario 2 — "they resolve through `metadataBase` (relative `alternates.canonical`)".)
- The existing `D15: canonical never reflects filter params` invariant is preserved — only `page` is read; `category`, `brand`, `price`, `sort` are ignored when building the canonical.
- Page body is unchanged.

import { buildPaginatedCanonical } from "@/lib/seo";

type SP = { [k: string]: string | string[] | undefined };

export async function generateMetadata({
  params,
  searchParams,
}: {
  params: Promise<Params>;
  searchParams: Promise<SP>;
}): Promise<Metadata> {
  const { category } = await params;
  const meta = getCategoryMeta(category);
  if (!meta) return {};
  const sp = await searchParams;
  return {
    title: `${meta.name} | aff-store`,
    description: meta.metaDescription,
    alternates: {
      canonical: buildPaginatedCanonical(`/danh-muc/${meta.slug}`, sp.page),
    },

- The existing inline comment "Pages 2+ append `?page=N` — handled client-side; metadata reflects page 1 canonical." must be **removed** — that invariant is exactly what US00099 reverses.
- `generateStaticParams` is **not** removed; static path generation still works. Only the per-request `?page` flips the rendering mode (see §3 trade-off).

## 6. What does NOT change

- **F0004/F0006 crawlable pagination:** `<Pagination>` continues to emit real `<a href="?page=N">` links. The grid components (`CatalogGrid`, `CategoryPageClient`, `PostListingGrid`) still read `?page` client-side via `useSearchParams()`. No change to the grid logic.
- **Filter-state invariants:** canonical never reflects `?category`, `?brand`, `?price`, `?sort`. The filter URL contract from F0004 is untouched.
- **Product/post/about/policy pages:** unaffected. Their `generateMetadata` (where it exists) keeps using `params` only.
- **Root metadata + `metadataBase`:** owned by US00091; this story consumes that foundation.

## 7. Edge cases (mapped to F0009 §5)

| Edge case | Behavior |
|---|---|
| `?page=1` (explicit) | Canonical = `/san-pham/` (no `?page=1`). Matches Scenario 2. |
| `?page=` (empty) | Treated as `undefined` → page 1 canonical. |
| `?page=abc` (non-numeric) | `Number.parseInt` → `NaN` → page 1 canonical (no crash). |
| `?page=-3`, `?page=0` | `< 2` branch → page 1 canonical. |
| `?page=99` on a 2-page listing | Canonical = `/san-pham/?page=99` (literal). Grid shows out-of-range/empty UI. Acceptable; see §4 open question. |
| `?page=2&page=3` (multi-value, becomes `string[]`) | Helper takes the first element → `?page=2`. |
| Unknown category slug | `generateMetadata` returns `{}` *before* reaching the canonical block (existing behavior). `notFound()` fires in the page component. No regression. |
| Missing `NEXT_PUBLIC_SITE_URL` at build | Already fails the build via `lib/env.ts` (US00091's `metadataBase` consumes it). Not our concern here. |

## 8. Tests

Add `lib/seo.test.ts` (or extend the file if US00091 created it) with a focused unit suite for `buildPaginatedCanonical`:

```ts
import { describe, it, expect } from "vitest";
import { buildPaginatedCanonical } from "@/lib/seo";

describe("buildPaginatedCanonical", () => {
  it.each([
    [undefined, "/san-pham/"],
    ["1", "/san-pham/"],
    ["", "/san-pham/"],
    ["2", "/san-pham/?page=2"],
    ["10", "/san-pham/?page=10"],
    ["abc", "/san-pham/"],
    ["-3", "/san-pham/"],
    ["0", "/san-pham/"],
    [["2", "3"], "/san-pham/?page=2"],
  ])("page=%j → %s", (input, expected) => {
    expect(buildPaginatedCanonical("/san-pham/", input as never)).toBe(expected);
  });

  it("preserves caller-supplied pathname (category, no trailing slash)", () => {
    expect(buildPaginatedCanonical("/danh-muc/chuot-gaming", "2"))
      .toBe("/danh-muc/chuot-gaming?page=2");
  });
});
```
