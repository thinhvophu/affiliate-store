# US00053 — Reusable inline affiliate-disclosure banner

**Feature:** F0005 (Compliance & Disclosure)
**Status:** Draft
**Type:** UI / Server Component

---

## 1. Story & Goal

> **As a** content reader, **I want to** see the affiliate disclosure at the top of every blog
> post, **so that** I know about affiliate links before reading product recommendations.

Deliver a reusable, presentational **`<AffiliateDisclosure />`** Server Component that renders the
canonical `AFFILIATE_DISCLOSURE_VI` string as a visually distinct, accessible note. This story
delivers **only the component and its placement contract**. Wiring it into the actual blog-post
template is explicitly F0006's job (see F0005 §2 Out of Scope and US00053 Scenario 2).

## 4. Component structure & decisions

```tsx
// components/AffiliateDisclosure.tsx (Server Component — no "use client")
import { AFFILIATE_DISCLOSURE_VI } from "@/lib/disclosures";
import styles from "./AffiliateDisclosure.module.css";

export function AffiliateDisclosure() {
  return (
    <aside className={styles.disclosure} aria-label="Công bố tiếp thị liên kết">
      <p className={styles.text}>{AFFILIATE_DISCLOSURE_VI}</p>
    </aside>
  );
}
```

**Decisions:**
1. **Semantic element — `<aside>`** (Scenario 3 "appropriate for an aside/note"). It is tangentially
   related to the article body, which is the canonical use for `<aside>`. An `aria-label` (in
   Vietnamese) gives it an accessible name so screen-reader users can identify the landmark; this
   keeps it a recognizable, skippable region without an extra visible heading.
2. **No props.** The component takes no inputs — F0006 imports and drops it in (`<AffiliateDisclosure />`)
   above the article body. Zero-config placement satisfies Scenario 2's "without re-implementing
   text or styling."
3. **Single source of truth (Scenario 1).** The only copy comes from `AFFILIATE_DISCLOSURE_VI`. No
   string literal of the disclosure text appears in the TSX or CSS. This guards against the
   "Disclosure text drift" edge case in F0005 §5.
4. **Named export `AffiliateDisclosure`**, matching the project's existing named-export components
   (`Footer`, `RelatedProducts`).
5. **No JSX-side affiliate-URL or analytics logic (Scenario 4).** No `<AffiliateLink>`, no
   `data-affiliate-*`, no GA hooks. It is purely presentational.

**Styling (`AffiliateDisclosure.module.css`) — accessible, themed, distinguishable from body copy:**
- `.disclosure`: `background: var(--color-bg-secondary)`; left accent `border-inline-start: 4px solid
  var(--color-primary)`; `border-radius: var(--radius-md)`; `padding: var(--spacing-md)`;
  `margin-block-end: var(--spacing-lg)` (sits above article body).
- `.text`: `color: var(--color-text)` (not the lighter `--color-text-secondary`) on the light-grey
  surface to safely clear WCAG AA 4.5:1 contrast; `font-size: 0.9rem`; `line-height:
  var(--line-height-base)`; system font stack inherited (no external fonts).
- Mobile-first; no media query needed (single text block reflows). No JS, no animation.

## 9. Resolved decisions

1. **Element choice** — RESOLVED: `<aside aria-label="Công bố tiếp thị liên kết">` (a landmark
   region), not `role="note"`. Most semantically appropriate "note alongside the article" and
   discoverable as a skippable region by screen readers.
2. **Visible heading** — RESOLVED: no visible heading; `aria-label` only, keeping the banner
   lightweight (matches the "banner with no heading" intent).
3. **F0006 import ergonomics** — RESOLVED: named export `AffiliateDisclosure` only (matches the
   `Footer`/`RelatedProducts`/`AffiliateLink` named-export convention). No default export.
