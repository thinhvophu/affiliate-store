# US00083 — Category highlights section on the homepage

**Feature:** F0008 (Homepage)
**Status:** Done
**Depends on:**
- US00081 (homepage shell at `app/page.tsx` — this story composes a section into that shell)
- US00045 (`lib/categories.ts` — `CATEGORIES`, `getCategoryLabels()`; `/danh-muc/[category]/` route already implemented)

---

## 1. Story summary

On the homepage (`/`), render a **"Danh mục"** section that lists every registered category from
`lib/categories.ts` as a link to `/danh-muc/[category]/`. Each link is labelled with the category's
Vietnamese display name from `getCategoryLabels()`.

**Source-of-truth contract:** adding a new entry to `lib/categories.ts` (and only that) must cause
the new category to appear on the homepage automatically after the next build. There is **no**
hard-coded category list inside the homepage or in this component — the list is derived from
`CATEGORIES` at build time (SSG).

This is a pure Server Component with no client interactivity, no new data loader, and no new route.

## 3. Component-reuse decision (CategoryNav vs. a new CategoryHighlights)

`components/CategoryNav.tsx` is a close shape match (it already maps `Object.values(CATEGORIES)` to
`<Link href="/danh-muc/${slug}/">{name}</Link>` items). Reusing it would be the cheapest path. After
inspecting it, however, **the reuse is a poor fit for the homepage**:

1. **`CategoryNav` requires a `currentSlug` prop** and is hard-coded to filter that slug out
   (`others = filter(c => c.slug !== currentSlug)`). On the homepage there *is* no "current" category,
   so we would have to pass a sentinel ("" / a non-existent slug) and rely on the filter no-op'ing.
   That couples homepage rendering to a sibling-page concept and is fragile if `CategoryNav` ever
   adds a "current" highlight style on top of the filter.
2. **Semantic role differs.** `CategoryNav` renders `<aside><nav aria-label="Danh mục">…</nav></aside>`
   intended for a left-panel sibling listing labelled *"Danh mục khác"* (literally "other categories").
   The homepage section needs a top-level page section with an `<h2>` heading *"Danh mục"* — a
   `<section>` landmark, not an `<aside>` and not a `<nav>` (the links are content, not site
   navigation chrome).
3. **Visual treatment differs.** `CategoryNav.module.css` is a compact, low-contrast vertical link
   list designed to live beside a content column. The homepage highlights row wants a more
   prominent, scannable layout (multi-column card-ish grid) that scales 2/3/4-ish across breakpoints
   alongside the featured-products section above and the latest-posts strip below.

**Decision:** build a new dedicated Server Component **`components/CategoryHighlights.tsx`** + co-located
`CategoryHighlights.module.css`. It reads `CATEGORIES` / `getCategoryLabels()` directly from
`lib/categories.ts` (same single source of truth `CategoryNav` uses, so there is still exactly one
registry). `CategoryNav` is left untouched — its left-panel role on `/danh-muc/[category]/` is
unchanged.

> If during implementation `CategoryNav` is refactored to be `currentSlug`-optional and given a
> `variant`/`heading` knob, `CategoryHighlights` could later collapse into a thin wrapper. That is
> **out of scope** for this story.

## 4. New component — `components/CategoryHighlights.tsx`

Server Component (no `"use client"`). Reads the registry at module/render time (synchronous, no
async I/O) — `CATEGORIES` is an in-process constant, evaluated by Next.js at build time for SSG.

```tsx
import Link from "next/link";
import { CATEGORIES, getCategoryLabels } from "@/lib/categories";
import styles from "./CategoryHighlights.module.css";

export function CategoryHighlights() {
  const labels = getCategoryLabels();
  // Stable iteration order: object insertion order in lib/categories.ts.
  const slugs = Object.keys(CATEGORIES);

  if (slugs.length === 0) return null; // defensive — registry is non-empty today

  return (
    <section className={styles.section} aria-labelledby="home-categories-heading">
      <header className={styles.header}>
        <h2 id="home-categories-heading" className={styles.heading}>
          Danh mục
        </h2>
      </header>
      <ul className={styles.grid}>
        {slugs.map((slug) => (
          <li key={slug} className={styles.item}>
            <Link href={`/danh-muc/${slug}/`} className={styles.link}>
              {labels[slug]}
            </Link>
          </li>
        ))}
      </ul>
    </section>
  );
}
```

Notes:
- **No hard-coded list.** The only inputs are `CATEGORIES` and `getCategoryLabels()`. Adding a new
  category in `lib/categories.ts` and re-running `next build` automatically renders the new tile —
  enforced structurally by deriving `slugs` from `Object.keys(CATEGORIES)`. This satisfies
  Scenario 3 by construction.
- **Vietnamese display names** come exclusively from `getCategoryLabels()` (already used by
  `CatalogFilters` and `PostFilters` — same chokepoint). Satisfies Scenario 1.
- **Href shape** is `/danh-muc/${slug}/` with the trailing slash, matching `CategoryNav` and the
  route map. Satisfies Scenario 2.
- **`<section>` + `aria-labelledby`** mirrors `RelatedProducts` / `RelatedPosts` precedent.
- **`<h2>`** keeps the homepage's single `<h1>` (owned by the US00081 hero) intact.
- **Server Component** — no client state, no event handlers, fully SSG.

## 9. Acceptance criteria (implementer-facing)

- **AC1 (Scenario 1 — category links render):** The homepage contains a `<section>` with `<h2>`
  "Danh mục" and one link per entry in `CATEGORIES`. Each link's visible text equals
  `getCategoryLabels()[slug]`. The number of links rendered equals
  `Object.keys(CATEGORIES).length`.
- **AC2 (Scenario 2 — navigation):** Each link's `href` is `/danh-muc/<slug>/` (trailing slash).
  Clicking a link navigates to that category page.
- **AC3 (Scenario 3 — source of truth):** No file other than `lib/categories.ts` is edited to add a
  new tile. Adding a new category to `CATEGORIES` and rebuilding causes the new tile to appear
  automatically. Verified by the smoke test in §7 step 7.
- **AC4 (SSG / Server Component):** `components/CategoryHighlights.tsx` does not declare
  `"use client"`, performs no `fetch`, and emits no client-side state. `next build` pre-renders `/`
  with the section baked in.
- **AC5 (Theming / a11y):** Section uses `--color-*` and `--spacing-*` tokens only. Link focus has
  a visible `:focus-visible` outline using `--color-primary`. Hover/focus animations honour
  `prefers-reduced-motion: reduce`. Heading hierarchy: page `<h1>` (US00081 hero) → this `<h2>`.

## 11. Open questions for reviewer

1. **Desktop column count vs. forward-compat.** Should the desktop grid pin at `repeat(5, …)` to
   match today's 5-category registry (clean rows now, awkward orphan row when a 6th is added) or use
   `repeat(auto-fit, minmax(180px, 1fr))` (forward-compatible but cells reflow as the registry
   grows)? §5 currently leans toward the explicit `repeat(5, …)` with a guard to revisit when the
   registry changes.
2. **Tile content beyond the name.** F0008 only requires the Vietnamese display name as the link
   label. Should the tile also surface a tiny secondary line (e.g., a 5-word teaser) or an icon /
   thumbnail? Strict reading of US00083: name only. Recommend deferring richer tiles to a follow-up.
3. **Refactoring `CategoryNav` to share with `CategoryHighlights`.** §3 deliberately keeps them
   separate. Confirm we should not pre-emptively unify them under a shared primitive in this PR.
4. **Ordering inside the section.** Tiles render in `Object.keys(CATEGORIES)` (i.e. registry
   insertion) order today. Want an alphabetical sort by Vietnamese display name instead for stable
   visual ordering as the registry grows? Recommend: keep insertion order (predictable for content
   ops, matches `CategoryNav`).
