**Status:** Done
**Feature:** F0011 — Build & Deploy Pipeline
**Spec ref:** `docs/specs/F0011.md` § US00115
**Predecessors:** US00111 (Vercel project connected to repo with `main` as Production branch). US00112/US00113/US00114 are independent and may land in any order.

## 1. Scope

Strictly:

1. Activate **Vercel Speed Insights** for this project so Core Web Vitals (LCP, CLS, INP, TTFB, FCP) are collected on every Production page view and visible in the Vercel dashboard.
2. Use **only Vercel-native instrumentation** — install `@vercel/speed-insights` and mount its `<SpeedInsights />` component inside the App Router root layout (`app/layout.tsx`). No third-party RUM scripts.
3. Enable Speed Insights in the Vercel project settings UI (operator step).
4. Document the dashboard location (URL pattern + navigation steps) in the README so the launch checklist's "Core Web Vitals passing in Vercel dashboard" item is verifiable by anyone with project access.
5. Verify after a real deploy + at least one Production page view that all five Core Web Vitals are reported.

**Explicitly out of scope** (owned by sibling stories or other features):

- Connecting the repo to Vercel / configuring Production branch -> **US00111**.
- Configuring `NEXT_PUBLIC_*` env vars in Vercel -> **US00112**.
- Scheduled rebuild workflow -> **US00113**.
- PR validation CI workflow -> **US00114**.
- GA4 / `useReportWebVitals` self-reporting pipeline -> **F0007** (cookie consent + GA4). Speed Insights is Vercel-managed RUM; GA4 is a separate, deferred analytics integration.
- Performance budgets that fail CI on regression — explicitly out of scope per F0011 §2.
- Custom domain / DNS — deferred to launch prep.

## 6. Step-by-step execution order

1. `git fetch origin && git checkout -b feat/US00115 origin/main`.
2. **Re-read the canonical activation steps** in `node_modules/@vercel/speed-insights/README.md` (it will only exist after step 3) or — if the file is not bundled — at https://vercel.com/docs/speed-insights/quickstart (open-web reference, not Claude training data). 6. Run local checks: `npm run typecheck` and `npm run lint` and `npm run build` — all must exit 0. The build will not actually emit beacons (no Production env), but it must succeed and not log warnings about missing peer deps.
7. Commit on the feature branch:
   `feat(US00115): enable Vercel Speed Insights for Core Web Vitals`
8. Push, open PR, wait for CI (US00114 if landed) to pass.
9. After PR merges to `main` and Vercel auto-deploys (US00111), in the Vercel UI:
   - Navigate to **Project -> Settings -> Speed Insights -> Enable** (if not already enabled at project creation).
10. Open the live Production URL once in a browser; reload to ensure at least one page view is recorded.
11. Wait ~1–5 minutes (per Vercel's documented latency), then open **Project -> Speed Insights** and confirm the five Core Web Vitals chart cards (LCP, CLS, INP, TTFB, FCP) display data for the latest deployment.
12. Run AC verification (Section 7). If everything passes, mark this plan's Status -> Done in a follow-up doc PR.

## 10. Risks / open items

- **Activation snippet drift.** The `@vercel/speed-insights` package may rename its Next.js subpath (`/next` vs `/react`) between versions. Per AGENTS.md, the implementer must read the installed package's README before wiring it up, not rely on this plan's snippet.
- **Project setting must be flipped.** Installing the component is necessary but not sufficient — the project-level toggle in Vercel must be ON. AC Scenario 1 explicitly tests this. If the operator forgets, the dashboard stays empty even though code is correct.
- **No data until live traffic.** Scenario 2 requires "a Production deploy has been live for at least one page view". On a brand-new project with no traffic, the implementer must self-visit the live URL once to seed a metric. Document this in the verification step (already noted in Section 6 step 10).
- **Bundle-size delta.** Roughly 1–2 KB gzipped per Vercel's docs. If a future performance story sets a budget, this needs to be in scope of that budget; for now, F0011 §2 explicitly excludes performance budgets.
- **Privacy / cookie consent.** Speed Insights does not set tracking cookies and does not collect personally identifying information per Vercel's docs — therefore it is **not** gated by F0007's cookie banner. Confirm this against the package privacy docs at implementation time; if Vercel changes the privacy posture, raise a follow-up story to gate it.
- **AdBlocker noise.** Some browser extensions block `_vercel/insights` endpoints. This is acceptable RUM noise — sample sizes will still be representative. No mitigation.
- **F0007 collision.** When F0007 lands GA4 + `useReportWebVitals`, both pipelines will report Web Vitals (Vercel dashboard + GA4 events). They are independent and can coexist. Plan for that future story to acknowledge the dual-reporting setup.
- **US00111 not yet landed.** If the Vercel project is not connected when this story is implemented, scenarios 1 and 2 cannot be verified. Implementer should pause and surface the blocker rather than half-shipping.
