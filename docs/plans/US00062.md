## 1. Story

> **As a** content writer, **I want** my MDX article body rendered with rich formatting and
> embeddable components, **so that** posts display headings, tables, lists, images, and affiliate
> cards correctly.

Source: `docs/specs/F0006.md` → `### US00062` (Scenarios 1–5), plus §2 In-Scope bullet 3, §5 Edge
Cases (post with no headings, missing/broken image path), §6 Technical Considerations (MDX
rendering paragraph).

This is the **MDX rendering engine** of the blog. It does **not** build the post page chrome
(US00066), the TOC UI (US00068), or the `<ProductCard slug>` lookup body (US00063) — but it MUST
expose the seams those stories plug into:
- a reusable `<PostBody content={...} />` Server Component (consumed by US00066),
- stable `h2`/`h3` `id` anchors (consumed by US00068 + hash deep-links),
- a component map that US00063 extends with `<ProductCard slug>`.

## 2. Decision: how we render the body

### Constraint
`lib/posts.ts` already uses **`gray-matter`** to split frontmatter from body and returns the body
as a **plain string** in `Post.content`. The native `@next/mdx` file-routing / `import Post from
'…mdx'` flow compiles the *whole file* through webpack and would (a) re-parse frontmatter as stray
MDX and (b) bypass the validated `getAllPosts()` loader. So we cannot use import-based rendering
for the body — we must compile the **string** at build time.

### Chosen approach — compile the string with `@mdx-js/mdx` (`evaluate`)
`@mdx-js/mdx@3.1.1` is **already installed**. We render `Post.content` inside a Server Component
using `evaluate()` (async, build-time under SSG), supplying the JSX runtime, the component map, and
the remark/rehype plugins. This is the officially documented "Deep Dive" path (`mdx.md` lines
762–791) for programmatic MDX where you don't own the file-routing pipeline.

```tsx
// components/PostBody.tsx  (Server Component, async)
import * as runtime from "react/jsx-runtime";
import { evaluate } from "@mdx-js/mdx";
import remarkGfm from "remark-gfm";
import { rehypeHeadingSlugs } from "@/lib/mdx-slug"; // shared chokepoint (see §4a)
import { getMdxComponents } from "@/components/mdx/mdx-components"; // shared map (see §4)

export async function PostBody({ content }: { content: string }) {
  const { default: MDXContent } = await evaluate(content, {
    ...runtime,
    remarkPlugins: [remarkGfm],
    rehypePlugins: [rehypeHeadingSlugs],
    useMDXComponents: getMdxComponents,
  });
  return <div className={styles.prose}><MDXContent /></div>;
}
```

Why `evaluate` over the alternatives:
- **vs. native `@next/mdx` import** — rejected: needs the raw file (frontmatter collides) and
  breaks the validated loader chokepoint.
- **vs. `next-mdx-remote`** — rejected: not installed, redundant given `@mdx-js/mdx` is present,
  and heavier.
- **vs. `compile()` + manual `run()`** — `evaluate()` is `compile`+`run` in one call; we prefer it
  for the build-time SSG path. (If a future RSC/serialization wrinkle appears, fall back to
  `compile()` to a string cached per-slug; noted as open question OQ-2.)

`remark-frontmatter` stays in `next.config.ts` for any future `.mdx`-as-page use, but the body
string handed to `evaluate()` has **already** had frontmatter stripped by gray-matter, so it is not
needed in the `evaluate` plugin list.

> Per `AGENTS.md`: confirmed against `node_modules/next/dist/docs/01-app/02-guides/mdx.md`
> (custom components, plugin-as-string for Turbopack, the remark/rehype deep dive) and
> `…/file-conventions/mdx-components.md`. Re-read before coding.

## 4. Component map (the seam US00063 extends)

Today `mdx-components.tsx` (root) is a placeholder exporting `useMDXComponents()` returning `{}`.
That file is the `@next/mdx` *file-convention* hook — but our `evaluate()` path passes its own
`useMDXComponents`. To avoid two diverging sources of truth, introduce a **single shared map** at
`components/mdx/mdx-components.tsx` (a new `components/mdx/` subfolder, keeping the existing
"`lib/` stays React-free" convention intact) and have **both** the root `mdx-components.tsx` and
`PostBody` consume it.

The map provides:
- **`img`** → wraps **`next/image`** (Scenario 2 — never raw `<img>`). Because markdown `![]()`
  gives no intrinsic dimensions, use `<Image fill>` inside a styled aspect wrapper, or a sensible
  default `width`/`height` with `sizes`; missing/broken `src` must degrade gracefully (Edge Case)
  — render nothing rather than crash. Decision D-IMG below.
- **`h2` / `h3`** (and `h1`/`h4`) → thin wrappers applying `.module.css` theme typography. **Do not
  hand-roll `id`s here** — the shared slugger in `lib/mdx-slug.ts` (see §4a) owns anchoring so it
  stays consistent with US00068's TOC. Optionally render a hover anchor link (`#`) but keep that
  minimal/deferred.
- **`table` / `thead` / `th` / `td`** → theme-styled (scrollable wrapper on mobile).
- **`ul` / `ol` / `li`, `blockquote`, `pre` / `code`, `a`** → theme typography + `<a>` gets
  `rel="noopener"` for external; **affiliate URLs are NOT allowed as raw anchors** (CLAUDE.md
  rule) — writers must use `<ProductCard slug>` (US00063). A lint/build note covers this.
- **`ProductCard`** → **placeholder/stub in this story**, real implementation in US00063. Register
  the key so the map's shape is stable; US00063 swaps the stub for the build-time
  `getProductBySlug()` lookup card. (Coordinate so US00063 only edits `components/mdx/mdx-components.tsx`.)

> D-IMG: use `<Image fill>` inside a styled aspect-ratio wrapper. Markdown `![]()` carries no
> intrinsic dimensions, so requiring writers to add width/height in raw MDX is fragile; the
> aspect wrapper is one-time CSS in `PostBody.module.css`. Sample fixtures use existing image
> paths under `static/images/blog/` (or `/static/images/products/*`) so `next build` doesn't fail
> on a missing file. If a writer omits/breaks an image the `img` mapper guards (`if (!src) return
> null`).

## 4a. Heading-slug chokepoint (shared with US00068)

`rehype-slug` uses `github-slugger` internally and is what we'd reach for by default — but it
leaves the slugging algorithm implicit, which means US00068's TOC builder has to trust that its
own `github-slugger` call matches the rehype plugin's output. That's a silent-drift surface.

**Decision:** add a tiny shared chokepoint, mirroring the `formatVnd` / `formatPostDate` /
`assertAffiliateUrl` pattern.

- Create `lib/mdx-slug.ts` exporting `createHeadingSlugger(): { slug(text: string): string }` —
  encapsulates a fresh `github-slugger` instance **per document** so dedupe state doesn't leak
  across posts.
- Replace `rehype-slug` with a small custom rehype plugin (≈15 LoC) `rehypeHeadingSlugs` (lives in
  `lib/mdx-slug.ts`) that walks `h1`–`h6` nodes and assigns `id` via `createHeadingSlugger()`.
- US00062 uses `rehypeHeadingSlugs` in `PostBody`'s `evaluate()` plugin list.
- US00068's TOC builder imports `createHeadingSlugger` from the same file. Identical output by
  construction (same function, same input order).

**Convention rule to document** (CLAUDE.md): *"Heading slugs for MDX bodies come from
`lib/mdx-slug.ts`. No other file may call `github-slugger` or hand-roll heading slugs."*

Dependency swap: **drop `rehype-slug`, add `github-slugger`** (the engine for the chokepoint).
`github-slugger` is the smaller, lower-level package and is already a transitive dep of much of
the rehype ecosystem.

   aspect wrapper styled in `PostBody.module.css`), heading/table/list/code/anchor wrappers, and a

   wrapper).

## 11. Resolved decisions

- **Map location → `components/mdx/mdx-components.tsx`** (B1). Preserves the existing
  *"`lib/` … (no React)"* convention by keeping JSX out of `lib/`; introduces one new subfolder
  `components/mdx/`.
- **`evaluate()` chosen** (B2). SSG renders each post once at build; caching `compile()` output
  adds plumbing for no runtime gain.
- **Syntax highlighting deferred** (B3). Code blocks render as plain semantic `<pre><code>`.
  Revisit `rehype-pretty-code` as a follow-up only when a post actually needs it.
- **`img` mapper → `<Image fill>` + aspect wrapper** (B4). Markdown `![]()` carries no intrinsic
  dimensions; requiring writers to author width/height is fragile. The wrapper is one-time CSS in
  `PostBody.module.css`.
- **Heading-slug chokepoint → `lib/mdx-slug.ts`** (A1). Drop `rehype-slug`, wrap `github-slugger`
  directly so US00062's rehype plugin and US00068's TOC builder share one source of truth (see §4a).
- **MDX component map ownership** (A2). US00062 ships the map with a `ProductCard` stub; US00063
  swaps the stub for the real build-time `getProductBySlug()` adapter. Lets US00062 verify
  fixtures end-to-end without waiting on US00063.
