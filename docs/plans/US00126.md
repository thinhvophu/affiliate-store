# Implementation Plan — US00126: Blog-post MDX scaffold for content-gap categories

**Status:** Draft
**Feature:** F0012 — Automated Content Ingestion Pipeline
**Spec ref:** `docs/specs/F0012.md` § US00126
**Predecessors:** **US00121** (shares the `scripts/` dir, `tsx` runner, and fatal-error style). Depends conceptually on products existing in `content/products/*.json` (ingested by US00124/US00125) so the named product slugs resolve — but the scaffold command itself only needs `getProductBySlug`/`getPostBySlug` and the slugifier.
**Successors:** none.

---

## 1. Scope

Strictly:

1. Add a **separate CLI**, `scripts/scaffold-post.ts`, wired as `npm run scaffold:post`, that writes an MDX **stub** for a content-gap category.
2. Inputs: a `--category` slug (registered) and **1–2 already-ingested product slugs** (`--products=slug-a,slug-b`), plus an optional `--slug`/`--title` for the post.
3. Output `content/posts/<slug>.mdx` with:
   - **Frontmatter**: `title` placeholder, `category` (validated via `assertCategoryRegistered`), `publishedAt` = **today**, `tags: []`, `summary` placeholder, `coverImage` sourced from **one of the named products' images**.
   - **Body**: one `<ProductCard slug="…" />` per named product; prose left for a human (empty/placeholder body is valid MDX).
4. It must **pass `next build` as-is** (Scenario 1) — frontmatter satisfies `lib/posts.ts` validation; embeds resolve via `MdxProductCard`.
5. **Cover images not reused across scaffolds** (Scenario 2): two runs naming different products get **different** `coverImage` values (each from its own products).
6. **Unknown product slug fails loudly** (Scenario 3): exit non-zero naming the unknown slug **before** writing any `.mdx`.

**Explicitly out of scope:**

- Article prose generation (spec §2 Out of Scope) — a human writes the body.
- Creating products — the named slugs must already exist (ingested by US00124/US00125).
- Auto-publishing/committing.

---

## 2. Pre-conditions / current state

- `lib/posts.ts::validatePostFrontmatter` requires non-empty strings: `title, summary, publishedAt, category, coverImage`, and `tags` must be an array. → The stub must fill all of these (placeholders for `title`/`summary` are fine as long as non-empty).
- `lib/posts.ts` calls `assertCategoryRegistered(category, slug)` per post at build → the scaffold's `--category` must be registered or `next build` fails; validate up front to fail before writing.
- MDX embeds: authors type `<ProductCard slug="…" />`; the map key resolves to `MdxProductCard` which calls `getProductBySlug` at build and **throws on unknown slug**. So a scaffold naming a bad slug would break `next build` — US00126 pre-checks with `getProductBySlug` to fail *at scaffold time* with a clear message (Scenario 3), not defer to build.
- `coverImage` in existing posts points at a product image path (e.g. `/static/images/products/keychron-k2-v2.jpg`) — the scaffold sources it the same way (from a named product's `images[0]`).
- `content/posts/*.mdx` slugs = filename; must be unique (build enforces).
- Reuse `slugifyProductName` from US00122? The **post slug** derives from `--title`/`--slug`; reuse the same slugifier (`scripts/ingest/slug.ts`) for consistency, or accept an explicit `--slug`.

## 3. Approved Decisions

| # | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| D1 | Separate entry point | New `scripts/scaffold-post.ts` + `npm run scaffold:post` (distinct from `ingest:products`) | Spec lists them as two commands; different inputs/outputs (posts vs products). |
| D2 | Product-slug input | `--products=slug-a,slug-b` (1–2 slugs); each **pre-validated** with `getProductBySlug` before any write | Scenario 3: unknown slug → exit non-zero naming it, before writing. |
| D3 | `coverImage` source | `images[0]` of the **first** named product (deterministic) | Scenario 2: since different runs name different products, their covers differ by construction. Ties cover to real, staged, local imagery. |
| D4 | Post slug | From `--slug` if given, else `slugifyProductName(--title)`; else derive from category + first product; fatal if the resulting `content/posts/<slug>.mdx` already exists (never overwrite) | Consistency with US00122's slug rules + no silent clobber of an existing post. |
| D5 | Frontmatter placeholders | `title: "TODO: …"`, `summary: "TODO: …"` (non-empty so validation passes), `tags: []`, `publishedAt` = `isoToday()` | Scenario 1: must build as-is; human fills real copy later. `tags: []` per spec. |
| D6 | Category validation | Call `assertCategoryRegistered(category, slug)` up front; fatal on failure | Mirror the build guard; fail before writing. |
| D7 | Body content | For each product: `<ProductCard slug="<slug>" />` on its own line, plus a `{/* TODO: viết nội dung bài viết */}` comment | Human writes prose; embeds already wired. Empty prose is valid MDX. |
| D8 | Cover-reuse guard (optional hardening) | Beyond D3's per-run determinism, optionally warn if the chosen `coverImage` already appears as another post's `coverImage` | Scenario 2's intent is "avoid one product's photo across every post." D3 satisfies the letter; the warning satisfies the spirit if an operator reuses the same lead product. |
| D9 | gray-matter for writing | Hand-write the frontmatter block (simple, deterministic) rather than round-tripping `gray-matter.stringify` | The stub is small and fixed; a literal template is clearer and avoids quoting surprises with Vietnamese text. (Reads still go through `gray-matter` in `lib/posts.ts`.) |

### 4.3 Template sketch

```ts
// scripts/scaffold/template.ts
export function renderPostStub(i: {
  title: string; summary: string; category: string;
  publishedAt: string; coverImage: string; productSlugs: string[];
}): string {
  const fm = [
    "---",
    `title: ${JSON.stringify(i.title)}`,
    `summary: ${JSON.stringify(i.summary)}`,
    `publishedAt: ${JSON.stringify(i.publishedAt)}`,
    `category: ${JSON.stringify(i.category)}`,
    `tags: []`,
    `coverImage: ${JSON.stringify(i.coverImage)}`,
    "---",
  ].join("\n");
  const body = [
    "",
    "{/* TODO: viết phần mở bài và nội dung đánh giá */}",
    "",
    ...i.productSlugs.map((s) => `<ProductCard slug="${s}" />`),
    "",
  ].join("\n");
  return fm + "\n" + body;
}
```

### 4.4 Entry sketch

```ts
// scripts/scaffold-post.ts (core)
assertCategoryRegistered(category, "<scaffold>");                 // D6 fatal
const products = productSlugs.map((s) => {
  const p = getProductBySlug(s);
  if (!p) throw new FatalError(`scaffold: unknown product slug "${s}" — ingest it first.`); // D2/Scenario 3
  return p;
});
const postSlug = slug ?? slugifyProductName(title ?? `${category}-${products[0].slug}`);    // D4
const dest = path.join("content", "posts", `${postSlug}.mdx`);
if (fs.existsSync(dest)) throw new FatalError(`scaffold: ${dest} already exists — refusing to overwrite.`);
const coverImage = products[0].images[0];                        // D3 → distinct per run (Scenario 2)
fs.writeFileSync(dest, renderPostStub({ ...placeholders, category, publishedAt: isoToday(), coverImage, productSlugs }));
```

## 9. Risks / open items

- **R1 — Cover-reuse spirit (Scenario 2).** D3 guarantees *per-run* distinct covers only because different runs name different products. If an operator names the **same** lead product twice, covers would match — D8's optional warning covers that; decide whether to make it a hard block (probably a warning is enough, since the operator is reviewing).
- **R2 — Placeholder frontmatter shipping to prod.** A `title: "TODO: …"` stub that gets committed+deployed unfinished would surface placeholder copy publicly. Mitigate: README states the stub is a draft; consider a `draft: true`-style convention — **out of scope** here (no draft support exists), so rely on human review before publish.
- **R3 — Empty-body MDX validity.** Confirm `@mdx-js/mdx evaluate()` (via `PostBody`) accepts a body that is only a comment + embeds. If it needs at least one prose node, add a placeholder paragraph. Verify in step 7 (`next build` + render).
- **R4 — Post slug collision with a product slug.** Namespaces differ (`content/posts/` vs `content/products/`), so no real collision; just ensure the post-file existence check targets `content/posts/`.
