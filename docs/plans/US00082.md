## 1. Scope

Strictly:

1. Add a **Server Component** at `components/FeaturedProducts.tsx` that renders the "Sản phẩm nổi bật" section on the homepage. Single prop: `{ products: Product[] }` — the caller (`app/page.tsx`) is responsible for sourcing and capping the array. Inside the component:
   - A `<section aria-labelledby="featured-products-heading">` wrapper.
   - A header row containing the `<h2 id="featured-products-heading">Sản phẩm nổi bật</h2>` and a "Xem tất cả" `<Link href="/san-pham/">` link, visually arranged as a heading-left / link-right row at tablet+ and stacking at mobile.
   - A responsive grid (2 cols mobile / 3 tablet / 4 desktop at the project F0001 breakpoints — 768 px and 1280 px) of `<ProductCard product={p} />`s, one per array entry.
   - **Empty state:** if `products.length === 0`, render a single Vietnamese paragraph `<p>Hiện chưa có sản phẩm nổi bật.</p>` inside the section AND still show the "Xem tất cả" CTA pointing at `/san-pham/`. The section is **not** unmounted (Scenario 4 permits either "empty message" OR "section omitted entirely"; this plan chooses the visible message variant so the page has stable visual rhythm even with zero featured products — see §3 D2).
2. Add a co-located CSS Module `components/FeaturedProducts.module.css` containing only the section chrome (heading row, grid, empty-state paragraph). Card-internal layout is owned by `ProductCard.module.css`.
3. Add a small build-time **filtering + capping helper** at the top of the component file (or inline in `app/page.tsx` — see §3 D5 for the decision):
   - `MAX_FEATURED_PRODUCTS = 8` exported as a named constant (Technical Considerations §6 of F0008 mandates "explicit constants, not magic numbers").
   - The filter applies `products.filter((p) => p.featured)` and slices to `MAX_FEATURED_PRODUCTS`. Order is whatever `getAllProducts()` already returns (publishedAt desc, then slug asc on tie) — deterministic per Scenario 3.
4. **Wire into the homepage** (`app/page.tsx`, created by US00081): call `getAllProducts()` once at the top of the Server Component, derive the featured array, pass it to `<FeaturedProducts products={featured} />` as one of the homepage sections.
5. **No new data loader.** `getAllProducts()` from `lib/products.ts` is the sole source. No new `getFeaturedProducts()` helper — the filter is trivial and adding a wrapper would just be indirection.
6. **No new card variant.** Reuse `components/ProductCard.tsx` exactly as-is. No prop additions, no new className paths.
7. **No raw affiliate anchors.** Affiliate semantics inherited from `<ProductCard />` via `<AffiliateLink>`. This story does not import `<AffiliateLink>` directly.
8. **Pure Server Component.** No `"use client"`, no hooks, no event handlers, no client-side data fetching. Zero hydration cost. SSG-only — runs once at `next build`.

**Explicitly out of scope** (owned by sibling stories or future features):

- **Homepage shell, hero/intro, baseline metadata** — **US00081**. This story only contributes one section; it does not author the page heading, hero block, `metadata` export, or layout shell.
- **Category highlights section** — **US00083**.
- **Latest blog posts strip** — **US00084**.
- **JSON-LD `Organization` / `WebSite` / `ItemList` schema on the homepage** — deferred to **F0009**. The product cards do not emit per-card JSON-LD here; the listing surface is informational only.
- **Open Graph / `og:image`** on `/` — **F0009**.
- **Personalization / ranking algorithms** — out of F0008 scope entirely. Featured ordering is `getAllProducts()` order, deterministic, build-time only.
- **A `featured` admin UI** — there is no admin; flipping `featured: true` in product JSON and pushing to `main` is the publishing flow.
- **Carousels / auto-rotating sliders** — F0008 §2 In Scope explicitly forbids them.
- **GA4 `affiliate_click` emission** — owned by F0007, already seamed via `<AffiliateLink>` data-attrs from US00033.
- **Sitemap inclusion** — `/` lands in `app/sitemap.ts` under F0009.

## 5. Concrete diff sketch

### 5.1 `components/FeaturedProducts.tsx` — new file

```tsx
import Link from "next/link";
import type { Product } from "@/types";
import { ProductCard } from "@/components/ProductCard";
import styles from "./FeaturedProducts.module.css";

/**
 * FeaturedProducts — F0008 / US00082
 *
 * Server Component. Renders the "Sản phẩm nổi bật" section on the homepage.
 * Accepts a pre-filtered, pre-capped Product[] from `app/page.tsx`. The cap
 * (`MAX_FEATURED_PRODUCTS`) is exported so the caller (and tests) can share it.
 *
 * Composition contract:
 *   - The caller derives the array via `getAllProducts().filter(p => p.featured).slice(0, MAX_FEATURED_PRODUCTS)`.
 *   - This component is a pure presenter — it never calls getAllProducts() itself.
 *
 * Empty state: when products.length === 0, renders a Vietnamese empty paragraph
 * and STILL shows the "Xem tất cả" CTA pointing at /san-pham/. The section is
 * not unmounted so the homepage rhythm stays stable (F0008 §4 Scenario 4 —
 * either-or; this implementation chose the visible-empty variant).
 *
 * No new data layer, no new card variant — reuses <ProductCard> (F0004 / US00042)
 * and inherits F0003 affiliate-link semantics through it.
 */

export const MAX_FEATURED_PRODUCTS = 8;

export interface FeaturedProductsProps {
  products: Product[];
}

export function FeaturedProducts({ products }: FeaturedProductsProps) {
  const hasProducts = products.length > 0;

  return (
    <section
      className={styles.section}
      aria-labelledby="featured-products-heading"
    >
      <header className={styles.header}>
        <h2 id="featured-products-heading" className={styles.heading}>
          Sản phẩm nổi bật
        </h2>
        <Link href="/san-pham/" className={styles.viewAll}>
          Xem tất cả
        </Link>
      </header>

      {hasProducts ? (
        <div className={styles.grid}>
          {products.map((product) => (
            <ProductCard key={product.slug} product={product} />
          ))}
        </div>
      ) : (
        <p className={styles.empty}>Hiện chưa có sản phẩm nổi bật.</p>
      )}
    </section>
  );
}
```

Notes on the JSX:

- **`Link` from `next/link`** for the "Xem tất cả" link — internal-route prefetch is desirable here (the catalog is the natural next destination).
- **`<header>`** wraps the heading + CTA row purely for layout grouping; it is a `<header>` element but lives inside a `<section>` so it is the section-level header per HTML5 spec (no accessibility-name conflict).
- **`key={product.slug}`** — slugs are unique by build-time invariant (`lib/products.ts` Map dedupe at lines 104–113). Index keys would be brittle if a category-edit reorders the list.
- **No `aria-label` on the grid** — the section's `aria-labelledby` already provides the accessible name; an extra `aria-label` on the inner `<div>` would be redundant.
- **The empty paragraph also lives inside the same `<section>`** — preserves the section's heading + CTA so screen-reader users still hear "Sản phẩm nổi bật, Xem tất cả link, Hiện chưa có sản phẩm nổi bật" in a coherent block.

## 10. Risks / open items

- **R1 — Dependency on US00081.** This story's `app/page.tsx` edit assumes US00081 has shipped the real homepage shell. If US00081 lands after this story, the wiring edit (§5.3) will conflict with the stub `<h1>aff-store</h1>` or with US00081's hero markup. **Mitigation:** verify US00081 status in step 1 of §6. Sequence the merges (US00081 first). If unavoidable, this story can ship the `<FeaturedProducts />` component standalone in one PR and follow up with a separate "wire into app/page.tsx" PR after US00081 lands.
- **R2 — Empty state choice.** F0008 §4 Scenario 4 explicitly permits either "render Vietnamese empty message" or "omit section entirely". This plan chose the visible empty variant (D2) for layout stability. If review prefers the omit-section variant, the change is one line: replace `return (<section …>…</section>)` with `if (!hasProducts) return null;` at the top of `FeaturedProducts`. Tracked as Q1 below.
- **R3 — Existing typo `--space-xl` in `RelatedProducts.module.css`.** While checking the precedent, this plan noticed `RelatedProducts.module.css` references `--space-xl` and `--space-md`, but the actual tokens in `app/globals.css` are named `--spacing-xl` / `--spacing-md`. The browser falls back to nothing for the undefined values, so visual layout currently "works" by coincidence. **Not introduced by this story.** This story uses the correct `--spacing-*` names. Tracked for a separate cleanup PR (or fold into a future F0008 polish story).
- **R4 — Filename ordering depends on POSIX `readdirSync` behavior.** `lib/products.ts` calls `fs.readdirSync(PRODUCTS_DIR)` without an explicit sort. On Linux/macOS, the typical result is alphabetical, but POSIX does not guarantee this — `readdir(3)` can return entries in any order. In practice Vercel's build images return sorted entries, so the existing site is stable; this story inherits that assumption. **Mitigation:** if a future story needs guaranteed determinism, add `.sort()` after `readdirSync` in `lib/products.ts`. Not a blocker for this story; the `publishedAt`-desc sort plus the production fixtures' distinct dates make the order stable.
- **R5 — Featured products and recency can disagree.** `getAllProducts()` sort is `publishedAt` desc. A product flagged `featured: true` but with an old publishedAt will sort to the bottom. If the cap of 8 is reached, that old-but-featured product may drop off. **Mitigation:** editorial control — the content owner should publish a fresh `publishedAt` for featured items if they want them surfaced. Documented in this plan; no code change. Q3 below opens it for review.
- **R6 — Second `getAllProducts()` call from `app/page.tsx`.** If US00081 / US00083 also call `getAllProducts()` (e.g., to count categories), the build does multiple disk reads. Cheap at this catalog size (25 fixtures) but architecturally noisy. **Mitigation:** if multiple sections need the array, call once at the top of `app/page.tsx` and pass the array to each section as a prop. Same pattern as `<RelatedProducts />` accepting a pre-filtered array. Coordinate with US00081/83/84 in the PR.
- **R7 — JSON-LD `ItemList` for the featured strip.** F0009 will own SEO schema for the homepage. If a future reviewer asks for `ItemList` JSON-LD here, do it in F0009, not in this story. Documented to prevent scope creep.
- **R8 — Accessibility: section heading + "Xem tất cả" pairing on screen readers.** VoiceOver / NVDA traversal order: heading → link → grid. Some screen-reader users may expect the link to be at the end of the section ("View all" after the items). The chosen layout (link in the header row) matches the visual hierarchy. If user testing reveals an issue, move the link to a footer `<a>` after the grid in a follow-up. Documented; not a blocker.

### Open questions for the reviewer

- **Q1 — Empty state: visible message or full omission?** This plan chose visible empty (D2) for layout stability. The spec accepts either. **Recommendation:** ship as visible empty; revisit if the homepage feels broken with a permanently empty section in some hypothetical state.
- **Q2 — Filter inside the component or at the caller?** D4 places `filter + slice` at `app/page.tsx`. Alternative: accept the full `getAllProducts()` array as a prop and filter inside `<FeaturedProducts />`. **Recommendation:** keep filtering at the caller (matches `<RelatedProducts />` precedent and keeps `<FeaturedProducts />` a pure presenter). If reviewer prefers the component to do the filtering, change the prop signature to `{ allProducts: Product[] }` and apply the filter internally — single-file refactor.
- **Q3 — Featured + recency disagreement.** If a product is `featured: true` but old, it can fall off the 8-cap. Acceptable? Or should the sort prefer featured items first? **Recommendation:** keep `publishedAt` desc as the only sort (editorial controls freshness by re-dating). Flagging an item without keeping its date fresh is an editorial choice, not a code concern.
- **Q4 — Visual divider between sections.** D15 uses padding on the section. Some homepages use a hairline (`border-block-start: 1px solid var(--color-border-hairline)`) between sections. **Recommendation:** ship without dividers in v1 (matches the project's minimalist tone). Easy to add in F0008 polish if needed.
