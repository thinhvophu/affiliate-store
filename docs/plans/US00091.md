# Implementation Plan: US00091 — Global metadata foundation

**Story:** US00091 (Feature F0009 — SEO & Discoverability)
**Status:** Done

## 1. Goal

Establish the **root-level metadata foundation** so every page on the affiliate storefront inherits:

- A correct absolute base URL (`metadataBase`) sourced from `NEXT_PUBLIC_SITE_URL` via the existing `lib/env.ts` helper.
- A consistent title template (`"%s | <SITE_NAME>"`) and default title.
- A default Vietnamese site description.
- Default Open Graph metadata (`type: "website"`, `siteName`, `locale: "vi_VN"`) with a default OG image at `/static/images/og-default.png`.
- Default Twitter card (`summary_large_image`).
- A small shared SEO helper (`lib/seo.ts`) that sibling stories (US00092, US00099) will reuse to build per-page metadata and resolve canonical paths consistently.

US00091 **owns the shared seams** the rest of F0009 depends on. The plan must establish them cleanly with no per-page boilerplate left to maintain.

## 5. Design — `lib/seo.ts`

```ts
// lib/seo.ts — no React, no JSX. Server-side and build-time only.
import type { Metadata } from "next";
import { env } from "@/lib/env";
import { SITE_NAME } from "@/lib/site";

// Validated absolute base URL, exposed as a getter so consumers can't typo it.
// Returns a string (not URL) for ergonomic concatenation in callers that need it
// (e.g., JSON-LD in US00094/US00095). For metadataBase we wrap in `new URL(...)`.
export function getSiteUrl(): string {
  return env.NEXT_PUBLIC_SITE_URL.replace(/\/+$/, ""); // strip trailing slash
}

/**
 * Compose an absolute URL from a path. Used by JSON-LD / sitemap / robots —
 * for Metadata fields, prefer the relative form which Next resolves via metadataBase.
 */
export function absoluteUrl(path: string): string {
  const base = getSiteUrl();
  const suffix = path.startsWith("/") ? path : `/${path}`;
  return `${base}${suffix}`;
}

/**
 * Build a canonical pathname for a route. Encodes the project's slug-and-trailing-slash
 * convention in one place. Returns a relative path (e.g. "/san-pham/abc/") that
 * per-page `alternates.canonical` can pass straight through.
 */
export function buildCanonicalPath(segments: ReadonlyArray<string>): string {
  if (segments.length === 0) return "/";
  const joined = segments
    .map((s) => s.replace(/^\/+|\/+$/g, ""))
    .filter(Boolean)
    .join("/");
  return `/${joined}/`;
}

export const DEFAULT_OG_IMAGE = "/static/images/og-default.png";
export const DEFAULT_OG_LOCALE = "vi_VN";

const DEFAULT_DESCRIPTION_VI =
  "Đánh giá, hướng dẫn mua và liên kết Shopee cho chuột, bàn phím, tai nghe gaming & gadget công nghệ — chọn nhanh sản phẩm tốt, đọc bài viết bằng tiếng Việt.";

/**
 * Root-layout metadata. Centralized here so the layout file stays declarative.
 */
export function buildRootMetadata(): Metadata {
  return {
    metadataBase: new URL(getSiteUrl()),
    title: {
      template: `%s | ${SITE_NAME}`,
      default: `${SITE_NAME} — Đồ chơi công nghệ & gaming Việt Nam`,
    },
    description: DEFAULT_DESCRIPTION_VI,
    openGraph: {
      type: "website",
      siteName: SITE_NAME,
      locale: DEFAULT_OG_LOCALE,
      images: [
        {
          url: DEFAULT_OG_IMAGE, // relative — resolved by metadataBase
          width: 1200,
          height: 630,
          alt: SITE_NAME,
        },
      ],
    },
    twitter: {
      card: "summary_large_image",
    },
  };
}

/**
 * Per-page metadata builder. Used by US00092 to standardize per-page exports.
 * Kept minimal — full per-page audit is US00092; this signature is the contract
 * sibling stories will consume.
 */
export interface BuildPageMetadataOpts {
  title: string;            // becomes `<title>` after template prefix
  description: string;      // ≤160 chars (caller is responsible; helper does not truncate)
  path: string;             // canonical relative path, e.g. "/san-pham/abc/"
  image?: string;           // absolute or relative path; omit → default OG image
  type?: "website" | "article" | "product";
  publishedTime?: string;   // ISO; for type === "article"
}

export function buildPageMetadata(opts: BuildPageMetadataOpts): Metadata {
  const { title, description, path, image, type = "website", publishedTime } = opts;
  return {
    title,
    description,
    alternates: { canonical: path },
    openGraph: {
      title,
      description,
      url: path,
      type,
      ...(publishedTime ? { publishedTime } : {}),
      ...(image ? { images: [{ url: image }] } : {}),
    },
    twitter: {
      title,
      description,
      ...(image ? { images: [image] } : {}),
    },
  };
}
```

**Design notes / decisions:**

- **Why a `buildRootMetadata()` function instead of inlining in `app/layout.tsx`?** Keeps `lib/seo.ts` as the single chokepoint and makes the root-defaults shape unit-testable. The layout becomes a one-liner: `export const metadata = buildRootMetadata();`.
- **Why expose both `absoluteUrl()` and `buildCanonicalPath()`?** `absoluteUrl()` is for JSON-LD and the sitemap (US00094–US00097) where absolute URLs are required. `buildCanonicalPath()` returns a relative path because Next.js resolves it against `metadataBase` — this avoids the deprecated `${process.env.NEXT_PUBLIC_SITE_URL}` pattern (per F0009 § 6 "Single source of truth for the base URL").
- **Why does `buildPageMetadata` not enforce ≤160 chars on description?** Truncation is a US00092 concern (the per-page audit will add a `truncateDescriptionVi()` helper at word boundary in the same `lib/seo.ts` file). Keeping it out of US00091 prevents scope creep.
- **`metadataBase` accepts `URL`, not string.** Next.js requires `new URL(...)`. We wrap `getSiteUrl()` once at construction time.
- **`DEFAULT_OG_IMAGE` is a relative path.** When `metadataBase` is set, Next.js resolves `/static/images/og-default.png` to `${base}/static/images/og-default.png` automatically — and per the docs, _"If a `metadata` field provides an absolute URL, `metadataBase` will be ignored,"_ which is exactly what we want for fall-throughs.

## Resolved cross-cutting decisions (F0009 review — 2026-06-21)

These are owner-approved and supersede the relevant "Open questions" above.

- **Trailing-slash policy:** every canonical / OG / JSON-LD / sitemap URL ends with a **trailing slash** (e.g. `/san-pham/`, `/danh-muc/chuot-gaming/`), except the root `/` and file routes (`/sitemap.xml`, `/robots.txt`). `buildCanonicalPath` / `absoluteUrl` in `lib/seo.ts` must normalize to exactly one trailing slash.
- **`absoluteUrl()` is THE single chokepoint** for composing absolute URLs across F0009. JSON-LD stories (US00094/US00095/US00096) and the sitemap (US00097) MUST import it — no re-deriving from `process.env` / `env` in those files.
- **SITE_NAME stays `"aff-store"`** for v1 (placeholder accepted). Title template + JSON-LD publisher/author use it as-is; swapping the brand later is a one-line `lib/site.ts` change.
- **Default OG image:** ship a real placeholder PNG ≥1200×630 at `/static/images/og-default.png` now; it can be replaced later with no code change.

## 13. Implementation notes (post-merge)

- **Title-doubling bug found and fixed in this PR.** Six existing pages (`/ve-chung-toi/`, `/danh-muc/[category]/`, `/chinh-sach-bao-mat/`, `/cong-bo-tiep-thi-lien-ket/`, `/san-pham/`, `/bai-viet/`) already baked `" | aff-store"` (or the stale `" | TechShop"`) directly into their `title` string. Adding the root `title.template: "%s | aff-store"` would have applied the template to those strings too, doubling the suffix in production (e.g. `"Về chúng tôi | aff-store | aff-store"`). User decision: strip the redundant suffix from those six `title` strings in this same PR (one-line edit each, no other metadata fields touched) rather than ship the regression and wait for US00092. `/san-pham/[slug]/` and `/bai-viet/[slug]/` already set bare titles (no baked suffix) and needed no change.
- **`BuildPageMetadataOpts.type` narrowed to `"website" | "article"`** (dropped `"product"` from the plan's §5 draft) — Next.js's typed `Metadata["openGraph"]["type"]` union has no `"product"` literal (confirmed against `node_modules/next/dist/lib/metadata/types/opengraph-types.d.ts`); the original signature did not compile under `tsc --noEmit`.
- **`vitest.config.ts` gained `test.env.NEXT_PUBLIC_SITE_URL`** — `lib/env.ts` validates the var eagerly at import time, but vitest doesn't load `.env.local`, so any test importing `lib/seo.ts` would crash on import without it. Set to the same `https://example.com` placeholder already used in `.env.local`/`.env.example`.
