# Implementation Plan — US00123: Local image staging (no hotlinking)

**Status:** Draft
**Feature:** F0012 — Automated Content Ingestion Pipeline
**Spec ref:** `docs/specs/F0012.md` § US00123
**Predecessors:** **US00121** (candidate model + accept/reject flow). Pairs with **US00122** (needs the final `slug` to name image files `<slug>-<n>.<ext>`).
**Successors:** consumed by US00124/US00125 output (their fixtures get local image paths).

---

## 1. Scope

Strictly:

1. Add the **image-staging step** (`scripts/ingest/images.ts`) that runs for every **accepted** candidate before its fixture is written.
2. Download each of a candidate's `imageUrls` to `public/static/images/products/<slug>-<n>.<ext>` (1-based `n`), and set the fixture's `images` array to the **local** `/static/images/products/<slug>-<n>.<ext>` paths only — **never** a remote URL (Scenario 1).
3. **Download failure rejects the candidate** (Scenario 2): a 404/timeout on any image means no partial fixture is written; the summary names the candidate + failure reason. (Staging is atomic per candidate.)
4. **Idempotent staging** (Scenario 3): if the target local file already exists (from a previous partial run), skip the download and treat it as already-staged, so a retried run completes.
5. Honor `--dry-run`: stage nothing, download nothing (guarded upstream in US00121).

**Explicitly out of scope:**

- Image optimization/resizing/format conversion — files are stored as downloaded (extension derived from source). `next/image` handles responsive delivery at render time.
- Slug derivation → **US00122** (image filenames depend on the final slug; this story consumes it).
- `next.config.ts` `images.remotePatterns` — deliberately **not** added; local staging is the whole point (spec §6).

## 2. Pre-conditions / current state

- `public/static/images/products/` is the established location (CLAUDE.md map; existing fixtures reference `/static/images/products/<slug>.jpg`).
  - **Path note:** existing fixtures use `/static/images/products/...` and the file lives at `public/static/images/products/...` (Next.js serves `public/` at root). The **fixture `images` value** is `/static/images/products/<slug>-<n>.<ext>`; the **on-disk write target** is `public/static/images/products/<slug>-<n>.<ext>`. Confirm the existing convention at implementation time (some existing images may be under `public/static/...` vs a top-level `static/...`; the map shows both `public/static/images/products/` and `static/images/{products,blog}/`). **Match whatever `next/image` currently resolves for existing products** — verify by loading an existing product page.
- US00121's flow yields `AcceptedCandidate` objects; staging mutates `images` from remote → local before `writeFixture`.
- Node ≥22.11 has global `fetch` — use it for downloads (no new dep).

## 3. Approved Decisions

| # | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| D1 | Filename scheme | `<slug>-<n>.<ext>`, `n` 1-based, in source order | Spec §2/§Scenario 1 exact scheme. Multi-image products get `-1`, `-2`, … Deterministic + collision-safe under a unique slug. |
| D2 | Extension source | Derive `<ext>` from the URL path (`.jpg/.png/.webp`); fall back to `Content-Type` header; if neither yields a known image type → **reject** | Avoids writing an extension-less or mislabeled file that breaks `next/image`. Guessing wrong is worse than rejecting. |
| D3 | Staging is atomic per candidate | Download **all** of a candidate's images to temp names first; only on full success rename into place + set `images`. Any failure → reject, clean up partials | Scenario 2: "no partial fixture written." Prevents a fixture that references a half-downloaded image. |
| D4 | Idempotency | Before downloading `<slug>-<n>.<ext>`, if the final file already exists, **skip** (treat as staged) | Scenario 3: retried run completes; no re-download, no error. |
| D5 | Failure = candidate rejection, not fatal | A 404/timeout rejects **that** candidate; the run continues for others | Consistent with US00121 D6 (per-candidate rejections are non-fatal). |
| D6 | Timeout | Per-image fetch timeout (e.g. 15s via `AbortSignal.timeout`); exceed → treated as download failure → reject | Scenario 2 names "times out." Bounds a hung run. |
| D7 | Download tool | Global `fetch` + stream to disk (`Readable.fromWeb` → `fs.createWriteStream`) | No new dependency; Node ≥22.11 has stable `fetch`. |
| D8 | Ordering vs. slug | Staging runs **after** slug is finalized (US00122), **before** `writeFixture` | Filenames embed the slug; the fixture's `images` must be set before serialization. |
| D9 | Dry-run | Staging entirely skipped under `--dry-run`; summary still reports what *would* be staged | Scenario 3 of US00121: dry-run downloads no image. |
| D10 | Content-type/extension allow-list | `.jpg/.jpeg/.png/.webp/.avif` (align with what `next/image` + the existing fixtures use) | Rejects HTML error pages served with 200 (a common Shopee-CDN failure mode) that would otherwise be written as a `.jpg`. Optionally sniff magic bytes for extra safety. |

### 4.3 Staging sketch

```ts
// scripts/ingest/images.ts
const OK_EXT = new Set(["jpg", "jpeg", "png", "webp", "avif"]);
const DEST_DIR = path.join(process.cwd(), "public", "static", "images", "products");

export async function stageImages(c: Candidate, slug: string): Promise<{ images: string[] } | Rejection> {
  const finalPaths: string[] = [];
  const temps: string[] = [];
  try {
    for (let i = 0; i < c.imageUrls.length; i++) {
      const ext = resolveExt(c.imageUrls[i]);                       // D2/D10 → reject if unknown
      const publicRel = `/static/images/products/${slug}-${i + 1}.${ext}`;
      const abs = path.join(DEST_DIR, `${slug}-${i + 1}.${ext}`);
      if (fs.existsSync(abs)) { finalPaths.push(publicRel); continue; } // D4 skip
      const tmp = abs + ".part";
      await downloadTo(c.imageUrls[i], tmp, { timeoutMs: 15_000 });    // D6/D7 → throws on 404/timeout/bad type
      temps.push(tmp);
      fs.renameSync(tmp, abs);                                          // D3 atomic per file
      finalPaths.push(publicRel);
    }
    return { images: finalPaths };
  } catch (e) {
    temps.forEach((t) => fs.existsSync(t) && fs.unlinkSync(t));         // clean partials
    return { ref: c.sourceRef, name: c.name, reason: `image staging failed: ${msg(e)}` };
  }
}
```

## 9. Risks / open items

- **R1 — Path convention ambiguity.** The map lists both `public/static/images/products/` and a top-level `static/images/{products,blog}/`. Step 2 must confirm which one `next/image` actually resolves for existing products and match it exactly, or staged images 404 at render.
- **R2 — Shopee CDN hotlink protection.** The CDN may 403 non-browser requests or require a `Referer`. If downloads fail broadly, add a `User-Agent`/`Referer` header (still just `fetch`); surface as a per-candidate rejection meanwhile. Note in README.
- **R3 — Duplicate images across products** (same CDN URL for two products under different slugs) → stored twice under different slug names. Acceptable (each product owns its files); no dedupe of image bytes.
- **R4 — Large images / disk.** No resizing; a run could add several MB. Acceptable for a manual, reviewed tool; the operator reviews the diff before commit.
- **R5 — Partial-file crash.** `.part` temp + rename-on-success avoids referencing a truncated file; a crash mid-download leaves only a `.part` which the next run ignores (final name absent → re-download).
