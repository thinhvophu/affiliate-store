## 1. Scope

Strictly:

1. Add a client-side filter + sort UI to the left panel of `/san-pham/` (both desktop inline panel and mobile drawer).
2. Filter dimensions: **Category** (checkboxes — multi-select), **Brand** (checkboxes — multi-select), **Price** (5 preset buckets — multi-select), and **Sort** (4 mutually exclusive options — radio or `<select>`).
3. Selections are reflected in the URL as query params (`?category=`, `?brand=`, `?price=`, `?sort=`) using `useRouter().replace(...)` for shareable, reloadable, back-button-friendly state. Multi-value params use comma-separated values (e.g., `?category=chuot-gaming,tai-nghe`).
4. Filter options are **derived at build time** from `getAllProducts()`: distinct categories and brands; price buckets are a fixed constant.
5. Filtering and sorting are **in-memory in the browser** against the full product payload that US00043 already shipped in the HTML — no extra network requests.
6. **Pagination resets to 1** whenever any filter or sort changes (the `?page` param is dropped from the URL).
7. **Empty filter result** renders a Vietnamese empty state with a "Xóa bộ lọc" action that clears all filters in one click.
8. **Active filter count + clear-all chip set** rendered at the top of the panel when ≥1 filter is active.
9. **Filtered views canonicalize to the bare listing.** The page emits `<link rel="canonical" href="…/san-pham/" />` (or `/san-pham/?page=N` for paginated unfiltered views) regardless of which filter params are present. Filter params do not influence the canonical.
10. **No free-text search input** — the UI exposes only enumerated checkboxes/radios. Search is v2 per spec §2 Out of Scope.
11. **Mobile drawer integration**: the same `<CatalogFilters />` component renders inside the F0002 mobile drawer for `< 768px`; selections take effect immediately and persist after the drawer closes.

**Explicitly out of scope** (owned by sibling stories or downstream features):

- Rendering of the product grid itself — `<ProductCard />` and the grid layout are US00042 + US00043.
- The all-products page route, the page slicing, and the unfiltered pagination control — US00043.
- The category-page filter UI at `/danh-muc/[category]/` — US00045 (it may reuse `<CatalogFilters />` but plumbing is its own concern).
- Per-page `<title>` / `<meta>` / OG / JSON-LD — F0009.
- GA4 events on filter changes — F0007 (and not in scope for this site at all per spec).
- Persisting filter state to `localStorage` between sessions — URL is the only state container.
- Free-text search (`<input type="search">`) — v2.
- Filter UI as a Server Component — by the nature of "client-side filtering / URL-driven state changes", `<CatalogFilters />` is necessarily a `"use client"` component.
- Animations on filter apply / clear — there are none; state changes are instant per `prefers-reduced-motion` defaults.

## 3. Approved Decisions

| # | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| D1 | State container | **URL search params only.** No React Context, no `useReducer`, no client store. `useSearchParams()` reads, `useRouter().replace()` writes. | Spec Scenario 5: "reloading the URL restores the same filter state" — URL is the source of truth. Avoids state-management dep (CLAUDE.md §Technical Considerations). Back/forward buttons restore state for free. |
| D2 | Multi-value param encoding | **Comma-separated values**: `?category=chuot-gaming,tai-nghe&brand=Logitech,Razer&price=under-200k,200-500k`. Empty / missing param = "no filter on this dimension". | Repeat-key syntax (`?category=a&category=b`) is heavier in the URL and slightly awkward to parse with `URLSearchParams.get()` (which returns only the first). Comma encoding is short, human-readable, and matches the project's "crawlable / shareable" SEO principle. Values are URL-safe slugs (categories are kebab-case already; brands are URL-encoded). |
| D3 | Sort param keys (URL values) | `relevance` (default, omitted from URL), `price-asc`, `price-desc`, `newest`. Labels: `Mặc định`, `Giá tăng dần`, `Giá giảm dần`, `Mới nhất`. | Spec Scenario 3 names the exact four labels. URL values are kebab-case English so they're stable across copy changes. `relevance` is the spec name for "Mặc định" / same stable order as US00043 Scenario 9. |
| D4 | Default sort order definition | **`publishedAt` desc, with `slug` asc as tiebreaker** — identical to US00043 Scenario 9's stable default. `lib/filters.ts` exports the comparator so US00043 and US00044 share it. | Spec Scenario 3 ties "Mặc định" to US00043 Scenario 9's stable order. Both consume the *same* comparator from one place; if US00043 picks a different stable order, this plan adopts it verbatim. |
| D5 | Price buckets (constant) | **5 buckets** with these `id` / VND range / label: `under-200k` / `[0, 200_000)` / `Dưới ₫200.000`; `200-500k` / `[200_000, 500_000)` / `₫200.000 – ₫500.000`; `500k-1tr` / `[500_000, 1_000_000)` / `₫500.000 – ₫1.000.000`; `1tr-3tr` / `[1_000_000, 3_000_000)` / `₫1.000.000 – ₫3.000.000`; `over-3tr` / `[3_000_000, Infinity)` / `Trên ₫3.000.000`. | Spec Scenario 2 names this set ("`<200k`, `200–500k`, `500k–1tr`, `1tr–3tr`, `>3tr`"). Vietnamese-shopper-appropriate. Half-open intervals avoid bucket overlap at boundaries. Labels generated *via `formatVnd()`* so number formatting stays centralized. |
| D6 | Multi-select semantics within a dimension | **OR** within a dimension (any selected category matches), **AND** across dimensions (must match every active dimension). | Standard e-commerce filter semantics. Matches user mental model: "show me Logitech *or* Razer mice (not specifically gaming)". |
| D7 | When all checkboxes in a dimension are unchecked | **No filter applied on that dimension** (all products pass). Equivalent to omitting the param from the URL. | Avoids the awkward "zero selections = zero products" state. Matches Amazon/Shopee behavior. |
| D8 | Filter-options source | **Build-time derivation in `lib/filters.ts`**. `getFilterOptions(products)` returns `{ categories: string[], brands: string[], priceBuckets: PriceBucket[] }` with categories and brands deduplicated and sorted alphabetically (Vietnamese collation via `localeCompare("vi")`). Page passes these to `<CatalogFilters />` as a serializable prop. | Spec Scenario 2: "Category and Brand options reflect the distinct values across all products … no hand-maintained list". Deriving once at build keeps the client component pure; the visitor's browser does no scanning. |
| D9 | Categories with display labels | **A small Vietnamese label map lives in `lib/categories.ts` (introduced by US00045, may be stubbed here if US00045 hasn't landed).** The filter UI shows the Vietnamese label (e.g., "Chuột gaming"); the URL still uses the kebab-case slug (`chuot-gaming`). If a slug has no label entry, fall back to the slug itself. | Spec Scenario 2 wants categories "derived from product JSON" — but raw slugs are not user-friendly. The label map is the natural place to keep the Vietnamese title-cased name. US00045's category-page intros need the same map. Coordinate: if US00045 lands first, reuse its file; if US00044 lands first, create a tiny `lib/categories.ts` with at least the slug→label map so US00045 only adds intro copy on top. |
| D10 | Brand display | **Brands are rendered verbatim from `product.brand`** (no slug indirection). Brand values like "Logitech", "Razer" already double as both display name and URL value. URL-encode them on emission so values with diacritics or spaces remain a single token. | Brands in the seed data are Latin-script ASCII (Logitech, Razer, HyperX, …). No label map needed for v1. |
| D11 | Component layout | **Single `<CatalogFilters />` Client Component.** It renders the four filter groups in a flat vertical stack with section headings (`<h2 class="srOnly">` for a11y, visible `<h3>` group titles `Danh mục`, `Thương hiệu`, `Giá`, `Sắp xếp`). | One client component is simpler than four, and the focus order is more predictable. Co-located styles in `CatalogFilters.module.css`. |
| D12 | Re-render strategy | **Compute filter result inline on each render** via `useMemo(() => filterAndSort(products, currentParams), [products, currentParams])`. Catalog payload is small (≤200 products v1); no virtualization or memoization beyond `useMemo` is needed. | `CLAUDE.md` "Technical considerations" sets v1 catalog scale at ≤200 products. Filtering 200 product objects per keystroke is sub-millisecond. Premature optimization avoided. |
| D13 | URL update style | **`router.replace(...)` with `{ scroll: false }`**, *not* `router.push(...)`. | Each filter tweak shouldn't push a new history entry — that would make the back button cycle through every checkbox toggle. `replace` keeps history clean; the visitor can still bookmark/share the current URL. `scroll: false` prevents the page from jumping to top on each toggle. |

### 5.1 `lib/filters.ts` (new — pure module, no React)

```ts
import type { Product } from "@/types";
import { formatVnd } from "@/lib/format"; // ships in US00041

export interface PriceBucket {
  id: "under-200k" | "200-500k" | "500k-1tr" | "1tr-3tr" | "over-3tr";
  min: number;
  max: number; // exclusive; Number.POSITIVE_INFINITY for the open-ended top bucket
  label: string;
}

export const PRICE_BUCKETS: readonly PriceBucket[] = [
  { id: "under-200k", min: 0, max: 200_000, label: `Dưới ${formatVnd(200_000)}` },
  { id: "200-500k", min: 200_000, max: 500_000, label: `${formatVnd(200_000)} – ${formatVnd(500_000)}` },
  { id: "500k-1tr", min: 500_000, max: 1_000_000, label: `${formatVnd(500_000)} – ${formatVnd(1_000_000)}` },
  { id: "1tr-3tr", min: 1_000_000, max: 3_000_000, label: `${formatVnd(1_000_000)} – ${formatVnd(3_000_000)}` },
  { id: "over-3tr", min: 3_000_000, max: Number.POSITIVE_INFINITY, label: `Trên ${formatVnd(3_000_000)}` },
] as const;

export interface SortOption {
  id: "relevance" | "price-asc" | "price-desc" | "newest";
  label: string;
}

export const SORT_OPTIONS: readonly SortOption[] = [
  { id: "relevance", label: "Mặc định" },
  { id: "price-asc", label: "Giá tăng dần" },
  { id: "price-desc", label: "Giá giảm dần" },
  { id: "newest", label: "Mới nhất" },
] as const;

export const DEFAULT_SORT_ID: SortOption["id"] = "relevance";

export interface FilterOptions {
  categories: string[]; // distinct slugs, sorted by Vietnamese collation
  brands: string[];     // distinct brand names, sorted by Vietnamese collation
  priceBuckets: readonly PriceBucket[];
}

export function getFilterOptions(products: readonly Product[]): FilterOptions {
  const categories = Array.from(new Set(products.map((p) => p.category)))
    .sort((a, b) => a.localeCompare(b, "vi"));
  const brands = Array.from(new Set(products.map((p) => p.brand)))
    .sort((a, b) => a.localeCompare(b, "vi"));
  return { categories, brands, priceBuckets: PRICE_BUCKETS };
}

export interface ActiveFilters {
  categories: string[];
  brands: string[];
  priceBucketIds: PriceBucket["id"][];
  sort: SortOption["id"];
}

const PRICE_BUCKET_IDS = new Set(PRICE_BUCKETS.map((b) => b.id));
const SORT_IDS = new Set(SORT_OPTIONS.map((s) => s.id));

export function parseFilterParams(
  searchParams: URLSearchParams,
  options: FilterOptions,
): ActiveFilters {
  const knownCategories = new Set(options.categories);
  const knownBrands = new Set(options.brands);

  const readList = (key: string): string[] =>
    (searchParams.get(key) ?? "")
      .split(",")
      .map((s) => s.trim())
      .filter((s) => s.length > 0);

  const categories = readList("category").filter((c) => knownCategories.has(c));
  const brands = readList("brand").filter((b) => knownBrands.has(b));
  const priceBucketIds = readList("price").filter(
    (id): id is PriceBucket["id"] => PRICE_BUCKET_IDS.has(id as PriceBucket["id"]),
  );

  const sortRaw = searchParams.get("sort") ?? "";
  const sort: SortOption["id"] = SORT_IDS.has(sortRaw as SortOption["id"])
    ? (sortRaw as SortOption["id"])
    : DEFAULT_SORT_ID;

  return { categories, brands, priceBucketIds, sort };
}

export function serializeFilterParams(filters: ActiveFilters): URLSearchParams {
  const out = new URLSearchParams();
  if (filters.categories.length > 0) out.set("category", filters.categories.join(","));
  if (filters.brands.length > 0) out.set("brand", filters.brands.join(","));
  if (filters.priceBucketIds.length > 0) out.set("price", filters.priceBucketIds.join(","));
  if (filters.sort !== DEFAULT_SORT_ID) out.set("sort", filters.sort);
  return out;
}

export function compareDefault(a: Product, b: Product): number {
  // publishedAt desc, slug asc — must match US00043 Scenario 9
  const t = new Date(b.publishedAt).getTime() - new Date(a.publishedAt).getTime();
  if (t !== 0) return t;
  return a.slug.localeCompare(b.slug);
}

export function applyFilters(
  products: readonly Product[],
  filters: ActiveFilters,
): Product[] {
  const catSet = new Set(filters.categories);
  const brandSet = new Set(filters.brands);
  const bucketSet = new Set(filters.priceBucketIds);
  const activeBuckets = PRICE_BUCKETS.filter((b) => bucketSet.has(b.id));

  const filtered = products.filter((p) => {
    if (catSet.size > 0 && !catSet.has(p.category)) return false;
    if (brandSet.size > 0 && !brandSet.has(p.brand)) return false;
    if (activeBuckets.length > 0) {
      const inAny = activeBuckets.some((b) => p.price >= b.min && p.price < b.max);
      if (!inAny) return false;
    }
    return true;
  });

  switch (filters.sort) {
    case "price-asc":
      return [...filtered].sort((a, b) => a.price - b.price || compareDefault(a, b));
    case "price-desc":
      return [...filtered].sort((a, b) => b.price - a.price || compareDefault(a, b));
    case "newest":
      // identical to compareDefault since default is publishedAt-desc — keep them as one branch
      return [...filtered].sort(compareDefault);
    case "relevance":
    default:
      return [...filtered].sort(compareDefault);
  }
}

export function countActiveFilters(filters: ActiveFilters): number {
  return (
    filters.categories.length +
    filters.brands.length +
    filters.priceBucketIds.length +
    (filters.sort !== DEFAULT_SORT_ID ? 1 : 0)
  );
}
```

```tsx
import { useCallback, useMemo } from "react";
import { useRouter, useSearchParams, usePathname } from "next/navigation";
import {
  PRICE_BUCKETS,
  SORT_OPTIONS,
  DEFAULT_SORT_ID,
  parseFilterParams,
  serializeFilterParams,
  countActiveFilters,
  type ActiveFilters,
  type FilterOptions,

import styles from "./CatalogFilters.module.css";

interface Props {
  options: FilterOptions;
  /** Optional: maps category slug → Vietnamese display label (from US00045 lib/categories.ts).
   *  Falls back to the raw slug when not provided. */

}

export function CatalogFilters({ options, categoryLabels }: Props) {
  const router = useRouter();
  const pathname = usePathname();
  const searchParams = useSearchParams();

  const active: ActiveFilters = useMemo(
    () => parseFilterParams(new URLSearchParams(searchParams.toString()), options),
    [searchParams, options],
  );

  const writeFilters = useCallback(
    (next: ActiveFilters) => {
      // D14: stripping ?page resets pagination. D13: replace, not push; no scroll.
      const params = serializeFilterParams(next);
      const query = params.toString();
      router.replace(query ? `${pathname}?${query}` : pathname, { scroll: false });
    },
    [router, pathname],
  );

  // toggle helpers for each dimension follow the same shape:
  const toggle = useCallback(
    <K extends "categories" | "brands" | "priceBucketIds">(
      key: K,
      value: ActiveFilters[K][number],
    ) => {
      const current = active[key] as string[];
      const exists = current.includes(value);
      const nextList = exists ? current.filter((v) => v !== value) : [...current, value];
      writeFilters({ ...active, [key]: nextList } as ActiveFilters);
    },
    [active, writeFilters],
  );

  const setSort = useCallback(
    (sort: SortOption["id"]) => writeFilters({ ...active, sort }),
    [active, writeFilters],
  );

  const clearAll = useCallback(
    () =>
      writeFilters({
        categories: [],
        brands: [],
        priceBucketIds: [],
        sort: DEFAULT_SORT_ID,
      }),
    [writeFilters],
  );

  const count = countActiveFilters(active);

  return (
    <aside className={styles.root} aria-label="Bộ lọc sản phẩm">
      {/* Active-filter status + clear-all (D17) */}

                  type="button"
                  onClick={() => toggle("categories", c)}

                >
                  {categoryLabels?.[c] ?? c} ×
                </button>
              </li>
            ))}
            {/* brand and price chips follow the same pattern */}
            {active.sort !== DEFAULT_SORT_ID && (
              <li>
                <button
                  type="button"
                  onClick={() => setSort(DEFAULT_SORT_ID)}
                  aria-label={`Bỏ chọn sắp xếp ${SORT_OPTIONS.find((s) => s.id === active.sort)?.label}`}
                >
                  {SORT_OPTIONS.find((s) => s.id === active.sort)?.label} ×
                </button>
              </li>
            )}
          </ul>
        </div>
      )}

      {/* Category group */}
      <fieldset className={styles.group}>
        <legend>Danh mục</legend>

              type="checkbox"

              onChange={() => toggle("categories", c)}
            />
            <span>{categoryLabels?.[c] ?? c}</span>
          </label>
        ))}
      </fieldset>

      {/* Brand group */}
      <fieldset className={styles.group}>

              type="checkbox"
              checked={active.brands.includes(b)}
              onChange={() => toggle("brands", b)}
            />
            <span>{b}</span>
          </label>
        ))}
      </fieldset>

      {/* Price group */}
      <fieldset className={styles.group}>
        <legend>Giá</legend>
        {PRICE_BUCKETS.map((b: PriceBucket) => (
          <label key={b.id} className={styles.option}>
            <input
              type="checkbox"
              checked={active.priceBucketIds.includes(b.id)}
              onChange={() => toggle("priceBucketIds", b.id)}
            />
            <span>{b.label}</span>
          </label>
        ))}
      </fieldset>

      {/* Sort group — D19 native <select> */}
      <div className={styles.group}>
        <label htmlFor="catalog-sort" className={styles.sortLabel}>
          Sắp xếp
        </label>
        <select
          id="catalog-sort"
          value={active.sort}

              {s.label}
            </option>

        </select>

  );
}
```

```tsx
"use client";

import { useMemo } from "react";
import { useRouter, useSearchParams, usePathname } from "next/navigation";
import { applyFilters, parseFilterParams, getFilterOptions } from "@/lib/filters";
import type { Product } from "@/types";
import { ProductCard } from "./ProductCard"; // from US00042
import styles from "./CatalogGrid.module.css";

interface Props {
  products: Product[];
  itemsPerPage?: number; // default 24 per US00043
}

export function CatalogGrid({ products, itemsPerPage = 24 }: Props) {
  const router = useRouter();
  const pathname = usePathname();
  const searchParams = useSearchParams();

  const options = useMemo(() => getFilterOptions(products), [products]);
  const active = useMemo(
    () => parseFilterParams(new URLSearchParams(searchParams.toString()), options),
    [searchParams, options],
  );
  const filtered = useMemo(() => applyFilters(products, active), [products, active]);

  const page = Math.max(1, Number(searchParams.get("page") ?? 1) || 1);
  const start = (page - 1) * itemsPerPage;
  const slice = filtered.slice(start, start + itemsPerPage);

  // Empty state (Scenario 9)
  if (filtered.length === 0) {
    return (
      <div className={styles.empty} role="status">
        <p>Không có sản phẩm phù hợp với bộ lọc.</p>
        <button
          type="button"
          className={styles.clear}
          onClick={() => router.replace(pathname, { scroll: false })}
        >
          Xóa bộ lọc
        </button>
      </div>
    );
  }

  return (
    <ul className={styles.grid}>
      {slice.map((p) => (
        <li key={p.slug}>
          <ProductCard product={p} />
        </li>
      ))}
    </ul>
  );
}
```

  This plan documents Option B so US00044 is not blocked; the follow-up swap is trivial (one component swap, no logic change). The `// TODO(US00025-drawer)` marker makes the swap discoverable.
- **US00041 coordination — `formatVnd()`.** This plan calls `formatVnd()` from `lib/format.ts` to build price-bucket labels. If US00041 has not landed, the implementer must either (a) inline a temporary local `formatVnd` in `lib/filters.ts` to keep US00044 unblocked or (b) reorder the F0004 batch so US00041 ships first. Recommend (b) — US00041 is trivial.
- **US00045 coordination — category labels.** The filter UI accepts an optional `categoryLabels` map for Vietnamese display names. US00045 owns the canonical map (`lib/categories.ts`). If US00045 ships first, US00044 consumes it; if US00044 ships first, the filter panel falls back to displaying raw slugs (acceptable v1 behavior — slugs are kebab-case Vietnamese already).
- **Canonical metadata vs. pagination.** D15 / §5.5 emit a static `<link rel="canonical" href="/san-pham/">` from `metadata` because Next.js does not vary `Metadata` per query string for a static page. This is **correct for filter combinations** (Scenario 6: "filtered view canonicalizes to the unfiltered listing"), but the **page-2+ self-canonical** required by US00043 Scenario 7 needs a different mechanism (likely a Client Component that mounts the canonical link via `next/head` equivalent, or — better — F0009's eventual `generateMetadata(searchParams)` for the dynamic case). **Action:** US00043's plan must own the page-2+ canonical concern; US00044 only ships the *bare-path* canonical. F0009 will harmonize both. Flag in PR description.
- **Param explosion / URL length.** With 5 categories × 8 brands × 5 price buckets selected, URL length is ~150 chars — well under any practical limit. Not a real risk; mentioned for completeness.
- **History stack.** D13 uses `router.replace` to avoid polluting the history stack. Trade-off: the back button does not "undo" a single checkbox toggle — it goes back to the previous *page* visit. Spec is silent on this; user behavior in catalog UIs strongly favors `replace` (try Shopee, Amazon — same pattern).
- **`prefers-reduced-motion`** — Filter UI has no animations; nothing to gate. Empty-state swap is instant. If a future story adds animation, the gate goes there.
- **Layout shift on filter apply.** Going from 24 cards → 0 cards (empty state) shrinks the grid container. Acceptable v1 behavior; if it ever flickers visibly, add a `min-height` to the grid container. Out of scope.
- **Accessibility — chip remove buttons.** Each chip's `aria-label` includes the dimension and the value (e.g., "Bỏ chọn danh mục Chuột gaming") so screen readers announce the action precisely. Verify via VoiceOver during browser check.
