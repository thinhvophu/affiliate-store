## 1. Scope

Strictly:

1. Add a **Server Component** at `components/ProductCard.tsx` whose single prop is `{ product: Product }` (the `Product` type already exists at `types/product.ts`). The component renders the standard product summary used by every product surface on the site: primary image, name, brand, formatted price, category badge, and a non-interactive "Mua ngay" CTA element — all wrapped in a single `<AffiliateLink>` styled as a whole card via US00032's `.card` pattern.
2. Add a **co-located CSS Module** at `components/ProductCard.module.css` containing only the card-internal layout (image aspect-ratio frame, two-line name clamp, brand-row, price-row, category badge chrome). The wrapping anchor's hover / focus-visible / reduced-motion behaviour is **inherited from `AffiliateLink.module.css`** — this story does not duplicate it.
3. The component is a **pure Server Component** — no `"use client"`, no hooks, no event handlers. It ships zero hydration cost. CLAUDE.md "Server Components by default" rule applies.
4. The card emits **exactly one** `<a>` (the wrapping `<AffiliateLink>`) and **zero** `<button>` or nested `<a>` elements per US00032's no-nested-interactives rule. The "Mua ngay" CTA is a styled `<span>` carrying `data-affiliate-cta`.
5. The card uses **`next/image`** for the primary image, with an explicit `sizes` attribute matching the 2 / 3 / 4-column grid breakpoints (US00043 §Scenario 2 / US00045 §Scenario 4 / US00047 §Scenario 5). Alt text is the product name (`product.name`).
6. The card respects **`<h1>` discipline** (F0004 §6) — the product name renders inside a non-`<h1>` element. Choosing `<h3>` because the listing/category pages use `<h1>` for the page title and may use `<h2>` for grid section headings; the card sits below them in the heading tree. Detail-page related-products consumers (US00047) also have a `<h2>` section heading above the row, so `<h3>` is correct there too.
7. The card surface is built so a screen reader announces, in order, the F0003 sr-only "Mở [productName] trên Shopee (mở tab mới)" label (provided by `<AffiliateLink>`) plus the visible product fields (name, brand, formatted price). Image `alt` is the product name; decorative chrome (category badge, "Mua ngay" CTA span) is not redundantly announced — the badge is presentational text inside the link and the CTA span is decorative.
8. Layout behaviour required by Scenario 7:
   - Name truncates at **two lines** via `-webkit-line-clamp: 2` (with `overflow: hidden`), so very long names do not push the card height.
   - The card maintains **consistent height within a row** of grid cells — image is a fixed aspect-ratio frame (1:1), text rows sit beneath, and the CTA pins to the bottom of the card via `margin-top: auto` on a flex column.
   - The image frame uses `aspect-ratio: 1 / 1` with `object-fit: cover` on the `<Image>` so different source aspect ratios still align cleanly across the grid.

**Explicitly out of scope** (owned by sibling stories):

- **Listing-page chrome and the grid container CSS** (2 / 3 / 4 columns, page heading, pagination control, empty-state copy) — **US00043**.
- **Filters and sort UI** — **US00044**.
- **Category page (route, intro paragraph, `generateStaticParams`)** — **US00045**.
- **Product detail page (gallery, specs table, big CTA, related row)** — **US00046**. Note: the detail page's prominent "Mua trên Shopee" CTA is a **different** `<AffiliateLink>` (button-styled, not whole-card) — this story does not author that.
- **Related-products computation (`getRelatedProducts(currentProduct)`)** — **US00047**. This story renders one card per product; the caller decides which products.
- **MDX inline `<ProductCard slug="…" />` wrapper** — **F0006**. F0006 will load the product by slug via `getProductBySlug()` and pass it to *this* `<ProductCard product={…} />`.
- **Homepage featured-picks horizontal row** — **F0008**.
- **JSON-LD `Product` schema, Open Graph, per-card metadata** — **F0009**.
- **GA4 `affiliate_click` event emission** — **F0007**; the seam is already on `<AffiliateLink>` (US00033).
- **A custom ESLint rule banning nested `<a>` / `<button>` inside `<AffiliateLink>` children** — explicitly P2 per F0003. Convention + JSDoc + review is v1 enforcement.

## 5. Concrete diff sketch

### 5.1 `components/ProductCard.tsx` — new file

```tsx
import Image from "next/image";
import { AffiliateLink } from "@/components/AffiliateLink";
import affiliateStyles from "@/components/AffiliateLink.module.css";
import { formatVnd } from "@/lib/format";
import type { Product } from "@/types";
import styles from "./ProductCard.module.css";

/**
 * Standard product summary card — F0004 (US00042).
 *
 * Renders the canonical product card used by:
 *   - the product listing grid (US00043)
 *   - the category page grid (US00045)
 *   - the related-products row on the product detail page (US00047)
 *   - F0006's inline blog `<ProductCard slug="…" />` (composes this primitive)
 *   - F0008's homepage featured-picks layout
 *
 * The entire card surface is a single affiliate link (whole-card pattern
 * established by US00032). Tab-focus stops ONCE on the card. The "Mua ngay"
 * CTA is a styled <span> — NEVER <button>, NEVER a nested <a>.
 *
 * Server Component — no client JS, no hooks, no event handlers. The grid
 * container (US00043/US00045/US00047) handles responsive column counts; this
 * card sizes itself to whatever grid cell it lands in.
 *
 * Image rendering uses `next/image` in `fill` mode inside a 1:1 aspect-ratio
 * frame; the `sizes` attribute targets the 2 / 3 / 4-col grid widths so the
 * browser picks an appropriately sized srcset candidate.
 */
export interface ProductCardProps {
  product: Product;
}

export function ProductCard({ product }: ProductCardProps) {
  return (
    <AffiliateLink
      className={`${affiliateStyles.card} ${styles.card}`}
      href={product.affiliateUrl}
      productName={product.name}
      productCategory={product.category}
    >
      <div className={styles.imageFrame}>
        <Image
          src={product.images[0]}
          alt={product.name}
          fill
          sizes="(min-width: 1280px) 25vw, (min-width: 768px) 33vw, 50vw"
          className={styles.image}
        />
        <span className={styles.categoryBadge}>{product.category}</span>
      </div>

      <h3 className={styles.name}>{product.name}</h3>
      <p className={styles.brand}>{product.brand}</p>
      <p className={styles.price}>{formatVnd(product.price)}</p>

      <span data-affiliate-cta className={styles.cta}>
        Mua ngay
      </span>
    </AffiliateLink>
  );
}
```

Notes on the JSX:

- `className={`${affiliateStyles.card} ${styles.card}`}` composes US00032's whole-card chrome with a card-local override class. `styles.card` exists only to apply `display: flex; flex-direction: column; gap;` (extends US00032's `display: block` for the card-internal layout). If US00032's `.card` declaration changes, this composition still works because CSS Modules emit unique class names and both are applied to the same anchor.
- `<h3>` per D9. The grid container's `<h2>` (if any) and the page `<h1>` sit above this in the heading tree.
- `<Image fill>` requires its parent (`.imageFrame`) to be `position: relative` and have a defined size — handled by CSS Module (§5.2).
- No `loading` prop on `<Image>` — `next/image` lazy-loads by default.
- The sr-only "Mở [name] trên Shopee (mở tab mới)" label is appended by `<AffiliateLink>` itself — this card does NOT add an extra one.

### 5.2 `components/ProductCard.module.css` — new file

```css
/* ==========================================================================
   ProductCard — card-internal layout (US00042)
   --------------------------------------------------------------------------
   Card chrome (border, padding, hover, focus-visible, prefers-reduced-motion)
   is INHERITED from components/AffiliateLink.module.css → .card. This file
   only owns the card-internal layout: flex column, image frame, category
   badge overlay, name clamp, brand/price rows, CTA pill.

   No new :root tokens — composes from F0001 spacing + F0002 surface tokens.
   ========================================================================== */

/* Extends affiliateStyles.card (which gives us border, padding, background,
   border-radius, hover, focus-visible, reduced-motion). We add flex-column
   layout so the children stack and the CTA can pin to the bottom. */
.card {
  display: flex;
  flex-direction: column;
  gap: var(--spacing-sm);
  height: 100%; /* fill the grid cell so all cards in a row align */
}

/* Image frame — fixed 1:1 aspect ratio, relative positioning so next/image
   in `fill` mode anchors correctly. overflow:hidden clips the badge / image
   inside the rounded corners of the parent card. */
.imageFrame {
  position: relative;
  width: 100%;
  aspect-ratio: 1 / 1;
  overflow: hidden;
  border-radius: var(--radius-sm);
  background: var(--color-bg-secondary); /* placeholder while image loads */
}

/* next/image in fill mode — object-fit:cover keeps every source aspect
   ratio rendering consistently inside the 1:1 frame. */
.image {
  object-fit: cover;
}

/* Category badge — overlay top-left, white pill with primary-coloured text.
   Sits visually above the image without obscuring more than the corner. */
.categoryBadge {
  position: absolute;
  top: var(--spacing-sm);
  left: var(--spacing-sm);
  padding: var(--spacing-xs) var(--spacing-sm);
  background: var(--color-on-primary); /* white */
  color: var(--color-primary);
  font-size: 0.75rem;
  font-weight: 600;
  border-radius: var(--radius-sm);
  box-shadow: 0 1px 2px rgba(0, 0, 0, 0.08);
  /* Slugs may contain hyphens; preserve them as-is for v1 */
  text-transform: none;
}

/* Product name — two-line clamp with reserved min-height so cards align
   even when one name is short and another wraps to two lines. */
.name {
  font-size: 1rem;
  font-weight: 600;
  line-height: 1.4;
  color: var(--color-text);
  /* Two-line clamp */
  display: -webkit-box;
  -webkit-line-clamp: 2;
  line-clamp: 2;
  -webkit-box-orient: vertical;
  overflow: hidden;
  /* Reserve vertical space for two lines so short names don't shrink the card */
```

## 10. Risks / open items

- **R1 — Dependency on US00041.** This story imports `formatVnd` from `@/lib/format`. If US00041 has not landed when implementation starts, `npm run build` fails at the import. **Mitigation:** Verify US00041 status in step 1 of §6. If US00041 is in flight, sequence the merges (US00041 first). The two stories are small and independent enough to land together.
- **R2 — `static/` vs `public/` asset path divergence.** `Product.images` references paths like `/static/images/products/<file>.jpg`. Next.js serves static assets from `public/` by default, not `static/`. This is an unresolved codebase-wide issue, **not introduced by this story**. The card just passes the path to `next/image`; if the path is broken the build / dev server surfaces a clear 404 on the image. Resolving the directory convention is US00043's concern (the first story that wires real product JSON to a real route). **Mitigation:** Browser-verify the card with a placeholder image inside `public/static/images/products/test.jpg`, then revert.
- **R3 — Card does not guard against empty `images[]`.** Per D19, the card trusts the loader. If the loader has not yet been hardened to fail on `images.length === 0`, a malformed product JSON would render `next/image` with `src={undefined}` and surface a runtime error. **Mitigation:** This is a F0004 §5 edge case explicitly assigned to `lib/products.ts`. Add a defensive note in the JSDoc that the card requires `product.images.length >= 1` and that the loader is the enforcement point. File a follow-up story (or fold it into US00043) to extend `validateProduct()` to enforce `images.length >= 1`.
- **R4 — Category badge displays a slug.** v1 renders `product.category` verbatim ("chuot-gaming" rather than "Chuột gaming"). Acceptable v1 behaviour because (a) the slug is short and Vietnamese-friendly, (b) humanization requires `lib/categories.ts` which US00045 introduces, (c) the card surfaces the category visually for scanning, not for canonical labelling. **Mitigation:** US00045 retrofits an optional `categoryLabel?: string` prop on `<ProductCard />` or — preferred — extends `Product` with a derived `categoryDisplay` field at load time. Either way the card's public surface remains compatible.
- **R5 — `next/image` `fill` mode requires a parent with intrinsic size.** Handled by `.imageFrame { aspect-ratio: 1 / 1; }`. Older browsers without `aspect-ratio` support (Safari < 15.4) would collapse the frame; per the project's mobile-first / current-browsers stance and `:focus-visible` already requiring Safari 15.4+, this is acceptable. Add a fallback `padding-bottom: 100%;` hack only if the implementation reveals a real issue.
- **R6 — sr-only label is appended INSIDE the anchor as a trailing `<span>`.** A screen-reader user encounters the link's accessible name as the visible card text (name + brand + price + "Mua ngay" + category badge) **followed by** the sr-only "Mở [name] trên Shopee (mở tab mới)" suffix. This is the F0003 design — the card inherits it without modification. If the announcement order proves awkward in user testing, fix in F0003, not here.
- **R7 — Card height may be inconsistent if the grid container does not use `align-items: stretch`.** The card uses `height: 100%` in its `.card` class. If the grid sets `align-items: start` or `align-items: center`, the cards will collapse to content height and the CTA `margin-top: auto` pin loses meaning. **Mitigation:** US00043 / US00045 / US00047 grid CSS must use `align-items: stretch` (the CSS Grid default). Document in the JSDoc.
- **R8 — Composing `affiliateStyles.card` with `styles.card`.** Both apply to the same `<a>`. CSS specificity is identical (both single-class selectors), so cascade order matters. Because `ProductCard.module.css` is imported *after* `AffiliateLink.module.css` in `ProductCard.tsx`, the card-local rules win on collisions. The only collision is `display`: `affiliateStyles.card` sets `display: block`, `styles.card` overrides to `display: flex`. Intentional — the card needs flex-column. Verified via DevTools Computed Styles tab. **Mitigation:** Document the override explicitly in `ProductCard.module.css` comment so a future reader knows the override is intentional.
- **R9 — `prefers-reduced-motion` honoured only for the inherited `.card` transitions.** This story adds no new transitions (CTA pill is static, badge is static). If a future enhancement adds an animation to any internal element, it must follow US00026's pattern (`@media (prefers-reduced-motion: no-preference)` gate). Document in `ProductCard.module.css`.

### Open questions for the reviewer

- **Q1 — Category badge: render the raw slug, or accept an optional human-readable label?** The plan recommends rendering `product.category` verbatim for v1 (slug "chuot-gaming") and letting US00045 retrofit humanization via `lib/categories.ts`. Alternative: add `categoryLabel?: string` prop now so listings can pass through a human name without a follow-up touch to the card's public API. **Recommendation:** ship v1 with verbatim slug; revisit in US00045.
- **Q2 — Should the card link to the detail page `/san-pham/[slug]/` somewhere?** Per the F0004 process flow §7, "the visitor clicks the product name / image inside the card and navigates to `/san-pham/[slug]/` for the detail view." But Scenario 2 mandates the whole card is **one** affiliate link — no nested `<a>`. These contradict. **Recommendation:** the whole-card affiliate link wins (Scenario 2 is unambiguous); navigation to the detail page happens via a separate explicit anchor on adjacent surfaces (US00046's affiliate CTA, US00047's related row each follow the same rule). The process flow §7 sentence is loose; the AC is binding. Confirm with reviewer.
- **Q3 — Test artefact placement.** Step 8 of §6 prefers a temporary `app/_test-product-card/page.tsx` route. Next.js does **not** automatically exclude `_test-*` directories from routing in the App Router; the `_` prefix is a Pages Router convention. To exclude, either use a `(_test)` route group (parentheses indicate non-route-segment) or place under `app/_test-product-card/` but **manually delete** before PR. **Recommendation:** use a `(_test)/test-product-card` route group, which prevents the path segment from appearing in the URL — except the children still render at `/test-product-card`, which is still production-leakable. Safest: delete the entire test directory before PR commit, verify via `git status`.
