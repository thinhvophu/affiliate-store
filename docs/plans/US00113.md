## 1. Scope

Strictly:

1. Create `.github/workflows/ci.yml` that runs on `pull_request` and on `push` to **non-`main`** branches.
2. Run three jobs (or three steps inside one job — see D1) using **`package.json` scripts**, never inline commands:
   - `npm run typecheck` (`tsc --noEmit`)
   - `npm run lint` (`eslint .`)
   - `npm run build` (`next build`)
3. Pin Node major version to **22** to match `package.json` `engines` (`>=22.11.0`) and `.nvmrc` (`22`), aligned with Vercel's runtime.
4. Use only `actions/checkout` + `actions/setup-node` + `npm ci` — **no secrets are read**, so the workflow runs unchanged on forked PRs.
5. Document the **branch protection click-path** for `main` so the operator can wire the CI jobs as required status checks (this part is GitHub UI only — it is not in the YAML).

**Explicitly out of scope** (owned by sibling stories or deferred):

- Vercel Production / Preview wiring (US00111).
- Vercel env vars and `.env.example` (US00112 / already created in US00018).
- Cron rebuild (`scheduled-deploy.yml`) (US00114).
- Speed Insights (US00115).
- E2E / visual regression / Lighthouse / performance budgets (per F0011 §2 "Out of Scope").
- Deploy notifications, build cache tuning, "Ignore Build Step".
- Format check (`npm run format`) — not requested by the AC; revisit if it becomes a quality-gate ask.
- Dependency caching beyond `actions/setup-node`'s built-in `cache: "npm"` (acceptable default).

## 5. Concrete diff sketch

### 5.1 `.github/workflows/ci.yml`

```yaml
name: CI

on:
  pull_request:
  push:
    branches-ignore:
      - main

permissions:
  contents: read

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  typecheck:
    name: typecheck
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version-file: ".nvmrc"
          cache: "npm"
      - run: npm ci
      - run: npm run typecheck

  lint:
    name: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version-file: ".nvmrc"
          cache: "npm"
      - run: npm ci
      - run: npm run lint

  build:
    name: build
    runs-on: ubuntu-latest
    env:
      # Placeholder URL satisfies lib/env.ts (US00018) — CI is verifying the
      # project compiles, not that production env is correct. Real values
      # live in Vercel per US00112; CI never reads any repo secret.
      NEXT_PUBLIC_SITE_URL: https://example.com
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version-file: ".nvmrc"
          cache: "npm"
      - run: npm ci
      - run: npm run build
```

Notes on the sketch:

- All three commands come from `package.json` scripts (`npm run …`), satisfying AC Scenario 6.
- No `secrets.*` reference appears anywhere — forked PRs run identically (AC Scenario 8).
- `node-version-file: ".nvmrc"` keeps Node version pinned to whatever `.nvmrc` says (currently `22`); upgrading Node is then a one-line change in `.nvmrc` (AC Scenario 5).

## 6. Branch protection click-path (manual, post-merge)

Branch protection is **not** part of the YAML — it lives in GitHub repo settings. The implementer must complete this after the workflow has run at least once on `main` (so GitHub knows the job names and they appear in the picker).

**Click-path (GitHub web UI):**

1. Open the repo on GitHub: `https://github.com/<owner>/aff-store`.
2. **Settings** (top tab) → **Rules** (left sidebar) → **Rulesets** → **New ruleset** → **New branch ruleset**.
   - _(Equivalent legacy path: Settings → Branches → "Branch protection rules" → "Add rule".)_
3. **Ruleset name:** `Protect main`.
4. **Enforcement status:** **Active**.
5. **Bypass list:** leave empty (no admin bypass).
6. **Target branches:** click **Add target** → **Include by pattern** → enter `main` → **Add Inclusion pattern**.
7. Under **Branch rules**, enable:
   - **Restrict deletions** (prevents accidental `main` deletion).
   - **Require a pull request before merging** → (optional, per project preference).
   - **Require status checks to pass** → click **Add checks** → search for and add **each of**:
     - `typecheck`
     - `lint`
     - `build`
   - Tick **Require branches to be up to date before merging**.
   - **Block force pushes**.
8. Click **Create**.

**Verification of branch protection (AC Scenario 7):**

- Open a throwaway PR that intentionally introduces a TS error.
- Confirm the `typecheck` check fails on the PR and the **Merge** button is disabled with "Required statuses must pass".
- Revert the breaking change, confirm checks go green, confirm Merge re-enables. Close the throwaway PR.

> If the job names in the picker do not appear, push any branch / open any PR first so the workflow runs once — GitHub only registers a check name after it has been reported at least once.

## 11. Risks / open items

- **First CI run won't gate the US00113 PR itself.** Branch protection is applied after merge, so the PR that introduces `ci.yml` cannot be blocked by `ci.yml` (chicken-and-egg). The PR's checks will still run and must be green; the merge gate just won't be enforced for this one PR. This is unavoidable and acceptable.
- **Job-name registration delay.** Required-check pickers in GitHub's branch-protection UI only show check names that have run at least once on the repo. If the picker is empty when the operator clicks through Section 6, push any branch first to register the names, then return to Settings.
- **Non-default-branch workflow caveat (US00114-related, mentioned for context).** Cron schedules only fire on the default branch; `pull_request` and `push` triggers do **not** have this restriction, so US00113 will work on any branch. No action needed.
- **`npm ci` requires `package-lock.json`.** Verified present at repo root. If it ever gets deleted, all three jobs will fail loudly with a clear error — fail-fast is fine.
- **`actions/checkout@v4` and `actions/setup-node@v4` major versions.** Current at time of writing. Future Dependabot upgrades to `@v5` etc. are routine; no special handling.
- **Cancellation policy (D7) means an in-flight run is killed on a new push.** Reviewers might briefly see "cancelled" status on superseded commits. Acceptable; the latest commit's run is what gates the merge.
- **Format check (`npm run format`) is intentionally not part of CI.** AC does not require it. If the team later wants Prettier to gate merges, add a fourth `format` job mirroring `lint`. Tracked as an open future enhancement, not blocking US00113.
- **Vercel-side Node version is not controlled by `.nvmrc` automatically.** Vercel reads its own Node version setting in project settings → Build & Development Settings, or honors `engines.node` if configured. Confirm via Vercel UI in US00111 that Vercel's Node major matches `22`. If they drift, AC Scenario 5 ("matches the version Vercel uses") is at risk — escalate to US00111 implementer.
- **Forked-PR pre-flight is a manual ask.** AC Scenario 8 is verified statically (no `secrets.*` in YAML) plus optionally with a real fork. The static check is sufficient for the story; the live test is a "nice to have".
