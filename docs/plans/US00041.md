**Explicitly out of scope** (owned by sibling stories or downstream features):

- `<ProductCard />` and its consumption of `formatVnd()` → **US00042**.
- Product detail page price slot → **US00046**.

## 5. Concrete diff sketch

### 5.1 `lib/format.ts` (new)

```ts
/**
 * Price & currency formatting helpers — F0004 / US00041.
 *
 * Single chokepoint for rendering monetary values in the storefront. Every
 * card, detail page, related-products row, inline MDX embed, and (future)
 * homepage featured-pick must call `formatVnd()` rather than reaching for
 * `Intl.NumberFormat` or hand-rolling a "₫" prefix. A future change (e.g.,
 * a thinner non-breaking space, a different separator, a compact "1,2tr"
 * shorthand) then touches exactly one file.
 *
 * SSG determinism (spec Scenario 7):
 *   `Intl.NumberFormat("vi-VN", …)` is intentionally NOT used. ICU data can
 *   differ subtly between Node builds (e.g. non-breaking-space vs. regular
 *   space between digits and currency symbol), which would make HTML
 *   non-deterministic across Vercel build pools and break snapshot review.
 *   Manual digit-grouping with the canonical "\B(?=(\d{3})+(?!\d))" regex
 *   produces byte-identical output on every machine.
 */

/**
 * Render an integer dong amount as the canonical Vietnamese price string,
 * `"₫1.200.000"` (₫ prefix, dot thousands separator, no decimal places).
 *
 * - Non-integer inputs are rounded to the nearest integer dong (`Math.round`).
 * - Zero renders as `"₫0"`.
 * - Sub-thousand amounts render without separator (`formatVnd(390) === "₫390"`).
 * - Non-finite inputs (NaN, ±Infinity) fall back to `"₫0"`; the price loader
 *   in `lib/products.ts` already rejects bad data at build time, so this is a
 *   belt-and-braces sentinel that should never appear in shipped HTML.
 * - Negative inputs render as `"-₫1.200.000"` (sign outside the currency mark).
 *   The loader already rejects `price < 0`, so this only surfaces for direct
 *   callers that bypass the loader path.
 *
 * @example
 *   formatVnd(1_200_000) // "₫1.200.000"
 *   formatVnd(390)       // "₫390"
 *   formatVnd(0)         // "₫0"
 *   formatVnd(390_000.7) // "₫390.001"
 */
export function formatVnd(amount: number): string {
  const rounded = Math.round(amount);

  if (!Number.isFinite(rounded)) {
    return "₫0";
  }

  const sign = rounded < 0 ? "-" : "";
  const digits = Math.abs(rounded)
    .toString()
    // Insert "." between every group of 3 digits from the right.
    // \B = non-word-boundary; (?=(\d{3})+(?!\d)) = followed by groups of 3
    // digits that are NOT themselves followed by a digit. Matches between
    // each thousand and leaves the leftmost group untouched.
    .replace(/\B(?=(\d{3})+(?!\d))/g, ".");

  return `${sign}₫${digits}`;
}
```

## 10. Risks / open items

- **Locale / non-breaking space.** Vietnamese sources sometimes use a non-breaking space (`U+00A0`) between the digits and the currency mark (e.g., `1.200.000 ₫`). The spec explicitly puts `₫` *before* the digits with no space (`₫1.200.000`); we honour that. If the design later wants the trailing-space style, this is a one-file edit.
- **Compact format (`1,2tr`, `500k`) for the filter buckets in US00044.** F0004 spec §6 lists predefined bucket labels (`<200k`, `200–500k`, `500k–1tr`, `1tr–3tr`, `>3tr`) and those are hard-coded enumerations — no helper needed for v1. If a future story wants to derive bucket labels from numeric bounds, that's a new `formatVndCompact()` export living alongside `formatVnd()`.
- **Negative-input rendering** (`"-₫1.200.000"`) is not in the spec. It exists only as defensive output; no AC depends on it. Decision: cheap and harmless, keeps the helper composable. If a reviewer disagrees, drop the `sign` lines — five-line removal.
- **`NaN` / `Infinity` sentinel** returning `"₫0"` rather than throwing is a deliberate choice (D6). If the project later adds Vitest, the first regression test should pin this behaviour explicitly so a "throw on bad input" refactor would fail a test rather than silently change shipped HTML.
- **No unit-test runner in CI.** D9 documented the rationale. The build itself doesn't exercise `formatVnd` (no consumer yet), so the §6 step 5 harness is the only one-shot verification. Once US00042 lands and renders a card, `next build` becomes a real consumer-side smoke test for the helper.
- **`Math.round` half-up behaviour.** JS `Math.round(0.5) === 1`, `Math.round(-0.5) === 0` (round-half-towards-positive-infinity). The spec example only covers `390000.7 → 390001`, which is unambiguous. Half-exact edge cases (`x.5`) are not specified; if anyone ever cares, document the round-half-up convention in the JSDoc.
- **Helper not exported via a barrel.** The project has `types/index.ts` as a barrel but no `lib/index.ts`. Following the existing pattern of importing each helper directly (`@/lib/affiliate`, `@/lib/posts`, `@/lib/products`), `@/lib/format` is the right path. Don't introduce `lib/index.ts` for one helper.
- **What if `<ProductCard />` (US00042) is built before the helper lands?** The card cannot pass an AC without a price-rendering call site. Resolution: this story should ship before or in the same wave as US00042. If sequencing slips, the card author can stub `formatVnd` temporarily — but the chokepoint convention then has a brief gap, which the §7 Scenario 6 grep would catch in code review. Cheap to avoid by sequencing.
