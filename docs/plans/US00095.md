## 4. Article schema shape (final emitted JSON)

```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "<post.title â€” verbatim from frontmatter>",
  "image": "<absolute URL of post.coverImage OR /static/images/og-default.png>",
  "datePublished": "<post.publishedAt â€” ISO 8601 from frontmatter>",
  "author": {
    "@type": "Organization",
    "name": "<SITE_NAME>",
    "url": "<NEXT_PUBLIC_SITE_URL>"
  },
  "publisher": {
    "@type": "Organization",
    "name": "<SITE_NAME>",
    "url": "<NEXT_PUBLIC_SITE_URL>"
  },
  "mainEntityOfPage": {
    "@type": "WebPage",
    "@id": "<NEXT_PUBLIC_SITE_URL>/bai-viet/<slug>/"
  }
}
```

Notes:
- `headline` is the full title untruncated (F0009 Â§6: "titles pass through the template untruncated"; spec scenario 2 says we emit the full title and let Google truncate display).
- `image` is **always** an absolute URL string (single string, not an array) â€” Article schema accepts string or array; we keep it simple.
- `datePublished` is the raw `publishedAt` ISO string (e.g., `"2026-05-20"`). No `Date` round-trip, no `formatPostDate()` â€” that helper is human-facing only.
- `author` and `publisher` both resolve to the same Organization until a posts-level author field exists (see Q1).

## 5. New helper: `lib/article-schema.ts`

Pure, no JSX. Exports one function:

```ts
import type { Post } from "@/types";
import { env } from "@/lib/env";
import { SITE_NAME } from "@/lib/site";

const DEFAULT_OG_IMAGE = "/static/images/og-default.png"; // owned by US00091

function toAbsolute(pathOrUrl: string): string {
  // If already absolute (http(s)://), return as-is.
  // Else, join NEXT_PUBLIC_SITE_URL + pathOrUrl (ensure single slash).
}

export interface ArticleSchema {
  "@context": "https://schema.org";
  "@type": "Article";
  headline: string;
  image: string;
  datePublished: string;
  author: { "@type": "Organization"; name: string; url: string };
  publisher: { "@type": "Organization"; name: string; url: string };
  mainEntityOfPage: { "@type": "WebPage"; "@id": string };
}

export function buildArticleSchema(post: Post): ArticleSchema {
  const siteUrl = env.NEXT_PUBLIC_SITE_URL.replace(/\/$/, "");
  const cover = post.coverImage?.trim() ? post.coverImage : DEFAULT_OG_IMAGE;
  return {
    "@context": "https://schema.org",
    "@type": "Article",
    headline: post.title,
    image: toAbsolute(cover),
    datePublished: post.publishedAt,
    author: { "@type": "Organization", name: SITE_NAME, url: siteUrl },
    publisher: { "@type": "Organization", name: SITE_NAME, url: siteUrl },
    mainEntityOfPage: {
      "@type": "WebPage",
      "@id": `${siteUrl}/bai-viet/${post.slug}/`,
    },
  };
}
```

Conventions honored:
- Imports `env` from `lib/env.ts` (not `process.env`) â€” single source of truth for the base URL (F0009 Â§6).
- Imports `SITE_NAME` from `lib/site.ts` â€” no string literal duplication.
- Uses `Post.slug`, `Post.title`, `Post.coverImage`, `Post.publishedAt` from `@/types` â€” frontmatter contract (US00066).
- Fallback to `DEFAULT_OG_IMAGE` mirrors US00092's "pages without an image fall back to the site default" rule (F0009 spec scenario 3 of US00092) and the F0009 edge case "Product with no images / post with no cover".

## 11. Open questions for reviewer

1. **Author identity.** Post frontmatter has no `author` field today (`types/post.ts` confirmed: only `title`, `summary`, `publishedAt`, `category`, `tags`, `coverImage`). This plan emits `author = { "@type": "Organization", name: SITE_NAME, url: siteUrl }` â€” i.e., the site itself is the author until per-post bylines exist. Acceptable, or should we (a) add an optional `author` string to frontmatter now and fall back to `SITE_NAME`, or (b) introduce a constant `DEFAULT_AUTHOR = SITE_NAME` in `lib/site.ts` for future override? Note also that `app/bai-viet/[slug]/page.tsx` already renders `<span className={styles.byline}>{SITE_NAME}</span>` in the visible meta row, so emitting `SITE_NAME` here keeps visible byline â‰ˇ structured-data author.
2. **`SITE_NAME` value.** `lib/site.ts` currently exports `SITE_NAME = "aff-store"`, which is a slug, not a brand name. Should we rename to a Vietnamese display name (e.g., `"Aff Store"` or whatever the launch brand is) before this lands, so the JSON-LD doesn't ship with `"aff-store"` as the author/publisher name? Either way, **this story will not change that value** â€” flagging it because the JSON-LD makes it externally visible.
3. **Default OG image timing.** US00091 owns `/static/images/og-default.png`. If US00091 hasn't merged when this story implements, do we (a) block on US00091, (b) commit a placeholder PNG inside US00095 and hand it off, or (c) introduce the fallback constant pointing at a path that doesn't yet exist (build won't fail because `next/image` is not involved here â€” it's just a string)? Recommend (a) to keep ownership clean.
4. **Image schema type.** Article schema accepts `image` as a string, an array of strings, or an `ImageObject`. We're emitting a single string for simplicity. Google's Rich Results guidance recommends `ImageObject` with explicit `width`/`height` for best results. Acceptable to defer to a follow-up, or include now?
5. **`dateModified`.** Not in spec. Posts don't have a `modifiedAt` field. Skipping. Confirm.
6. **Trailing slash on `mainEntityOfPage["@id"]`.** Plan emits `/bai-viet/<slug>/` with a trailing slash to match the existing `alternates.canonical` pattern in `app/bai-viet/[slug]/page.tsx` (line 32). Confirm this is the project's canonical convention. (Spec Â§6 implies yes â€” "self-referencing canonical â€¦ `/san-pham/?page=2`" uses trailing slashes elsewhere.)

## 12. Risks / non-risks

- **Non-risk:** No runtime behavior change for users â€” JSON-LD is invisible.
- **Non-risk:** No client-side JS added.
- **Risk:** If US00094's `<JsonLd />` API differs from `<JsonLd data={...} />` (e.g., takes `id` or `children`), this plan's import line must be adjusted. Mitigation: review US00094's plan/PR before implementing.
- **Risk:** A post with malformed `publishedAt` would emit a broken `datePublished`. The post loader (`lib/posts.ts`) validates `publishedAt` as a non-empty string but does **not** validate ISO format. Out of scope to fix here; flag for a future hardening story if Rich Results Test complains.
