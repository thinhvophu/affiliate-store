# US00094 — Product JSON-LD

## 1. Business intent

Emit a valid `Product` JSON-LD `<script>` block in the statically generated HTML for every `/san-pham/[slug]/` page so Google can qualify the page for product rich results. Same build-time seam (`<JsonLd>`) is reused by US00095 (Article) and US00096 (BreadcrumbList).

## 2. Codebase touchpoints (quoted from `CLAUDE.md` structure doc)

This story owns one new component, one new lib helper, and a single edit to the product detail page. It introduces the **shared `<JsonLd>` seam** that US00095 + US00096 will consume.

Affected rows in the structure doc:

- `app/san-pham/[slug]/page.tsx` — "Product detail page — SSG per slug, generateStaticParams + notFound() (US00046)". We add a JSON-LD `<script>` inside the existing `<article>` (or as a sibling at the page root).
- `components/` — new file `JsonLd.tsx` (Server Component) following the **PascalCase.tsx + co-located `<Name>.module.css`** convention. No CSS module here (the component renders only a `<script>`; no visible markup).
- `lib/` — new file `lib/product-schema.ts` (pure, no JSX) that builds the schema object. Follows the **"no JSX in lib/"** rule (per `## Module conventions`).
- `lib/env.ts` — already exposes `env.NEXT_PUBLIC_SITE_URL` and fails the build loudly when missing (§5 Edge Cases). We import from it; no edit.
- `lib/products.ts` — already exposes `getAllProducts()` and guarantees `images.length ≥ 1` at build time (§5 Edge Cases). We consume it; no edit.
- `types/product.ts` — canonical `Product` shape with `price: number` (raw VND, no formatting). We consume it; no edit.

No changes to `lib/format.ts` (`formatVnd` is human-display only — §6 Technical Considerations).
No changes to `lib/categories.ts`.
No changes to `<AffiliateLink>` or other product surfaces.

## 3. Architecture decisions

### 3.1 Shared `<JsonLd>` component lives in `components/`, schema builders live in `lib/`

- **`components/JsonLd.tsx`** holds the JSX (`<script type="application/ld+json" dangerouslySetInnerHTML={…}>`) and the safe-serialization logic. Server Component — no `"use client"`.
- **`lib/product-schema.ts`** is a pure data-shaping module: `buildProductSchema(product, { siteUrl }) → object`. No JSX, complies with the **"no JSX in lib/"** rule.

This split lets US00095 reuse `<JsonLd>` with an `lib/article-schema.ts` builder, and US00096 reuse it with the breadcrumb items array — without duplicating the escaping logic.

### 3.2 Safe serialization (per Next.js docs + §5 Edge Cases)

The Next.js JSON-LD guide (`node_modules/next/dist/docs/01-app/02-guides/json-ld.md`) is explicit: `JSON.stringify` alone does **not** sanitize against XSS or `</script>` injection inside the script block. `<JsonLd>` must:

1. Call `JSON.stringify(data)` once.
2. Replace **every** `<` with the unicode escape `<`. This covers the `</script>` close-tag attack (the `<` becomes `<`, so `</script>` becomes `</script>` — inert inside the JSON literal but still re-parsed by Google as `</script>`).
3. Replace ` ` and ` ` (line and paragraph separators — valid in JSON, invalid in legacy JS contexts) with their escaped forms `\\u2028` / `\\u2029`. Defensive; cheap.
4. **No need to escape `>` or `&`** — the `<` escape alone defeats the closing-tag attack, and JSON.stringify already escapes `"`. Diacritics (Vietnamese tones) are preserved by `JSON.stringify`'s default behavior (UTF-8, no `\uXXXX` for printable chars by default) and are valid inside a `<script>` element — no extra handling required.

The component sets `<script type="application/ld+json">`, so the browser does not execute it as JS; the escaping protects Google's parser, not the runtime.

### 3.3 Schema shape (US00094 Scenario 1)

```ts
{
  "@context": "https://schema.org",
  "@type": "Product",
  name: product.name,
  image: <absolute URL, first image>,            // see 3.4
  description: product.description,              // full text, untruncated
  brand: { "@type": "Brand", name: product.brand },
  url: <absolute canonical URL>,                 // <siteUrl>/san-pham/<slug>/
  offers: {
    "@type": "Offer",
    price: product.price,                        // RAW number, NO formatVnd, NO separator
    priceCurrency: "VND",
    availability: "https://schema.org/InStock",  // v1 default; see 3.6
    url: <absolute canonical URL>,               // mirror url; per Google's Product rich-result guidance
  },
}
```

Notes:

- `brand` uses the nested `{ "@type": "Brand", name }` form (Google's preferred shape; the bare-string form is also accepted but the nested form passes the Rich Results Test cleanly).
- `description` is the **full** product description (no truncation). Google truncates display-side; truncation only applies to `<meta name="description">` (US00092, not this story).
- `image` is a **single absolute URL string** (the first item of `product.images`). The schema also allows `string[]`; we keep it scalar for now and revisit if multi-image rich results are needed.

### 3.4 Absolute URL resolution (§6 Technical Considerations)

Per the spec:

> *Single source of truth for the base URL: canonical, OG, sitemap, and robots URLs all derive from `NEXT_PUBLIC_SITE_URL` via `metadataBase`.*

US00091 is the story that wires `metadataBase` into the root layout. **US00094 must not assume US00091 has shipped first.** To stay decoupled:

- The schema builder accepts an explicit `siteUrl: string` argument; the page reads it from `env.NEXT_PUBLIC_SITE_URL` and passes it in.
- Absolute URL construction lives in one tiny helper inside `lib/product-schema.ts`: `toAbsoluteUrl(siteUrl, path)`. It trims a trailing `/` from `siteUrl`, ensures `path` starts with `/`, and joins them. No `new URL()` allocation per call needed.
- The product image path (e.g., `/static/images/products/<slug>.jpg`) is joined the same way.

When US00091 lands and `metadataBase` is set, this code does **not** change — JSON-LD bodies are JSON strings, not `Metadata` entries that benefit from Next.js's metadataBase resolution. We always emit fully absolute URLs in the JSON.

### 3.5 Canonical URL shape

Mirror the existing convention in `app/san-pham/[slug]/page.tsx` `generateMetadata`:

```
<siteUrl>/san-pham/<slug>/
```

(trailing slash, matches the route segment). This is the same string used for `url` and `offers.url` in the schema.

### 3.6 `availability` default

The product JSON has no stock field today. Per F0009 §5 Edge Cases, we ship with a constant default of `https://schema.org/InStock`. If a future story adds stock tracking, swap the constant for a per-product value — single chokepoint in `buildProductSchema`. This is called out under **Open Questions** below in case the reviewer prefers `OnlineOnly` or a different default.

### 3.7 Where the `<script>` lives in the React tree

Render `<JsonLd data={schema} />` as a direct child of the page's outermost element (the `<div className={styles.container}>` in `app/san-pham/[slug]/page.tsx`). Placing it inside `<article>` is also valid, but a top-level sibling is the conventional Next.js placement (per the official docs example) and keeps the structured-data block visually separable from product copy.

## 4. Files to create / modify

### Create

1. **`components/JsonLd.tsx`** — Server Component.

   ```tsx
   interface JsonLdProps {
     data: Record<string, unknown>;
   }

   export function JsonLd({ data }: JsonLdProps) {
     const json = JSON.stringify(data)
       .replace(/</g, "\\u003c")
       .replace(/ /g, "\\u2028")
       .replace(/ /g, "\\u2029");
     return (
       <script
         type="application/ld+json"
         dangerouslySetInnerHTML={{ __html: json }}
       />
     );
   }
   ```

   No `"use client"`. No CSS module (no visible markup).

2. **`lib/product-schema.ts`** — pure data shaper. Exports:
   - `buildProductSchema(product: Product, opts: { siteUrl: string }): Record<string, unknown>`
   - (internal) `toAbsoluteUrl(siteUrl: string, path: string): string`
   - (exported, for reuse + testing) `PRODUCT_AVAILABILITY_IN_STOCK = "https://schema.org/InStock"` constant.

3. **`lib/product-schema.test.ts`** — Vitest unit tests (see §6 Test plan).

4. **`components/JsonLd.test.tsx`** *(optional)* — Vitest unit test verifying `<` → `<` escaping. Pure string assertion via `renderToStaticMarkup` from `react-dom/server`; keep small. If the existing repo has no React renderer wired into Vitest yet, fold the escape test into `lib/product-schema.test.ts` by exposing a small `escapeJsonForScriptTag(str)` helper from the component file — see Open Questions.

### Modify

5. **`app/san-pham/[slug]/page.tsx`**:
   - Add `import { JsonLd } from "@/components/JsonLd";`
   - Add `import { buildProductSchema } from "@/lib/product-schema";`
   - Add `import { env } from "@/lib/env";`
   - Inside `ProductDetailPage`, after resolving `product`, compute `const productSchema = buildProductSchema(product, { siteUrl: env.NEXT_PUBLIC_SITE_URL });`
   - Render `<JsonLd data={productSchema} />` as the first child of the existing `<div className={styles.container}>`.
   - **No other changes** — keep US00046's existing structure intact.

### Not modified (intentional)

- `lib/format.ts` — `formatVnd` is human-display only. Raw `product.price` (number) is used directly.
- `lib/products.ts` — `images.length ≥ 1` invariant is already enforced; we trust it.
- `app/san-pham/[slug]/product-detail.module.css` — no visual change.
- `app/layout.tsx` — `metadataBase` belongs to US00091; out of scope.
- Other product surfaces (cards, listings) — no JSON-LD on those (per spec).

## 6. Test plan (write tests first)

### `lib/product-schema.test.ts` (Vitest)

Following the `lib/format.test.ts` style (Vitest + `describe`/`it`/`expect`):

- **Builds a complete Product schema**: given a fixture `Product` with name, brand, description, price `390000`, one image at `/static/images/products/foo.jpg`, and `siteUrl: "https://example.com"`, the output contains:
  - `"@context": "https://schema.org"`
  - `"@type": "Product"`
  - `name`, `description`, `brand: { "@type": "Brand", name: "Logitech" }`
  - `image: "https://example.com/static/images/products/foo.jpg"` (absolute, scalar)
  - `url: "https://example.com/san-pham/foo/"` (trailing slash)
  - `offers.price === 390000` (number, **not** string, **not** `"390.000 ₫"`)
  - `offers.priceCurrency === "VND"`
  - `offers.availability === "https://schema.org/InStock"`
  - `offers.url === url`
- **`offers.price` is the raw number** — explicitly `expect(typeof schema.offers.price).toBe("number")` + `expect(schema.offers.price).toBe(390000)`. Guards against any future `formatVnd` regression.
- **Trailing-slash siteUrl** — `siteUrl: "https://example.com/"` produces identical output (no double `//`).
- **Multi-image product** — `images: [a, b, c]` still produces `image: a` (first image, scalar string).
- **Brand nested form** — `brand` is an object with `@type: "Brand"`, not a bare string.
- **Long Vietnamese description with diacritics & quotes** — `description: 'Chuột "siêu nhẹ" với độ trễ thấp'` round-trips through `JSON.stringify` correctly (smoke test for §5 Edge Case "special characters").

### `<JsonLd>` escape test

## 9. Edge cases (from F0009 §5)

| Case | Handling |
| --- | --- |
| Missing `NEXT_PUBLIC_SITE_URL` | `lib/env.ts` throws at build (already). No new handling. |
| Product with no images | Cannot occur — `lib/products.ts` enforces `images.length ≥ 1` at build. |
| Special chars / Vietnamese diacritics in name/description | `JSON.stringify` preserves them; `<` escape covers script-tag injection. |
| Long description | No truncation in JSON-LD (description truncation belongs to US00092 / `<meta>`). |
| Out-of-stock product | Out of scope today; default `InStock` constant. See Open Questions. |
| Unknown slug | `notFound()` runs before the schema is built; no schema emitted. |

## 11. Risks & open questions for reviewer

1. **`availability` default.** Spec says `availability` is required but does not pin a value. Plan ships `https://schema.org/InStock`. Acceptable, or should we use `OnlineOnly` since we never hold inventory ourselves? *(Recommend: `InStock` — matches reality from the user's perspective; Shopee fulfils the order.)*
2. **`<JsonLd>` test framework.** The repo currently tests pure lib functions (Vitest, no React renderer). Should we wire `@testing-library/react` for the `<JsonLd>` JSX test, or expose a tiny `escapeJsonForScriptTag` helper from the component and unit-test that? *(Recommend: expose the helper — keeps the test surface pure and avoids dragging in a React renderer just for one assertion.)*
3. **Should `<JsonLd>` accept an array of objects** (so future pages emitting both `Article` + `BreadcrumbList` can wrap them in one `<script>` block with `@graph`)? *(Recommend: ship as single-object now; the @graph wrapper can be added without a breaking change since `<JsonLd>` is typed as `Record<string, unknown>`. US00095/US00096 can each emit their own `<script>` tag — Google explicitly supports multiple JSON-LD blocks per page.)*
4. **No `schema-dts` typing.** The Next.js doc suggests it; we skip it to avoid a dependency. The `Record<string, unknown>` return type is sufficient since the data is consumed only by `JSON.stringify`. Confirm OK.
5. **F0009 sibling ordering.** US00094 introduces the shared seam. If US00095 or US00096 implementation lands first, they will need to create `components/JsonLd.tsx` themselves. The plans for those two stories should explicitly note "depends on US00094 (or whoever creates `components/JsonLd.tsx` first)" so the seam is created only once.

## Resolved cross-cutting decisions (F0009 review — 2026-06-21)

- **Single absolute-URL chokepoint:** build `image`, `url`, and `offers.url` via `absoluteUrl()` from `lib/seo.ts` (US00091). Do **not** re-derive absolute URLs from `env` / `process.env` in `lib/product-schema.ts` — that would fork the chokepoint. (This overrides the draft's "decoupled from US00091" note.)
- **Trailing slash:** every URL in the Product schema carries a trailing slash per the site-wide policy.
- **`availability`** defaults to `https://schema.org/InStock` (no stock data in v1).
