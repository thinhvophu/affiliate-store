# Implementation Plan — US00121: Ingestion CLI foundation & candidate validation

**Status:** Done
**Feature:** F0012 — Automated Content Ingestion Pipeline
**Spec ref:** `docs/specs/F0012.md` § US00121
**Predecessors:** none (this is the foundation story of F0012).
**Successors:** US00122 (slug/dedupe), US00123 (image staging), US00124 (scrape source), US00125 (file source) all build on the candidate model + validation engine established here.

---

## 1. Scope

Strictly:

1. Create a new top-level **`scripts/`** directory and the CLI entry point `scripts/ingest-products.ts`, wired as `npm run ingest:products`.
2. Add the **`tsx`** devDependency so the script can run TypeScript with the `@/*` path alias resolved (decision D2).
3. Define the **shared in-memory candidate model** (`scripts/ingest/candidate.ts`) that both future sources (scrape / file) normalize into.
4. Build the **validation engine** — one function that, per candidate, reuses the existing build-time chokepoints **without re-implementing them**:
   - `assertAffiliateUrl(url, slug)` from `lib/affiliate.ts` (host allow-list),
   - `assertCategoryRegistered(category, slug)` from `lib/categories.ts` (registry),
   - a required-`Product`-field presence check mirroring `types/product.ts` (name, brand, price, images, specs, description).
5. **Argument parsing** (`--category`, `--source`, `--dry-run`, plus source-specific flags passed through) with a **category preflight** that calls `assertCategoryRegistered` *before any source runs* (Scenario 1).
6. A **summary reporter** that prints `added / skipped-duplicate / rejected-invalid`, each line naming the candidate and the reason.
7. **`--dry-run`** mode: run the full pipeline (parse → validate) and print the summary, but write **no** `content/products/*.json` and download **no** image (Scenario 3).
8. **Non-zero exit** with the offending candidate/line/slug on any validation-halting failure, mirroring the existing `next build` failure style.

**Explicitly out of scope** (owned by sibling stories):

- The **scrape** source adapter → **US00124**.
- The **file** source adapter → **US00125**. (US00121 ships with a trivial stub/in-memory source so the engine is testable end-to-end; the real adapters replace it.)
- **Image download/staging** → **US00123** (in US00121 the "write fixture" step is stubbed behind the same accept/reject flow; dry-run never writes anyway).
- **Slug generation & dedupe** → **US00122** (US00121 uses a naive slug placeholder only far enough to name candidates in the summary; the canonical slugifier + dedupe land in US00122).
- **MDX scaffold** (`scaffold:post`) → **US00126**.

> **Sequencing note:** US00121 delivers the *skeleton that runs*: arg parse → preflight → (stub source) → validate → report → dry-run. Real fixture writing is only exercised once US00122 (slug/dedupe) + US00123 (staging) land. To keep US00121 independently verifiable, its non-dry-run write path may write a fixture from an already-local candidate (no remote image), and the summary/exit-code behavior is fully testable now.

## 3. Approved Decisions

| #   | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| D1  | Script location | New top-level **`scripts/`** dir; kept out of `lib/`, `app/`, `components/` | Spec §6: this is build tooling, not app code or a rendered route. The "no JSX in lib/" rule and the app/components conventions don't apply to a dev-only CLI. |
| D2  | TS runner | **Add `tsx` devDep**; run via `tsx scripts/ingest-products.ts` | Resolves `@/*` paths so the script imports the real chokepoints; ESM-native, no build step. (User-approved.) |
| D3  | Candidate model | One `Candidate` interface (`scripts/ingest/candidate.ts`) that is **source-agnostic** and pre-slug/pre-staging | Spec §6 "Shared candidate model": both sources normalize into it so validation/staging/dedupe run once, source-agnostic. |
| D4  | Reuse, don't reimplement | Validation **imports** `assertAffiliateUrl` + `assertCategoryRegistered` | Spec §6 hard rule: no file outside `lib/affiliate.ts` may validate affiliate URLs; no file outside `lib/categories.ts` may validate categories. Importing keeps pipeline ⇄ build in lock-step. |
| D5  | Preflight ordering | Category checked **before** invoking any source | Scenario 1: unregistered `--category` must halt non-zero before a source runs or a file is written. |
| D6  | Rejection is per-candidate, non-fatal | An invalid candidate is **collected as a rejection** and reported; the run continues for the rest | Scenario 2 + US00124 Scenario 3: one bad candidate must not abort ingesting the good ones. Distinct from D5's fatal preflight and D7's fatal input errors. |
| D7  | Fatal vs. per-candidate failures | **Fatal (exit non-zero, halt):** unregistered category, unparseable CLI args, malformed input file (US00125). **Per-candidate (collect + continue):** missing field, bad host, image 404 (US00123). | Mirrors existing build-time style: structural errors halt; content errors are itemized. |
| D8  | `--dry-run` placement | Guard the **write + download** step only; everything up to the summary runs identically | Scenario 3: dry-run must still parse, validate, dedupe, and report — it just performs no side effects. |
| D9  | Summary shape | `{ added[], skippedDuplicate[], rejected[{candidate, reason}], requested, shortfall }`; printed as three labelled groups + a footer line | Human reviews before committing (spec §3 step 7). Reason strings name the offending field/host/slug. |
| D10 | Arg parser | Hand-rolled minimal `--key=value` / `--flag` parser in `scripts/ingest/args.ts` (no new runtime dep) | The arg surface is tiny and fixed; a dependency (yargs/commander) is overkill for a dev tool and adds supply-chain surface. Node's built-in `util.parseArgs` is the fallback if hand-rolling proves fiddly. |
| D11 | Exit codes | `0` success (incl. "some candidates rejected but run completed"); **non-zero** only on fatal (D7) errors | Scenario 2 rejects a candidate but the run itself succeeded — that is exit 0 with rejections in the summary. A fatal arg/category/input error is exit 1. |

### 4.3 Candidate model (concrete)

```ts
// scripts/ingest/candidate.ts
export interface Candidate {
  name: string;
  brand: string;
  price: number;                    // VND integer; missing → rejection, never a guessed default
  affiliateUrl: string;             // raw, validated by assertAffiliateUrl (NOT parsed here)
  description: string;
  specs: Record<string, string>;
  imageUrls: string[];              // remote URLs pre-staging (US00123 turns these local)
  category: string;                 // injected from --category (single source of truth)
  sourceRef: string;                // "scrape#3" or "file:line 12" — for summary/error messages
}

export interface AcceptedCandidate extends Candidate {
  slug: string;                     // derived in US00122
  images: string[];                 // local /static/... paths after US00123 staging
}

export interface Rejection {
  ref: string;                      // candidate.sourceRef
  name: string;                     // best-effort human label
  reason: string;                   // "missing required field: price" / "host \"foo.com\" not allowed"
}
```

### 4.4 Validation engine (concrete)

```ts
// scripts/ingest/validate.ts
import { assertAffiliateUrl } from "@/lib/affiliate";
import { assertCategoryRegistered } from "@/lib/categories";

const REQUIRED_STRINGS = ["name", "brand", "affiliateUrl", "description"] as const;

export function validateCandidate(c: Candidate, provisionalSlug: string): Rejection | null {
  for (const f of REQUIRED_STRINGS) {
    if (typeof c[f] !== "string" || c[f].trim() === "")
      return rej(c, `missing required field: ${f}`);
  }
  if (typeof c.price !== "number" || !Number.isFinite(c.price) || c.price < 0)
    return rej(c, "missing or invalid required field: price");
  if (c.specs == null || typeof c.specs !== "object" || Object.keys(c.specs).length === 0)
    return rej(c, "missing required field: specs");
  if (!Array.isArray(c.imageUrls) || c.imageUrls.length === 0)
    return rej(c, "missing required field: images");
  try { assertAffiliateUrl(c.affiliateUrl, provisionalSlug); }
  catch (e) { return rej(c, e instanceof Error ? e.message : String(e)); }
  try { assertCategoryRegistered(c.category, provisionalSlug); }
  catch (e) { return rej(c, e instanceof Error ? e.message : String(e)); }
  return null;
}
```

> Note: the field checks intentionally mirror `lib/products.ts::validateProduct` so a candidate that passes here also passes `next build`. They are *pre-write* guards, not a second source of truth for URL/category rules — those stay imported.

## 10. Risks / open items

- **`tsconfig include`**: verify `scripts/**` is type-checked/linted. If `include` is narrow, add `scripts` (or a `scripts/tsconfig.json`) so `npm run typecheck` covers the CLI.
- **ESM/`@/*` at runtime**: `tsx` resolves tsconfig paths; confirm with a smoke run. If a path fails to resolve, the fix is a `tsx` version bump or `--tsconfig` flag, not switching to relative imports (which would break D4's clean-import intent).
- **Stub source coupling**: the temporary in-memory source in US00121 must be trivially removable so US00124/US00125 slot their adapters in without refactoring the entry point. Keep the source behind a `loadCandidates(args): Promise<Candidate[]>` seam.
- **`process.exitCode` vs `process.exit`**: prefer setting `process.exitCode` and returning so buffered stdout flushes before exit.
