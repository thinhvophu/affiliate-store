# Implementation Plan — US00047: Related products section (3–4 same-category)

**Status:** Approved
**Feature:** F0004 — Product Catalog
**Spec ref:** `docs/specs/F0004.md` § US00047
**Predecessors / siblings / consumers:**

- **Upstream dep — US00041** (`lib/format.ts` → `formatVnd()`). Indirect: each related card renders a price via `<ProductCard />`, which uses `formatVnd()`.
- **Upstream dep — US00042** (`components/ProductCard.tsx`). This story renders exactly N instances of `<ProductCard />` inside a section block. US00047 owns **no** card chrome — it owns selection + section layout only.
- **Upstream dep — US00046** (`app/san-pham/[slug]/page.tsx`). The detail page is the host. Per spec Scenario 7 of US00046 (“the related-products section (US00047) renders below the primary content, before the F0002 Footer”), US00047 is composed by US00046 — not the other way around.
- **Upstream dep — F0001** (`lib/products.ts`). `getAllProducts()` is the single data source. The selection helper added by this story is a pure derivation on the array `getAllProducts()` returns — already sorted by `publishedAt` desc with file-read order as the implicit tiebreaker.
- **Upstream dep — F0003** (`components/AffiliateLink.tsx` + `lib/affiliate.ts`). Indirect — every card in the row is a whole-card `<AffiliateLink>` (per US00042), so the F0003 ↔ F0007 contract is preserved transparently.
- **Sibling stories sharing the same wave (F0004):** US00041 (formatter), US00042 (card), US00043 (listing), US00044 (filters), US00045 (category page), US00046 (detail page). This plan is **selection logic + a section component**; the visual unit is delegated to US00042.

This story is the **catalog cross-link layer**. It increases time-on-site, gives Google a second-tier internal link surface for each detail page, and gives the visitor a path to compare alternatives without bouncing back to `/san-pham/`. None of that requires new chrome — only a deterministic selection of 3–4 sibling products and a tidy heading + grid wrapper rendered by the detail page.

## 1. Scope

Strictly:

1. Add a pure selection helper **`getRelatedProducts(currentProduct, allProducts?)`** to `lib/products.ts` (co-located with `getAllProducts()` and `getProductBySlug()` so the catalog data API stays in one file). Returns an array of `Product` of length `0 | 3 | 4`:
   - Prefer **same-category siblings** ordered by `publishedAt` desc, `slug` asc tiebreaker (Scenario 1, 2).
   - If fewer than 3 same-category siblings exist, **top up** from any other category, still excluding the current product, still ordered `publishedAt` desc / `slug` asc (Scenario 3).
   - If the catalog has fewer than 3 other products total (i.e. ≤ 3 products in the whole store including the current one), return an **empty array** so the consumer omits the section entirely (Scenario 4).
   - Target row size: **4 when ≥ 4 candidates are available, otherwise exactly 3**. No length 1, 2, or `> 4` row is ever returned.
2. Create a new Server Component **`components/RelatedProducts.tsx`** + co-located **`RelatedProducts.module.css`** that takes `products: Product[]` as a prop and renders:
   - A `<section>` with a Vietnamese heading "Sản phẩm liên quan" (Scenario 5 wording; US00046 § Scenario 10 cites this exact string).
   - A grid wrapper that mirrors the listing-grid breakpoints documented in F0004 (`2 cols < 768px`, `3 cols 768–1279px`, `3–4 cols ≥ 1280px` depending on count) — Scenario 5.
   - One `<ProductCard product={p} />` per array entry. **No** other markup, no extra anchors, no “Xem tất cả” link (out of scope; the listing link lives in the header nav).
   - Renders **nothing** (`return null`) when `products.length === 0`. Empty heading + empty grid is explicitly disallowed by Scenario 4.
3. Wire the helper into **`app/san-pham/[slug]/page.tsx`** (US00046’s file) so that the detail page calls `getRelatedProducts(product)` at render time (build time, since the route is statically generated) and passes the result to `<RelatedProducts />`. The mount point sits below the primary content `<article>` (gallery + heading + price + specs + CTA) and above the F0002 Footer (`<Footer />` is in `app/layout.tsx`, so "above the Footer" simply means it is the **last child of `<main>` for the detail route**, after the article).
4. Verification (no new test runner introduced — consistent with US00034 D8): three deterministic build-time fixture scenarios run by hand against the production build (§7). The implementer adds and removes the test fixtures locally; nothing test-related ships in git.
5. **Structure-doc update:** add `components/RelatedProducts.tsx` + `components/RelatedProducts.module.css` rows to the `components/` tree in `CLAUDE.md` and `README.md`, and bump the "Last updated" line.

**Explicitly out of scope** (owned by sibling stories or downstream features):

- The card visual itself (`<ProductCard />`) → **US00042**. This story imports the card; it does **not** restyle it, change its props, or add a `compact` variant for the related-row.
- The detail page chrome — `<h1>`, gallery, specs table, CTA, JSON-LD — → **US00046**. This story does not touch any of those, only the bottom-of-page mount.
- Listing grid / filters / category page / pagination → **US00043, US00044, US00045**. The related row deliberately does **not** reuse a "RelatedProductsContainer" abstraction with those screens — it is a small standalone grid because the consumer is a different page chrome (no `<ShellLayout />`, no left panel; see US00046 Scenario 8).
- Cross-category recommendation algorithms (collaborative filtering, brand-similarity, price-bucket-similarity) → out of scope by spec §2 ("related is strictly same-category in v1"). The top-up rule in Scenario 3 is the only cross-category logic and is purely chronological.
- “Sản phẩm phổ biến / Bán chạy” surfaces, “Sản phẩm cùng thương hiệu” surfaces → not in spec. v2+.
- The MDX `<ProductCard slug="…" />` wrapper used by blog posts (F0006).
- A *standalone* page like `/san-pham/[slug]/lien-quan/` — the related row is strictly a section inside the detail page, not its own route.
- `Math.random()`-based variation between builds — explicitly disallowed by Scenario 2 ("no randomness in SSG output").

## 6. Selection-algorithm spec (pseudocode)

For reviewers and the implementer, the helper is small enough to commit to memory:

```ts
function getRelatedProducts(current: Product, all: Product[] = getAllProducts()): Product[] {
  const byRecency = (a: Product, b: Product) => {
    const t = new Date(b.publishedAt).getTime() - new Date(a.publishedAt).getTime();
    return t !== 0 ? t : a.slug.localeCompare(b.slug); // slug asc tiebreaker
  };

  const notSelf = all.filter(p => p.slug !== current.slug);
  const sameCat = notSelf.filter(p => p.category === current.category).sort(byRecency);
  const otherCat = notSelf.filter(p => p.category !== current.category).sort(byRecency);

  const merged = [...sameCat, ...otherCat].slice(0, 4);
  return merged.length >= 3 ? merged : [];
}
```

Invariants (proved by inspection):

- Return length ∈ `{0, 3, 4}`. (Slice caps at 4; the `>= 3` gate excludes 1 and 2; 0 is the explicit empty case.)
- Order is total and deterministic across processes — `Date.getTime()` and `String.localeCompare` are both pure.
- Same-category candidates always rank before cross-category candidates, regardless of dates (Scenario 1 + 3 contract).
- The current product is never returned (`notSelf` excludes it before either bucket sees it).
- The helper never reads files — it operates on the in-memory array, so the per-build cost is O(N log N) per call (sort) and O(N²) total across all detail pages; trivial at v1 scale.

## 10. Risks / open items

- **US00046 not yet landed at planning time.** US00047 is meaningless without a detail page to mount in. The implementer must verify US00046's PR is merged before opening this one. If US00046's `app/san-pham/[slug]/page.tsx` evolves between plan approval and US00047 implementation, the §5 mount step may need a small tweak. Acceptable v1 risk.
- **`<ProductCard />` (US00042) prop shape not yet finalized.** This plan assumes `<ProductCard product={product} />` per the spec text of US00042. If US00042 ends up exposing additional required props (e.g., `priority`, `loading`, `variant`), the related-row consumer will need to pass them. Mitigation: when US00042 lands, re-read its prop signature before starting US00047.
- **Empty catalog during development.** `content/products/` is empty at planning time. `getRelatedProducts()` returns `[]` for any input over an empty catalog (the slug-based filter yields nothing). The detail-page route also has no `generateStaticParams` output, so the page never renders. No bug surface, but the §7 verification must seed at least 4 fixtures to exercise the algorithm. The implementer should add and remove seeded JSON fixtures during local testing.
- **`publishedAt` parsing.** Dates are ISO strings in JSON. `new Date(str).getTime()` returns `NaN` for malformed dates, which sorts unpredictably. Mitigation: `lib/products.ts` already requires `publishedAt` to be a string but does not parse-validate it; if a future story tightens validation, the sort becomes safer. For US00047, we rely on the input being well-formed (CLAUDE.md Product JSON shape requires ISO dates).
- **Same `publishedAt` across many products.** `slug` asc is the tiebreaker — guaranteed unique by `lib/products.ts`'s duplicate-slug detection (lines 94–102). Order is always total.
- **Catalog grows past v1 expectations.** O(N²) cost across the build (`getRelatedProducts` called once per detail page, each call sorts N items). At 1000 products, ~10⁶ ops total — still fast. At 100k, revisit; F0004 §6 already flags this as a future concern.
- **Cross-category top-up may surprise editors.** A visitor on a `tai-nghe` page may see `chuot-gaming` cards in the related row when the `tai-nghe` catalog is sparse. This is the documented Scenario 3 behavior; if editors find it confusing in practice, the v2 follow-up is to source from a wider sibling set (e.g., the brand) before falling back to "any other category". Out of scope for v1.
- **No `featured` weighting.** Per D5 we ignore `Product.featured` in the related row. If business wants featured products promoted everywhere, that's a v2 story touching this helper. Documented; not implemented.
- **No analytics differentiation between related-row clicks and detail-page-CTA clicks.** Both surfaces emit `affiliate_click` (F0007). If the team later wants `affiliate_click_source = related | cta | listing | card`, that's a new attribute on `<AffiliateLink>` (a US00033 follow-up, not US00047). Documented; not implemented.
