**Status:** Done

## 1. Scope

This story is the foundational Vercel ↔ GitHub wiring. The deliverable is **operational, not code**: a working Vercel project linked to `github.com/thinhvophu/aff-store` with `main` as the Production branch, such that any commit pushed to `main` auto-deploys, and any PR opened produces a Preview deployment.

Strictly:

1. Create / claim a Vercel project for this repository under the operator's Vercel account.
2. Connect the GitHub repo `thinhvophu/aff-store` via the Vercel GitHub App.
3. Set the **Production branch** to `main` in Vercel project Git settings.
4. Confirm the framework preset is **Next.js**, the build command is `next build` (default), and the output directory is the Next.js default — produces SSG output per F0001.
5. Configure the **minimum** environment variable required for the build to succeed: `NEXT_PUBLIC_SITE_URL` in Production (set to the Vercel-assigned `*.vercel.app` URL until a custom domain lands). Without this, US00018's validation crashes the build.
6. Verify Scenarios 1–5 of US00111 by triggering a real Production deploy from `main` and a real Preview deploy from a throwaway PR.
7. Capture the Vercel project URL, project ID, org ID, and Production deploy URL in `README.md`'s "Hosting" section so future operators can find them.

**Explicitly out of scope** (owned by sibling stories in F0011):

- Full env-var matrix (`NEXT_PUBLIC_GA_MEASUREMENT_ID`, Preview overrides, `.env.example` audit) → **US00112**. This story sets only the single var needed to make the build pass.
- Scheduled rebuild workflow (`.github/workflows/scheduled-deploy.yml`) and `VERCEL_DEPLOY_HOOK_URL` secret → **US00113**.
- PR validation CI (`.github/workflows/ci.yml`) and branch-protection required-checks list → **US00114**.
- Speed Insights enablement and `@vercel/speed-insights` wiring → **US00115**.
- Custom domain / DNS (deferred to launch prep).
- Deploy notifications (Slack/email).

## 6. Step-by-step execution order

1. `git checkout -b feat/US00111` (sibling subagents may be running; do not touch `main` directly).
2. **Vercel side — create and link the project:**
   1. Sign in to vercel.com.
   2. "Add New…" → "Project" → "Import Git Repository" → select `thinhvophu/aff-store`. If the repo is not visible, click "Adjust GitHub App Permissions" and grant access.
   3. Project name: `aff-store`. Framework preset auto-fills as `Next.js`. Leave Build/Output/Install commands at defaults.
   4. Expand "Environment Variables" → add `NEXT_PUBLIC_SITE_URL` = `https://aff-store.vercel.app` (Production scope). This URL will be re-confirmed once Vercel assigns the canonical hostname; if it differs, edit the var afterward and redeploy.
   5. Click "Deploy". Wait for the first Production build to complete. **If it fails on `NEXT_PUBLIC_SITE_URL` validation, the env var was not saved correctly — fix it and redeploy.**
3. **Vercel side — confirm Git settings:**
   1. Project → Settings → Git → confirm Production Branch is `main`.
   2. Project → Settings → Git → confirm "Commit Comments" and "Pull Request Comments" toggles are ON.
   3. Project → Settings → General → confirm Node.js Version is `22.x`. If it shows `20.x` or older, set to `22.x` and trigger a redeploy.
4. **Verify Scenario 1 (Production branch is `main`):** screenshot or note the Git settings panel.
5. **Verify Scenario 3 (build produces SSG output):** in the deployment's "Build Logs" panel, confirm lines like `Generating static pages (X/X)` and `Route (app) … ○ /` (the `○` symbol marks static routes per Next.js build output). Confirm `λ` (server function) markers are absent for routes that should be static. The deployment should be served from Vercel's CDN — confirm by `curl -I https://<production-url>/` and looking for the `x-vercel-cache` / `cache-control: s-maxage=…` headers.
6. **Verify Scenario 2 (push-to-main triggers a deploy):** make the README edit (§5.1, §5.2) on `feat/US00111`, push the branch, open a PR, merge it to `main` after Scenario 5 verification. Confirm in the Vercel dashboard that the merge commit triggers a fresh Production deployment within ~30 s.
7. **Verify Scenario 5 (Preview deploys + PR comment):** the PR opened in step 6 itself satisfies this. Within ~30 s of opening, the Vercel bot must comment with a Preview URL. Click the URL — it must resolve to a working build of the branch. If the bot does not comment, confirm "Pull Request Comments" is enabled in Project → Settings → Git → Comments and re-push to retrigger.
8. **Verify Scenario 4 (failed build does not replace live deploy):** this is destructive to verify directly on `main`, so do it on the PR's Preview track instead — push a deliberate breaking commit (e.g. `throw new Error("intentional")` in a top-level module) to `feat/US00111`, observe that the Preview build fails in the Vercel dashboard, observe that the existing Production deployment continues serving traffic on the Vercel URL. Then revert the breaking commit before merging. **Document this verification in the PR description rather than committing the breaking change to history** — use a short-lived commit, then `git reset` it locally, OR push it as a separate "verify" commit on the branch and remove via squash-merge cleanup.
9. **AC verification round** (Section 7).
10. Update PR description with screenshots/links to: the Vercel project page, the Production deployment, the Preview deployment, and the Build Logs showing static-page generation.
11. Merge the PR (squash). The merge commit will trigger another Production deploy — confirm green.
12. Optional follow-up after merge: edit `NEXT_PUBLIC_SITE_URL` in Vercel if the assigned canonical Vercel URL differs from the placeholder, and trigger a redeploy.

## 10. Risks / open items

- **Vercel account ownership.** The plan assumes the operator has admin access to a Vercel scope (personal or team) that they want this project to live under. If a team scope is desired (e.g. for shared billing later), creating it in a personal scope first and migrating later is non-trivial. **Open question for the implementer:** which Vercel scope hosts the project? Personal or team? This must be settled before step 2.1.
- **`NEXT_PUBLIC_SITE_URL` chicken-and-egg.** The plan sets the var to `https://aff-store.vercel.app` before the project exists. If Vercel assigns a different canonical hostname (e.g. `aff-store-thinhvophu.vercel.app` because the simple slug is taken), the env var must be edited post-creation and a redeploy triggered. This is a 30-second fix but worth flagging.
- **First-deploy URL placeholder.** The README documents the Production URL as `https://aff-store.vercel.app`; the implementer must replace this with the actual URL that Vercel assigns. This is a non-issue if the slug is available, but the implementer must check.
- **Custom domain not in scope.** `NEXT_PUBLIC_SITE_URL` will need to change again when launch prep configures DNS. Not a blocker; documented in §1 out-of-scope and in `docs/specs/F0011.md` §6.
- **Branch protection not set up yet.** US00114 will require status checks before merge. Until then, this PR can merge with only the Vercel deploy as a check. Acceptable because CI does not yet exist.
- **No `vercel.json`.** Defaults are sufficient for a vanilla Next.js SSG project. If a future story needs custom headers, redirects, or function regions, that story can introduce `vercel.json` and document why.
- **GitHub App permissions.** If the Vercel GitHub App is not already installed for `thinhvophu`, step 2.2 requires accepting the install prompt. This requires GitHub organization admin (or personal-account owner) authorization. Not a code issue — just a click prompt the implementer must handle.
- **Speed Insights, Cron, CI all deferred.** They're separate stories (US00115, US00113, US00114) so the launch checklist won't be fully green after this story alone — that's by design.
- **Verifying Scenario 4 destructively.** §6 step 8 pushes a deliberately-broken commit to a feature branch. This is safe because it lives only on the feature branch, not on `main`. The implementer must remember to revert before merging.
- **Build minutes on free tier.** Each push to `feat/US00111` triggers a Preview build. Verifying Scenario 4 + the merge to `main` triggers ~3 builds total — well within free-tier monthly minutes.
