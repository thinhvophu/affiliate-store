**Status:** Approved
**Feature:** F0002 ‚Äî Site Shell & Theme
**Spec ref:** `docs/specs/F0002.md` ¬ß US00024
**Predecessors:**
- **US00023** (hard dep) ‚Äî adds the Shopee surface tokens `--color-surface`, `--color-border-hairline`, `--radius-md`, and the body `--color-page-bg` that this layout's card surface relies on. - **US00016** (hard dep, already shipped) ‚Äî owns the breakpoint tokens (`--breakpoint-tablet: 768px`, `--breakpoint-desktop: 1280px`) and the layout tokens (`--max-width: 1280px`, `--sidebar-width: 280px`) consumed verbatim by this component.

**Sibling / consumer relationships:**

## 1. Scope

Strictly:

1. Create one new Server Component, `components/ShellLayout.tsx`, that renders a two-column shell: a ~280px left panel (rendered as a white card surface) on the left and the main content on the right, wrapped in a centered max-1280px container with consistent horizontal gutter padding.
2. Expose a **slot-based API**: a `leftPanel: React.ReactNode` prop and the standard `children: React.ReactNode` prop for the main content slot. The component dictates layout (column widths, gutter, card chrome) but **never** dictates the internal markup of either slot.
3. Use only the **CSS custom-property tokens** introduced by US00023 (`--color-surface`, `--color-border-hairline`, `--radius-md`) for the left-panel card chrome, and the **layout / breakpoint tokens** already shipped by US00016 (`--max-width`, `--sidebar-width`, `--breakpoint-tablet`) ‚Äî no hardcoded hex values, no inline magic numbers for the breakpoint.
4. Ship the layout's styles as a **co-located CSS Module** (`components/ShellLayout.module.css`) so the global stylesheet stays untouched and the styles are tree-shaken for any route that does not opt in.
5. Implement the **desktop-only inline two-column behavior** (‚â•768px): below 768px, the inline left-panel column collapses (`display: none`) so the main content occupies the full container width ‚Äî leaving a clean hook for US00025 to render a drawer trigger + drawer surface without rewriting this component.
6. Use **CSS Grid** for the column structure (`grid-template-columns: var(--sidebar-width) 1fr` at ‚â•768px), with a documented gap derived from existing spacing tokens (`--spacing-lg` / `--spacing-xl`).
7. Pure **Server Component** ‚Äî no `"use client"`, no hooks, no state. All responsiveness is CSS-only. Client-only behavior (drawer toggle, focus trap, scroll lock) is explicitly deferred to US00025.

**Explicitly out of scope** (owned by sibling stories):

- Mobile drawer trigger, drawer surface, overlay, focus trap, scroll lock, Esc handling ‚Äî **US00025**.
- Sticky-on-scroll Header ‚Äî **US00026**.
- Header / Footer themselves ‚Äî **US00021 / US00022**.
- Mounting the layout into routes that should adopt it (`/san-pham`, `/bai-viet`, `/bai-viet/[slug]`) ‚Äî **F0004 / F0006** consumers.
- Filter UI, TOC UI, category sidebar UI ‚Äî whatever ends up inside the `leftPanel` slot is owned by the consumer feature.
- Modifying root `app/layout.tsx` ‚Äî this is an **opt-in** wrapper, not a root-layout change.
- Adding new tokens ‚Äî US00024 is a strict **consumer** of tokens shipped by US00016 + US00023.

## 3. Approved Decisions

| #   | Decision | Choice | Rationale |
| --- | --- | --- | --- |
| D1  | File location & name | **`components/ShellLayout.tsx`** + co-located **`components/ShellLayout.module.css`** | Matches `CLAUDE.md` module convention "Components in `components/<Name>.tsx`". Name reads as a layout primitive (parallels Next.js's own "layout" vocabulary). |
| D2  | Component kind | **Server Component** (no `"use client"`) | Story is purely structural; all interactivity (drawer) is US00025's responsibility. Server Component keeps client JS at zero for this primitive. |
| D3  | API shape | **`{ leftPanel: React.ReactNode; children: React.ReactNode }`** ‚Äî both required | Slot-based per AC Scenario 4. Two named slots are simpler than a polymorphic `slots` map. Making both required prevents accidental no-panel misuse (consumers that want no panel should not import this component at all ‚Äî AC Scenario 5). |
| D4  | Layout primitive | **CSS Grid**, `grid-template-columns: var(--sidebar-width) 1fr` at ‚â•768px; single-column at <768px | Grid is the canonical primitive for fixed-sidebar + fluid-main layouts. Avoids float/flex gymnastics; trivially swaps to single-column via a media query. |
| D5  | Gap between columns | **`var(--spacing-lg)`** (24px) on tablet, **`var(--spacing-xl)`** (32px) at ‚â•1280px | Reuses the US00016 spacing scale. The wider gap at desktop matches the visual breathing room a Shopee-style grid expects when the viewport approaches `--max-width`. |
| D6  | Container padding (horizontal gutter) | **`var(--spacing-md)`** (16px) on mobile, **`var(--spacing-lg)`** (24px) at ‚â•768px | Standard responsive gutter scale. Symmetric left/right padding keeps the centered max-width container visually balanced. |
| D7  | Container max-width | **`var(--max-width)`** (1280px) | Token already exists; spec requires "max-width ~1280px, centered horizontally" (AC Scenario 3). |
| D8  | Left-panel card chrome | **`background: var(--color-surface); border: 1px solid var(--color-border-hairline); border-radius: var(--radius-md);`** | Verbatim from AC Scenario 2. All three values come from US00023's tokens. |
| D9  | Left-panel internal padding | **`var(--spacing-md)`** (16px) | Provides the card a default inner gutter so consumers don't have to remember to pad. Consumers can override by giving their slot content its own inner wrapper. Cheap to remove if any consumer needs zero padding (low risk; punt to follow-up). |
| D10 | Breakpoint mechanism | **`@media (min-width: 768px)` with a documenting comment** referencing `--breakpoint-tablet` | Per US00016's known constraint, CSS custom properties cannot be used inside `@media` conditions. | D11 | CSS scoping | **CSS Module** (`ShellLayout.module.css`) | Component-scoped classes; no risk of global selector collisions with future stories (Header, Footer, filter panel, etc.). First CSS Module in the repo ‚Äî sets the convention for component-scoped styles going forward. |
| D12 | Markup landmarks | **No `<aside>` / `<main>` / `<section>` landmarks emitted by this component** ‚Äî root is a `<div>`, left panel is a `<div>`, main slot is a `<div>` | The root layout already wraps every page in a single `<main>` (US00023). Emitting another `<main>` here would break the "one `<main>` per page" rule. The left panel's semantic role (filter region, blog TOC, category nav) is **consumer-specific** ‚Äî the consumer wraps its slot content in `<aside>`/`<nav>`/`<form>` as appropriate. Component stays semantically neutral. |
| D13 | Drawer hand-off seam for US00025 | The left-panel column carries a **stable, documented class** (e.g., `styles.leftPanel`) and is hidden via `display: none` below 768px in the same module. US00025 will add a sibling client component (e.g., `LeftPanelDrawer.tsx`) that renders **only** at `<768px` (mirror media query: `display: none` at ‚â•768px), takes the same `leftPanel` ReactNode, and renders the drawer trigger + drawer surface | Keeps US00024 a pure server-component layout primitive. | D14 | Reduced-motion / animation | **None** ‚Äî no transitions on the layout itself | Layout is static at any viewport width above/below the breakpoint. Drawer animations belong to US00025 (which must honor `prefers-reduced-motion`). |
| D15 | TypeScript types | **Local `interface ShellLayoutProps`** in the component file | No need for a `types/` entry ‚Äî this prop type isn't shared. Aligns with how `React.FC`-style local prop interfaces are written elsewhere in the Next.js docs. |

## 5. Concrete diff sketch

### 5.1 `components/ShellLayout.tsx`

```tsx
import styles from "./ShellLayout.module.css";

export interface ShellLayoutProps {
  /**
   * Content for the left panel column. On desktop (‚â•768px) this renders as a
   * white card surface (background: var(--color-surface), 1px hairline border,
   * var(--radius-md) corner radius). On mobile (<768px) the inline column is
   * hidden ‚Äî US00025 plugs in a drawer trigger + drawer surface that consumes
   * the same node.
   *
   * Consumers wrap their slot content in <aside>/<nav>/<form> as appropriate;
   * ShellLayout stays semantically neutral.
   */
  leftPanel: React.ReactNode;

  /**
   * Main content column. Fills the remaining container width to the right of
   * the left panel on desktop, and the full container width on mobile.
   */
  children: React.ReactNode;
}

/**
 * Two-column shell layout ‚Äî Shopee card surfaces.
 *
 * Opt-in layout primitive: consumer pages (product list, blog list, blog post)
 * wrap their per-page content in <ShellLayout leftPanel={...}>...</ShellLayout>.
 * Pages that don't need the panel (homepage, About) do NOT import this.
 *
 * Pure Server Component. All responsiveness is CSS-only. Drawer / interactivity
 * for the <768px viewport ships in US00025.
 */
export function ShellLayout({ leftPanel, children }: ShellLayoutProps) {
  return (
    <div className={styles.container}>
      <div className={styles.grid}>
        <div className={styles.leftPanel}>{leftPanel}</div>
        <div className={styles.main}>{children}</div>
      </div>
    </div>
  );
}
```

### 5.2 `components/ShellLayout.module.css`

```css
/* ==========================================================================
   ShellLayout ‚Äî two-column shell, Shopee card surfaces (US00024)
   --------------------------------------------------------------------------
   Breakpoints sourced from --breakpoint-tablet (768px) and --breakpoint-desktop
   (1280px) defined in app/globals.css :root. CSS spec forbids var() inside
   @media conditions, so the literal pixel values appear here. The token is
   the canonical source of truth ‚Äî keep these in sync if it ever changes.
   ========================================================================== */

/* Centered container with max-width and responsive horizontal gutter. */
.container {
  width: 100%;
  max-width: var(--max-width);
  margin-inline: auto;
  padding-inline: var(--spacing-md);
}

@media (min-width: 768px) {
  .container {
    padding-inline: var(--spacing-lg);
  }
}

/* Two-column grid: single column below 768px (left panel hidden), fixed
   sidebar + fluid main at ‚â•768px. */
.grid {
  display: grid;
  grid-template-columns: 1fr;
  gap: var(--spacing-lg);
}

@media (min-width: 768px) {
  .grid {
    grid-template-columns: var(--sidebar-width) 1fr;
    gap: var(--spacing-lg);
  }
}

@media (min-width: 1280px) {
  .grid {
    gap: var(--spacing-xl);
  }
}

/* Left panel ‚Äî white card surface using US00023 tokens. Hidden inline below
   768px; US00025 surfaces this content via a drawer at that viewport. */
.leftPanel {
  display: none;
}

@media (min-width: 768px) {
  .leftPanel {
    display: block;
    background: var(--color-surface);
    border: 1px solid var(--color-border-hairline);
    border-radius: var(--radius-md);
    padding: var(--spacing-md);
  }
}

/* Main slot ‚Äî no chrome; consumer pages own their own cards / typography. */
.main {
  min-width: 0; /* allow flex/grid children to shrink below intrinsic width */
}
```

## 10. Risks / open items

- **US00023 ordering.** Hard dependency on the three new tokens. Implementer must verify they're in `app/globals.css` at step 6.1; otherwise pause. The plan does not gracefully degrade if the tokens are missing ‚Äî `border: 1px solid var(--color-border-hairline);` would resolve to no border, and `background: var(--color-surface);` would fall back to transparent, but the card chrome would be visually broken. Better to block than to half-ship.
- **US00025 hand-off seam.** The current design assumes US00025 will either (a) wrap `ShellLayout` in a consumer-side client component that injects its own drawer trigger/surface alongside, or (b) extend `ShellLayout` with an optional `mobileDrawer?: React.ReactNode` prop in a follow-up. If US00025 instead needs to fundamentally restructure the markup (e.g., requires the `leftPanel` to be a render-prop function for SSR-safe state lift-up), the seam designed here may not be enough. **Mitigation:** keep the `leftPanel` prop typed as `React.ReactNode` (the most flexible shape); US00025's planner should review this plan and flag any incompatibility before implementation.
- **Breakpoint literal duplication.** The `768px` and `1280px` literals appear in the CSS Module. If US00016's breakpoint tokens are ever changed, this file must be updated in lockstep. **Mitigation:** the documented comment block at the top of the CSS Module is the contract; flagged in the AC checklist.
- **No `<aside>` landmark.** D12 chose to leave the left-panel wrapper as a `<div>` because the semantic role is consumer-specific. This means an a11y reviewer expecting an `<aside>` landmark on every product-list page must inspect the consumer code, not ShellLayout. **Mitigation:** consumers (F0004's product-list, F0006's blog-post) wrap their `leftPanel` content in the appropriate landmark.
