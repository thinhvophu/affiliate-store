**Status:** Approved
**Feature:** F0004 — Product Catalog

## 0. Predecessors, siblings, downstream consumers

US00046 ships the **product detail surface** of F0004 — one statically generated page per product slug. It is the most monetization-load-bearing surface in the entire storefront because it hosts the prominent "Mua trên Shopee" CTA next to the price.

**Hard upstream deps (must be on `main` when implementation starts):**

- **US00031 / US00032 / US00033 / US00034** — `components/AffiliateLink.tsx` + `components/AffiliateLink.module.css` + `lib/affiliate.ts`. The detail-page CTA is a prominent button-styled `<AffiliateLink>` (not the whole-card pattern); the F0003 ↔ F0007 data-attribute contract is honored transparently.
- **US00041** — `lib/format.ts` → `formatVnd(amount: number): string`. The detail page renders the price exactly once via this helper; no other formatter is permitted.
- **US00042** — `components/ProductCard.tsx`. Consumed by the related-products row (US00047) which US00046 mounts at the bottom of the page. US00046 imports `<ProductCard />` only transitively through `<RelatedProducts />`.
- **F0001** — `lib/products.ts` already exposes `getAllProducts()` / `getProductBySlug()`. US00046 uses both: `getAllProducts()` to feed `generateStaticParams()` and to pass into `getRelatedProducts()`; `getProductBySlug()` (or `find()` over the loaded array — see D11) to resolve the route param to a single product.
- **F0002 — US00021 / US00022** — Header + Footer chrome inherited via `app/layout.tsx`. No `<ShellLayout />` on this surface (Scenario 8).

**Sibling stories sharing this wave:**

- **US00047 — Related products section.** US00047 owns the `getRelatedProducts()` helper, the `<RelatedProducts />` component, and the section's styling. US00046 **mounts** the section but does not implement it. Per US00047's plan §5 step 5, US00046's page must call `getRelatedProducts(product, all)` (passing the already-loaded array to avoid a second JSON read) and render `<RelatedProducts products={related} />` as the last child of the page tree.
- **US00043 / US00044 / US00045** — listing, filters, category page. Independent surfaces; they share `<ProductCard />` and the SSG / canonical conventions but do not constrain US00046's file shape.

**Downstream consumers (not modified in this story):**

- **F0006 (Editorial blog posts)** — inline `<ProductCard slug="…">` MDX wrapper composes US00042; no direct dependency on this detail page beyond the canonical URL pattern `/san-pham/[slug]/` which F0006 may link to in prose.
- **F0007 (Analytics & Click Tracking)** — picks up the CTA click via the F0003 ↔ F0007 data-attribute contract on `<AffiliateLink>`. No CTA-specific instrumentation in US00046.
- **F0008 (Homepage featured picks)** — links to detail pages by slug; no shape coupling.
- **F0009 (SEO & Discoverability)** — will harden JSON-LD `Product`, `BreadcrumbList`, Open Graph tags, and sitemap entries on top of this story. US00046 ships the **canonical URL strategy** and the **`<h1>` discipline** (exactly one `<h1>` containing `product.name`) now so F0009 has the right base.

**Sequencing.** This story may land in parallel with US00043 and US00045 once US00041 and US00042 are merged. It must land **before** US00047 because US00047 mounts inside this page's file.

## 1. Scope

Strictly:

1. Create the dynamic route file **`app/san-pham/[slug]/page.tsx`** — an async Server Component that:
   - Implements `generateStaticParams()` returning `{ slug: string }[]` for every product in `getAllProducts()` (Scenario 1).
   - Resolves the route's `params.slug` to a single `Product` and calls `notFound()` from `next/navigation` if no product matches (Scenario 1).
   - Renders, in document order: a single `<h1>` with `product.name`, the brand, the formatted price via `formatVnd()`, the image gallery (single-image vs. multi-image variants), the short `product.description`, a `<dl>`-based specs table (omitted entirely when `product.specs` is empty), and the prominent button-styled "Mua trên Shopee" `<AffiliateLink>` CTA (Scenarios 2–6, 10).
   - Mounts `<RelatedProducts products={related} />` (from US00047) as the last child of the page's main tree, where `related = getRelatedProducts(product, all)` (Scenario 7).
   - Sits inside the F0002 Header/Footer chrome (inherited via `app/layout.tsx`) but **does not** use `<ShellLayout />` (Scenario 8). The page uses a full-width container that lets the gallery + specs breathe.
2. Implement `generateMetadata({ params })` returning `Metadata` with at minimum:
   - `title` — `${product.name} | <site name>` (the site-name suffix matches whatever existing routes ship; if no convention yet, just `product.name`).
   - `description` — sourced from `product.description` (truncated to ≤160 chars at the first sentence boundary if longer; see D14).
   - `alternates.canonical` — `${NEXT_PUBLIC_SITE_URL}/san-pham/${slug}/` (trailing slash matches the route map; see D13).
   - Open Graph / JSON-LD intentionally **deferred to F0009** — Scenario 2 does not require them and a per-page metadata helper that emits them would conflict with F0009's hardening strategy.
3. Create a new client component **`components/ProductGallery.tsx`** (`"use client"`) that:
   - Accepts `{ images: string[]; productName: string }` as props.
   - When `images.length === 1`, renders a single `next/image` (no thumbnail row, no client state used) — but the component is still a client component to keep the gallery API single-shape (Scenario 4 + 5). Alternative discussed in D6.
   - When `images.length > 1`, manages local state `[activeIndex, setActiveIndex] = useState(0)`, renders the active image at the top and the full `images.map(…)` row as clickable thumbnails beneath; clicking a thumbnail swaps the main image without a navigation (Scenario 5).
   - Falls back gracefully under JS-off: the first image still renders as static HTML; thumbnail clicks are ignored (Scenario 9 + spec §5 JS-disabled edge case).
4. Create **`components/ProductGallery.module.css`** — co-located styles for the main image frame (aspect-ratio 1:1 with `object-fit: contain` so product photography is not cropped), the thumbnail row layout (horizontal scroll on narrow widths, fixed grid on wide), and active-thumbnail focus / aria-pressed visual state.
5. Create **`app/san-pham/[slug]/product-detail.module.css`** — page-level layout: a 2-column grid on `≥1024px` (gallery left ~55%, info column right ~45%), single-column stack below `1024px`. Specs table styling. Prominent CTA pill (full-width on mobile, large button on desktop). Section spacing tokens.
6. Update **`lib/products.ts`** to enforce two additional product invariants demanded by F0004 § Edge Cases:
   - `images.length >= 1` — non-empty array of strings. Bad product fails the build with the offending slug named.
   - (Already enforced: `price >= 0`, `affiliateUrl` allow-listed, unique slug, valid `publishedAt` string.) No change to those.
   - Note: spec also lists "missing price" failing the build; the current code rejects non-numeric / negative `price` (lines 37–39) which covers both. No additional change needed beyond `images.length`.
7. Update **`CLAUDE.md`** and **`README.md`**:
   - Add `app/san-pham/[slug]/page.tsx` to the route map's "implemented" column (the route map table in CLAUDE.md currently lists it as planned).
   - Add `components/ProductGallery.tsx` and `components/ProductGallery.module.css` rows to the `components/` tree.
   - Note the new `app/san-pham/[slug]/product-detail.module.css` co-located CSS file.
   - Extend the description of `lib/products.ts` to mention the new `images.length >= 1` invariant.
   - Bump the "Last updated" line to include US00046.

**Explicitly out of scope** (owned by sibling stories or downstream features):

- The all-products listing `/san-pham/` and its grid — **US00043**.
- Client-side filter / sort UI — **US00044**.
- Category page `/danh-muc/[category]/` — **US00045**.
- `getRelatedProducts()` helper, `<RelatedProducts />` component, related-section CSS — **US00047**. US00046 only **mounts** `<RelatedProducts />` at the bottom of its page.
- JSON-LD `Product` / `BreadcrumbList`, Open Graph tags, breadcrumb UI, sitemap entry — **F0009**.
- GA4 `affiliate_click` event emission — **F0007** (seam already in `<AffiliateLink>` via US00033).
- `<ProductCard />` itself — **US00042**.
- `formatVnd()` itself — **US00041**.
- Image zoom-on-hover, lightbox, fullscreen modal, video — spec §2 explicitly out of scope.
- Cross-category recommendation, brand-affinity, "featured" sort — spec §2 / US00047 D5 already exclude.
- Search input on the detail page — spec §2 / out of scope for v1.

```tsx
/**
 * Product detail page — F0004 / US00046.
 *
 * Statically generated per product slug. Renders the canonical detail view:
 *   - <h1>{name}</h1>
 *   - brand, formatted price
 *   - image gallery (single- or multi-image via <ProductGallery>)
 *   - short description

 *   - prominent "Mua trên Shopee" <AffiliateLink> CTA
 *   - <RelatedProducts /> (mounted; selection logic owned by US00047)
 *
 * No <ShellLayout /> — the page uses a full-width container inside <main>
 * (Scenario 8). Header + Footer come from app/layout.tsx.
 */

interface PageProps {
  params: Promise<{ slug: string }>;
}

/** Truncate a description to ≤160 chars at the nearest sentence/word boundary
 *  for use in <meta description>. Falls back to a hard slice + "…" if no
 *  boundary exists below the limit. */
function truncateDescription(text: string, max = 160): string {
  if (text.length <= max) return text;
  const sliced = text.slice(0, max);
  const sentenceEnd = sliced.lastIndexOf(". ");
  if (sentenceEnd >= 80) return sliced.slice(0, sentenceEnd + 1);
  const wordEnd = sliced.lastIndexOf(" ");
  return (wordEnd > 0 ? sliced.slice(0, wordEnd) : sliced) + "…";
}

export async function generateStaticParams() {
  const products = getAllProducts();
  return products.map((p) => ({ slug: p.slug }));
}

export async function generateMetadata({ params }: PageProps): Promise<Metadata> {
  const { slug } = await params;
  const product = getAllProducts().find((p) => p.slug === slug);
  if (!product) return {};

  const siteUrl = process.env.NEXT_PUBLIC_SITE_URL ?? "";
  return {
    title: product.name,
    description: truncateDescription(product.description),
    alternates: {
      canonical: `${siteUrl}/san-pham/${product.slug}/`,
    },
  };
}

export default async function ProductDetailPage({ params }: PageProps) {
  const { slug } = await params;
  const all = getAllProducts();
  const product = all.find((p) => p.slug === slug);
  if (!product) notFound();

  const related = getRelatedProducts(product, all);
  const specEntries = Object.entries(product.specs);

  return (
    <div className={styles.container}>
      <article className={styles.detail}>
        <div className={styles.galleryColumn}>
          <ProductGallery images={product.images} productName={product.name} />
        </div>

        <div className={styles.infoColumn}>
          <p className={styles.brand}>{product.brand}</p>
          <h1 className={styles.name}>{product.name}</h1>
          <p className={styles.price}>{formatVnd(product.price)}</p>

          <AffiliateLink
            className={styles.ctaButton}
            href={product.affiliateUrl}
            productName={product.name}
            productCategory={product.category}
          >
            <span className={styles.ctaLabel}>Mua trên Shopee</span>
          </AffiliateLink>

          {product.description && (
            <p className={styles.description}>{product.description}</p>
          )}

          {specEntries.length > 0 && (
            <section className={styles.specsSection} aria-labelledby="product-specs-heading">
              <h2 id="product-specs-heading" className={styles.sectionHeading}>
                Thông số kỹ thuật
              </h2>
              <dl className={styles.specsList}>
                {specEntries.map(([key, value]) => (
                  <div key={key} className={styles.specRow}>
                    <dt className={styles.specKey}>{key}</dt>
                    <dd className={styles.specValue}>{value}</dd>
                  </div>
                ))}
              </dl>
            </section>
          )}
        </div>
      </article>

      <RelatedProducts products={related} />
    </div>
  );
}
```

Notes on the JSX:

- The outer `<div className={styles.container}>` is the page's full-width wrapper (max-width, centered, horizontal padding). Inside it, `<article>` carries the primary content and `<RelatedProducts />` is a sibling so it can break out of the article's typographic flow.
- `<p className={styles.brand}>` precedes `<h1>` visually but the `<h1>` is still the unambiguous page heading. Brand renders as small uppercase secondary text above the name (typography choice in §5.2).
- The CTA is a single `<AffiliateLink>` with a wrapping `<span>` for the label (no nested anchor or button — preserves US00031's link semantics). `data-affiliate-link`, `data-product-name`, `data-product-category`, `data-destination-url` are emitted automatically by the primitive.
- The specs section is conditional on `specEntries.length > 0` (Scenario 6). No empty `<dl>`, no empty heading.
- Description is conditional on truthiness (defensive; the loader already requires it to be a non-empty string, so this branch is effectively always taken — but cheap to guard).
- `<RelatedProducts products={related} />` is unconditional; the component itself returns `null` when `related.length < 3` (US00047 D10). The conditional is therefore inside the component, not the page.

### 5.3 `components/ProductGallery.tsx` (new file)

```tsx
"use client";

import Image from "next/image";
import { useState } from "react";
import styles from "./ProductGallery.module.css";

/**
 * Product image gallery — F0004 / US00046.
 *
 * Single-image case: renders just the main image, no thumbnail row.
 * Multi-image case: renders the active image as the main slot and the full
 * image array as a thumbnail row beneath. Clicking a thumbnail swaps the
 * main image without a navigation.
 *
 * JS-off behaviour: the first image still renders as static HTML; thumbnail
 * buttons are inert but visible (spec § Edge Cases — visitor with JavaScript
 * disabled: "the first image still renders, which is sufficient").
 *
 * "use client" is required for useState. The component is below-the-fold
 * priority (the page LCP is the main image which still server-renders
 * because next/image emits an <img> in the initial HTML before hydration).
 */

export interface ProductGalleryProps {
  images: string[];
  productName: string;
}

export function ProductGallery({ images, productName }: ProductGalleryProps) {
  const [activeIndex, setActiveIndex] = useState(0);
  const activeSrc = images[activeIndex] ?? images[0];
  const isMulti = images.length > 1;

  return (
    <div className={styles.gallery}>
      <div className={styles.mainFrame}>
        <Image
          src={activeSrc}
          alt={productName}
          fill
          sizes="(min-width: 1024px) 55vw, 100vw"
          priority
          className={styles.mainImage}
        />
      </div>

      {isMulti && (
        <ul className={styles.thumbnails} role="list">
          {images.map((src, i) => (
            <li key={src} className={styles.thumbnailItem}>
              <button
                type="button"
                className={`${styles.thumbnailButton} ${i === activeIndex ? styles.thumbnailActive : ""}`}
                aria-label={`Ảnh ${i + 1}`}
                aria-pressed={i === activeIndex}
                onClick={() => setActiveIndex(i)}
              >
                <Image
                  src={src}
                  alt=""
                  fill
                  sizes="80px"
                  className={styles.thumbnailImage}
                />
              </button>
            </li>
          ))}
        </ul>
      )}
    </div>
  );
}
```

## 10. Risks / open items

- **R1 — US00041 / US00042 / US00047 sequencing.** The page imports `formatVnd` (US00041), depends on `<ProductCard />` transitively (US00042), and mounts `<RelatedProducts />` (US00047). The detail page itself cannot ship without all three. **Mitigation:** Step 1 of §6 verifies each dependency. If US00047 is not landed but US00041 + US00042 are, the implementer can stub the `<RelatedProducts />` mount with a TODO and circle back — the rest of the page is independently valuable.
- **R2 — Internal 1024px breakpoint deviates from F0001's 768/1280.** The two-col detail layout snaps to single-col below 1024px because at 768–1023px the gallery + info-column would be cramped. Documented in D8. If the design review prefers strict adherence to 768/1280, switch the breakpoint to 1280 (info column stacks below at 1024 / wide tablet). **Recommendation:** keep 1024px; it is a common breakpoint and the spec does not mandate the listing-grid breakpoints for the detail page (the listing-grid breakpoints govern the *card grid*, not arbitrary page layouts).
- **R3 — `next/image` `priority` on the main image.** Setting `priority` triggers a `<link rel="preload">` for the LCP image; appropriate here because the detail page's hero IS the gallery main image. Side-effect: the preload is emitted even for the multi-image case where the visitor may swap to a different image after hydration — but the initial LCP target is still index 0, so the preload is correct.
- **R4 — `object-fit: contain` vs. `cover` on the main image.** D15 chose `contain` to preserve product geometry. Downside: products with non-square photos leave letterbox bars on the sides. Acceptable v1 — better than cropping a wide keyboard or tall headset photo. If editorial complains, revisit per-product (a `Product` field for `imageAspect: "square" | "wide" | "tall"` could control the frame in v2).
- **R5 — Specs values long enough to break the row.** `word-break: break-word` in `.specValue` handles long words (model numbers, URLs). The 2-col grid template `minmax(0, 2fr)` allows the value cell to shrink properly. Verified with the §7 long-spec-value edge-case test.
- **R6 — Description rendering as a single `<p>`.** Scenario 2 calls for "the short description". If editorial writes a multi-paragraph description in JSON (`\n\n` line breaks), the single `<p>` will render it as one paragraph. v1 acceptable — editors should keep descriptions short and link to a blog post for long-form content. v2 may introduce Markdown rendering inside the description field.
- **R7 — Client gallery is hydrated even when the visitor never interacts.** `<ProductGallery>` is `"use client"` regardless of `images.length` (D6). Bundle cost is small (≤ 2KB minified for the gallery alone, plus the React useState runtime which is already on every page that uses any client component). Acceptable trade-off for code simplicity. If LCP regression is observed, split into a server-rendered single-image path + a client-rendered multi-image path.
- **R8 — Missing root `app/not-found.tsx`.** If no root 404 page exists, calling `notFound()` renders Next.js's default 404 UI without the project's Header/Footer chrome. **Mitigation:** add a per-route `app/san-pham/[slug]/not-found.tsx` (Vietnamese 404 surface inside the F0002 chrome) if root 404 is missing at implementation time. Flagged as Q1.
- **R9 — `NEXT_PUBLIC_SITE_URL` may be empty in dev.** D13's canonical URL falls back to `""` (empty string) → canonical becomes `/san-pham/<slug>/`. Per CLAUDE.md "Env vars", `NEXT_PUBLIC_SITE_URL` is **Required**. The Vercel deployment sets it; local dev may not. The page does not block on the value; it just emits an empty-host canonical, which is acceptable in dev.
- **R10 — Image alt text for product photos.** D16 uses `product.name` as alt. For multi-image products, every image uses the same alt. v1 acceptable; v2 could introduce per-image captions in JSON (`{ src: string, alt: string }`). Out of scope.
- **R11 — `useState` re-hydration when query params change.** The gallery uses local state for `activeIndex`. It does not read from URL. Bookmarking a specific image is therefore impossible (always opens at index 0). Acceptable v1 — the page is short enough that scrolling and clicking is trivial. v2 could add `?image=N` query param.
- **R12 — Multi-image thumbnails inside the affiliate-anchor area.** The thumbnails are `<button>` elements inside `<ProductGallery>`, which is *outside* the CTA `<AffiliateLink>`. There is NO nesting violation. The detail page does NOT wrap the whole article in `<AffiliateLink>` (Scenario 3 explicit). Documenting here to head off the misreading of US00032's whole-card pattern.
- **R13 — `getRelatedProducts()` called twice if dev mode triggers two render passes.** Pure derivation; idempotent. No I/O penalty (same in-memory array).

### Open questions for the reviewer

- **Q1 — Per-route Vietnamese 404 surface?** D21 defaulted to no `app/san-pham/[slug]/not-found.tsx`, relying on a global root 404 (which does not yet exist). Should US00046 ship its own per-route 404 page with Vietnamese copy like "Sản phẩm không tồn tại" inside Header/Footer chrome? **Recommendation:** ship the per-route file as ~15 lines of Vietnamese 404 inside the global `<main>`. Cheap, lifts the visitor experience, decouples from whatever root 404 strategy lands later.
- **Q2 — Display category as slug or humanized label?** Brand is rendered as plain text (`product.brand` is already a human-readable string like "Logitech"). Category is a slug (`chuot-gaming`). The detail page does NOT currently render the category text anywhere (`<h1>` is the name, brand is above it, price below). If category should appear (e.g., a breadcrumb "Chuột gaming › G102 Lightsync"), that's F0009's breadcrumb UI — out of scope here. **Recommendation:** do not render category text on the detail page in v1; let F0009 add the breadcrumb.
- **Q3 — Should the CTA also appear at the bottom of the page (sticky CTA on mobile)?** Spec asks for "a prominent CTA". One CTA in the right-column info section is sufficient for the AC. A sticky bottom CTA on mobile would lift conversion but is not required by AC. **Recommendation:** ship one CTA in v1; revisit if F0007 analytics show mobile bounce before scroll-to-CTA.
- **Q4 — Image preloading for all images vs. just the LCP?** D15 sets `priority` on the main image only. Thumbnails are `next/image` defaults (lazy). Switching to a multi-image preload would speed up thumbnail-swap but cost bandwidth on mobile. **Recommendation:** keep the LCP-only preload.
