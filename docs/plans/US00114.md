## 1. Scope

Strictly:

1. Create `.github/workflows/scheduled-deploy.yml` — a GitHub Actions workflow that POSTs to a Vercel deploy hook on a weekly cron (`0 2 * * 1`, Mon 02:00 UTC) and on demand via `workflow_dispatch`.
2. Read the deploy hook URL from `secrets.VERCEL_DEPLOY_HOOK_URL` so the URL never appears in the workflow file or in run logs.
3. Set the workflow's `permissions:` block to `contents: read` only — minimum required for a webhook POST.
4. Document for the operator: the cron only fires once the workflow exists on the **default branch** (`main`); manual `workflow_dispatch` works from any branch the workflow is committed to.

**Explicitly out of scope** (owned by sibling stories in F0011 or by operator/Vercel config):

- Creating the deploy hook in Vercel — operator action, prerequisite to this story (US00111).
- Adding `VERCEL_DEPLOY_HOOK_URL` to GitHub repo secrets — operator action, verified in this story but performed in the GitHub UI.
- PR validation CI workflow `.github/workflows/ci.yml` → **US00113**.
- Branch protection / required status checks → **US00113** (configured in repo settings).
- Vercel Speed Insights → **US00115**.
- Notifications (Slack / email) on success or failure — explicitly out of scope per F0011 § 2.
- Idempotency / retry / concurrency tuning beyond GitHub Actions defaults.

## 5. Concrete diff sketch

### 5.1 `.github/workflows/scheduled-deploy.yml`

```yaml
name: Scheduled deploy

# Weekly rebuild to refresh affiliate metadata + manual fallback (F0011 / US00114).
# NOTE: GitHub Actions only honors the `schedule:` trigger from workflows on the
# repository's default branch (`main`). Until this file is merged to `main`, only
# `workflow_dispatch` will work.

on:
  schedule:
    # Mon 02:00 UTC (09:00 ICT). UTC-only; cron skew of a few minutes is acceptable.
    - cron: "0 2 * * 1"
  workflow_dispatch:

# Minimum permissions — this job only POSTs to an external webhook.
permissions:
  contents: read

# Avoid two runs racing if a manual dispatch overlaps a cron run.
concurrency:
  group: scheduled-deploy
  cancel-in-progress: false

jobs:
  trigger-vercel-rebuild:
    name: POST Vercel deploy hook
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - name: Trigger Vercel rebuild
        # The hook URL is injected via `env:` from secrets so GitHub Actions
        # auto-masks it in logs. Do NOT echo $DEPLOY_HOOK_URL anywhere.
        env:
          DEPLOY_HOOK_URL: ${{ secrets.VERCEL_DEPLOY_HOOK_URL }}
        run: |
          set -euo pipefail
          if [ -z "${DEPLOY_HOOK_URL:-}" ]; then
            echo "VERCEL_DEPLOY_HOOK_URL secret is not configured on this repository." >&2
            exit 1
          fi
          # -f: fail on HTTP >= 400; -s: silent; -S: still show errors; -X POST: explicit method.
          # We intentionally do not pass -v or print the URL.
          curl -fsS -X POST "$DEPLOY_HOOK_URL"
          echo "Vercel deploy hook triggered."
```

Key safety points (each maps to an AC):

- Hook URL only ever lives in `secrets.VERCEL_DEPLOY_HOOK_URL` and the masked `DEPLOY_HOOK_URL` env var → **Scenario 5**.
- `curl -fsS` is silent on success and does not log the URL → **Scenario 5**.
- `permissions: contents: read` at workflow level → **Scenario 6**.
- `schedule` cron string and `workflow_dispatch` trigger present → **Scenarios 2, 3**.
- Path is `.github/workflows/scheduled-deploy.yml` → **Scenario 1**.
- POST request to the hook URL → **Scenario 4**.

## 10. Risks / open items

- **Cron requires default-branch presence.** Until the PR merges to `main`, the schedule trigger does **not** fire. AC Scenario 3 (manual run button) is also gated on this — the button only appears for workflows on the default branch. Verification steps in § 7 are explicit about post-merge testing.
- **GitHub Actions cron drift.** Schedules can lag 5–15 minutes (or, rarely, skip a run during platform incidents). F0011 § 5 calls this out as acceptable; `workflow_dispatch` is the manual fallback.
- **Operator must add the secret.** If `VERCEL_DEPLOY_HOOK_URL` is missing or misnamed when the first scheduled fire happens, the run fails with the explicit guard message. Operator updates the secret and re-runs manually.
- **No retry on hook failure.** If Vercel transiently 5xx's, the run fails. Cost of missing a Monday rebuild is low (content is mostly static); operator can re-dispatch manually. Adding retry is out of scope for v1.
- **No notification on failure.** Out of scope per F0011 § 2. Operator monitors via GitHub Actions email defaults / Actions tab.
- **Rotating the deploy hook.** When the hook is regenerated in Vercel, operator updates `VERCEL_DEPLOY_HOOK_URL` in GitHub secrets. No code change needed.
- **Forks cannot trigger this workflow.** GitHub does not expose secrets to fork PRs. Acceptable — the workflow is operator-facing and not part of PR validation (US00113 owns secret-free PR CI).
- **Single source of truth for the secret name.** Search for the literal `VERCEL_DEPLOY_HOOK_URL` across the repo before any future rename — currently it appears only in this workflow + this plan + `docs/specs/F0011.md`.
