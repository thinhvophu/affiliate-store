# Implementation Plan: US00096 — BreadcrumbList JSON-LD

**Status:** Approved
**Story:** US00096 (Feature F0009 — SEO & Discoverability)
**Spec source:** `docs/specs/F0009.md` (US00096 section + Edge Cases + Technical Considerations)

---

## 1. Summary

Emit a Google-compliant `BreadcrumbList` JSON-LD `<script>` block on **every page that already renders the visible `<Breadcrumb>` from US00093** — namely product detail (`/san-pham/[slug]/`), category (`/danh-muc/[category]/`), and blog post detail (`/bai-viet/[slug]/`). The structured data MUST be built from the **same `BreadcrumbItem[]` array** the visible breadcrumb consumes (US00093 owns the type + builder), so the visible trail and the structured data can never diverge — Google flags name/order mismatches.

This story is a **pure consumer**:
- It does **not** define `BreadcrumbItem` (US00093 owns it in `lib/breadcrumbs.ts`).
- It does **not** define a JSON-LD serializer (US00094 owns `<JsonLd data={...} />` in `components/JsonLd.tsx`).
- It does **not** wire `metadataBase` / `NEXT_PUBLIC_SITE_URL` resolution (US00091 owns that).

What this story **does** add:
- A pure data shaper `breadcrumbListJsonLd(items)` in `lib/breadcrumbs.ts` (co-located with US00093's `BreadcrumbItem` + items builder — same file, single source of truth, no JSX so it belongs in `lib/`).
- Three integration points in `app/san-pham/[slug]/page.tsx`, `app/danh-muc/[category]/page.tsx`, `app/bai-viet/[slug]/page.tsx` that import `breadcrumbListJsonLd` + `<JsonLd>` and emit the script alongside the visible `<Breadcrumb>`.

All output must appear in the **statically generated HTML** (SSG-only — no client injection).

## 3. Dependencies (read carefully — do not duplicate)

This plan assumes the following deliverables from sibling F0009 stories. **If their plans land with different file paths or signatures, update this plan before implementing.**

### From US00093 (visible Breadcrumb component) — REQUIRED

US00093 must export from `lib/breadcrumbs.ts`:

```ts
export interface BreadcrumbItem {
  /** Vietnamese label (e.g., "Trang chủ", "Chuột gaming", product.name). */
  name: string;
  /** Site-relative path (e.g., "/", "/danh-muc/chuot-gaming/", "/san-pham/foo/"). */
  href: string;
}

/** Build the trail for a product detail page. */
export function buildProductBreadcrumb(product: Product): BreadcrumbItem[];
/** Build the trail for a category page. */
export function buildCategoryBreadcrumb(category: CategoryMeta): BreadcrumbItem[];
/** Build the trail for a blog post detail page. */
export function buildPostBreadcrumb(post: Post): BreadcrumbItem[];
```

The shape (`name` + `href`) and the included final/current crumb come from US00093. This story consumes them as-is.

### From US00094 (Product JSON-LD) — REQUIRED

US00094 must export a shared serializer from `components/JsonLd.tsx`:

```tsx
import type { Thing, WithContext } from "schema-dts"; // or inline minimal types

interface JsonLdProps {
  data: object; // shape varies by schema type
}

export function JsonLd({ data }: JsonLdProps): JSX.Element;
```

The serializer is responsible for:
- Rendering `<script type="application/ld+json" dangerouslySetInnerHTML={...} />`.
- Escaping `<` to `<` (per Next.js docs) to prevent XSS through inlined HTML in Vietnamese product names / titles.

This story consumes `<JsonLd data={...} />` only — no second implementation.

### From US00091 (Global metadata foundation) — REQUIRED for absolute URL resolution

US00091 wires `metadataBase = new URL(env.NEXT_PUBLIC_SITE_URL)` in `app/layout.tsx`. Per-page `metadata.alternates.canonical` becomes a relative path that Next.js resolves.

JSON-LD does **not** flow through `metadataBase` — Next.js only auto-resolves `metadata`-object URLs, not strings emitted into a `<script>` tag. So this story resolves absolute URLs **manually**, using a small helper:

```ts
// in lib/breadcrumbs.ts (this story)
function absoluteUrl(path: string): string {
  return new URL(path, env.NEXT_PUBLIC_SITE_URL).toString();
}
```

`env.NEXT_PUBLIC_SITE_URL` is already required by `lib/env.ts` and throws at build time if missing — matches the F0009 "fail the build loudly" edge case.

## 4. Data model

### 4.1 New helper: `breadcrumbListJsonLd(items)`

Add to **`lib/breadcrumbs.ts`** (the file US00093 creates):

```ts
import { env } from "@/lib/env";

/**
 * Schema.org BreadcrumbList payload shape (intentionally a structural type,
 * not a full schema-dts import — the runtime shape is what matters for SEO).
 */
interface BreadcrumbListJsonLd {
  "@context": "https://schema.org";
  "@type": "BreadcrumbList";
  itemListElement: Array<{
    "@type": "ListItem";
    position: number; // 1-indexed
    name: string;     // matches BreadcrumbItem.name exactly
    item: string;     // absolute URL built from BreadcrumbItem.href
  }>;
}

/**
 * Build a BreadcrumbList JSON-LD object from the SAME items array used by
 * the visible <Breadcrumb> (US00093). Order and `name` MUST match the
 * visible trail (Google flags mismatches).
 *
 * The final/current crumb is included with its canonical URL as `item`
 * (per US00096 Scenario 3).
 */
export function breadcrumbListJsonLd(items: BreadcrumbItem[]): BreadcrumbListJsonLd {
  return {
    "@context": "https://schema.org",
    "@type": "BreadcrumbList",
    itemListElement: items.map((it, i) => ({
      "@type": "ListItem",
      position: i + 1,
      name: it.name,
      item: absoluteUrl(it.href),
    })),
  };
}

function absoluteUrl(path: string): string {
  // env.NEXT_PUBLIC_SITE_URL is validated as a URL by lib/env.ts.
  return new URL(path, env.NEXT_PUBLIC_SITE_URL).toString();
}
```

**Rationale:**
- Lives in `lib/` (no JSX) per project convention.
- Consumes the SAME `BreadcrumbItem[]` US00093's visible component consumes → impossible to diverge.
- No truncation, no formatting — `name` is passed through verbatim (Google compares character-for-character to visible text).
- `item` is an absolute URL — Google requires it for `BreadcrumbList.ListItem.item`.

### 4.2 No new types

`BreadcrumbItem` is owned by US00093. This story does not add anything to `types/`.

## 5. UI / integration changes

For each of the three pages, the change is the same shape:

```tsx
// 1. Import the new helper alongside US00093's items builder
import {
  buildProductBreadcrumb,   // (or buildCategoryBreadcrumb / buildPostBreadcrumb)
  breadcrumbListJsonLd,
} from "@/lib/breadcrumbs";
import { Breadcrumb } from "@/components/Breadcrumb"; // owned by US00093
import { JsonLd } from "@/components/JsonLd";          // owned by US00094

export default async function Page(...) {
  // ...
  const crumbs = buildProductBreadcrumb(product); // single source of truth

  return (
    <>
      <Breadcrumb items={crumbs} />
      <JsonLd data={breadcrumbListJsonLd(crumbs)} />
      {/* ...existing page body... */}
    </>
  );
}
```

Per-page wiring:

| File | Items builder | Notes |
| --- | --- | --- |
| `app/san-pham/[slug]/page.tsx` | `buildProductBreadcrumb(product)` | Place `<Breadcrumb>` + `<JsonLd>` above the existing `<article className={styles.detail}>` inside the existing `<div className={styles.container}>`. |
| `app/danh-muc/[category]/page.tsx` | `buildCategoryBreadcrumb(meta)` | Place inside the existing `<article>`, above the `<header>`. The `<ShellLayout>` shell stays untouched. |
| `app/bai-viet/[slug]/page.tsx` | `buildPostBreadcrumb(post)` | Place above the existing `<header className={styles.postHeader}>` inside `<article>`. Works in both `shellWithToc` and `container` layouts. |

Where exactly the visible `<Breadcrumb>` lives in the DOM is US00093's call; this story only ensures `<JsonLd>` is rendered in the same Server Component tree (so it appears in the SSG'd HTML).

**Important:** the `<JsonLd>` block is invisible — it does not affect layout. It can sit immediately before or after `<Breadcrumb>` without CSS implications.

## 8. Acceptance criteria (for implementer + reviewer)

Mapped 1:1 to F0009 §US00096:

- [ ] **Scenario 1 — BreadcrumbList emitted alongside visible breadcrumb.** Every page that renders `<Breadcrumb>` (product detail, category, blog post detail) also renders a `<script type="application/ld+json">` block containing a `BreadcrumbList` whose `itemListElement` is an ordered array of `ListItem`s with `position`, `name`, `item`.
- [ ] **Scenario 2 — Structured data matches visible trail.** Both visible and structured data are derived from the same `BreadcrumbItem[]` (via `buildProductBreadcrumb` / `buildCategoryBreadcrumb` / `buildPostBreadcrumb`). The `name` and order of `ListItem`s exactly match the visible crumbs character-for-character.
- [ ] **Scenario 3 — Last item handling.** The current/last crumb is included as the final `ListItem`. Its `item` is the canonical absolute URL of the current page.
- [ ] **Edge case — Vietnamese characters.** Diacritics in `name` (e.g., "Chuột gaming", "Tai nghe gaming") survive serialization intact; `<` characters are escaped to `<` (via US00094's `<JsonLd>`); the JSON parses cleanly in Google's Rich Results Test.
- [ ] **Edge case — Missing `NEXT_PUBLIC_SITE_URL`.** Build fails loudly via `lib/env.ts`; no relative or malformed `item` URLs are ever emitted.
- [ ] **SSG-only.** JSON-LD `<script>` is present in the initial HTML response (verify via "view source", not DevTools-rendered DOM).

## 10. Open questions for the reviewer

1. **Trailing slashes in `item` URLs.** Next.js default is `trailingSlash: false`, but the existing app code uses both shapes (e.g., product canonical `/san-pham/${slug}/` with slash; category canonical `/danh-muc/${meta.slug}` without). US00093's items builder will pick one; this story passes through whatever it returns. Should we coordinate with US00091 to standardize? (Recommendation: defer to US00091 — it owns canonical URL resolution.)
2. **Should `breadcrumbListJsonLd` live in `lib/breadcrumbs.ts` (this plan) or `lib/jsonld.ts` (a new dedicated module)?** Co-locating with the items builder makes "shared source of truth" obvious in one file and matches the project's "single chokepoint" pattern (`lib/format.ts`, `lib/affiliate.ts`). Plan goes with co-location; flagging for explicit reviewer sign-off because US00094 (`Product`) and US00095 (`Article`) may push us toward a `lib/jsonld.ts` aggregator.
3. **No `Organization` / `publisher` schema in v1** — confirmed out of scope per F0009 §2 (Out of Scope), so we do not embed an `Organization` ref inside `BreadcrumbList`. (None is required by Google for `BreadcrumbList` anyway.)

## 11. Risks

- **Dependency drift.** If US00093's `BreadcrumbItem` shape changes (e.g., renames `name` → `label`), this story's helper + tests must update in lock-step. Mitigation: import the type rather than redeclaring it; TypeScript compile-time error if the shape drifts.
- **Visible/structured mismatch via stale cache.** If a future story bypasses `buildXBreadcrumb` and hand-builds an `items` array for only one of the two consumers, Google will flag a mismatch. Mitigation: the structure-doc convention (§7 item 3) forbids hand-rolled `BreadcrumbList` payloads.
- **XSS in `name`.** Vietnamese product titles can contain `<`, `>`, quotes. Mitigated by US00094's `<JsonLd>` which escapes `<` → `<` per Next.js docs.

## Resolved cross-cutting decisions (F0009 review — 2026-06-21)

- **Single absolute-URL chokepoint:** build each `ListItem.item` via `absoluteUrl()` from `lib/seo.ts` (US00091).
- **Trailing slash:** every `item` URL ends with `/`, matching the visible breadcrumb hrefs (US00093) and page canonicals — so visible trail and structured data never diverge.
- **Shaper location:** keep `breadcrumbListJsonLd(items)` co-located in `lib/breadcrumbs.ts` (next to US00093's `BreadcrumbItem` + builders). No separate `lib/jsonld.ts` aggregator in v1.
